DR · Part 1

First — RTO & RPO

Every DR question is really a trade-off between these two numbers and cost.

MetricMeans"How much…"
RTO (Recovery Time Objective)How long to recover after a disaster…downtime can you tolerate?
RPO (Recovery Point Objective)How far back the last good copy is…data loss can you tolerate?
🔑 The core trade-off

Lower RTO/RPO = faster recovery & less data loss = more cost. The four strategies are simply points along that cost-vs-speed line. The exam gives you the tolerance (e.g. "RTO of a few hours is fine" or "near-zero downtime required") and you pick the matching strategy.

DR · Part 3

The Pilot Light vs Warm Standby Trap

⚠️ The most-tested DR distinction

Pilot Light: only the core (usually the database) is running/replicated; the application/compute tier is off and must be started and scaled up at failover. Warm Standby: a scaled-down but already-running full stack — it's serving (just small) and you only scale it up. Key phrase: "core/DB on, app servers off" → Pilot Light; "scaled-down running stack" → Warm Standby.

🔑 Building blocks

DR uses tools you already know: S3 CRR & AWS Backup (backup/restore), RDS cross-region read replica / Aurora Global Database (pilot light/warm), DynamoDB Global Tables (active-active), and Route 53 / Global Accelerator for failover routing.

💵 DR tooling is mostly cheap to run, costly to over-provision: MGN and DRS are free to use (you pay only for the replication/staging EC2 + EBS); DMS bills a small replication-instance hourly rate; AWS Backup bills per-GB stored. The real cost lever is the strategy: Active-Active duplicates full production (most expensive); Pilot Light keeps only the DB warm (cheap).
🏢 Dublin Industry — Verified

Fenergo runs "Always On — Active-Active-Active"; Ryanair modernised backup & DB

Fenergo (Dublin) operates its platform active-active across 3 Availability Zones ("Always On") with event-sourcing on DynamoDB + EventStoreDB (leader/follower/read-only across AZs) — the near-zero-RTO end of the DR spectrum for a regulated SaaS.

Ryanair modernised its DR estate: it migrated Microsoft SQL Server → Amazon Aurora (a heterogeneous migration: SCT + DMS territory) and replaced unreliable physical tape backup with Storage Gateway Tape Gateway (~65% savings) — the Backup & Restore foundation done right.

Matching strategy to need

Fenergo's banking SaaS demands active-active (highest cost, ~0 RTO). Most workloads sit lower on the ladder — Backup & Restore or Pilot Light — because the architect matches RTO/RPO tolerance to cost, not gold-plating everything.

DB Migration · Part 1

AWS Database Migration Service (DMS)

DMS migrates databases to AWS while the source stays fully operational — minimal downtime.

It does a full load of existing data, then continuous replication (CDC — Change Data Capture) to keep the target in sync until you cut over. Works homogeneous (same engine, e.g. Oracle → Oracle on RDS) or heterogeneous (different engine, paired with SCT).

DB Migration · Part 2

AWS Schema Conversion Tool (SCT)

When migrating between different database engines (e.g. Oracle → Aurora PostgreSQL), the schema, stored procedures, and code aren't compatible. SCT converts the schema (and flags what needs manual work); then DMS moves the data.

🔑 The pairing

Same engine (Oracle → Oracle, MySQL → Aurora MySQL) → DMS alone. Different engine (Oracle → Aurora PostgreSQL, SQL Server → MySQL) → SCT (convert schema) + DMS (move data with CDC).

DB Migration · Part 3

The Database Migration Decision

⚠️ Files vs databases

DataSync, Snowball, and Storage Gateway move files/objects — NOT live databases. To migrate a live, transactional database with minimal downtime, the answer is DMS (+ SCT if the engine changes). Don't pick a file-transfer service for a database.

Migration · Part 1

AWS Application Migration Service (MGN) — Lift & Shift

AWS Application Migration Service (MGN) is the primary lift-and-shift (rehost) tool: it does block-level replication of your physical/VM/cloud servers into AWS, then launches them as EC2 instances — with minimal downtime and no app changes.

"Migrate existing servers to EC2 with minimal changes / minimal downtime, automated" → Application Migration Service (MGN). (It replaced the older Server Migration Service and CloudEndure Migration.)

Planning · Part 1

AWS Elastic Disaster Recovery (DRS)

AWS Elastic Disaster Recovery (DRS) continuously block-level replicates your servers (on-prem or cloud) into a low-cost staging area in AWS, ready to spin up full capacity in minutes on disaster — a managed pilot-light-style DR for whole servers (not just RDS). "Replicate physical/VM servers for fast DR" → DRS. (MGN = one-time migration; DRS = ongoing disaster recovery.)

🔑 MGN vs DRS

Same underlying tech, different goal: MGN = migrate servers once (then run on AWS). DRS = ongoing replication for recovery (servers stay primary elsewhere until disaster).

Planning · Part 2

Discovery, Migration Hub & Backup

  • AWS Application Discovery Service — inventory on-prem servers and map dependencies before migrating (plan with data, not guesses).
  • AWS Migration Hub — a single place to track migration progress across tools.
  • AWS Backup — centralized cross-service/cross-region backup with Vault Lock (the backbone of the Backup & Restore DR strategy).
  • Resilience Hub / Fault Injection Service (FIS) — assess & test resilience (chaos engineering / game-days).
