SQS · Part 1

Why Decoupling — Don't Let Services Crash Together

When service A calls service B directly, B's bad day becomes A's bad day. A queue breaks that chain.

Picture an order website. The web tier takes an order and directly calls the payment/processing service. This is synchronous coupling — and it's fragile.

If the processing service is slow, the web request hangs and users wait. If it's down, the order is lost and the user sees an error. On Black Friday, a flood of orders overwhelms the processor and the whole site falls over. The two tiers are chained together — one's failure drags down the other.

💡 Why SQS Was Built

Amazon SQS (Simple Queue Service) is a fully-managed message queue that sits between your services as a buffer. The web tier (the producer) drops an order message into the queue and instantly moves on — the user gets a fast "thanks, order received." The processing tier (the consumer) pulls messages from the queue at its own pace.

Now: if the processor is slow, messages simply wait safely in the queue. If it's down, nothing is lost — work resumes when it's back. On Black Friday, the queue absorbs the spike and the processor drains it steadily. The two tiers are decoupled — each can fail, scale, or deploy independently. This is the single most important resilience pattern in cloud architecture.

SQS · Part 2

How a Queue Works

Producers send; messages persist in the queue; consumers poll, process, then delete. No message is lost if a consumer dies.
  • A consumer polls the queue, processes the message, then explicitly deletes it. If it crashes before deleting, the message reappears for another consumer — at-least-once delivery.
  • Max message size 256 KB (use the S3 + SQS Extended Client Library for bigger payloads — store the blob in S3, pass a pointer).
  • One queue can have many consumers sharing the load (a competing-consumers pool) — but each message goes to one consumer. (For "everyone gets a copy," that's SNS, next tab.)
SQS · Part 3

Standard vs FIFO — the core exam distinction

Standard Queue

Scale over order

  • Nearly unlimited throughput
  • At-least-once delivery (rare duplicates possible)
  • Best-effort ordering (can arrive out of order)
  • Default — use when order doesn't matter
FIFO Queue

Order & exactly-once

  • Strict ordering (First-In-First-Out)
  • Exactly-once processing (dedup)
  • Throughput 300 msg/s (3,000 with batching)
  • Name must end in .fifo; uses Message Group ID
⚠️ Exam trap

"Order matters / no duplicates / exactly-once" → FIFO. "Maximum throughput, order doesn't matter" → Standard. Don't pick FIFO just because it sounds "better" — its 300/s limit can be a bottleneck, and Standard is cheaper and more scalable when ordering isn't required.

SQS · Part 5

Decouple + Auto-Scale on Queue Depth

The classic resilient pattern: put an Auto Scaling Group of consumers behind the queue and scale it on the queue length (CloudWatch metric ApproximateNumberOfMessages, ideally as a backlog-per-instance target). Spike of orders → queue grows → ASG adds workers → backlog drains → ASG scales back down. The queue buffers; the fleet elastically matches demand.

💵 SQS is $0.40 per million requests (Standard; FIFO $0.50/M), first 1M free. A serverless buffer for a busy app is often a few dollars a month — and it scales to zero when idle. (Each request carries ≤64 KB, so a 256 KB message = 4 requests.)
🏢 Industry — Verified

Starling Bank & Dublin fintech: idempotent, queue-backed microservices

Starling Bank (the AWS-native UK neobank) is the canonical example: microservices communicate through SQS with idempotent handlers and "at-least-once" semantics, deliberately avoiding distributed transactions. Dublin's AccessPay (B2B payments) uses SQS to buffer payment files arriving via SFTP for PCI-compliant downstream processing. The recurring theme: a queue lets each microservice fail, retry, and scale on its own without losing work — which is exactly why banks trust the pattern.

SNS · Part 1

Why SNS — One Event, Many Interested Parties

SQS is one-to-one (one message, one consumer). But what if many systems all need to react to the same event?

An order is placed. Now several things must happen: email the customer, update inventory, notify the warehouse, refresh analytics, alert fraud detection. Five different systems, all interested in the same event.

You could make the order service call all five directly — but that's tight coupling again (and you'd edit the order service every time you add a sixth listener). With a single SQS queue, only one consumer gets each message, not all five.

💡 Why SNS Was Built

Amazon SNS (Simple Notification Service) is publish/subscribe (pub/sub) messaging. A publisher sends one message to a topic; SNS instantly pushes a copy to every subscriber of that topic. The order service publishes "OrderPlaced" once; email, inventory, warehouse, analytics, and fraud each receive their own copy. Add a new listener? Just subscribe it to the topic — the publisher never changes. One-to-many, push-based, fully decoupled.

SNS · Part 2

How SNS Works

  • Topic — the channel publishers send to.
  • Subscribers — endpoints that receive every message: SQS queues, Lambda functions, HTTP/S endpoints, email, SMS, mobile push, and Kinesis Data Firehose.
  • Push, not poll — SNS delivers to subscribers (unlike SQS, where consumers pull).
  • Up to ~12.5 million subscriptions per topic; messages encrypted in transit, optional KMS at rest.
SNS · Part 3

The Fan-Out Pattern — SNS + SQS Together

The single most important architecture in this whole section.

Publish once to SNS → every subscribed SQS queue / Lambda gets its own copy to process independently, at its own pace.
🔑 Why fan-out (SNS → multiple SQS) beats SNS → Lambda directly

Subscribing SQS queues to the topic gives each consumer its own durable buffer: if a consumer is down, its messages wait safely in its queue (and can retry / land in a DLQ). It combines SNS's one-to-many with SQS's persistence + retry. This SNS → multiple SQS fan-out is a top-tested pattern for "one event must trigger several independent, reliable consumers."

SNS · Part 4

Message Filtering & SNS FIFO

  • Message filtering — each subscription can have a filter policy so a subscriber only receives the messages it cares about (e.g. only orders over €1,000), instead of every message. Avoids consumers filtering in code.
  • SNS FIFO — pairs with SQS FIFO for ordered, deduplicated fan-out (ordering preserved across the fan-out).
💵 SNS is $0.50 per million publishes (first 1M free). Deliveries to SQS/Lambda are cheap; SMS and email deliveries cost extra per message — a common "why is my SNS bill high?" answer.
🔑 SNS vs EventBridge (quick contrast)

SNS is fast, simple, high-throughput pub/sub. EventBridge adds richer content-based routing, SaaS partner event sources (Zendesk, Datadog…), a schema registry, and archive/replay. "Route third-party SaaS events by content to many AWS targets" → EventBridge; "simple high-throughput fan-out to subscribers" → SNS.

🏢 Industry

Event-driven backends: publish once, react everywhere

Dublin platforms like Ryanair use SNS/SQS/EventBridge for booking-confirmation and schedule-change events — one "schedule changed" event fans out to notifications, re-accommodation logic, and analytics simultaneously. The pattern lets teams add new reactions (a new loyalty-points service, say) by subscribing to the existing topic, with zero changes to the publisher.

Kinesis · Part 1

Why Streaming — When You Need Replay & Many Readers

SQS deletes a message once it's consumed. But some data needs to be re-read, by many systems, in order.

Think of a firehose of events: every click on a website, every IoT sensor reading, every stock tick — thousands to millions per second, continuous, and you want to analyse it in real time.

A queue isn't the right tool here. In SQS, once a consumer processes a message it's deleted — you can't replay it, and you can't easily have multiple independent consumers each read the whole stream. For real-time analytics you often want exactly that: the analytics team, the ML team, and the archival job all reading the same ordered stream, and the ability to rewind and re-process after a bug fix.

💡 Why Kinesis Was Built

Amazon Kinesis ingests and stores massive real-time data streams. Records are kept for a retention window (default 24h, up to 365 days), so multiple consumers can each read the stream independently and you can replay from any point. Data is ordered within a shard by partition key. It's built for high-volume, ordered, replayable, multi-consumer streaming — the things queues don't do.

Kinesis · Part 2

Kinesis Data Streams — Shards, Ordering & Replay

  • Shards are the unit of capacity: each ingests 1 MB/s or 1,000 records/s and emits 2 MB/s. Scale by adding shards.
  • Partition key decides which shard a record goes to — and records with the same key stay ordered within that shard.
  • Retention & replay: records persist (24h → 365 days) so consumers can re-read; multiple consumers can read the same stream.
  • Enhanced Fan-Out gives each consumer its own dedicated 2 MB/s per shard push pipe — for many parallel consumers without throttling each other.
  • Capacity modes: Provisioned (you manage shards, cheapest at steady scale) vs On-Demand (auto-scales, no shard management).
💵 Provisioned Streams: ~$0.015/shard-hour (~$11/shard/month) + $0.014 per million PUT units — you pay for shards 24/7 even when idle. On-Demand removes shard management at a higher per-GB rate.
Kinesis · Part 3

Kinesis Data Firehose — Stream → Storage, Zero Ops

Often you don't want to process a stream — you just want to land it in storage for analytics. Firehose is the fully-managed, serverless delivery service: point a stream at it and it buffers and writes (near-real-time) to S3, Redshift, OpenSearch, or Splunk — with optional Lambda transformation and format conversion on the way.

Data Streams

Real-time, you build consumers

  • Sub-second, custom processing
  • Shards, retention, replay
  • Many consumers, ordering per key
Data Firehose

Near-real-time delivery, no ops

  • Fully serverless, no shards
  • Buffers → S3/Redshift/OpenSearch/Splunk
  • No storage / no replay — it just delivers
🔑 The canonical least-ops trigger

"Stream data to S3 / Redshift / OpenSearch with the least operational overhead, near-real-time" → Kinesis Data Firehose. (For real-time SQL/aggregates on the stream, that's Managed Service for Apache Flink, a.k.a. Kinesis Data Analytics.)

💵 Firehose ≈ $0.029/GB ingested (tiers down) — pure pay-per-use, no shards to provision.

🏢 Dublin Industry — Verified

Ryanair ingests clickstream and operational events through Kinesis in real time into its S3 data lake, then analyses with downstream tools (and SageMaker for demand forecasting). Amazon MQ · Part 1

Why Amazon MQ — Lift-and-Shift Existing Brokers

SQS/SNS are brilliant — but they use AWS's own APIs. What about an old app that already speaks standard messaging protocols?

A company has a 10-year-old Java application that talks to a message broker using the industry-standard JMS and AMQP protocols. They want to move it to AWS without rewriting the messaging code.

SQS and SNS would mean rewriting the app to use AWS SDK calls — significant work and risk. They need something that speaks the same protocols their app already uses.

💡 Why Amazon MQ Was Built

Amazon MQ is a managed message broker for Apache ActiveMQ and RabbitMQ. It speaks the industry-standard protocols — JMS, AMQP, MQTT, STOMP, OpenWire, WebSocket — so an existing on-prem app can connect with little or no code change. AWS runs and patches the broker; you keep your messaging code. It's the migration / compatibility answer, not the cloud-native one.

Amazon MQ · Part 2

What It Is

  • Managed ActiveMQ or RabbitMQ brokers — both queues and topics in one service.
  • Standard protocols: JMS, AMQP, MQTT, STOMP, OpenWire, WSS.
  • Deployment: Single-instance (dev) or Active/Standby across two AZs for high availability.
  • Runs on a broker instance — so it's always-on (billed per broker-hour), unlike serverless SQS/SNS.
  • Doesn't scale as limitlessly as SQS/SNS — it's a traditional broker, sized to your throughput.
💵 Billed per broker-instance-hour — e.g. mq.m5.large ~$0.30/hr (~$220/month) + storage, running 24/7. That fixed always-on cost is the trade-off for protocol compatibility.
Amazon MQ · Part 3

MQ vs SQS/SNS — When to Pick Which

🔑 The one rule

New, cloud-native application → SQS/SNS (serverless, limitless scale, pay-per-use). Migrating an existing app that already uses JMS/AMQP/MQTT and you don't want to rewrite it → Amazon MQ. If a question mentions an existing on-prem broker, "least code change," or specific protocols like JMS/AMQP → it's Amazon MQ. If it's a brand-new design → don't pick MQ.
