Lambda · Part 1

Why Serverless — Stop Paying for Idle Servers

With EC2 you rent a server 24/7 even when no one's using it. Lambda flips that: you pay only while your code runs.

Run a small API on EC2 and you pay for the instance all month — even at 3am when traffic is zero. You also patch the OS, scale the fleet, and handle failures. That's a lot of work for "run my code when something happens."

AWS Lambda runs your code in response to events, with no servers to manage. You upload a function; AWS runs it when triggered, automatically scaling from zero to thousands of concurrent executions, and you pay only for the milliseconds it runs. Idle = $0. No provisioning, no patching, no capacity planning.

💡 What "serverless" really means

There are still servers — you just never see or manage them. AWS handles the infrastructure, scaling, and availability. You focus purely on code. Serverless shines for event-driven, spiky, or unpredictable workloads where paying for always-on capacity is wasteful.

Lambda · Part 2

How It Works & What Triggers It

An event fires → Lambda spins up, runs your function, scales per-event, then shuts down. You pay per invocation + run time.
  • Synchronous triggers (API Gateway, ALB) — caller waits for the response.
  • Asynchronous triggers (S3, SNS, EventBridge) — Lambda retries twice on failure, then sends to a DLQ / Destination.
  • Event source mappings (SQS, Kinesis, DynamoDB Streams) — Lambda polls the source and processes records in batches.
  • Execution role — the IAM role giving the function its AWS permissions (read S3, write DynamoDB…).

⚠️ The #1 Lambda trap

"The job runs longer than 15 minutes" → Lambda is the WRONG answer. Reach for Fargate (long-running container), Step Functions (orchestrate steps), or AWS Batch. Lambda is for short, event-driven units of work.

Lambda · Part 4

Concurrency & Cold Starts

  • Cold start — the first invocation (or a scale-up) has to initialise the runtime, adding latency. Subsequent "warm" invocations are fast.
  • Provisioned Concurrency — pre-warms N execution environments so there's no cold start (predictable low latency) — but you pay for them always-on. "Guarantee no cold start / consistent latency" → Provisioned Concurrency.
  • Reserved Concurrency — caps the max concurrent executions for a function (protects downstream systems / other functions from being starved). "Limit concurrency / protect a database from a flood" → Reserved Concurrency.
  • SnapStart — for Java/Python/.NET, snapshots an initialised environment to cut cold starts dramatically, at no extra charge (no Provisioned Concurrency needed).
🔑 Reserved vs Provisioned (don't confuse)

Reserved = a ceiling on concurrency (cost cap / throttle protection). Provisioned = pre-warmed instances (kills cold starts). Different problems, similar names.

Lambda · Part 5

💰 Cost & Lambda vs Fargate

💵 Lambda = $0.20 per 1M requests + $0.0000167 per GB-second of run time. Free tier: 1M requests + 400,000 GB-s every month. A modest API often runs $0–a few dollars/month and scales to zero. Cost = requests × memory × duration — and since more memory also means more CPU, a function can finish faster and cost less at higher memory (tune it).
🔑 Lambda vs Fargate

Both serverless. Lambda = short, event-driven functions (≤15 min). Fargate = long-running containers, >10 GB RAM, GPU, persistent connections, big images. "Process each upload" → Lambda; "run a service / 45-min job" → Fargate.

🏢 Dublin Industry — Verified

Fenergo runs a serverless-first, event-sourced platform on Lambda + DynamoDB

Fenergo (Dublin RegTech for KYC/AML) is single-cloud AWS, serverless-first: per-microservice least-privilege Lambda execution roles, DynamoDB as the state/read-model store, API Gateway WebSockets + Lambdas, and EventBridge fan-out — running active-active across 3 AZs for regulated banks. Ryanair uses Lambda as glue across its estate (e.g. S3 Object Lambda for the COVID travel-wallet). Lambda is the connective tissue of modern AWS apps.

DynamoDB · Part 1

Why DynamoDB — A Database That Scales Itself

A relational DB on one big server hits a ceiling. DynamoDB is built to scale horizontally to any size with steady single-digit-ms speed.

Amazon DynamoDB is a fully-managed, serverless NoSQL key-value & document database delivering single-digit millisecond performance at any scale.

No servers to provision, no patching, automatic replication across 3 AZs, virtually unlimited throughput and storage. It's the go-to database for serverless apps, high-velocity workloads (gaming leaderboards, IoT, shopping carts, session stores), and anything needing massive scale with predictable low latency. The trade-off vs SQL: you design around access patterns, not flexible ad-hoc joins.

DynamoDB · Part 2

Tables, Keys & Capacity Modes

  • Primary key: a partition key (decides which partition stores the item) — optionally plus a sort key for ordering/range queries within a partition.
  • Items & attributes: rows and columns, but schema-flexible per item.
On-Demand capacity

Pay per request

  • No capacity planning — scales instantly
  • Pay per read/write request
  • Best for spiky / unpredictable / new workloads
Provisioned capacity

Reserve RCU/WCU

  • Set Read/Write Capacity Units (+ auto-scaling)
  • Cheaper per unit at steady, predictable volume
  • Can buy reserved capacity for more savings
🔑 Trigger

"Unpredictable / spiky traffic, no capacity planning" → On-Demand. "Steady, predictable high volume, lowest cost" → Provisioned + auto-scaling. (Adaptive capacity automatically handles hot partitions.)

DynamoDB · Part 3

Indexes — GSI vs LSI

DynamoDB queries are fast only on keys. Indexes let you query on other attributes — and the exam tests the difference.

Global Secondary Index (GSI)Local Secondary Index (LSI)
Partition keyCan be any attribute (different from table)Same as the table's partition key
Sort keyAny attributeAn alternate sort key
When createdAnytime (add/remove later)Only at table creation
CapacityOwn provisioned throughputShares the table's throughput
ConsistencyEventually consistent onlyCan be strongly consistent
⚠️ Tested gotcha

LSI must be created when the table is created and uses the same partition key. A GSI can be added anytime and uses any key. "Need to query on a new attribute after the table exists" → GSI.

DynamoDB · Part 4

Streams, DAX & Global Tables

  • DynamoDB Streams — a change log of item-level modifications. Trigger a Lambda on every change (CDC) for event-driven processing, replication, or aggregation.
  • DAX (DynamoDB Accelerator) — an in-memory cache giving microsecond reads for hot data, fully managed and DynamoDB-specific (unlike ElastiCache). "Microsecond read latency for DynamoDB" → DAX.
  • Global Tables — multi-region, active-active replication. Users in every region read/write locally; great for global apps and multi-region DR.
  • TTL — auto-expire items by timestamp (sessions, temp data). PITR — point-in-time recovery (last 35 days). Transactions — ACID across multiple items.

🏢 Dublin Industry — Verified

Fenergo & Intercom: DynamoDB for high-velocity state and CDC

Fenergo uses DynamoDB as its event-sourced state/read-model store — with TTL, optimistic locking via condition expressions, and DynamoDB Streams for change-data-capture into EventBridge. The pattern: DynamoDB for scale & speed on known access patterns; SQL where you need joins.

API Gateway · Part 1

Why API Gateway — A Managed Front Door for Your APIs

Your Lambda functions need a secure, scalable HTTP endpoint clients can call. That's API Gateway.

A Lambda function isn't a public web API by itself. You need something to receive HTTPS requests, authenticate them, throttle abuse, and route them to the right backend — without running servers.

Amazon API Gateway is a fully-managed service to create, publish, secure, and monitor APIs at any scale. It's the front door: clients call API Gateway, which authenticates, validates, throttles, optionally caches, and forwards to Lambda (or HTTP backends, Step Functions, other AWS services). It scales automatically and you pay per call.

API Gateway · Part 2

REST vs HTTP vs WebSocket APIs

TypeUse forNote
REST APIFull-featured: API keys, usage plans, request validation, per-method caching, WAF$3.50/M — pick when you need these features

⚠️ The key trap

"Need API keys, usage plans, request validation, or per-method caching" → REST API (HTTP API lacks these). "Just a cheap, fast proxy to Lambda" → HTTP API. "Real-time two-way" → WebSocket API.

API Gateway · Part 3

Features & Authentication

  • Throttling & usage plans — rate-limit per client; issue API keys with quotas.
  • Caching — cache responses (REST) to cut backend calls and latency.
  • Request validation & transformation — reject bad requests before they hit Lambda; map payloads.
  • Authorization — Cognito User Pool authorizer (validate user JWT), Lambda (custom) authorizer (your own logic / 3rd-party tokens), or IAM auth.
  • Endpoint types — Edge-optimized (global users, via CloudFront), Regional, or Private (VPC-only, via interface endpoint).
  • WAF attaches to REST APIs for SQLi/rate/bot protection.

Cognito · Part 1

Why Cognito — Don't Build Auth Yourself

Sign-up, sign-in, password resets, MFA, social login, token management — Cognito does it so you don't have to.

Every app needs users to sign in. Building that securely — password hashing, MFA, email verification, token issuance, social/enterprise login — is hard and risky to get wrong.

Amazon Cognito is managed identity for your apps: user directories, sign-up/sign-in, MFA, and federation with Google/Facebook/Apple and enterprise SAML/OIDC providers. It issues standard JWT tokens your API can verify. Fully managed, scales to millions of users.

Cognito · Part 2

User Pools vs Identity Pools — The Core Distinction

The single most-tested Cognito fact.

User Pools

Authentication (who are you?)

  • A user directory: sign-up / sign-in
  • MFA, password policies, email/phone verification
  • Federation (social, SAML, OIDC)
  • Issues JWT tokens → verify at API Gateway / ALB
Identity Pools (Federated Identities)

Authorization (what can you access?)

  • Exchange a token for temporary AWS credentials
  • Lets users directly access AWS services (e.g. their own S3 prefix)
  • Maps identities → IAM roles
  • Often used with a User Pool
🔑 Memory hook

User Pool = authentication (logging users in, issuing tokens). Identity Pool = authorization to AWS (swapping a token for temporary AWS credentials to call AWS services directly). "Sign users in / manage the directory" → User Pool. "Give signed-in users temporary AWS access (e.g. upload to S3)" → Identity Pool.

Section 19 · The Whole Stack

The Classic Serverless Web/API Architecture

How all four services click together.

Cognito authenticates → API Gateway validates the token + routes → Lambda runs logic → DynamoDB stores data (+ S3 for files/static site via CloudFront). Zero servers, scales to zero.
💵 This whole stack for a small/medium app: Cognito free under 50,000 monthly active users; Lambda + DynamoDB on-demand + HTTP API often $0–a few $/month, scaling automatically with no idle cost. The serverless promise: pay for usage, not capacity.
