The Six Policy Types

Permissions in AWS come from several layers. Knowing which is which is half the battle.

Policies · Part 2

How a Request Is Evaluated

The single most-tested IAM rule.

🔑 The golden rule

Explicit Deny > Allow > implicit Deny. By default everything is denied (implicit deny). An Allow in any applicable policy grants access — unless an explicit Deny exists anywhere, which always wins. One Deny overrides any number of Allows.

⚠️ Cross-account & boundaries

Cross-account: access is granted only if both the identity-based policy (in the caller's account) AND the resource-based policy (in the resource's account) allow it. Permission boundary / SCP: they don't grant anything — they cap. The effective permission is the intersection: an action is allowed only if the identity policy allows it AND it's within the boundary AND within the SCP.

Policies · Part 3

Permission Boundaries — The Ceiling

A permission boundary sets the maximum permissions an IAM user or role can ever have, regardless of how generous their identity policies are. The effective permission = identity policy ∩ boundary.

Classic use: safe delegation. You let developers create IAM roles for their apps, but attach a boundary so they can never grant more than (say) S3 + DynamoDB — preventing privilege escalation. "Allow a team to create roles but cap what those roles can do" → permission boundaries.

Policies · Part 4

Conditions, Policy Variables & ABAC

  • Condition keys — restrict by IP (aws:SourceIp), MFA (aws:MultiFactorAuthPresent), time, requested region, tags, VPC endpoint, etc.
  • ABAC (Attribute-Based Access Control) — grant access based on tags (e.g. allow if resource tag Project == principal tag Project). Scales better than creating a policy per team (RBAC) for large/dynamic orgs.
  • "Require MFA for sensitive actions" → a condition on aws:MultiFactorAuthPresent.
💵 IAM is free. Users, groups, roles, policies, permission boundaries, SCPs, and IAM Identity Center cost $0 — you pay only for what the identities consume. That's exactly why "least privilege via many fine-grained roles" never costs more than a few broad ones — there's no penalty for creating 50 tightly-scoped roles.
🏢 Dublin Industry — Verified / Inferred

Enterprise identity governance — AIB and Fenergo patterns

AIB (Allied Irish Banks) federates IAM Identity Center to an external IdP (Okta/Entra), with permission sets mapped to job roles; developers get just-in-time access via a ServiceNow request that triggers a Lambda to provision a temporary role, and PCI auditors assume a cross-account read-only SecurityAudit role via STS — no standing privileged access.

Fenergo (multi-tenant SaaS for banks) enforces hard tenant isolation with IAM policies using per-tenant condition keys and least-privilege execution roles per microservice.

The enterprise pattern

No IAM user per person — federate via Identity Center. No standing admin — assume roles via STS with session limits. No broad blast radius — SCPs guardrail accounts, permission boundaries cap delegated roles, and ABAC scales access by tag.

🔑 STS & Cross-Account — On This Page
Why Roles & STS Cross-Account Access External ID & Confused Deputy Triggers
STS · Part 1

Why IAM Roles & STS — Temporary Credentials

Long-lived access keys are a liability. Roles + STS give short-lived, auto-rotating credentials instead.

AWS STS (Security Token Service) issues temporary, time-limited credentials when an identity assumes a role (AssumeRole).

Instead of embedding static access keys, an EC2 instance, Lambda, or user assumes a role and gets short-lived credentials that expire automatically — far safer. This is the foundation of EC2 instance roles, Lambda execution roles, and cross-account access.

🔑 A role has two policies

A trust policy (who is allowed to assume it) and permission policies (what it can do once assumed). To delegate access you grant the principal sts:AssumeRole and list them in the role's trust policy.

STS · Part 2

Cross-Account Access — The Pattern

Account B creates a role trusting Account A; a principal in A calls AssumeRole → STS returns temporary creds → A acts in B. No shared long-lived keys.
⚠️ The most-tested IAM trap

For cross-account access, the answer is almost always an IAM role assumed via STS — never an IAM user with shared access keys. "Give a partner/another account temporary access" → cross-account role + AssumeRole, not IAM user credentials.

STS · Part 3

External ID — Preventing the Confused Deputy

When you let a third-party (e.g. a SaaS vendor) assume a role in your account, you add an External ID to the trust policy. The vendor must present that secret ID when assuming the role — preventing the confused deputy problem (where the vendor could be tricked into accessing the wrong customer's account).

🔑 Other STS flavours

AssumeRoleWithSAML (enterprise SAML federation), AssumeRoleWithWebIdentity (OIDC / Cognito / social login), GetSessionToken (MFA-protected temporary creds). Roles also enable IAM Roles Anywhere for on-prem servers.

Federation · Part 1

Why Federation — Don't Create an IAM User Per Person

A company with 5,000 employees shouldn't create 5,000 IAM users. Federation lets people sign in with their existing corporate or social identity and receive temporary AWS access.

Their identity provider (IdP) authenticates them; AWS trusts the IdP and (via STS) issues temporary credentials mapped to an IAM role. No per-person IAM users, no extra passwords.

Federation typeFor
SAML 2.0Enterprise IdPs (Active Directory / ADFS, Okta) — legacy direct federation
Web Identity (OIDC)Social / OIDC logins — usually via Cognito Identity Pools
IAM Identity CenterThe modern, recommended way for workforce SSO across many AWS accounts & apps
Federation · Part 2

IAM Identity Center (formerly AWS SSO)

  • One place for workforce single sign-on into multiple AWS accounts (across an Organization) and business apps.
  • Connect to an external IdP (Okta, Entra/Azure AD, AD) or use its built-in directory.
  • Permission sets map groups → IAM roles in each account. Users get a portal listing their accounts/roles.
  • The modern, recommended replacement for hand-rolled SAML federation across accounts.
🔑 Trigger

"Single sign-on for employees across all our AWS accounts, federated from our corporate directory, least admin" → IAM Identity Center.

Section 25 · The Big Distinction

Workforce vs Application Users

⚠️ Identity Center vs Cognito — heavily tested

IAM Identity Center = your workforce (employees) signing in to AWS accounts & internal apps. Cognito = your application's external end-users (customers of your mobile/web app). "Employees access AWS" → Identity Center. "Customers log in to our app" → Cognito. Don't swap them.

🏢 Organizations & SCP — On This Page
Why Organizations SCPs — Guardrails Control Tower & More Triggers
Organizations · Part 1

AWS Organizations — Managing Many Accounts

Mature companies use many AWS accounts (per team/environment) for isolation and blast-radius control. AWS Organizations manages them centrally.

  • Consolidated billing — one bill; volume discounts & shared Savings Plans/RIs across accounts.
  • Organizational Units (OUs) — group accounts (e.g. Prod, Dev, Sandbox) to apply policies.
  • A management (payer) account + member accounts.
Organizations · Part 2

Service Control Policies (SCPs) — Guardrails

The most-tested Organizations feature.

🔑 SCPs set the ceiling, they don't grant

An SCP defines the maximum permissions for accounts in an OU/account — a guardrail. It never grants permissions; identities still need IAM Allow policies. The effective permission = SCP ∩ IAM policy. SCPs do not apply to the management account.

Classic uses: "no account may disable CloudTrail," "only eu-west-1 and us-east-1 regions allowed," "block leaving the Organization," "deny the root user from doing X." "Centrally restrict what any account can do, regardless of its admins" → SCP.

⚠️ SCP vs IAM policy

SCP = org-wide boundary (can only restrict). IAM policy = grants to identities. A user needs an IAM Allow and the SCP must not deny it. SCPs alone never let anyone do anything.

Organizations · Part 3

Control Tower, Tag Policies & RAM

  • AWS Control Tower — sets up a secure, multi-account landing zone with best-practice guardrails (built on Organizations + SCPs + Config).
  • Tag policies — enforce consistent tagging across accounts.
  • AWS RAM (Resource Access Manager) — share resources (subnets, Transit Gateway, etc.) across accounts.
  • Delegated administrator — let a member account manage a service org-wide (e.g. GuardDuty).
