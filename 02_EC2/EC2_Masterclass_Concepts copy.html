CH 01

What even is a server?

Before we talk EC2, let's make sure the floor is solid.

A server is just a computer. That's it. The thing that makes it a "server" is its job: instead of one human typing on its keyboard, a server sits there 24/7 listening for requests over a network and serving them back. Your laptop could be a server. A Raspberry Pi can be a server. A massive machine in a data center is a server.

The restaurant analogy

Think of a busy restaurant. The kitchen is the server — it cooks food when orders come in. The customers are clients sending requests ("I'd like a pizza!"). The waiter is the network carrying the request from customer to kitchen and back.

One kitchen can serve hundreds of customers. If the kitchen catches fire, the restaurant closes. If you want to handle more customers, you need a bigger kitchen — or several kitchens working together.

Why this matters

Before cloud, if you wanted to launch a website, here's what you did:

  • Buy a physical computer — anywhere from $5,000 to $50,000+
  • Find a place to put it — usually rented rack space in a data center, or a closet in your office
  • Install an operating system — Linux or Windows
  • Install your software — a web server, a database, your application
  • Pay for electricity, cooling, internet, IT staff — every month, forever
  • Hope nothing breaks — and when it does, drive over to fix it

The whole process took weeks or months. And if your website got popular and needed more capacity? Buy another machine. Wait weeks. Set it up. Pray.

Why it matters in industry

Companies used to maintain entire teams of engineers whose only job was buying, installing, racking, and replacing servers. Hardware costs were measured in millions. Provisioning took 6–12 weeks. This was a ceiling on how fast a business could grow.

Then came cloud computing. And EC2 was the moment the world fundamentally changed.

CH 02

Enter EC2 — the big idea.

A computer in the cloud, ready in 60 seconds, paid by the second.

EC2 stands for Elastic Compute Cloud. Three words that capture the whole story. Elastic — it can grow and shrink. Compute — it gives you CPU and RAM, the raw stuff of computation. Cloud — it lives on AWS's planet-spanning infrastructure, accessible from anywhere.

EC2 is renting, not buying

Owning a server is like owning a car. Big upfront cost. Yours forever. You handle maintenance. If you stop using it, you still pay.

EC2 is like renting a car by the second. You walk up, pick the model (a small economy car, an SUV, a sports car, a moving truck), drive it for as long as you want, and return it. You only pay for the time you used. Need a different car next time? Easy.

And the rental lot has every car ever made, available instantly, at any moment. That's EC2.

What you get when you launch an EC2 instance

A virtual computer
Not physical hardware — a slice of a large server, isolated from other slices. Behaves exactly like a real machine.
An operating system
Linux (Amazon Linux, Ubuntu, Red Hat, etc.) or Windows. You pick.
CPU + RAM
From tiny (1 vCPU, 0.5 GB RAM) to massive (448 vCPU, 24 TB RAM).
Storage
A disk attached to it (we'll deep-dive on this in Chapter 8).
Network
An IP address and the ability to talk to the internet (or stay private).
Per-second billing
Run it for 30 seconds, pay for 30 seconds. Run it for a year, pay for a year.

Two EC2 instances in different Availability Zones, each with attached EBS storage. This is the canonical highly-available baseline.

For absolute beginners

An "instance" is just AWS's word for a running EC2 virtual machine. When you "launch an instance" you're saying "AWS, please boot up a server for me right now." Sixty seconds later, you can SSH into it like any Linux box.

EC2 was launched in 2006. Before EC2, "the cloud" wasn't really a thing yet. Spinning up a server in minutes — instead of months — was so transformative that it created entire new categories of companies. Netflix, Airbnb, Pinterest, Reddit, Stripe, Slack — none of them could have existed at the speed they grew without something like EC2.

CH 03

Where your EC2 actually lives.

Regions, Availability Zones, and why geography matters.

When you launch an EC2 instance, it has to live somewhere. AWS owns hundreds of data centers around the world, organized in a hierarchy: Regions contain Availability Zones, and AZs contain physical data centers. You always pick where your instance goes.

Regions

A Region is a geographic area, like a country or sub-continent. Each Region has a code name:

  • us-east-1 — Northern Virginia (the biggest, oldest, cheapest)
  • us-west-2 — Oregon
  • eu-west-1 — Ireland
  • eu-central-1 — Frankfurt
  • ap-south-1 — Mumbai
  • ap-northeast-1 — Tokyo

There are 30+ Regions worldwide as of 2026. You pick a Region for three reasons:

  1. Latency — Closer to your users = faster response. A user in London hitting eu-west-1 sees ~10ms latency. Hitting us-east-1, they see ~80ms. Big difference.
  2. Cost — Prices vary slightly between regions. us-east-1 tends to be cheapest. São Paulo is one of the most expensive.
  3. Compliance / data residency — Some laws (GDPR in Europe, data localization in India) require you keep customer data in-country. Pick a Region that satisfies those rules.

Availability Zones (AZs)

Each Region is divided into 3 or more Availability Zones — labelled us-east-1a, us-east-1b, us-east-1c, and so on.

An AZ is one or more physical data centers, located miles apart from other AZs in the same region. They have separate power, cooling, and network connections — but they're connected to each other by ultra-fast, low-latency private fiber (single-digit milliseconds).

Why this design exists

If a single data center has a problem — a fire, a power outage, a flood, even a backhoe cutting a fiber line — only that one AZ goes down. Your application running in two AZs at once will keep serving users from the surviving AZ. This is the foundation of "high availability" on AWS.

Real industry pattern

Production workloads always span at least 2 AZs. Critical systems often span 3. A single-AZ deployment is fine for development and testing but is considered an architectural sin in production — your boss will not be happy when the AZ goes down and the app is offline.

Quick mental model:

1 AZ goes down
Your other AZs handle the load. App keeps running. (HA — High Availability)
1 entire Region goes down
If you only deployed to one Region, you're offline. To survive this, you need multi-Region setup. (DR — Disaster Recovery)

Edge Locations & Local Zones (briefly)

Beyond Regions and AZs, AWS also has:

  • Edge Locations (400+) — small POPs near users, used by CloudFront and Global Accelerator for caching and routing. Not for EC2.
  • Local Zones — extensions of a Region in specific cities for ultra-low-latency workloads (like LA, Boston). EC2 can run here.
  • Wavelength Zones — embedded inside 5G carrier networks for mobile-edge applications.
  • Outposts — actual AWS hardware racks installed in your own data center.

For 95% of cases, you'll work with Regions and AZs only.

CH 04

The AMI — your instance template.

A "save game" of a fully configured computer.

Every EC2 instance starts from an AMI (Amazon Machine Image, pronounced "ay-em-eye"). An AMI is a snapshot of an entire system: the operating system, system files, pre-installed software, and configuration — frozen in time, ready to be cloned into new instances.

Like a "save state" for your computer

Imagine you spent hours setting up your laptop perfectly: the OS configured, your apps installed, your dev tools ready. Now imagine you could press a button and clone that entire setup into a brand new laptop in 60 seconds. That's an AMI.

Where AMIs come from

AWS-provided

Bare-bones official images: Amazon Linux 2023, Ubuntu, Red Hat, Windows Server, SUSE, Debian. Free (you only pay for the EC2). Your default starting point.

AWS Marketplace

Pre-built AMIs from vendors — a fully-configured WordPress, a hardened Linux, a security appliance. You pay an extra hourly software fee on top of EC2. Saves setup time.

Community AMIs

Free, made by other AWS users. Use with caution — security audit them first. Mostly for niche use cases.

Your own custom AMIs

You take a configured EC2 instance and "bake" it into an AMI. Now any new instance launched from this AMI starts already configured. This is the industry pattern.

The "Golden AMI" pattern

Mature engineering teams build a "golden AMI" CI/CD pipeline. Every week, a tool like Packer (by HashiCorp) starts a fresh Amazon Linux instance, installs all the company's standard software, applies security patches, runs tests, then bakes the result into a new AMI version. Auto Scaling Groups and Launch Templates point to this AMI. Result: launching a new server takes 30 seconds and the server is production-ready immediately — no slow boot scripts.

AMIs are region-specific

An AMI you create in us-east-1 only exists in us-east-1. To use it elsewhere, you have to copy it to other regions. AWS makes this easy, but it's a one-time copy operation.

Exam-style scenario

"A company wants to launch identical EC2 instances in multiple regions for disaster recovery." → Build the AMI once, then copy the AMI to each target region. Use Launch Templates pointing to the regional AMI ID.

What's actually inside an AMI

  • One or more EBS snapshots (the disk contents)
  • Launch permissions — who is allowed to use this AMI
  • Block device mapping — what disks attach where on launch
  • Architecture — x86_64, ARM64 (Graviton), etc.
  • Virtualization type — HVM (modern) or PV (legacy, dying out)

You'll see AMIs identified by IDs like ami-0abcdef1234567890. These IDs are unique per region — the same image, copied to a different region, gets a new ID.

CH 05

Instance types — decoding the names.

Once you crack the naming code, picking the right one is easy.

EC2 offers over 750 different instance types. Names like t3.micro, m5.large, c6i.4xlarge, r7g.16xlarge, p4d.24xlarge. It looks like alphabet soup. But there's a clean, consistent naming convention that makes it easy.

Every EC2 instance name follows this pattern: familygenerationattributes.size

The optional letter suffix

You'll see letters like m6i, m6a, m6g, m5d, m5n:

  • i — Intel processor
  • a — AMD processor (typically ~10% cheaper than Intel for similar perf)
  • g — AWS Graviton ARM processor (cheapest, most power-efficient — but x86 software might need re-compilation)
  • d — has local NVMe SSD instance store (more on this in Ch 8)
  • n — enhanced networking (higher bandwidth)
  • e — extended memory

Sizes — what "xlarge" actually means

Each step doubles the resources (roughly):

nano  → micro → small → medium → large → xlarge → 2xlarge → 4xlarge → 8xlarge → 16xlarge → 24xlarge → 32xlarge → metal

An m5.large has 2 vCPU and 8 GB RAM. An m5.xlarge has 4 vCPU and 16 GB RAM. Each step roughly doubles cost and resources. .metal means you get the entire physical server (no virtualization).

Pro tip — picking instance size

Don't guess. Use AWS's Compute Optimizer service. It looks at your actual CPU, RAM, network, and disk metrics and recommends the right instance type. Most companies discover they're 40–60% over-provisioned when they first run it. Translates to massive savings.

Exam pattern

The exam will give you a workload description and you pick the family. Memorize these triggers:

  • "Cache" / "in-memory database" → R family
  • "High CPU" / "batch processing" → C family
  • "Web app, balanced load" → M family
  • "Low cost, sporadic traffic" → T family
  • "Machine learning training" → P family
  • "Video encoding / graphics" → G family
  • "NoSQL with massive IOPS" → I family
  • "SAP HANA" → X family
  • "Tightly-coupled HPC" → Hpc family

Burstable instances (T-series) — read carefully

The T family is special. T instances have a low baseline CPU (e.g., 20% on a t3.medium). When idle, they accumulate CPU credits. When they need to burst above baseline, they spend credits.

Two modes:

  • Standard mode — Run out of credits = throttled to baseline. Free.
  • Unlimited mode — Burst freely; if you exceed credits over a 24-hour rolling window, you pay extra per CPU-hour.
The classic T-instance trap

A team puts a production web server on a t3.medium in Unlimited mode. Traffic spikes one weekend. Monday morning, the AWS bill shows hundreds of dollars in CPU credit overages. Lesson: for production with sustained CPU demand, use M, C, or R — not T.

CH 06

The EC2 lifecycle.

Every state your instance can be in, and what you pay for each.

EC2 instances move through a small set of well-defined states. Knowing what each state means — what's preserved, what's lost, what you're billed for — is foundational. It also shows up on the exam constantly.

EC2 instance lifecycle. Stopping → Stopped → Running is your normal "pause" loop. Terminate is the one-way door.

The states explained

Pending
The instance is starting up. AWS is provisioning hardware, attaching storage, booting the OS. Takes ~30–90 seconds. Not billed yet.
Running
Live and serving traffic. Billed per second (Linux) or per hour (some Windows). EBS volumes attached are also billed.
Stopping → Stopped
You issued "Stop." OS shuts down cleanly. Not billed for compute, but EBS volumes still cost you. Public IP is released (unless you have an Elastic IP). You can start it again later.
Hibernated
RAM contents are saved to the encrypted EBS root volume, then instance stops. When you start again, RAM is restored — applications resume exactly where they were. Boot in seconds.
Terminating → Terminated
The instance is permanently deleted. By default, root EBS volume is also deleted (configurable). You cannot recover a terminated instance.

Stop vs Terminate vs Reboot — the critical distinctions

ActionWhat happensEBS data?Instance store data?Public IP?
RebootOS restart, same instancePreservedPreservedPreserved
StopInstance powered offPreservedLOSTLost (unless Elastic IP)
HibernateRAM dumped to EBS, then stoppedPreservedLOSTLost (unless EIP)
TerminateInstance deleted foreverDeleted (default)LOSTReleased
The most common student mistake

Confusing stop with reboot. A reboot is a quick OS restart — same hardware, same IP, no data loss. A stop powers off the VM, may move it to different hardware on next start, releases the public IP, and wipes any instance store data. Both feel like "turning off and on" but they're very different.

Exam scenario

"A user reports that their instance store data disappears every Monday morning." → Someone is stopping/starting the instance over the weekend. Tell them to use EBS for persistent data, or to stop the stop/start cycle.

Termination Protection

You can enable termination protection on critical instances. With it on, you can't terminate the instance via the API or console without first disabling protection. Cheap insurance against an oops.

CH 07

Pricing models — how to pay less.

Master this and you save your company millions. Seriously.

EC2's biggest weapon is its pricing flexibility. You can pay full price (On-Demand) or take massive discounts in exchange for commitment (Reserved, Savings Plans) or interruptibility (Spot). Real cloud architects don't pick one — they orchestrate a portfolio. This is also one of the heaviest exam topics.

If you put 100% of your EC2 fleet on On-Demand, you're a beginner. If you put 100% on Spot, you're reckless. Real architects mix.

1. On-Demand

The default. Pay per second (Linux) or per hour (Windows). No commitment, no upfront. Most expensive — but most flexible.

  • Use for: Unpredictable workloads, short-term experiments, dev environments that turn off at night, infrastructure with traffic spikes
  • You can launch and terminate any time without penalty
  • Industry rule: Anything running < 25% of the time is cheaper on On-Demand than on Reserved

2. Reserved Instances (RI)

Commit to a specific instance type for 1 or 3 years. In exchange, get up to 72% off the On-Demand price. The longer the commitment and the more you pay upfront, the bigger the discount.

RI flavors

TypeLocked toDiscountNotes
Standard RISpecific instance typeUp to 72%Cheapest. Can't change instance family.
Convertible RISpecific instance, but exchangeableUp to 66%Can swap to bigger/different family later.

Payment options: All Upfront (biggest discount) → Partial Upfront → No Upfront (smallest discount).

Bonus: there's a Reserved Instance Marketplace where you can sell unused RIs to other AWS customers if your needs change.

3. Savings Plans — the modern Reserved

Released in 2019. Same idea as RIs but more flexible: instead of committing to a specific instance type, you commit to a dollar amount per hour for 1 or 3 years.

Compute Savings Plan

Most flexible. Up to 66% off. Applies to EC2, Fargate, AND Lambda across any region, instance family, OS, or tenancy. Switch from m5 to c6g and the discount follows. Industry favorite.

EC2 Instance Savings Plan

Less flexible, deeper discount. Up to 72% off. Locked to a specific instance family (e.g., M5) in a specific region. You can change size and OS within the family.

Industry reality

Most modern engineering teams use Compute Savings Plans, not Reserved Instances. The 6% smaller discount is worth the dramatically improved flexibility — you don't have to predict your exact instance types 3 years in advance, which nobody can really do anyway.

4. Spot Instances — up to 90% off, with a catch

AWS sells unused EC2 capacity at a steep discount — up to 90% off. Catch: AWS can take it back at any time with a 2-minute warning.

This sounds scary. It isn't, if you architect for it.

Good fits for Spot

  • Batch processing jobs that can resume from a checkpoint
  • Big data crunching (Spark, Hadoop)
  • Machine learning training (save model checkpoints, resume on new instance)
  • CI/CD build farms
  • Stateless web servers behind a load balancer (lose one, ASG launches another)
  • Video transcoding

Bad fits for Spot

  • Databases (data corruption risk on sudden termination)
  • Stateful services holding important user sessions
  • Anything with hard real-time deadlines

Spot Fleet & Spot pools

You define a Spot Fleet: "give me 100 vCPUs across these instance types and AZs." AWS picks whichever pool is cheapest. If your AZ-a m5.large gets reclaimed, the fleet automatically shifts to AZ-b c6i.xlarge. Resilient, cheap.

A real Spot story

Lyft runs much of its ML and batch workloads on Spot, saving an estimated $10M+ per year. Pinterest uses Spot for image processing pipelines. Netflix uses Spot heavily for its video encoding pipeline — when an interruption happens, the job just resumes on a new instance from the last checkpoint.

5. Dedicated Hosts & Dedicated Instances

Most EC2 instances run on shared hardware (you have your own VM, but the physical server is shared with other AWS customers). Two ways to not share:

Dedicated Instance

Your VM runs on hardware dedicated to your AWS account. You don't see or control the hardware. Cheaper than Dedicated Host. Use: regulatory isolation requirements.

Dedicated Host

You rent the entire physical server. You see CPU sockets, can place instances on specific cores. Use: Bring-Your-Own-License (Oracle, SQL Server, Windows) where licenses are tied to physical CPU sockets. Most expensive.

6. Capacity Reservations

Reserve capacity in a specific AZ at On-Demand pricing. You're not getting a discount — you're guaranteeing the capacity is available when you need it. Combine with a Savings Plan to also get the discount.

Use case: Disaster recovery. You want to know that during a regional crisis when everyone is rushing to scale up, your DR fleet is guaranteed available.

The portfolio approach

Typical mature company mix: a Savings Plan baseline, On-Demand for variable, Spot for elastic batch.

Exam decision tree
  • "Steady, predictable workload, 24/7" → Reserved or Savings Plan
  • "Need maximum flexibility, applies to Lambda & Fargate too" → Compute Savings Plan
  • "Cheapest possible, batch job, fault-tolerant" → Spot
  • "Unpredictable, can't commit, occasional use" → On-Demand
  • "BYOL Oracle/Windows licenses tied to hardware" → Dedicated Host
  • "Need to guarantee capacity is available, no discount needed" → Capacity Reservation
CH 08

EC2 storage deep-dive.

EBS, Instance Store, EFS — and which to pick when.

An EC2 instance without storage is useless. AWS gives you three fundamentally different storage models for EC2 — and one of the most common architectural mistakes is picking the wrong one. By the end of this chapter, you'll always know which to use.

Three storage worlds. Instance Store lives inside the host. EBS and EFS are network-attached.

EBS — Elastic Block Store

The default, most-used EC2 storage. EBS gives you a virtual hard drive (a "volume") that's network-attached to your instance. You can format it, partition it, mount it — it behaves exactly like a physical disk.

Persistence
Survives independently of the EC2 instance. Stop your instance, the EBS volume keeps your data. Terminate the instance, the volume is deleted by default — but you can flip a setting to keep it.
AZ-bound
An EBS volume lives in one specific AZ. You can only attach it to instances in the same AZ. To move across AZs, take a snapshot, then create a new volume from snapshot in the new AZ.
One instance at a time
Normally, an EBS volume is attached to one instance. Exception: io1 and io2 volumes support Multi-Attach — up to 16 instances in the same AZ can share one volume (but you need a cluster-aware filesystem).
Resizable
You can grow an EBS volume on the fly with no downtime. Shrinking? Not possible — you'd take a snapshot, create a smaller new volume, copy data over.

Exam triggers for EBS types
  • "General-purpose, cost-effective" → gp3
  • "High-performance database, > 16,000 IOPS" → io1 / io2
  • "Big data, sequential reads, cheap" → st1
  • "Cheapest, rarely accessed" → sc1
  • "Multi-instance shared block storage in same AZ" → io1/io2 Multi-Attach

EBS Snapshots

A snapshot is a point-in-time backup of an EBS volume, stored in S3 (you don't see the bucket — AWS manages it). Snapshots are incremental: only blocks changed since the last snapshot are stored, but every snapshot logically represents the full volume.

  • You can copy snapshots across regions (for DR)
  • You can copy snapshots across AWS accounts
  • You can restore a snapshot into a new EBS volume in any AZ in that region
  • You can create an AMI from a snapshot
  • Snapshot Archive — 75% cheaper, 24–72-hour restore time. For long-term backups you rarely touch.
  • Recycle Bin — protects against accidental deletion (configurable retention period)

EBS Encryption

Free, easy, KMS-backed. When enabled:

  • Data at rest on the volume is encrypted
  • Data in flight between EC2 host and EBS is encrypted
  • Snapshots are encrypted
  • AMIs created from encrypted volumes are encrypted

Quirk: a snapshot of an unencrypted volume is unencrypted. To encrypt it, copy the snapshot with the encrypt flag — that creates an encrypted copy.

Best practice: enable EBS encryption by default at the account level. Set it once, every new volume is encrypted automatically.

Instance Store — the wild fast one

Some EC2 instance types have local NVMe SSDs physically attached to the host server (instance types ending in d: m5d, r5d, i3, i4i, etc.). This is called Instance Store or ephemeral storage.

Speed
Insanely fast. Millions of IOPS, multi-GB/s throughput. Faster than any EBS option.
Cost
Included in the instance price. No extra charge.
Persistence
NONE. Data is wiped when you stop, hibernate, terminate, or if the host hardware fails. The disk is physically tied to one piece of hardware.
Detach?
You cannot. The disk is welded to the host.
The Instance Store trap

A junior engineer mounts the instance store, copies their database files there for "speed," doesn't realize that next time anyone stops the instance, all the data is gone. Instance store is for ephemeral data only: caches, temporary buffers, scratch space, distributed databases that handle their own replication.

Good fits for Instance Store

  • Local cache layers
  • Temporary processing space (video editing, batch crunching)
  • Distributed databases that replicate across nodes themselves: Cassandra, Elasticsearch, MongoDB shards, Kafka brokers. They lose one node and seamlessly use another.
  • Scratch disk for ML training

EFS — quick mention (covered in its own topic)

EFS is a managed NFS file share. Many EC2s — even across multiple AZs — can mount and share the same EFS file system. Linux only. Pay for what you use.

If a question says "shared file storage for multiple EC2s" → EFS. We'll go deep when we cover the storage topic.

A real industry layout

A typical 3-tier web app on EC2:

  • Root volume — gp3 EBS, 30 GB, encrypted (the OS)
  • App data — gp3 EBS, 100 GB, snapshotted nightly
  • Database — io2 Block Express EBS for hot OLTP
  • Cache layer (Redis on EC2) — memory + instance store backing
  • Static assets shared across web tier — EFS
CH 09

EC2 networking essentials.

Just the EC2-specific bits — the full VPC topic gets its own chapter.

VPC, subnets, route tables, IGWs — those belong to the Networking topic. But there are a few networking concepts specific to EC2 that you absolutely need: Security Groups, Elastic Network Interfaces, Elastic IPs, and the public-vs-private IP behavior.

Security Groups (SG)

A Security Group is a virtual firewall at the instance level. Each EC2 instance has at least one (and up to 5).

  • Allow rules only. No "deny" — if something isn't explicitly allowed, it's blocked. (For deny rules, you use a Network ACL at the subnet level.)
  • Stateful. If you allow inbound on port 80, the response traffic is automatically allowed back out. You don't write outbound rules for return traffic.
  • Rules can reference IP ranges (CIDR) like 0.0.0.0/0 (anywhere) or other Security Groups ("allow traffic from any instance in this other SG").
  • Default: deny all inbound, allow all outbound.
Concrete example

A web server SG might say:

  • Inbound: TCP 443 from 0.0.0.0/0 (HTTPS from anywhere)
  • Inbound: TCP 22 from 203.0.113.0/24 (SSH only from your office's IP range)
  • Outbound: all (default — server can call out wherever)
SG-to-SG references — the pro move

Best practice: don't whitelist IPs of your other servers. Reference the other server's SG instead. "Allow port 3306 (MySQL) from web-tier-sg." Now any new web server you add automatically gets database access — no firewall rule edits needed.

Elastic Network Interface (ENI)

An ENI is a virtual network card. It has:

  • A primary private IP
  • One or more secondary private IPs
  • An Elastic IP per private IP (optional)
  • A MAC address
  • One or more security groups
  • A source/destination check flag

ENIs can be created independently and attached to instances later. They can also be detached from one instance and reattached to another. This is sometimes used as a low-tech HA trick: keep an ENI's IP/MAC stable while moving it to a standby instance during failover.

Public IP vs Private IP vs Elastic IP

Private IP
Internal IP within your VPC. Always present, never changes for the life of the instance. Used for instance-to-instance traffic.
Public IP
Optional. Auto-assigned to instances launched in public subnets (configurable). Changes every time you stop and start the instance. Free.
Elastic IP
A static public IPv4 you reserve. Tied to your account. You assign it to an instance, and it stays the same across stop/start. Free while attached to a running instance, charged when idle (~$3.60/month) — AWS encourages you to release them when not used.

Default limit: 5 Elastic IPs per region (request a higher limit if needed).

Best practice

Avoid Elastic IPs when possible. Instead, put your servers behind a load balancer and give users the load balancer's DNS name. The LB handles instance failures gracefully — no IP juggling required.

Hyperplane & Source/Dest Check

For NAT instances, transparent firewalls, and routing appliances, you need to disable the "Source/Destination check" on the ENI. Otherwise, AWS drops packets that don't have the instance's own IP as source or dest. This is a niche but exam-relevant detail.

CH 10

Placement Groups.

Tell AWS exactly how to physically arrange your instances.

Normally, AWS decides where on its hardware to place your instances — randomly across racks, hosts, etc. For most workloads, that's fine. But sometimes you want specific control: pack them tight for speed, or spread them out for resilience. That's a Placement Group.

Three strategies: Cluster (pack), Spread (isolate), Partition (group + isolate).

Cluster

Pack instances close together — same rack, same AZ. Lowest network latency between instances (single-digit microseconds), highest bandwidth (up to 10 Gbps single-flow, 100 Gbps aggregate).

  • Use: HPC, scientific simulations, financial trading systems, ML training nodes that must talk to each other constantly.
  • Risk: If the rack fails, all instances fail together. No HA inside a Cluster group.

Spread

Each instance on completely different physical hardware. Maximum hardware-failure isolation.

  • Limit: Max 7 instances per AZ per Spread group.
  • Use: Critical instances that must never fail together — e.g., 3 controllers of a system, primary/standby DB pairs, distinct master nodes of a small cluster.
  • Span AZs? Yes — use multi-AZ Spread groups for ultimate isolation.

Partition

The hybrid. You define up to 7 partitions per AZ. Instances within a partition can share hardware, but partitions don't share with each other. Hadoop-style systems love this.

  • Use: Distributed workloads that are aware of partition topology — Hadoop, Cassandra, Kafka, HBase. They're designed so a single partition failure doesn't take out a quorum.
  • Hundreds of instances can be in a single Partition group.
Exam triggers
  • "Lowest latency between instances" / "HPC" → Cluster
  • "Maximum availability, can't have correlated failure" → Spread
  • "Hadoop / Cassandra / Kafka / partition-aware system" → Partition
CH 11

Bootstrap, metadata & access.

User Data, IMDS, IAM Roles, Session Manager — the daily-driver toolkit.

Once an instance is launched, how do you set it up automatically? How does it learn about itself? How does it get permission to call AWS APIs without storing passwords? These are everyday operational concerns — and they're heavily tested.

User Data — the boot script

When you launch an EC2, you can paste in a script (Bash for Linux, PowerShell for Windows) that runs once on first boot, as the root user. Use it to install packages, fetch config, register with monitoring, etc.

#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "<h1>Hello from $(hostname)</h1>" > /var/www/html/index.html

Limits: max 16 KB of User Data. Runs once unless you reset it. Logs to /var/log/cloud-init-output.log.

Industry pattern: don't do too much in User Data

For complex setups, User Data should be minimal: maybe pull a configuration from an S3 bucket, or run an Ansible/Chef script. The bulk of the configuration should live in a baked AMI (golden image). Boot scripts are slow, fragile, and hard to debug.

Instance Metadata Service (IMDS)

From inside any EC2 instance, there's a magic URL: http://169.254.169.254/latest/meta-data/. Hit it from the instance and it tells you about itself:

curl http://169.254.169.254/latest/meta-data/instance-id
# → i-0abc123def456789

curl http://169.254.169.254/latest/meta-data/instance-type
# → m5.large

curl http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole
# → temporary AWS credentials (JSON)

IMDSv1 vs IMDSv2 — pay attention

  • IMDSv1 — simple HTTP GET. Vulnerable to SSRF attacks (a flaw in your web app could let an attacker fetch credentials). Deprecated.
  • IMDSv2 — requires you first PUT to get a session token, then send the token as a header on subsequent GETs. Defeats SSRF. Use this. Make it required at the account level.

IAM Roles for EC2 — never store credentials on the instance

Your EC2 instance often needs to call AWS APIs: read from S3, write to DynamoDB, publish to SQS. The wrong way: store an access key + secret key in a config file on the instance. (If the instance gets compromised, your keys leak. If you rotate the keys, every instance breaks.)

The right way: attach an IAM Role to the instance. The role has policies. The instance gets temporary credentials from IMDS that auto-rotate. AWS SDKs pick them up automatically.

A core SAA exam concept

Any time the question is about how an EC2 instance authenticates to AWS, the answer is IAM Role attached to the instance — never IAM User access keys. Hardcoded credentials on EC2 = automatically wrong answer.

How to actually log into your instance

1. SSH (the old classic)

Generate a key pair when you launch the instance. AWS keeps the public key, you keep the private key (a .pem file). Then:

ssh -i my-key.pem ec2-user@<instance-public-ip>

2. EC2 Instance Connect (browser-based)

From the AWS console, click "Connect." A web SSH session opens. AWS injects a temporary key for 60 seconds. No need to manage .pem files. Works on instances with the EC2 Instance Connect agent installed (default on recent Amazon Linux).

3. AWS Systems Manager Session Manager — the modern way

The industry standard

Mature teams have stopped using SSH entirely in production. They use SSM Session Manager:

  • No SSH port (22) open — your servers are completely closed off from the internet
  • No bastion host needed
  • No SSH keys to manage
  • Authenticated via IAM — same identity as everything else
  • Every session is audit-logged to S3 / CloudWatch
  • Works in private subnets via VPC Endpoints

This pattern is dramatically more secure and is now the default expectation for production AWS environments.

Hibernate

Hibernate is a special variant of stop. Instead of just turning the OS off, AWS dumps the contents of RAM to the encrypted root EBS volume. When you resume, RAM is restored and your processes pick up exactly where they left off (like closing a laptop lid).

Constraints:

  • Specific instance types only (Nitro-based, RAM < 150 GB)
  • EBS root volume must be encrypted
  • Maximum 60 days hibernated
  • Linux and some Windows

Use case: long-running interactive workloads where boot/setup is expensive (data scientist's GPU instance, dev workstation).

EC2 Auto Recovery

If the underlying physical hardware fails, AWS can automatically recover your instance: launch a new instance on different hardware, attach the same EBS volumes, keep the same private/public IPs, and resume. Free, just enable it via a CloudWatch alarm.

Note: this recovers from hardware failure, not OS-level issues. For OS-level resilience you still want ASG + Load Balancer.

CH 12

Nitro, ENA, EFA & performance.

The modern hardware platform under EC2 — and why it matters.

EC2 is in its second major hardware generation. The first generation used a Xen-based hypervisor — solid, but with overhead. Around 2017 AWS started rolling out the Nitro System, a custom hypervisor + custom hardware cards that handle networking, storage, and security in dedicated silicon. By 2026, almost every modern instance type is Nitro-based.

The Nitro System

  • Higher performance — near-bare-metal speeds, lower latency, more throughput
  • Better security — security functions moved to dedicated hardware, smaller hypervisor attack surface
  • Faster networking — up to 100+ Gbps
  • Faster EBS — Multi-Attach, EBS Block Express, no encryption performance penalty
  • Required for newer features — ENA, EFA, Hibernate, EBS encryption performance

ENA — Elastic Network Adapter

Nitro's modern network interface. Up to 100 Gbps on supported instance types. Default on all Nitro instances. You don't have to do anything special — just pick a Nitro instance.

EFA — Elastic Fabric Adapter

A specialized network interface for HPC and ML workloads. Bypasses the OS kernel for networking ("OS bypass" via Libfabric), achieving sub-microsecond latency between nodes. Used with cluster placement groups for tightly-coupled HPC.

  • Use case: distributed ML training across many GPU nodes; CFD simulations; weather modeling
  • Linux only
  • Available on specific instance types (c5n, c6i, p4d, hpc6a, etc.)

Graviton — AWS's ARM processor

The little g in instance names like m6g, c7g, r7g. Graviton is AWS's custom ARM processor. Compared to x86:

  • ~20% better price/performance
  • Lower power consumption
  • Competitive raw performance
  • Catch: Your software has to be ARM-compatible. Most major languages and frameworks (Java, Python, Node, Go, Rust) just work. Some compiled C/C++ binaries need rebuilding. Most container images need an ARM build.
The industry trend

By 2026, Graviton runs significant percentages of AWS's largest customers' fleets. AWS itself runs much of its own infrastructure on Graviton. If your workload is portable, switching to Graviton is one of the biggest cost wins available — often 20–40% off your bill with minimal code changes.

Enhanced Networking & SR-IOV

"Enhanced networking" is an umbrella term for low-overhead, high-throughput networking. On older instances this was achieved via SR-IOV; on Nitro instances it's the default via ENA. The exam may mention "Enhanced Networking" — it just means high-bandwidth, low-CPU networking, on by default for modern instances.

CH 13

Industry patterns & best practices.

What separates a junior cloud engineer from a senior architect.

You can pass the SAA-C03 by memorizing services. You can do real-world architecture by understanding the patterns. Here are the unwritten rules that experienced architects live by.

1. Tag everything

Every resource — instance, EBS volume, snapshot, AMI, security group — gets tags. The standard tag set most companies enforce:

  • Name — human-readable
  • Environment — prod / staging / dev
  • Owner — team or person responsible
  • Project or Application
  • CostCenter — for chargeback

Without tags you cannot do cost allocation, cleanup, or compliance. Companies enforce tag policies via SCPs and AWS Config.

2. Don't run standalone EC2 in production

If a production-bound EC2 instance is not behind an Auto Scaling Group, you've probably designed it wrong. ASG gives you:

  • Auto-replacement of failed instances
  • Multi-AZ deployment
  • Auto-scaling for traffic
  • Rolling deploys

"ASG of one" is even an acceptable pattern: a single instance that auto-replaces itself if it dies.

3. Infrastructure as Code, always

Don't click around in the console for production resources. Use Terraform (the de-facto industry standard) or CloudFormation / CDK (AWS-native). Reasons:

  • Reviewable in pull requests
  • Version-controlled history of every change
  • Reproducible across environments
  • No "snowflake" servers

4. The "cattle, not pets" mindset

Servers are cattle, not pets. You don't name them. You don't nurse them when they're sick. You replace them.

Old-school operations: Each server has a name (Apollo, Athena, Zeus). When one acts up, you SSH in and fix it. You hand-tune its config over years.

Cloud-native operations: Servers are interchangeable. They're auto-launched from a baked AMI by ASG. If one is misbehaving, terminate it — ASG will spawn a fresh one in 30 seconds. You never SSH into them in production.

5. Right-size with Compute Optimizer

AWS Compute Optimizer analyzes 14 days of metrics and recommends optimal instance types. Companies running it for the first time typically find 20–50% over-provisioning. Run it quarterly.

6. Mix your pricing models

The portfolio approach (Ch 7): Savings Plans for the predictable baseline, On-Demand for elasticity, Spot for fault-tolerant batch. Most mature shops are 60–80% covered by Savings Plans / RIs.

7. Patch via Systems Manager Patch Manager

Don't manually run yum update across hundreds of servers. SSM Patch Manager applies OS patches on a schedule, with reporting, rollback, and maintenance windows. Or, better yet, bake new AMIs and roll them out via ASG instance refresh — "immutable infrastructure."

8. Encrypt everything at rest, by default

Account-level setting: EBS encryption by default — ON. Use customer-managed KMS keys for sensitive workloads (so you can audit who used the key). Snapshots inherit encryption.

9. CloudWatch is your eyes

CloudWatch metrics for CPU, network, disk are free and automatic. RAM and disk usage from inside the instance require the CloudWatch Agent — a common gap. Install the agent on every instance via User Data or AMI baking.

10. Use Session Manager, not SSH

Already covered, but worth repeating. The audit-log alone justifies it for any compliance-bound shop.

The 12 anti-patterns to avoid

  • ❌ Hardcoding AWS credentials in code on EC2
  • ❌ Single-AZ production deployment
  • ❌ Standalone EC2 not in an ASG (for prod)
  • ❌ Persistent data on Instance Store
  • ❌ T-instance in unlimited mode for sustained-load production
  • ❌ SSH port 22 open to 0.0.0.0/0
  • ❌ IMDSv1 still enabled
  • ❌ Click-ops in prod (no IaC)
  • ❌ "Pet" servers with hand-tuned config
  • ❌ Missing tags
  • ❌ All On-Demand pricing for steady workloads
  • ❌ Unencrypted EBS volumes
CH 14

How real companies use EC2.

Pattern recognition from the giants.

EC2 was launched in 2006. Twenty years in, the patterns are well-understood. Here's how some of the largest companies on AWS architect their EC2 fleets — and what you can learn from each.

Netflix

The poster child of cloud-native. Netflix runs tens of thousands of EC2 instances across multiple regions, all serving 200M+ subscribers' streaming.

  • Custom AMIs baked via their open-source tool Spinnaker. Every code change → new AMI → ASG rolling deploy. Truly immutable infrastructure.
  • Heavy Spot usage for video encoding. Encoding a movie in 100+ qualities is fault-tolerant by nature — losing a Spot instance just retries that segment.
  • Reserved capacity for the streaming serving fleet, which has steady predictable load.
  • Multi-region active-active for streaming services — full DR by design.
  • Open-sourced Chaos Monkey — a tool that randomly terminates EC2 instances in production to validate resilience.

Airbnb

Migrated from a Ruby monolith to microservices on AWS over many years.

  • Mix of EC2 and ECS containers
  • Aggressive Reserved/Savings Plan coverage for steady workloads
  • Auto Scaling Groups for the booking-traffic-driven fleets (highly seasonal!)

Lyft

Heavy Spot user — saves an estimated $10M+ annually on machine learning workloads.

  • GPU instances (p3, p4) for ML model training
  • Spot Fleets for batch ML training jobs
  • Custom autoscaling logic for ride-demand prediction

Pinterest

Image-heavy infrastructure.

  • R-family for in-memory caches and Redis
  • I-family with instance store for high-IOPS image search indexes
  • Spot for batch image processing pipelines

The general production layout

The "boring 3-tier app." Web (M-family) → App (C-family) → Data (R-family RDS + ElastiCache).

The decision-making muscle

What separates an architect from someone who memorized services? Pattern recognition. When you see a problem, you should be matching it to a pattern, not searching through a list of services.

SituationPattern
"Web app handling unpredictable traffic"ALB + ASG (M-family) across 2+ AZs, On-Demand or Compute Savings Plan
"Internal high-CPU batch processing"Spot Fleet on C-family, fault-tolerant via SQS queue + retries
"In-memory cache layer"R-family or ElastiCache (managed)
"Highly performant DB on EC2"R-family with io2 EBS, in private subnet, Multi-AZ via DB-level replication
"GPU ML training"P-family with EFA + Cluster Placement Group, Spot Fleet
"Distributed Hadoop/Cassandra"I-family with instance store, Partition Placement Group
"Windows app with vendor licensing"Dedicated Host (BYOL)
"Dev environment that turns off at night"On-Demand, scheduled stop/start via EventBridge
EC2 doesn't reward memorization. It rewards judgment about trade-offs: speed vs cost, isolation vs density, managed vs self-managed.

The takeaway

Master EC2 and you've mastered the foundation of AWS. Almost every other AWS service is either built on EC2, talks to EC2, or is a managed alternative to EC2. RDS is managed databases on EC2. Lambda is serverless EC2. ECS is containers on EC2. Auto Scaling Groups orchestrate EC2. Load Balancers route to EC2. CloudWatch monitors EC2.

If you internalize the chapters above — instance families, pricing models, storage trade-offs, placement, lifecycle, and the network surroundings — you're already 60% of the way to the SAA-C03 because EC2 patterns echo through every other domain.
