Section OneWhy ELB & ASG carry exam weight

ELB and ASG sit inside the Resilient Architectures (26%) and High-Performing Architectures (24%) domains — almost half the exam combined. They also overlap with Security (Security Groups in front of ALB, ALB+Cognito auth, ALB+WAF) and Cost (Spot in ASG, Savings Plans for steady ASG fleets), which means they touch every domain.

From pass-story analysis, expect 4–7 questions that hinge directly on:

High-frequency triggers

ALB vs NLB

  • "static IP" / "preserve client IP"
  • "path-based routing" / "host-based"
  • "millions of requests per second"
  • "WebSocket" / "TCP/UDP traffic"
High-frequency triggers

ASG scaling decisions

  • "keep CPU at 50%" → Target Tracking
  • "scale at known peak times" → Scheduled
  • "forecast future load" → Predictive
  • "different magnitudes by threshold" → Step
High-frequency traps

The classic gotchas

  • ALB has NO static IP
  • EC2-only health checks miss broken apps
  • NLB cross-zone is OFF by default + costs money
  • Sticky sessions ≠ shared session store
High-frequency traps

Don't pick these on the exam

  • Classic Load Balancer — deprecated
  • Launch Configuration — deprecated
  • Simple Scaling — Step Scaling supersedes it
  • Vertical scaling — never the HA answer

Section ThreeALB deep cheat

Routing rule types

  • Path — /api/*
  • Host header — shop.example.com
  • HTTP header — X-Tenant: gold
  • Query string — ?version=v2
  • HTTP method — POST only
  • Source IP — CIDR-based
  • Weighted target groups — for blue/green

Targets ALB supports

  • EC2 instances
  • IP addresses (incl. on-prem via VPN/DX)
  • Lambda functions
  • Containers (ECS / EKS / Fargate)

Lambda targets enable serverless behind a corporate URL.

Built-in features

  • HTTPS w/ ACM certs (free, auto-renew)
  • SNI for multiple certs on one LB
  • Cognito / OIDC / SAML authentication
  • WAF integration
  • HTTP/2 + WebSocket
  • Sticky sessions (cookie-based)
  • Access logs to S3

ALB will NOT

  • Give you a static IP
  • Handle non-HTTP protocols
  • Preserve source client IP (uses XFF)
  • Replace WAF — it integrates with WAF
  • Work without subnets in ≥2 AZs
The most-tested ALB trap

Question describes "customers connect via a hardcoded IP from a corporate firewall whitelist". ALB has no static IP — wrong answer. Two correct paths: use NLB (static IPs per AZ, can attach Elastic IPs), OR keep ALB + put Global Accelerator in front (gives 2 static anycast IPs and routes to ALB). On the exam, both can appear; pick the one that matches the rest of the question.

Section FourNLB deep cheat

What only NLB does

  • Static IP per AZ (assignable Elastic IPs)
  • TCP / UDP / TLS traffic
  • Source IP preservation by default
  • Millions of requests/sec sustained
  • ~Microsecond latency (Hyperplane)
  • PrivateLink endpoint service creation

Use NLB for

  • Gaming servers (UDP)
  • IoT / MQTT / SMTP / FTP
  • Financial / trading apps
  • Whitelist-bound clients
  • Workloads needing stable IPs forever
  • Exposing services via PrivateLink

NLB will NOT

  • Do path / host / header routing
  • Authenticate with Cognito
  • Integrate with WAF directly
  • Have cross-zone on by default
  • Work for HTTP routing (use ALB instead)

NLB + ALB combos

  • NLB → ALB (static IP front, HTTP routing back)
  • NLB exposed via PrivateLink to other VPCs
  • NLB for Kubernetes ingress with raw TCP
NLB cross-zone — billable!

Unlike ALB (cross-zone always on, free), NLB cross-zone is OFF by default and turning it on may incur regional data-transfer charges. If a question mentions "cost-effective" alongside NLB and uneven AZ distribution, the answer may be to keep cross-zone OFF and rebalance instances per AZ instead.

Section FiveGWLB deep cheat

Memorise these GWLB facts

  • Layer 3 / 4 — works with raw IP packets
  • Uses GENEVE protocol on port 6081
  • Pairs with a GWLB Endpoint (GWLBe) in your VPC
  • Designed to insert third-party virtual appliances in-line
  • Works transparently — no changes to applications

Trigger phrases on the exam

  • "third-party firewalls"
  • "transparent traffic inspection"
  • "deep packet inspection appliances"
  • "intrusion detection / prevention systems"
  • Mention of Palo Alto / Fortinet / Check Point
  • "GENEVE" / "port 6081"
GWLB trap

GWLB is NOT for load balancing your application servers. If the question says "balance traffic to my web fleet," GWLB is wrong even if it looks tempting. GWLB is plumbing for security appliances, full stop.

Section SixHealth checks & stickiness

Health check parameters worth memorising

Success codesHTTP codes that count as healthy200

Sticky sessions — what to know

  • ALB uses cookies. Two flavours: application-based cookie (your app sets it) or load-balancer-generated cookie (AWSALB).
  • NLB stickiness is based on source IP (since there's no HTTP).
  • Stickiness keeps a user pinned to one target for a configurable duration.
Decision rule

Question says "shopping cart disappears between requests" or "login state lost":

  • Quick fix answer → enable sticky sessions
  • Best long-term / scalable / "best practice" answer → store session in ElastiCache for Redis (or DynamoDB)

Read the qualifier carefully — both can be "right."

EC2 vs ELB health checks (ASG setting)

EC2 health check (default)

Checks only the instance's status — running, stopped, terminated. Doesn't know if your app is actually responding.

An EC2-healthy but app-broken instance keeps receiving traffic.

ELB health check (turn this on)

If the LB marks the target unhealthy, ASG also treats the instance as unhealthy and replaces it.

This is the reliability-correct setting — almost always pick this.

Section SevenSSL/TLS, SNI, security groups for LBs

SSL/TLS termination

  • Use ACM certificates — free + auto-renew
  • Public certs: ACM (US East 1 for CloudFront)
  • Internal certs: ACM Private CA
  • Backend connection can be HTTP or HTTPS — your choice

SNI — multiple certs, one LB

  • ALB and NLB both support SNI
  • Classic LB does NOT support SNI (one of many reasons CLB is dead)
  • Trigger: "host multiple HTTPS sites with separate certificates"

Security group pattern

  • ALB SG: inbound 443 from 0.0.0.0/0
  • EC2 SG: inbound 80/443 ONLY from ALB SG (reference the SG, not an IP range)
  • Result: instances unreachable except via ALB

NLB & security groups

  • Older NLBs had NO security group — SGs lived only on the targets
  • 2023+: NLB now supports security groups (newer feature)
  • Older exam questions may still reflect the no-SG behaviour

Section NineScaling policies — the 6 flavours mapped

PolicyHow it worksPick when…
Manual You adjust desired capacity yourself Testing only — never the exam answer for "automatic"
Target Tracking 🟢 default "Keep CPU at 50%" — ASG creates alarms automatically "automatically maintain performance," 90% of scaling questions
Step Scaling Alarms with thresholds & step adjustments — different magnitudes at different breach levels "add 1 at 60%, add 3 at 80%, add 5 at 90%" pattern
Simple Scaling One alarm → one action; cooldown then repeat Legacy — Step Scaling supersedes it. Don't pick.
Scheduled Scaling Set capacity by clock — "every Monday 9am, min=10" Predictable load by time-of-day / day-of-week
Predictive Scaling ML forecasts load up to 48h ahead, pre-warms capacity Cyclical patterns + slow-launching instances
Combining policies

You can combine policies. Common production pattern: Predictive handles the daily curve (proactive) + Target Tracking handles unexpected deviations (reactive). Some exam questions reward this combo as the "best" answer for known-pattern workloads with occasional spikes.

Section TenLifecycle Hooks · Warm Pools · Instance Refresh

Lifecycle Hooks

  • Pause instance during launch or termination
  • States: Pending:Wait / Terminating:Wait
  • Use to: install software, register w/ external service, drain connections, flush logs
  • Default timeout: 1 hour

Trigger phrase: "run a script before traffic / during termination"

Warm Pools

  • Pre-initialised instances kept stopped/hibernated
  • Used FIRST when ASG scales out
  • Cuts startup from minutes to seconds
  • Stopped instances don't bill compute

Trigger phrase: "reduce launch time during spikes"

Instance Refresh

  • Roll out a new launch template version
  • Gradually replaces old instances with new
  • Configurable: minimum healthy %, instance warmup
  • Native, no detach/reattach gymnastics

Trigger phrase: "deploy new AMI / new config to existing fleet"

Section ElevenMixed instances & Spot inside ASG

The mixed instances policy

Lets one ASG run multiple instance types AND mix On-Demand with Spot. Two key knobs:

KnobWhat it doesExample
On-Demand base capacityNumber of instances always On-Demand (never Spot)"Always have at least 5 On-Demand"
On-Demand percentage above baseOf additional capacity, what % is On-Demand vs Spot"30% On-Demand, 70% Spot above base"
Instance types listAllowed instance types (capacity flexibility)m5, m5a, m5n, c5, c5n
Spot allocation strategyHow to choose Spot pools — capacity-optimized recommended for stabilitycapacity-optimized = lowest interruption rate
Cost-optimisation answer pattern

"Reduce cost for a stateless web fleet behind an ALB while maintaining reliability" → ASG with mixed instances policy: On-Demand base + Spot above base, capacity-optimized strategy. Common SAA-C03 cost-optimisation winner.

Section ThirteenDecision trees

Tree 1 — Which load balancer?

Q1. Is the question about inserting third-party security appliances?
YES → GWLB. Stop.
NO → continue.
Q2. Is the traffic HTTP / HTTPS?
NO (TCP / UDP / TLS / non-HTTP) → NLB. Stop.
YES → continue.
Q3. Are there hard requirements only NLB satisfies — static IP, source IP preservation, millions req/sec, ultra-low latency?
YES → NLB (or NLB → ALB chain if you also need HTTP routing).
NO → continue.
Q4. Default → ALB. Path/host routing, Cognito auth, WAF, containers, Lambda — everything HTTP-shaped.

Tree 2 — Which scaling policy?

Q1. Is the trigger time-based ("every Monday 9am", "during business hours")?
YES → Scheduled Scaling. Stop.
NO → continue.
Q2. Does the question say "forecast" / "predict" / "anticipate based on history"?
YES → Predictive Scaling. Stop.
NO → continue.
Q3. Does the question describe different scaling magnitudes at different breach levels (multi-step alarms)?
YES → Step Scaling. Stop.
NO → continue.
Q4. Default → Target Tracking. "Keep metric X at value Y" — simplest and best for most.

Tree 3 — "Instances are misbehaving"

Q1. New instances fail health check at launch?
→ Increase Health Check Grace Period.
Q2. Running instances pass EC2 check but app is broken?
→ Switch ASG health checks to EC2 + ELB. Make /health meaningful.
Q3. ASG keeps adding then removing instances rapidly?
→ Increase Cooldown; review Target Tracking thresholds.
Q4. Need to update AMI / launch template across the fleet without downtime?
→ Instance Refresh.
Q5. Need to run setup script before traffic / cleanup before termination?
→ Lifecycle Hooks.
Q6. Launches too slow during traffic spikes?
→ Warm Pool (or pre-baked AMI).

Section Fourteen"Two technically correct" tie-breakers

SAA-C03 questions almost always leave 2 answers that could work. The qualifier picks the winner. Here are the universal tie-breakers for ELB/ASG questions.

Qualifier reads…

"least operational overhead"

  • Pick managed / serverless every time
  • Target Tracking > Step / Simple
  • Launch Template > Launch Configuration
  • ALB w/ Cognito > rolling your own auth
  • Instance Refresh > manual rolling
Qualifier reads…

"most cost-effective"

  • ASG + Spot for stateless workloads
  • Mixed instances policy with capacity-optimized
  • NLB cross-zone OFF if uneven AZs are fine
  • Reserved/Savings Plans for steady baseline
Qualifier reads…

"highly available" / "resilient"

  • Multi-AZ ASG (≥2 AZs always)
  • ELB health checks ON in ASG
  • ASG min ≥ 2
  • Public ALB → private EC2 across AZs
Qualifier reads…

"performant" / "low latency"

  • NLB for non-HTTP / extreme rates
  • Warm Pool for fast scale-out
  • CloudFront in front of ALB for caching
  • ElastiCache for session/data cache

Section Fifteen25 SAA-C03-style practice questions

Each question is shaped like a real Tutorials Dojo / SAA-C03 item: long scenario, "MOST cost-effective / LEAST operational overhead / HIGHLY available" qualifier, four plausible options, one clearly best answer. Question 01
Domain: Resilient ArchitecturesTopic: ALB vs NLB

A fintech company is building a stock-trading API used by partner banks. The partners' corporate firewalls require two static IP addresses to be whitelisted. The API uses standard HTTPS, and the company also wants the backend EC2 instances to see the original client IP for fraud detection. Which load balancer choice meets these requirements with the LEAST operational overhead?

  1. Use an Application Load Balancer (ALB) and rely on the X-Forwarded-For header for client IPs.
  2. Use a Network Load Balancer (NLB) with Elastic IP addresses attached, one per AZ.
  3. Use a Classic Load Balancer with cross-zone load balancing enabled.
  4. Use an ALB and front it with a CloudFront distribution to obtain stable IPs.
Reveal full breakdown
Qualifier & correct answer

The qualifiers are "two static IP addresses" + "see the original client IP".

B — NLB with Elastic IPs, one per AZ

NLB is the only LB that gives you static IPs (and lets you bring your own Elastic IPs). NLB also preserves the source client IP by default — no XFF parsing needed.

Why the others are wrong
  • A — ALB has no static IP; partners cannot whitelist a moving DNS target.
  • C — Classic LB is deprecated; never the SAA-C03 answer.
  • D — CloudFront IPs change continuously and span hundreds of edges; not whitelistable.
The trap

D is tempting because CloudFront feels "global and stable," but its edge IPs are not static. Global Accelerator gives 2 static anycast IPs and would also work — if it were an option here, it'd compete with NLB. Since it isn't, NLB wins.

Rule: static IP + source IP preservation + non-HTTP-specific = NLB.
Question 02
Domain: High-PerformingTopic: ALB routing

A retail company runs a single domain shop.example.com serving three back-end services: a customer-facing web frontend, a mobile API, and an admin portal. Traffic for paths starting with /api/* must go to the mobile API fleet, /admin/* to the admin portal, and everything else to the web frontend. The architecture team wants to minimise the number of load balancers. Which solution is BEST?

  1. Three Network Load Balancers, one per service, with Route 53 weighted routing.
  2. One Network Load Balancer with three listeners on different ports.
  3. One Application Load Balancer with path-based listener rules and three target groups.
  4. One Classic Load Balancer with URL pattern matching enabled.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "minimise number of load balancers" + path-based routing.

C — ALB with path-based listener rules

ALB is built for exactly this — one LB, multiple paths, multiple target groups. Each target group has its own health check and instances.

Why the others are wrong
  • A — three NLBs is operational overhead; NLB has no path awareness anyway.
  • B — NLB doesn't route by URL path; ports are not URL paths.
  • D — Classic LB is deprecated; even when used, its routing is far weaker.
The trap

Path routing tempts people toward "use Route 53 to split traffic" — but Route 53 routes by hostname/region/health, not by URL path. Path-based routing is exclusively at L7, and that's ALB.

Rule: path-based / host-based routing = ALB, always.
Question 03
Domain: ResilientTopic: ASG health checks

A SaaS company runs an Auto Scaling group of EC2 instances behind an ALB. Recently, several instances have become unresponsive at the application level — the operating system was running fine, but the application returned HTTP 500 errors. The ALB stopped routing traffic to them, but the ASG kept the instances around for hours. What change should the architect make so the ASG automatically replaces these instances?

  1. Set the ASG health-check type to "EC2 and ELB" so ELB-reported failures trigger replacement.
  2. Reduce the ALB unhealthy threshold to 1.
  3. Enable instance status checks on each EC2 instance.
  4. Add a CloudWatch alarm on HTTP 5xx errors that terminates the instance.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "ASG keeps the instance around for hours" while the ALB has marked it unhealthy.

A — set ASG health check type to EC2 + ELB

By default, ASG only uses EC2 status checks. To make ASG act on ELB-reported failures, you must explicitly enable ELB health checks in the ASG. Once enabled, an ELB-unhealthy instance is replaced automatically.

Why the others are wrong
  • B — already triggered (ALB stopped routing); doesn't make ASG act.
  • C — instance status check is OS-level; that's already passing — that's the whole problem.
  • D — possible workaround but adds operational overhead and isn't the intended mechanism.
The trap

Many candidates assume ASG is "smart" enough to listen to ELB health by default. It is not. This is a one-checkbox setting that's responsible for many real-world outages.

Rule: Always enable EC2 + ELB health-check type on ASGs behind a load balancer.
Question 04
Domain: High-PerformingTopic: Scaling policy

A streaming service notices its CPU utilisation rises predictably from 8am, peaks around 8pm, and falls overnight — every single day. Provisioning is currently reactive (Target Tracking on CPU), but instances take ~4 minutes to launch and bootstrap, and viewers experience buffering at the start of the daily ramp. The team wants compute capacity to be ready BEFORE the load arrives. Which scaling approach is MOST appropriate?

  1. Increase the maximum capacity of the ASG and lower the cooldown period.
  2. Switch from Target Tracking to Step Scaling with a smaller scale-up step.
  3. Add a Predictive Scaling policy in addition to the existing Target Tracking policy.
  4. Replace the ASG with manual scaling executed by a CRON-driven Lambda function.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "predictable daily pattern" + "capacity ready BEFORE the load arrives" + slow boot.

C — Add Predictive Scaling alongside Target Tracking

Predictive uses ML on up to 14 days of history to forecast the next 48h and pre-launches capacity proactively. Pairing it with Target Tracking handles unexpected deviations reactively. This combo is the recommended pattern for cyclical workloads.

Why the others are wrong
  • A — raising max doesn't pre-warm; cooldown changes don't help reactive policies launch faster.
  • B — still reactive; doesn't solve the "before load arrives" problem.
  • D — a Lambda+CRON is a homemade Scheduled Scaling, with more overhead and no forecasting.
The trap

D feels "elegant" because it's serverless. But Predictive Scaling exists precisely to remove that custom solution. SAA-C03 rewards using the managed feature.

Rule: Cyclical pattern + slow launch + "ready before load" = Predictive Scaling (often combined with Target Tracking).
Question 05
Domain: Cost-OptimisedTopic: ASG with Spot

A media company runs a stateless image-resizing fleet behind an ALB. The fleet runs 24×7 with variable load. The CFO wants to reduce compute costs significantly while keeping the service highly available. Each request takes only seconds and can be retried by clients. What is the MOST cost-effective architecture?

  1. Convert the entire fleet to On-Demand Reserved Instances on a 3-year, all-upfront term.
  2. Configure the ASG with a mixed-instances policy: a small On-Demand base, the remainder on Spot using the capacity-optimized allocation strategy, across multiple instance types.
  3. Run the fleet entirely on Spot Instances with a single instance type for simplicity.
  4. Migrate to Lambda regardless of payload size or duration.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "stateless" + "highly available" + "most cost-effective".

B — Mixed instances: On-Demand base + Spot above base, capacity-optimized, multiple types

Captures up to 90% Spot savings while preserving an On-Demand reliability floor. Multiple instance types reduce Spot interruption risk. capacity-optimized picks pools with the lowest interruption rate.

Why the others are wrong
  • A — locks 100% to On-Demand-priced Reserved; misses Spot savings.
  • C — pure Spot is risky for HA; one instance type concentrates interruption risk.
  • D — Lambda may be wrong for image-resizing depending on size/duration; switching architectures wholesale is overkill.
The trap

C looks like the cheapest option, but "single instance type Spot" is the textbook way to get hammered by interruptions in one Spot pool. Capacity-optimized + multiple types is what makes Spot reliable enough for production.

Rule: Stateless + cost-effective + HA = ASG mixed instances policy with On-Demand base + Spot, capacity-optimized.
Question 06
Domain: High-PerformingTopic: NLB / TLS

An IoT vendor receives ~5 million MQTT messages per second from devices over TCP on port 8883 (TLS). Backend EC2 servers process each message and need to see the device's source IP for geofencing. Which load balancer satisfies these requirements?

  1. Application Load Balancer with TLS listener and X-Forwarded-For header.
  2. Network Load Balancer with TLS listener (or TCP pass-through) and source IP preservation.
  3. Gateway Load Balancer.
  4. CloudFront distribution with custom origin.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "MQTT (TCP)" + "5M messages/sec" + "see device source IP".

B — NLB with TLS listener (or TCP pass-through)

MQTT is TCP, not HTTP — ALB is out. Volumes match NLB's strengths. Source IP preserved by default.

Why the others are wrong
  • A — ALB doesn't handle MQTT/TCP; XFF only works with HTTP.
  • C — GWLB is for inserting security appliances, not for app load balancing.
  • D — CloudFront is HTTP-only.
The trap

The mention of "TLS" might tempt people to think "ALB with TLS." Many protocols use TLS — what matters is the application protocol underneath. MQTT-over-TLS is still TCP at L4, not HTTP at L7.

Rule: Non-HTTP protocols (MQTT, SMTP, FTP, custom TCP) = NLB, not ALB.
Question 07
Domain: SecurityTopic: ALB + auth

A company is moving an internal HR application behind a load balancer and wants users to sign in with their corporate identity provider (SAML / OIDC) before reaching the application servers. The team wants to AVOID writing authentication code in the application. Which solution requires the LEAST custom development?

  1. Front the EC2 fleet with an ALB and configure ALB authentication using Cognito or OIDC.
  2. Front the EC2 fleet with an NLB and have each instance handle SAML directly.
  3. Use API Gateway with Lambda authorizers for the entire HTTP traffic.
  4. Use CloudFront signed URLs to gate access.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "AVOID writing authentication code" / "least custom development".

A — ALB with Cognito / OIDC authentication

ALB has built-in authentication actions for Cognito User Pools, OIDC, and (via Cognito) SAML federation. The LB challenges the user, completes the OAuth/OIDC dance, and only forwards authenticated requests to the backend. Zero auth code in your app.

Why the others are wrong
  • B — NLB has no auth; would need custom code on every instance.
  • C — possible but heavy: rewriting the front tier on API Gateway + Lambda is more work than turning on a feature on ALB.
  • D — signed URLs are for static asset access, not interactive sign-in.
The trap

API Gateway + Lambda authorizers (C) sounds modern but means rebuilding the front tier. The qualifier "least custom development" rules it out.

Rule: "Authenticate at the LB without writing code" = ALB + Cognito / OIDC.
Question 08
Domain: ResilientTopic: Lifecycle Hooks

A company's Auto Scaling group launches EC2 instances that must register themselves with an external monitoring service AND warm a local cache from S3 BEFORE serving any user traffic. Currently, instances start receiving traffic immediately and serve cold-cache responses for the first few minutes. What is the BEST solution?

  1. Use a Lifecycle Hook in autoscaling:EC2_INSTANCE_LAUNCHING that holds the instance in Pending:Wait until registration and cache warm-up complete.
  2. Increase the ASG's health-check grace period to 10 minutes.
  3. Increase the ALB's healthy threshold to 100.
  4. Add a CloudWatch alarm to detect cold-cache responses.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "register with external service" + "warm a cache" + "BEFORE serving traffic".

A — Lifecycle Hook on instance launching

Lifecycle Hooks are the documented mechanism: instance pauses in Pending:Wait, your script runs the registration + cache warm-up, signals CONTINUE, instance moves to InService and joins the target group.

Why the others are wrong
  • B — grace period suppresses health checks, doesn't run setup work.

  • D — alarms detect the problem after it's hurt users.

Grace period (B) sounds related but is the wrong tool — it's about ignoring health checks during boot, not about pausing for setup. Lifecycle Hooks are the correct mechanism for "do something before serving traffic."

Question 09
Domain: ResilientTopic: Multi-AZ

A startup runs an ALB and an Auto Scaling group of 6 EC2 instances. All 6 are currently in a single Availability Zone for simplicity. The CTO requires the application to remain available even if a single AZ fails. What is the SIMPLEST architectural change to meet this requirement?

  1. Configure the ASG and ALB across at least two Availability Zones, with instances spread across them.
  2. Move the EC2 instances to a different Region and use Route 53 failover.
  3. Increase the EC2 instance type to a larger size to add resilience.
  4. Place the ALB behind CloudFront for edge replication.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "survive single-AZ failure" + "simplest".

A — Spread ASG and ALB across ≥2 AZs

This is the textbook HA pattern. ALB requires subnets in ≥2 AZs anyway; ASG can spread automatically.

Why the others are wrong
  • B — multi-Region is overkill for "single-AZ resilience" and far from the simplest option.
  • C — bigger instance type still in one AZ is no more resilient.
  • D — CloudFront is a CDN, not an HA mechanism for compute.
The trap

Multi-Region (B) is the classic over-engineering trap. SAA-C03 separates Multi-AZ (HA inside one region) from Multi-Region (cross-region DR). When the question only asks for AZ resilience, the answer is Multi-AZ.

Rule: Multi-AZ = HA within a Region. Multi-Region = DR across Regions. Don't conflate.
Question 10
Domain: ResilientTopic: Sticky sessions vs ElastiCache

An e-commerce site stores user shopping carts in the local memory of each EC2 instance behind an ALB. Customers complain that their cart sometimes disappears between page loads. Which is the BEST long-term solution?

  1. Enable sticky sessions on the ALB target group.
  2. Move shopping-cart state to Amazon ElastiCache for Redis, accessible from all instances.
  3. Increase the EC2 instance type so the cache lives longer.
  4. Replace the ALB with an NLB to preserve client IP.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "BEST long-term solution".

B — Move cart state to ElastiCache for Redis

The root cause is statefulness on individual instances. Externalising state to a shared store solves it for good — any instance can serve any request, scaling/replacement become trivial.

Why the others are wrong
  • A — sticky sessions is a quick fix, but state on a single instance is still lost when the instance dies. "Best long-term" rules it out.
  • C — bigger instance still loses memory when replaced.
  • D — source IP has nothing to do with cart persistence.
The trap

If the qualifier had said "fastest fix," sticky sessions (A) would be correct. The exam carefully chooses qualifiers — read them precisely.

Rule: Quick fix → sticky sessions. Best long-term / scalable → externalise session state to ElastiCache or DynamoDB.
Question 11
Domain: SecurityTopic: GWLB

A regulated bank requires every packet entering its production VPC to be inspected by a Palo Alto Networks virtual firewall before reaching application servers. The architecture must be transparent to applications and scale automatically. Which AWS service achieves this?

  1. Application Load Balancer with WAF rules.
  2. Network Load Balancer with the firewall as a target.
  3. Gateway Load Balancer with a fleet of Palo Alto firewall appliances behind it, paired with a GWLB endpoint in the VPC.
  4. VPC Traffic Mirroring sending copies to a single firewall instance.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "third-party firewall" + "every packet inspected in-line" + "transparent + scaling".

C — Gateway Load Balancer with the firewall fleet + GWLB endpoint

This is the textbook GWLB use case. GWLB encapsulates traffic in GENEVE on port 6081 and bounces it through the appliances; GWLBe in the VPC routes traffic transparently.

Why the others are wrong

Question 12
Domain: High-PerformingTopic: Warm Pool

A gaming company experiences flash spikes when streamers go live — instance launch + bootstrap takes ~3 minutes, but the spike lasts only 5 minutes, by which point new instances arrive too late. The team wants the ASG to absorb spikes within seconds without paying for fully running idle instances 24×7. What is the BEST approach?

  1. Configure an ASG Warm Pool with stopped (or hibernated) pre-initialised instances.
  2. Multiply the minimum capacity by 5 to keep headroom always running.
  3. Switch to Lambda for the entire game backend.
  4. Disable health-check grace period to reduce launch latency.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "absorb spikes in seconds" + "don't pay for idle 24×7".

A — Warm Pool with stopped instances

Warm Pool keeps pre-initialised instances stopped — so you don't pay for compute, only EBS — and brings them online in seconds during scale-out.

Why the others are wrong
  • B — keeping headroom always running violates "don't pay for idle."
  • C — wholesale architecture change; Lambda may not fit a real-time game backend.
  • D — grace period doesn't change actual launch time.

B is the natural reflex — "just keep more running." Warm Pool was built precisely to avoid that cost while keeping the speed.

Question 13
Domain: ResilientTopic: Instance Refresh

A financial services company has a 50-instance ASG running a critical web application. They need to deploy a new AMI version with a security patch. Replacement must be gradual, with a minimum of 80% of the fleet healthy at all times, and require minimal manual operations. Which solution is BEST?

  1. Manually terminate instances one by one and let the ASG launch replacements.
  2. Use ASG Instance Refresh with the new launch template version and a minimum healthy percentage of 80.
  3. Detach all instances, launch a new ASG with the new AMI, then re-attach.
  4. Increase desired capacity to 100, wait for new instances, then drop back to 50.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "gradual" + "≥80% healthy" + "minimal manual operations".

B — ASG Instance Refresh, min healthy 80%

Instance Refresh is purpose-built for this: pick the launch template version, set min healthy %, click start. ASG handles draining, replacement, and verification.

Why the others are wrong
  • A — manual = high operational overhead and error-prone at scale.
  • C — detach/re-attach is operationally painful; Instance Refresh is the supported path.

D is a tempting "double then halve" pattern but doesn't replace the old instances — it only adds new ones, then terminates them when scaling in (because the default termination policy may pick newer instances).

Question 14
Domain: ResilientTopic: Cross-zone

A team runs an ALB with target groups containing 8 instances in AZ-A and 2 instances in AZ-B. They notice instances in AZ-B receive disproportionately high traffic and exhibit higher latency during peak hours. The team has not changed default ALB settings. What is the MOST likely cause and the simplest fix?

  1. Cross-zone load balancing is off; enable it on the ALB.
  2. Cross-zone load balancing is on by default for ALB and is already balancing. The issue is uneven instance count between AZs; rebalance instances.
  3. NLB cross-zone is off; switch to NLB and enable cross-zone.
  4. Disable health checks to allow more traffic through.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "ALB" + "default settings" + "AZ-B instances overloaded".

B — ALB cross-zone is already on; rebalance instances across AZs

For ALB, cross-zone is always on and free. With 8/2 distribution and cross-zone on, every instance gets roughly equal traffic, but AZ-B's 2 instances each receive ~4× the work of an "AZ-A-share" instance because there are simply fewer of them. Fix: deploy more instances in AZ-B (or fewer in AZ-A) for symmetry.

Why the others are wrong
  • A — cross-zone is already on by default for ALB.
  • C — switching to NLB doesn't fix the imbalance and would need cross-zone manually.
  • D — disabling health checks is dangerous and unrelated.
The trap

People mix up cross-zone defaults: ALB on (free), NLB off (paid). And even with cross-zone on, if instance counts are asymmetric, per-instance load is asymmetric.

Rule: ALB cross-zone = always ON, free. NLB cross-zone = OFF by default, billable when ON. Rebalance instance counts for fair per-instance load.
Question 15
Domain: Cost-OptimisedTopic: Scaling policy choice

A SaaS billing system runs jobs only during business hours (8am–6pm, Mon–Fri). Outside those hours, the system is idle. The team currently runs a fixed fleet of 12 EC2 instances 24×7. Which approach reduces cost the MOST while keeping business-hours capacity unchanged?

  1. Use Target Tracking with CPU at 50%.
  2. Use Scheduled Scaling: scale up to 12 instances at 8am Mon–Fri, scale down to 0 at 6pm.
  3. Switch all 12 instances to Spot.
  4. Reserved instances for all 12 on a 3-year all-upfront term.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "reduces cost the MOST" + "only business hours" + "keep business-hours capacity unchanged".

B — Scheduled Scaling: 12 at 8am, 0 at 6pm, weekdays

The workload pattern is fully predictable by clock. Eliminating night/weekend capacity entirely is the largest possible saving — roughly 70% reduction.

Why the others are wrong
  • A — Target Tracking responds to utilisation but you'd still pay for some baseline 24×7.
  • C — Spot helps but doesn't eliminate the 14h/day idle window.
  • D — Reserved 3-year all-upfront commits you to 24×7 capacity you don't need.

Reserved Instances feel like the textbook "cost optimisation" answer, but they're wrong here because the workload is intermittent. Reserved is for steady, predictable, always-on workloads — not for systems idle 60%+ of the time.

Question 16
Domain: High-PerformingTopic: SNI

A SaaS provider hosts 25 different customer-branded HTTPS websites, each with its own domain and its own ACM certificate. They want to consolidate to a single load balancer for cost and management simplicity. Which feature makes this possible?

Qualifier & correct answer

Qualifier: "25 HTTPS sites, separate certificates, ONE load balancer".

A — SNI with multiple certificates on the listener

SNI lets the LB pick the correct certificate per hostname during the TLS handshake. ALB and NLB both support it. Classic LB does not — that's one of CLB's many limitations.

Question 17
Domain: ResilientTopic: ALB target types

A company wants to migrate a small REST API from EC2 to AWS Lambda but keep the existing public DNS endpoint and TLS certificate served by their corporate ALB. Which configuration achieves this with the LEAST change to clients?

  1. Add a Lambda target type to the ALB target group; route HTTPS traffic to the Lambda function.
  2. Replace the ALB with an API Gateway and re-issue a new public DNS.
  3. Front the Lambda with CloudFront and abandon the ALB.

Reveal full breakdown
Qualifier & correct answer

Qualifier: "keep existing DNS endpoint and certificate" + "least change to clients".

A — ALB with Lambda target

ALB supports Lambda as a direct target type. The public DNS, listener, and certificate are unchanged; only the target group's backend changes. Clients see no difference.

Why the others are wrong
  • B — replaces the public endpoint; clients must change.

API Gateway (B) is the "obvious" Lambda fronting service, but it can't preserve an existing ALB's public endpoint. ALB's Lambda target type is the lesser-known feature that nails this scenario.

Question 18
Domain: ResilientTopic: Health check grace period

An ASG launches new EC2 instances that need 4 minutes to start a Java application. During the first few minutes, ALB health checks fail and the ASG terminates the instance, replacing it endlessly in a loop. What is the simplest correct fix?

  1. Increase the ASG's health-check grace period to a value greater than 4 minutes.
  2. Disable ELB health checks on the ASG.
  3. Switch to NLB to bypass health checks.
  4. Add a Lifecycle Hook in EC2_INSTANCE_TERMINATING to delay termination.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "app needs 4 min to start" + "health check fails during boot" + "simplest fix".

A — Increase Health Check Grace Period

Grace period tells ASG to ignore health-check failures for the first N seconds after launch. Set it above the slow-start time (e.g., 300+ seconds) and the loop stops.

Why the others are wrong
  • B — disabling ELB checks loses real failure detection.
  • C — NLB still does health checks; doesn't solve slow boot.
  • D — terminating-hook delays termination but doesn't fix why it's terminating.
The trap

People often confuse Lifecycle Hooks with grace period. Hook = run setup code. Grace = ignore health failures during boot. Different problems.

Rule: "App takes a while to start; health checks fail at boot" = increase Health Check Grace Period.
Question 19
Domain: ResilientTopic: Blue/green

A company wants to deploy a new version of its web application with the ability to roll back instantly if errors increase. They want to send 5% of traffic to v2 first, monitor, then ramp up to 100%, all without changing DNS. Which ALB feature supports this directly?

  1. ALB weighted target groups within a single listener rule.
  2. Route 53 latency-based routing.
  3. NLB connection draining.
  4. CloudFront cache invalidation.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "send X% to v2 then ramp up" + "without changing DNS".

A — ALB weighted target groups

ALB supports forwarding a percentage of traffic to multiple target groups within a single listener rule. Adjust weights gradually; instant rollback by flipping weight back. Native blue/green / canary primitive.

Why the others are wrong
  • B — Route 53 splits at DNS, not at the ALB; needs DNS-level changes and TTL waits.
  • C — connection draining is termination handling, not deployment.
  • D — invalidation is for cached content; unrelated.
The trap

Route 53 weighted routing (B) does support traffic-splitting, but acts at DNS level. It's a coarser tool with TTL caching delays. ALB weighted target groups are immediate and request-level.

Question 20
Domain: ResilientTopic: Termination policy

A company runs an ASG with multiple Launch Template versions. When scaling in, they want the ASG to terminate the OLDEST launch-template-version instances FIRST so that new versions persist. Which setting controls this?

  1. The ASG's termination policy, set to OldestLaunchTemplate.
  2. The ALB's deregistration delay.
  3. The CloudWatch metric used by Target Tracking.
  4. A Lifecycle Hook in EC2_INSTANCE_TERMINATING.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "terminate oldest launch-template-version FIRST".

A — Termination policy = OldestLaunchTemplate

ASG termination policies determine which instance to kill on scale-in. OldestLaunchTemplate kills oldest-LT instances first; other options include OldestInstance, NewestInstance, ClosestToNextInstanceHour, AllocationStrategy, and Default (a sensible mix).

Why the others are wrong
  • B — deregistration delay controls the in-flight request grace, not which instance is chosen.
  • C — the metric controls when to scale, not which instance to kill.
  • D — Lifecycle Hook delays termination but doesn't choose the victim.
The trap

Termination policy is one of those "I read it once and forgot" settings. Worth knowing because real-world deploys depend on it.

Rule: "Pick which instance to terminate on scale-in" = ASG termination policy.
Question 21
Domain: SecurityTopic: SG between ALB & EC2

A solutions architect wants to ensure that EC2 instances in private subnets accept traffic ONLY from the company's public-facing ALB and never directly from the internet. What is the BEST security group configuration?

  1. EC2 SG: inbound 80 from the ALB's security group ID. ALB SG: inbound 443 from 0.0.0.0/0.
  2. EC2 SG: inbound 80 from 0.0.0.0/0.
  3. EC2 SG: inbound 80 from the ALB's specific public IP addresses.
  4. Disable security groups; use NACLs only.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "accept ONLY from the ALB; never directly from the internet".

A — Reference the ALB's security group, not an IP range

Security groups can reference other security groups, not just IP ranges. This is the canonical pattern: only traffic that came through the ALB's SG can reach the EC2 SG, regardless of which AZ or what IP the ALB happens to be using right now.

Why the others are wrong
  • B — opens the EC2 fleet to the entire internet, defeating the design.
  • C — ALB IPs change; whitelisting them breaks every time AWS replaces an ALB node.
  • D — NACLs alone are insufficient and not how AWS recommends layering security.
The trap

People IP-whitelist the ALB (option C) and it works for a few days — until ALB scales or replaces nodes, and traffic starts being denied with no obvious cause. SG-to-SG references prevent that.

Rule: EC2-behind-ALB security: reference the ALB's security group in the EC2 SG, not its IP.
Question 22
Domain: High-PerformingTopic: Global static IPs

A global SaaS application runs ALBs in three AWS Regions. Customers' enterprise networks require two static IPs to be whitelisted globally, and want failover to a healthy Region within seconds — without waiting for DNS TTLs. Which AWS service provides this?

  1. Route 53 with health checks and a 60-second TTL.
  2. AWS Global Accelerator with the regional ALBs as endpoints.
  3. CloudFront with multi-origin failover.
  4. Three NLBs with Elastic IPs and Route 53 latency-based routing.
Reveal full breakdown
Qualifier & correct answer

Qualifiers: "2 static IPs globally" + "failover in seconds without DNS".

B — AWS Global Accelerator with ALBs as endpoints

Global Accelerator gives you 2 static anycast IPs that route to the optimal healthy regional endpoint over AWS's private backbone. Failover happens at the network layer in seconds — no DNS TTL involved.

Why the others are wrong
  • A — DNS-based; clients cache TTLs, slowing failover.
  • C — CloudFront is HTTP caching, not L4 global static IP service.
  • D — gives static IPs but per Region, not globally; failover is DNS-based.
The trap

Route 53 with low TTL (A) feels like the answer for "fast failover" but DNS resolvers worldwide cache stale records aggressively, and "60-second TTL" rarely behaves like 60 seconds in the wild. Global Accelerator dodges that entire layer.

Question 23
Domain: ResilientTopic: Connection draining

A company's web application runs behind an ALB. Long file uploads occasionally fail when the ASG scales in and instances are terminated mid-upload. Which setting helps the existing connections complete gracefully before termination?

  1. Increase the deregistration delay (connection draining) on the target group.
  2. Reduce the cooldown period.
  3. Switch to NLB with TLS pass-through.
  4. Change the termination policy to NewestInstance.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "in-flight uploads must complete before termination".

A — Increase deregistration delay

Deregistration delay tells the LB to stop sending new requests to the target while letting in-flight ones up to N seconds to finish. Default 300s; raise it for long uploads.

Why the others are wrong
  • B — cooldown is between scaling actions, not for in-flight requests.

Question 25
Domain: ResilientTopic: ASG flapping

An ASG with a Target Tracking policy on average CPU utilisation keeps flipping rapidly between adding and removing instances every few minutes during fluctuating load. The team wants stability without sacrificing responsiveness. Which combination of settings addresses this BEST?

  1. Increase the cooldown period and adjust the Target Tracking metric to a more stable signal (e.g., ALB request count per target) where appropriate.
  2. Switch to Simple Scaling and set the cooldown to zero.
  3. Change to a single instance type and reduce the maximum capacity.
  4. Disable health checks during scaling events.
Reveal full breakdown
Qualifier & correct answer

Qualifier: "flapping" + "stable but responsive".

A — Increase cooldown + use a more stable scaling metric

Flapping comes from over-eager scaling on a noisy signal. Two-pronged fix: cooldown lets each scaling action settle before another fires; choosing a smoother metric (request count per target, ALBRequestCountPerTarget, or longer-window CPU averages) reduces noise.

Why the others are wrong
  • B — Simple Scaling + zero cooldown is a recipe for worse flapping.
  • C — single instance type is unrelated to flapping.
  • D — disabling health checks is dangerous and unrelated.
The trap

People sometimes blame Target Tracking and want to switch to Step or Simple. But Target Tracking with a sensible metric and cooldown is correct for most cases — just tune the inputs.

Rule: ASG flapping = (a) increase cooldown, (b) pick a smoother metric for Target Tracking. Don't downgrade to Simple Scaling.

Read this the morning of your exam. Recite without notes.

Load balancer pick

  • HTTP routing → ALB
  • Static IP / TCP / UDP / extreme rate → NLB

Scaling policy pick

  • Default → Target Tracking
  • Time-based → Scheduled

ASG hygiene

  • Health-check type = EC2 + ELB
  • Use Launch Template, not Launch Configuration
  • Spread across ≥2 AZs
  • Min ≥ 2 for HA
  • Mixed instances + Spot for cost-effective stateless

  • ALB has NO static IP
  • EC2-only health check ≠ application healthy
  • NLB cross-zone OFF by default + paid
  • Sticky sessions ≠ shared session store (read qualifier)
  • Multi-AZ ≠ Multi-Region (HA vs DR)

Tie-breakers

  • "Least operational overhead" → managed/serverless
  • "Most cost-effective" → Spot/Scheduled/Reserved
  • "Highly available" → Multi-AZ + ELB health checks
  • "Best long-term" → fix root cause, not workaround
  • "Real-time / sub-second failover" → Global Accelerator

Phrase → answer reflexes

  • "X-Forwarded-For" → ALB
