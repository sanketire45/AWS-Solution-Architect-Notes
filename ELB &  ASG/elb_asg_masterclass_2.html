01

The Big Picture — What Problem Does This Solve?

Real-world analogy — the restaurant

Imagine a restaurant (your app) that gets 10 customers on a quiet Monday and 10,000 on a busy Saturday night. Without planning: one overworked server, long queues, customers leave angry.

A Load Balancer = a head waiter who stands at the door and sends each new customer to whichever server (table/staff member) is currently least busy. No single person gets overwhelmed.

An Auto Scaling Group = a smart manager who hires extra waiters before the Saturday rush arrives and sends them home after closing time. Staff count matches demand automatically.

Why you need both together

  • A load balancer alone doesn't help if all your servers are still overwhelmed — it just spreads the pain evenly across them.
  • An ASG alone that adds new servers doesn't help if all traffic still hits only the original server — nobody told the other servers new ones exist.
  • Together: the ALB distributes traffic, the ASG adjusts how many targets exist, and the ALB automatically registers or deregisters instances as they launch or terminate. This is the canonical HA pattern on AWS.
Problem 1
High Availability

If one server, AZ, or instance dies — traffic automatically goes to a healthy one. Users never see downtime.

Problem 2
Scalability

As load increases, add capacity automatically. As load drops, remove it. Pay only for what you use.

The exam HA pattern: "highly available + cost-effective web application" almost always = ALB + ASG deployed across minimum 2 Availability Zones.

Elastic Load Balancing (ELB)

There are 4 types of AWS load balancer. Three are current; one is legacy. The exam heavily tests which type fits which scenario. Here's your mental model before diving in:

The OSI Model in 30 seconds — why "Layer 4 vs Layer 7" matters

Think of network traffic like a parcel. Layer 4 (Transport) only sees the address written on the outside of the box — the IP address and port number. Layer 7 (Application) opens the box and reads what's inside — the HTTP method, the URL path, the headers, the cookies.

A Layer 7 load balancer can make much smarter routing decisions because it understands what's actually being requested. A Layer 4 LB is faster but "blind" to the content.

ALB — Application Load Balancer

High Frequency
Type
ALB
Layer 7 load balancer. Understands HTTP, HTTPS, WebSockets, and gRPC. The smart router that can read every detail of an HTTP request before deciding where to send it.
Layer 7 HTTP/HTTPS WebSocket gRPC Cognito WAF
  • Routes by: URL path, hostname, HTTP headers, query strings, source IP
  • Integrates natively with Cognito (add user auth without app code changes), WAF, Lambda, ECS/Fargate
  • One ALB serves many apps via target groups — each listener rule points to a different group
  • Supports sticky sessions (cookie-based — routes same user to same instance)
  • Supports connection draining — waits for in-flight requests to complete before deregistering an instance
  • Supports Server Name Indication (SNI) — multiple SSL certificates on one listener for multiple domain names
  • Can target: EC2 instances, IP addresses, Lambda functions, and even another ALB

NLB — Network Load Balancer

High Frequency
Type
NLB
Layer 4 load balancer. Raw speed at massive scale. Handles millions of requests/second with ultra-low latency. Provides static Elastic IPs per AZ.
Layer 4 TCP/UDP TLS Static IP Source IP preserved
  • One static Elastic IP per AZ — these are fixed and whitelistable in external firewalls
  • Preserves source IP natively — the backend EC2 instance sees the real client IP address
  • Handles millions of requests per second with ultra-low latency (compared to ALB's higher-level processing)
  • Can be exposed as an AWS PrivateLink endpoint
  • TLS passthrough supported — encrypted traffic can reach backend unchanged (end-to-end encryption)
  • Cross-zone load balancing is OFF by default (costs extra per-AZ when enabled)
NLB trigger phrases: "static IP," "whitelist IP in firewall," "TCP/UDP traffic," "ultra-low latency," "millions of req/sec," "non-HTTP traffic," "source IP preservation."
NLB cannot route by URL path or headers — it doesn't open the packet. If you need path-based routing → ALB. If you need static IPs + path routing → NLB in front of ALB (chaining).

GWLB — Gateway Load Balancer

Medium Frequency
Type
GWLB
Layer 3 network-level load balancer. Designed exclusively to insert third-party virtual appliances (firewalls, IDS/IPS) transparently into traffic flow using GENEVE on port 6081.
Layer 3 GENEVE port 6081 3rd-party appliances IDS/IPS
  • Traffic flow: Internet → GWLB → Appliance (inspect/allow/block) → GWLB → Application servers
  • The appliance sees all traffic but is transparent to the source and destination
  • Uses GENEVE encapsulation on port 6081 — this specific detail appears on the exam
  • Supports horizontal scaling of the appliance fleet
GWLB trigger phrases: "inspection appliances," "third-party firewall," "IDS/IPS system," "transparent network inspection," "GENEVE," "virtual appliances from AWS Marketplace."
GWLB is NOT for balancing your own application servers. If you pick GWLB for a regular web app question, that is wrong. GWLB exists solely for inserting security appliances into the traffic path.

Advanced Features to Know Cold

Sticky Sessions

  • ALB inserts a cookie so the same user is always routed to the same instance
  • Two types: duration-based (LB-generated cookie, e.g. AWSALB) and application-based (your app sets its own cookie)
  • NLB uses source IP for stickiness (no cookie)
Design trap: Sticky sessions cause uneven load distribution. One heavy user's session pins all their traffic to one instance. The better pattern: stateless app + store sessions in ElastiCache Redis. Sticky sessions are an anti-pattern for truly scalable systems.

Connection Draining / Deregistration Delay

  • When an instance is deregistered (or fails health check), the LB stops sending new requests to it
  • In-flight requests are given time to complete before the instance is removed
  • Default: 300 seconds. Range: 1–3600 seconds. Set to 0 for instant deregistration
Set connection draining to 0 for fast rolling deployments. Keep it higher for long-lived connections or streaming.

SSL/TLS Termination & SNI

  • ALB/NLB can terminate SSL at the load balancer and forward plain HTTP to backend instances
  • Certificates managed via AWS Certificate Manager (ACM) — free SSL certs
  • SNI (Server Name Indication): allows multiple SSL certificates on one ALB listener — one cert per domain name. No extra cost
  • NLB can do TLS passthrough — encrypted all the way to the backend (end-to-end TLS)
Multiple SSL certs on one load balancer = ALB with SNI. One listener, many domains, many certificates.

Cross-Zone Load Balancing

Load BalancerDefaultCost
ALBAlways ONFree

GWLBOFFCharged per AZ when enabled
With cross-zone LB on: traffic is distributed equally across ALL instances in ALL AZs. Without it: each AZ node distributes only to instances in its own AZ (unequal if AZ instance counts differ).

Auto Scaling Groups (ASG)

An ASG maintains a fleet of EC2 instances and automatically adjusts the count based on demand, health checks, or a schedule. It always works alongside a load balancer for traffic distribution.

Core Parameters

Launch Template vs Launch Configuration

High Frequency
Always pick this

Launch Template

  • Supports multiple versions — can update without recreation
  • Supports Spot + On-Demand mix in one ASG
  • Supports T2/T3 unlimited burst mode
  • Supports placement groups
  • Full feature parity with latest EC2 options
Legacy — avoid unless forced

Launch Configuration

  • Immutable — cannot be modified after creation
  • No Spot + On-Demand mix
  • No T2/T3 unlimited support
  • AWS is deprecating this
  • Pick only if question says "legacy"
If both appear in an answer choice — always pick Launch Template. No exceptions on the exam.

Scaling Policies

High Frequency
Policy Type

Target Tracking

Define a target metric value. ASG automatically adds/removes instances to maintain it.

Example: "Keep average CPU at 50%." Simplest to configure. Lowest operational overhead.

Trigger: "automatically maintain performance," "least operational overhead"

Policy Type

Step Scaling

Define CloudWatch alarms at multiple thresholds, each triggering a different step.

Example: CPU 60–80% → add 1 instance. CPU >80% → add 3 instances.

Trigger: "graduated response based on alarm severity"

Policy Type

Scheduled Scaling

Scale up or down at a specific time using a cron schedule.

Example: Add 10 instances every Friday at 5pm. Perfect for predictable load patterns.

Trigger: "known busy times," "Black Friday," "end-of-month reports"

New in C03 ★

Predictive Scaling

Uses ML to forecast load based on historical patterns and pre-scales before demand arrives. Removes the reactive lag of other policies.

Trigger: "forecast-based," "scale before demand arrives," "ML scaling"

Predictive Scaling was added to SAA-C03 and is specifically called out as a new topic. If you see "anticipate demand and pre-scale automatically" — this is the answer over Target Tracking.

Lifecycle Hooks

High Frequency
  • Pause an instance at two key moments: during launch (before it serves any traffic) or during termination (before it is killed)
  • During the pause: run whatever script you need — install software, drain connections, push final logs to S3, notify monitoring systems
  • The instance stays in Pending:Wait (launch) or Terminating:Wait (terminate) state during the hook
  • Default timeout: 1 hour. At the end, hook either CONTINUE (normal flow) or ABANDON (terminate immediately)
Lifecycle hook trigger: "run a custom script when an instance launches before it gets traffic" OR "gracefully drain connections before termination" OR "send instance logs to S3 before the instance is deleted."

Health Checks — Critical Distinction

Health Check TypeWhat it testsWhen to use
EC2 (default)Is the EC2 instance running at OS level?Not sufficient for production apps
ELBIs the application responding correctly? (e.g. HTTP 200)Always enable in production
CustomYour application marks instance healthy/unhealthy via APIComplex health logic
This trap appears on the exam: An EC2 instance can be "running" at the OS level while serving HTTP 500 errors. The EC2 health check passes ✓, the ELB health check fails ✗. If you only use EC2 checks, the ASG will NOT replace the broken instance. Always enable ELB health checks on production ASGs.

Scaling Cooldown

  • After a scaling event, ASG waits a cooldown period (default 300s) before acting again
  • Prevents "thrashing" — rapid scale-up/scale-down oscillation caused by metric spikes
  • Best fix for excessive scaling: bake software into a pre-configured AMI to reduce instance boot time
If instances take 8 minutes to boot, the ASG sees elevated CPU for 8 minutes and over-scales. Fix: pre-baked AMI reduces boot to ~30 seconds. Also configure instance warm-up period so booting instances don't skew the aggregate metrics.

Multi-AZ for True High Availability

  • Configure ASG across 2+ AZs minimum. ASG automatically rebalances instances across AZs
  • If an entire AZ fails: ASG automatically launches replacement instances in remaining healthy AZs
  • ALB distributes traffic only to healthy registered targets in healthy AZs
  • ASG minimum of 2 ensures at least one instance survives any single AZ failure
True HA pattern = ALB (multi-AZ) + ASG (min 2, spanning 2+ AZs) + ELB health checks enabled. This 3-part combination is the default answer for "highly available web application."

The 5 Most Common Candidate Mistakes

01

Choosing ALB when a static IP is required

ALB does NOT have a static IP address. The question says "whitelist our load balancer IP in a partner firewall" → you pick ALB thinking "it's the main one" → wrong. The answer is NLB (which has one Elastic IP per AZ) or put Global Accelerator in front of the ALB to get 2 static anycast IPs.

02

Only using EC2 health checks on production ASGs

An instance can be "running" at the OS level while serving HTTP 500 errors. EC2 health check = OS alive. ELB health check = app is healthy. If you only use EC2 checks and your app crashes, the ASG never replaces it. Always enable ELB health checks on ASGs behind a load balancer.

03

Choosing GWLB for application load distribution

GWLB is specifically and exclusively for inserting 3rd-party security appliances. It is not a replacement for ALB or NLB. If you see "Palo Alto," "IDS/IPS," or "transparent inspection" → GWLB. Any other web app traffic routing → ALB or NLB.

04

Forgetting ALB replaces source IP with its own

When ALB forwards a request to a backend instance, it replaces the client's source IP with its own IP. The backend must read the X-Forwarded-For HTTP header to get the real client IP. With NLB, source IP is preserved natively — no header needed.

05

Picking Launch Configuration instead of Launch Template

Launch Configuration is immutable (can't edit after creation), doesn't support Spot + On-Demand mix, and AWS is deprecating it. Launch Template is always the right answer unless the question specifically uses the word "legacy." If you see both in the options, pick Launch Template every time.

Question 1 of 10 Easy
A company runs a web application on multiple EC2 instances. They need to route /api/* requests to a set of backend API servers, and all other requests to their frontend servers. Which load balancer should they use?
Qualifier: "route /api/* requests" → path-based routing signal
A — NLB: Layer 4 only — has no concept of URL paths. Cannot route by /api/*. Wrong.
C — GWLB: For 3rd-party security appliances only. Not for app traffic routing. Wrong.
D — CLB: Legacy with very limited routing. Wrong.
B is correct. ALB is the only load balancer that understands HTTP URL paths and can route to different target groups based on /api/* vs /* rules. Path-based routing is an ALB-exclusive feature.

Trap tested: NLB is a load balancer but Layer 4 only — it can't read HTTP paths.

Question 2 of 10 Easy
A financial services company needs their AWS load balancer to have a fixed, static IP address so they can whitelist it in their partner company's firewall. Which load balancer supports this requirement?
Qualifier: "static IP address," "whitelist in firewall"
A — ALB: Has NO static IP. This is the #1 most common mistake on this topic. Wrong.
C — GWLB: For security appliances, not for general app traffic. Wrong.
D — CLB: Legacy, no static IPs. Wrong.
B is correct. NLB provides one Elastic IP per Availability Zone — these are fixed, static IP addresses that can be whitelisted in external firewalls.

Trap tested: Most candidates assume ALB has a static IP because it's the "main" load balancer. It does not.

Question 3 of 10 Easy
A company wants their Auto Scaling Group to automatically add EC2 instances when average CPU exceeds a target, and remove them when it drops. They want the simplest configuration with the least operational overhead. Which scaling policy should they use?
Qualifier: "least operational overhead" + "automatically maintain performance"
A — Scheduled: For known time-based patterns, not dynamic metric response. Wrong.
B — Step Scaling: Works but requires defining multiple alarm thresholds manually — more complex. Wrong.
D — Simple Scaling: Older basic type requiring individual CloudWatch alarm config — not the simplest. Wrong.
C is correct. Target Tracking is AWS's recommended default — you set a target metric value (e.g. CPU 50%) and AWS handles all the alarm creation and scaling logic automatically. Lowest operational overhead of all policies.
Question 4 of 10 Medium
A company uses an Auto Scaling Group. Each new EC2 instance must download a configuration file from S3 and register with a monitoring tool before it receives any production traffic. How should they implement this requirement?
Qualifier: "before it receives any traffic" — must guarantee the instance is NOT yet registered when scripts run
A — User Data: Runs at launch but the instance may be registered with the ALB before the script finishes. No guarantee of "before traffic." Wrong.
B — CloudWatch Events + Lambda: Doesn't pause the instance. It could receive traffic immediately after launch. Wrong.
D — Baked AMI: Good for static config but can't handle dynamic config (download current file) or external tool registration. Wrong.
C is correct. An ASG lifecycle hook holds the instance in "Pending:Wait" state — it is NOT registered with the ALB and receives zero traffic until the hook sends a CONTINUE signal. This is the only mechanism that guarantees "before traffic."
Question 5 of 10 Medium
An ASG has EC2 instances behind an ALB. Occasionally, instances are shown as healthy by the ASG but are actually serving HTTP 500 errors to users. The application process has crashed but the OS is still running. What change will fix this?
Qualifier: "healthy by ASG but serving HTTP 500 errors" → EC2 alive, app dead
A: More instances doesn't fix the unhealthy ones. Wrong.
B: Detailed monitoring increases metric frequency but doesn't trigger instance replacement. Wrong.
D: ALB doesn't have a native "retry on 500" rule. Wrong.
C is correct. EC2 health checks only verify the instance is running at OS level. ELB health checks verify the application is responding correctly (e.g. HTTP 200 on /health). Switching to ELB checks means the ASG terminates and replaces instances whose apps have crashed.

Trap tested: EC2 health check ≠ application health check. The default is EC2 — always change it.

Question 6 of 10 Medium
A company wants to perform a canary deployment — sending exactly 10% of production traffic to a new application version while 90% continues to the current version. They want to use their existing ALB with the least possible operational overhead.
Qualifier: "10% to new version," "existing ALB," "least operational overhead"
A: Creates a second ALB and Route 53 complexity. Not least overhead. Wrong.
C: Geolocation routes by geography, not by traffic percentage. Wrong concept. Wrong.
D: Mixing versions in one target group loses precise 10/90 control. Sticky sessions add unrelated complexity. Wrong.
B is correct. ALB natively supports weighted target groups — you set TG-v1 weight to 90 and TG-v2 weight to 10 on a single listener rule. No extra infrastructure. This is the canary deployment pattern at the load balancer layer.
Question 7 of 10 Medium
An application behind an Application Load Balancer needs its backend EC2 instances to log the real IP address of each client request for security auditing. The instances are receiving all requests from the ALB's IP address. What is the correct solution?
Qualifier: "ALB in use," "need real client IP on backend"
B: ALB has no "source IP preservation" toggle. It always replaces the source IP. Wrong.
C: Switching to NLB would work technically but is a major infrastructure change — the question asks for a solution, not a rebuild. Wrong in context.
D: Flow logs are for network analysis, not real-time per-request logging in an application. Wrong.
A is correct. ALB always replaces the source IP with its own when forwarding requests. The original client IP is appended to the X-Forwarded-For HTTP header. The backend application must be configured to read this header for logging.

Trap tested: ALB does NOT preserve source IP natively — only NLB does. With ALB you must use X-Forwarded-For.

Question 8 of 10 Medium
A company is deploying a network security solution using third-party virtual firewall appliances from AWS Marketplace. All traffic entering their VPC must pass through these appliances for deep-packet inspection before reaching application servers. Which AWS service enables this architecture with the least operational overhead?
Qualifier: "third-party virtual firewall appliances," "deep-packet inspection," "transparently inserted"
A — WAF: AWS-native, rules-based — not for routing through 3rd-party EC2 appliances. Wrong.
