Chapter OneThe first big correction

Before anything else makes sense, we have to fix one mental model. You wrote:

You said

"1a and 1b are different subnets of the same AZ."

→
Actually

1a and 1b are different AZs, not different subnets.

us-east-1a and us-east-1b are two different physical data-center zones. Subnets live inside AZs — never the other way around.

This single mix-up was making everything downstream confusing. Once we lock in the correct hierarchy, everything else clicks.

Chapter TwoThe AWS hierarchy

Hold this picture in your head before you click anything in AWS. It's a strict tree — bigger things contain smaller things, and never the reverse.

Region contains AZs. AZs contain Subnets. Subnets contain Instances. Strict tree.
Region

A large geographic area like us-east-1 (Virginia), eu-west-1 (Ireland), ap-south-1 (Mumbai). Choose one per workload.

Availability Zone (AZ)

A physically separate data-center cluster inside a Region. Each Region has typically 3–6 AZs. They have independent power, cooling, networking. us-east-1a, us-east-1b, us-east-1c are different AZs in the same Region.

Subnet

A slice of your network inside one specific AZ. A subnet belongs to exactly one AZ — never spans across AZs. One AZ can contain many subnets.

Instance

An EC2 (or RDS, or ALB node) launched inside a subnet. Inherits everything about that subnet — which AZ it's in, what route table applies, etc.

Chapter ThreeWhat a VPC actually is

A VPC (Virtual Private Cloud) is your own private, isolated network inside AWS. Think of it as your company's private wing inside AWS's gigantic data centers — nobody outside your VPC can reach inside it unless you explicitly open a door.

A VPC spans an entire Region. That means one VPC can stretch across all the AZs in that Region — that's how the same network can have resources in us-east-1a, 1b, 1c simultaneously.

When you create a VPC, you give it a CIDR block — a range of IP addresses you control, like 10.0.0.0/16. That's the address space all your resources will live in.

Real-world parallel

Imagine AWS is a city. Your VPC is a private gated society inside that city. You own the entire society. Inside, you can lay out streets (subnets), put houses (EC2s), decide who can come in through which gate (route tables + security groups). The world outside can only enter if you explicitly open a gate.

Chapter FourWhat a subnet actually is

A subnet is a portion of the VPC's IP range, pinned to one specific AZ. You carve up 10.0.0.0/16 into smaller chunks like 10.0.1.0/24 and 10.0.2.0/24, and assign each chunk to an AZ.

When you create an EC2 instance, you don't pick an AZ directly — you pick a subnet, and the subnet's AZ becomes the instance's AZ automatically. This is why your hands-on screen showed VPC + subnet, not VPC + AZ + subnet.

Important

A subnet cannot span two AZs. If you want resources spread across AZs, you need multiple subnets, one per AZ. This is why ALB and RDS require subnets in at least 2 AZs — the design forces you to think about multi-AZ from day one.

Chapter FiveThe Default VPC mystery

You noticed something important:

You said

"There is no mention of 'public' anywhere when I selected the default subnet for 1b."

→
Actually

That's because "public" and "private" are not labels AWS sticks on a subnet. They're a consequence of the subnet's route table. AWS doesn't show the word "public" because, internally, no such property exists.

Every AWS account gets a Default VPC per Region, with one Default Subnet per AZ. These default subnets have an interesting property: they all have a route to an Internet Gateway in their route table. That makes them de facto public subnets, even though the console doesn't say "public" anywhere obvious.

That's why the EC2s you launched in default subnets could get public IPs, and the ALB you created in default subnets could face the internet. Not because AWS labeled them public — but because their route tables contain the magic line that makes them public.

Chapter SixPublic vs Private — it's the route table

This is the single most important concept in AWS networking. Internalize this and you'll never be confused again.

Public Subnet

Its route table contains an entry like:

0.0.0.0/0 → igw-xxxxx

This route says: "any packet going to a destination not in this VPC should be forwarded to the Internet Gateway." That's what makes the subnet reachable from the internet (when combined with an EC2 having a public IP).

Private Subnet

Its route table has no route to the Internet Gateway. It might have:

0.0.0.0/0 → nat-xxxxx

That sends outbound traffic through a NAT Gateway (so instances can reach the internet outbound for updates), but inbound from the internet has no way in.

The truth nobody tells beginners

An EC2 instance is not "public" or "private" by its own nature. The subnet's route table decides whether the subnet can reach the internet. The EC2's public IP decides whether the internet can reach the EC2. Both must align.

Even if you put an EC2 in a "public" subnet but don't give it a public IP, it's effectively private — the internet has no way to address it.

Chapter SevenInternet Gateway — the only legitimate door

The Internet Gateway (IGW) is the single physical door between your VPC and the public internet. One per VPC. Without it, your VPC is hermetically sealed from the internet.

Two conditions must both be true for internet traffic to reach an EC2:

  1. The subnet's route table has 0.0.0.0/0 → igw-xxx
  2. The EC2 has a public IP address (auto-assigned or Elastic IP)

Either one missing? No internet reachability. This is the network-level isolation that makes "private" actually private — independent of any security group rules.

Chapter EightReplay of your ALB hands-on

Now let's walk through what you actually did, step by step, with what was really happening behind the screen.

1

You launched 2 EC2 instances, one in "1a" and one in "1b"

What you saw: pick a subnet → AWS shows default subnets for each AZ → you picked one in 1a, one in 1b.

What was actually happening: each subnet you picked was a default subnet in its respective AZ. Both have IGW routes. Both EC2s got public IPs. That's why you could SSH/HTTP to them directly.

2

You created a target group and added both EC2s

The target group is just a list — "here are the backends ALB can forward traffic to." It also defines the health check and the protocol/port used to reach the backends.

The target group isn't bound to a subnet or AZ — it can hold targets from any AZ in the VPC.

3

You created the ALB and selected subnets in 1a, 1b, 1c

What you thought: "did AWS create new public subnets for me?"

What actually happened: AWS did not create any subnets. You picked from existing default subnets in each AZ. AWS then deployed an ALB node (an ENI — Elastic Network Interface) inside each of the selected subnets. So you ended up with one ALB node in us-east-1a, one in 1b, one in 1c — all sitting in those default-public subnets.

4

You configured the listener — HTTP/HTTPS

The listener is "the door the ALB opens to the world." Port 80 or 443. The ALB starts accepting connections on that port at each of its nodes.

5

AWS gave you an ALB DNS name

Something like my-alb-1234.us-east-1.elb.amazonaws.com. When the world resolves this DNS name, AWS returns the public IPs of the ALB nodes — usually multiple IPs, one per AZ where the ALB has a node. That's how DNS-level load balancing across AZs happens.

6

You hit the ALB URL — traffic flowed

Your browser resolved the ALB DNS, picked one of the returned IPs, sent the request. The ALB node received it, picked a healthy EC2 from the target group (could be in any AZ — cross-zone load balancing is on by default for ALB), and forwarded the request.

7

You changed EC2 SG to allow only from ALB-SG

The clever move. Now your EC2 security group says "only allow port 80 if the source is the ALB's security group." This means even if someone discovered your EC2's public IP, they couldn't reach it directly. Only traffic that came through the ALB is allowed.

BUT — and this is the subtle part — the EC2 is still in a public subnet with a public IP. The SG is what's making it private-in-practice. In a real production setup, you'd also move the EC2 to a private subnet so the network itself blocks direct access.

Chapter NineHow ALB spans multiple AZs for HA

Your intuition was exactly right:

You said

"It must have made a copy of this ALB to each AZ (1a, 1b, 1c..) so that if ALB goes down in one AZ, others will work."

✓
Confirmed

Yes — that's exactly what AWS does. ALB is a managed service. When you select 3 subnets in 3 AZs, AWS deploys an ALB node (effectively a network interface + load-balancing logic) in each AZ. The DNS name resolves to multiple IPs, one per AZ.

One ALB = many nodes, one per selected AZ. Each node can route to any healthy target in any AZ.

Now if AZ 1a has an outage:

  • DNS stops returning the 1a node's IP (it's failing health checks).
  • Users get redirected to the 1b and 1c nodes.
  • Those nodes forward traffic to surviving EC2s in 1b and 1c.
  • End users see no outage — they just transparently moved to healthy AZs.

This is why ALB requires you to select subnets in at least 2 AZs. AWS will not let you create an ALB with only one AZ — they're forcing the HA pattern on you.

Chapter TenThe packet's full journey

Let's trace one HTTP request end-to-end through your architecture. Every hop, every gate, every check.

1

Browser resolves DNS

You type my-alb-1234.us-east-1.elb.amazonaws.com/home. Your browser asks DNS to resolve this name → DNS returns 2–3 IPs (one per AZ where the ALB has a node). Browser picks one (usually the closest).

2

Packet enters AWS via the Internet Gateway

The packet leaves your ISP, travels the internet, and enters your VPC through the Internet Gateway attached to that VPC. The IGW knows: "this destination IP belongs to an ALB node sitting in this public subnet."

3

Route table forwards to the ALB node's subnet

The IGW hands the packet to the public subnet's route table, which routes it locally to the ALB node's network interface.

4

ALB security group check

The ALB's security group says "allow inbound 443 from 0.0.0.0/0" (or 80 for HTTP). The packet passes.

5

ALB terminates TLS, reads the HTTP request

If HTTPS, the ALB decrypts using its ACM certificate. It now sees the HTTP request — path, headers, host. It evaluates listener rules to pick the right target group.

6

ALB picks a healthy target from the target group

From the registered targets, ALB picks one that's currently healthy. It may be in the same AZ as the ALB node, or in a different AZ — cross-zone load balancing doesn't care.

7

ALB sends a fresh HTTP request to the EC2's private IP

This new request originates from the ALB node's IP (inside the VPC). Travels through VPC routing to the EC2's subnet.

8

EC2's security group check

The EC2 SG says "allow inbound 80 from source = ALB's security group." The packet's source is the ALB node, which belongs to the ALB SG. ✓ Allowed.

If someone tried to hit the EC2's IP directly from the internet (not via ALB), the source wouldn't match the ALB SG — denied at the firewall.

9

EC2 application responds

Web server renders the page, sends HTTP 200 back. Response retraces the path: EC2 → ALB → IGW → internet → browser. Done.

Chapter ElevenSG vs Route Table — two different layers

This is the part you were mixing up. They sound similar — both are "rules about traffic" — but they live in completely different places and do completely different things.

Route Table — Layer 3 (Network)

Question it answers: "Can the packet physically reach this destination?"

Attached to: a subnet (or VPC).

If there's no route, the packet has nowhere to go. It's dropped at the network level. Doesn't matter what SG rules say — the packet never arrives.

Example: Private subnet has no 0.0.0.0/0 → IGW route. Internet packets simply have no path to reach instances in this subnet.

Security Group — Layer 4 (Firewall)

Question it answers: "If a packet arrives, is it allowed?"

Attached to: an ENI (network interface on an instance).

SGs are stateful. They evaluate incoming packets. If a rule allows it → packet reaches the instance. If no rule allows it → dropped at the firewall.

Example: EC2 SG allows port 80 only from the ALB's SG. Any other source is dropped, even if the packet successfully traveled to the EC2.

Both must approve

For traffic to reach your instance, both the route table AND the security group must allow it. They're independent gates in series. SG is not a substitute for route table design, and vice versa.

This is why "putting the EC2 in a private subnet" is genuinely different from "blocking everything in the SG." The private subnet stops the packet at the network layer — there's nothing to filter. The SG approach lets the packet arrive at your instance, then refuses it. Both work, but the network-layer approach is more robust and idiomatic.

Chapter TwelveYour insight: same subnet for ALB & EC2

You wrote:

You said

"In my hands-on, the ALB node and the EC2 were both in the same default subnet for 1b. It's not like the subnet has a property of public/private — it's what's inside it and how rules give it to it. So if I only allow traffic from ALB on this instance, it works even though the ALB is taking public requests."

✓
Mostly correct

Yes — same-subnet works. ALB doesn't require its targets to be in a different subnet, or even a different AZ, than itself.

What you built is functionally OK and very common in beginner labs. It's just not the production-grade topology. Let's see why.

Why same-subnet works

  • The subnet is "public" (has IGW route).
  • ALB node listens on the public side, accepts internet traffic.
  • ALB then sends a fresh request to the EC2's IP (still inside the subnet — local routing, no IGW involved for this hop).
  • EC2's SG allows traffic only from ALB-SG → the only requests that reach the EC2 are ALB-relayed ones.
  • Direct internet → EC2 attempts are dropped by the SG.

Why production doesn't do this

  • Network-layer protection in addition to SG. If your EC2 sits in a public subnet with a public IP, the network can reach it. You're relying purely on the SG to keep it private. One SG misconfiguration and you're exposed. With the EC2 in a private subnet, the network itself blocks direct access — defense in depth.
  • No public IPs on app servers. Production app servers should never have public IPs. The blast radius of any compromise is reduced.
  • Clean operational boundaries. Public subnet routing rules are different from private subnet routing rules (NAT, VPC endpoints, etc.). Mixing makes operations harder.
  • NACLs (subnet firewalls) can apply different rules at the subnet level. You can be stricter at the network layer for private subnets.

Chapter ThirteenWhat a TRUE private setup looks like

Production setup uses a 2-subnet pattern per AZ:

Per AZ: one public subnet (ALB + NAT) and one private subnet (app servers). The EC2s have no public IP and are network-isolated from the internet.

What changes vs your hands-on:

  • EC2s sit in a private subnet with no IGW route. Internet has no network path to them at all.
  • EC2s have no public IP. There's no public address to attack.
  • EC2s still need to reach the internet outbound (for OS updates, S3, etc.) — that's where the NAT Gateway in the public subnet comes in. NAT allows outbound but blocks inbound.
  • ALB stays in the public subnet — it's the only thing the internet can talk to.
  • EC2 SG still allows only from ALB-SG (defense in depth).

Now you have two independent layers of isolation: the network (no route) and the firewall (SG rules). Either one alone would block direct access. Both together is what real architects mean by "private."

Chapter FourteenYour other question: app + DB in same subnet?

You asked:

You said

"You mentioned Private App Subnet and Private DB Subnet as two different subnets. But I can create both in the same subnet, right? I can just divide them by target group — one for app servers, one for DB. The only disadvantage I see is the same hardware might fail and take both down."

~
Partially right, mostly wrong

Yes, technically you can put app and DB in the same subnet — networking doesn't forbid it.

But the reason production splits them is not about hardware separation. That's actually a misconception. Subnets in the same AZ share the AZ's underlying hardware pool — splitting them into different subnets doesn't put them on physically different hardware.

The real reasons are about routing, policy, and managed-service requirements.

The misconception about hardware

Two subnets in the same AZ are in the same physical data-center cluster. They share AZ-level infrastructure: power, cooling, network gear. If us-east-1b has a fire, both subnets in 1b are gone together. Splitting into multiple subnets inside one AZ does not protect against AZ-level failure.

If you want hardware-level fault tolerance, you need different AZs, not different subnets.

Chapter FifteenDifferent subnets in the SAME AZ — what they actually buy you

So if not hardware separation, why split app and DB into different subnets at all? Five real reasons.

1. Different route tables

Each subnet has its own route table. App servers might need a route through NAT for outbound internet (to download packages, hit external APIs). The DB subnet might have zero internet routes — outbound blocked at the network level. The DB literally has no way to reach the internet, even if compromised.

2. Different NACL boundaries

NACLs (Network ACLs) are stateless firewalls applied per subnet. You can have a stricter NACL on the DB subnet (e.g., only allow port 3306/5432 from the app subnet CIDR, deny everything else) than on the app subnet.

3. Cleaner IAM and tagging boundaries

You can write IAM policies like "developers can launch EC2 only in the app subnet, not the DB subnet." Tag-based access control. Auditing becomes simpler.

4. Managed service requirements

RDS won't let you put a database in a target group behind an ALB — RDS uses a DB Subnet Group instead (we'll cover this next). Even if you wanted to "put DB in same subnet as app," RDS forces you into a specific subnet model.

5. Conceptual clarity

Six months from now, when a new engineer joins your team and sees private-app-subnet and private-db-subnet, they immediately understand the architecture. If everything's in one big misc-subnet, nobody knows what should live where. Subnet naming is documentation.

The big takeaway

Different subnets in the same AZ = different routing, NACLs, and policy boundaries. NOT different hardware. Not different fault domains. If you need real fault tolerance, you need different AZs.

Chapter SixteenDifferent subnets in DIFFERENT AZs — real fault tolerance

This is what gives you actual resilience against an AZ outage.

SetupSurvives AZ outage?Why
Two subnets, both in 1bNoSame AZ failure domain
Two subnets in different AZs (1a + 1b)YesIndependent failure domains
ALB across 1a + 1b + 1cYesALB DNS skips the failed AZ

The rule of thumb: HA = at least 2 AZs. Anything less is a single point of failure dressed up to look fancy.

Chapter SeventeenRDS & the DB Subnet Group model

Databases on AWS use a different model from EC2 + ALB. You don't put RDS behind a target group. Instead:

  1. You create a DB Subnet Group — a named collection of subnets that RDS is allowed to place itself in.
  2. The DB Subnet Group must contain subnets in at least 2 AZs. AWS enforces this.
  3. When you create an RDS instance, you point it at a DB Subnet Group. RDS picks which subnet (and AZ) to deploy in.
  4. If you enable Multi-AZ, RDS automatically deploys a standby in a different AZ from the subnet group, and handles failover for you.
  5. Your app server connects to the RDS endpoint (a DNS name). RDS handles the network routing.
The clean mental model

ALB & ASG = target groups (you control which instances are in there). RDS & ElastiCache = subnet groups (you tell AWS the pool of subnets, AWS handles placement and failover).

Different services, different patterns. Don't try to force RDS into an ALB target group — it doesn't work that way.

Chapter EighteenCross-zone load balancing

You asked: "If I make a request, ALB will call the ALB node in default subnet c... and then it will call that instance in c which is in a different subnet."

Close, but with an important addition. By default, ALB has cross-zone load balancing turned on. This means:

Cross-zone means

An ALB node in AZ-1a can forward traffic to a target in AZ-1b or AZ-1c. The ALB doesn't try to keep traffic "local" to one AZ. It picks healthy targets across all AZs evenly.

So in your example: even if the ALB node receiving the request is in 1c, it might forward to an EC2 in 1a or 1b. The traffic flow is:

  1. User → DNS → IP of ALB node (say in 1c)
  2. ALB node in 1c receives request
  3. ALB picks a healthy target — could be any EC2 in any AZ in the target group
  4. ALB forwards via VPC internal routing

This is why ALB doesn't care whether your EC2 is in the same subnet/AZ as the ALB node. VPC routing handles cross-subnet, cross-AZ traffic seamlessly within one VPC.

Watch out

Cross-zone is free and always on for ALB — no choice, you can't turn it off, you don't pay extra. But for NLB, cross-zone is OFF by default and turning it on incurs inter-AZ data transfer charges. This is a real exam question and a real production gotcha.

Chapter NineteenThe 3 mystery options at ALB creation

You mentioned: "then 3 other options which I didn't understand at all — CloudFront + WAF, WAF, Global Accelerator."

These are not load balancers — they're services that sit in front of your ALB to add capabilities. AWS shows them at ALB creation because they're commonly added together. Let's unpack each.

CloudFront — a CDN

A global content delivery network. Caches your HTTP responses at 600+ edge locations worldwide. Users hit the nearest edge, get cached content fast, and only cache misses go to your ALB. Used for static assets, images, video, and API caching. Reduces ALB load and improves global latency.

WAF — Web Application Firewall

An L7 firewall that inspects HTTP requests and blocks attacks: SQL injection, XSS, bot traffic, suspicious patterns. Attaches to ALB, CloudFront, API Gateway. Comes with managed rule sets (AWS-curated, OWASP top 10) plus your own custom rules. Doesn't replace ALB — it sits in front of it.

Global Accelerator — static IPs + private backbone

Gives you 2 static anycast IP addresses that work globally. Traffic enters AWS at the nearest edge, then travels over AWS's private backbone (faster, lower jitter than the public internet) to your ALB. Useful for: (1) when you need static IPs to whitelist, (2) when you want fast failover across regions without DNS, (3) when global users need consistent low latency.

Chapter TwentyReal industry architecture

Let's put it all together with a "Netflix-style" pattern that real companies use.

The actual shape of a production-grade web app on AWS. Notice the 3-tier subnet pattern per AZ.

What each tier does and why it's separate

Public Subnet

Contains ALB nodes and NAT Gateways. The only tier exposed to the internet. Has a route to IGW. Tightly secured at SG level (only ALB ports, only NAT outbound).

Private App Subnet

Application servers. No public IPs. Internet outbound via NAT (for updates, external APIs). Inbound only from ALB SG. Where business logic runs.

Private DB Subnet

Databases, caches. Inbound only from app subnet SG. May have zero internet route — totally network-isolated. Used for RDS DB subnet groups, ElastiCache subnet groups.

Isolated Subnet (optional)

For workloads that should never touch the internet at all — internal microservices, secrets stores, compliance-sensitive data. No route to IGW, no NAT. Only reachable from within the VPC.
