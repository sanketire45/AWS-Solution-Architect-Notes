Chapter OneThe problem nobody told you about

Imagine you build a website. You put it on one server. It works. You're happy.

Then one of three things happens.

One: the server crashes at 3am. Your site is down. You wake up to angry emails.

Two: your website goes viral. A thousand people show up at the same time. Your one server, designed for ten people, melts. The site loads in 47 seconds — if it loads at all.

Three: you need to update the software. To install the update, you have to turn the server off. While it's off, the site is down. Customers see an error.

Each of these is a different problem, but they share a root cause: you only have one of the thing. One of anything is fragile. One of anything is a bottleneck. One of anything is a single point of failure.

Real-world parallel

Picture a popular restaurant with one chef. If the chef gets sick — the restaurant closes. If 200 people show up for dinner — they wait two hours. If the chef wants a holiday — no dinner that week.

The fix isn't a faster chef. The fix is more chefs, a host at the door directing customers, and a manager who hires more chefs when the queue grows. That's exactly what we're going to build in AWS.

The host at the door is the Load Balancer. The manager who hires and fires chefs based on how busy it is — that's the Auto Scaling Group. Together they solve all three problems: failures, traffic spikes, and zero-downtime updates.

This entire document is about understanding those two pieces deeply enough that you can sketch this architecture for any business problem someone throws at you — including problems on the SAA-C03 exam.

Chapter TwoFour words you must understand

Architects throw around four words that sound similar but mean different things. Mixing them up is one of the most common reasons people miss exam questions and design bad systems. Let's nail them down.

1. High Availability (HA)

The system keeps working when something breaks. We typically measure this in "nines" — 99.9% (three nines), 99.99% (four nines), and so on. Four nines = roughly 52 minutes of downtime per year.

HA usually means: have more than one of each thing, in separate failure zones, with automatic failover when one breaks.

Restaurant analogy

Two chefs in two kitchens, in two different buildings. If a fire shuts down one kitchen, the other keeps cooking.

2. Scalability

The system can handle more load when needed. Two flavours:

  • Vertical scaling — make the existing server bigger (more CPU, more RAM). Like upgrading from a small van to a big truck.
  • Horizontal scaling — add more servers running the same thing. Like adding more vans to your fleet.

3. Elasticity

The system scales up AND down automatically based on demand. Scalability says "we can grow." Elasticity says "we grow when needed and shrink when not — without anyone pressing a button."

Elasticity is what saves you money. A non-elastic but scalable system can grow but stays big — you pay for the peak even at 3am when nobody's there.

4. Fault Tolerance

The system continues working with zero impact even when something fails. Stronger than HA — HA might have a 30-second blip during failover; fault tolerance has none. Achieved by running redundant active components, often across multiple regions.

The cleanest one-liners

HA = stays up when things break.
Scalable = can handle more load.
Elastic = grows AND shrinks automatically.
Fault Tolerant = HA with zero downtime, even mid-failure.

Chapter FourWhat is a Load Balancer, really?

So we've decided to scale horizontally. We have ten web servers running our app. There's an immediate problem: which server does the user's request go to?

If we hand out IP addresses to users randomly, half the servers might be slammed and half might be idle. Worse, if one server dies, users who were "assigned" to it just see errors.

We need something that sits in front of the servers, takes every incoming request, and intelligently picks a healthy server to handle it. That something is a load balancer.

Users always hit one address — the load balancer — which spreads the work across many servers.

What a load balancer actually does for you

  1. Spreads the load across many servers so no one server gets crushed.
  2. Hides failed servers — if a server stops responding, the LB stops sending traffic to it. Users never notice.
  3. Gives you a single endpoint — one DNS name (or IP). You can add or remove servers behind it without changing anything users see.
  4. Handles SSL/TLS — decrypts HTTPS so your servers don't have to. Faster, cheaper, simpler.
  5. Enables zero-downtime deploys — drain traffic from old servers, route to new ones, swap.
Restaurant analogy

The host at the door of a busy restaurant. You don't pick a chef — you give your name to the host, and the host walks you to a chef who's free, whose station is clean, and who isn't on break. If a chef burns out mid-shift, the host stops sending tables to them. The customer never knows.

Chapter FiveThe OSI detour — Layer 4 vs Layer 7

You'll keep seeing "Layer 4" and "Layer 7" in AWS docs and exam questions. If you understand this in 90 seconds, the entire ELB family snaps into place.

The OSI model is a 7-layer way of describing how computers talk to each other. We only need two layers:

Layer 4 — the Transport Layer
This is the level of TCP and UDP. It deals with raw connections — IP addresses, ports, packets. It does NOT understand what's inside the connection. It doesn't know if the data is HTTP, video, a game, or noise.
Layer 7 — the Application Layer
This is the level of HTTP, HTTPS, gRPC, WebSocket. It understands the meaning of the data — URLs, headers, cookies, hostnames. It can read "GET /api/users" and decide based on it.
Postal analogy

Layer 4 is like a postal worker who only reads the address on the envelope. They can deliver fast. They don't know or care what's inside.

Layer 7 is like a personal assistant who opens every letter and decides where to forward it based on the contents. Slower, but smarter — "this is a job application, send it to HR; this is an invoice, send it to Finance."

The headline

A Layer 4 load balancer is fast and protocol-agnostic, but dumb about content. A Layer 7 load balancer is slower and HTTP-only, but can route based on URL, host, headers, and cookies. AWS gives you one of each — plus a third for security inspection.

Chapter SixThe Elastic Load Balancing family

AWS's load balancing service is called ELB — Elastic Load Balancing. It's not one product. It's a family of four:

For the SAA-C03 exam and for real-world architecture today, you only care about the first three. Let's go through them one at a time.

Chapter SevenApplication Load Balancer (ALB) Layer 7

The ALB is the workhorse of AWS web architecture. If you're building a website, an API, a microservice, or anything HTTP — ALB is almost always the answer.

💵 An ALB costs ~$0.0225/hr ≈ $16/month even with zero traffic (the fixed hour charge), plus a usage charge (LCUs) on top — figure ~$20–30/mo for a busy small app. NLB is the same base; Gateway LB is cheaper (~$9/mo). Ireland ≈ +12%. Idle load balancers in dev accounts are a classic waste.

What makes ALB special: it understands HTTP

Because ALB operates at Layer 7, it can read every incoming request and route it differently based on what's inside. This unlocks patterns that no Layer 4 LB can do:

  • Path-based routing. Send /api/* to one set of servers, /images/* to another, /admin/* to a third. One domain, many backends.
  • Host-based routing. shop.example.com goes to one app, blog.example.com to another — same load balancer.
  • Header-based routing. Route by user-agent, custom headers, or anything in the HTTP request.
  • Query-string & method routing. Treat ?version=v2 requests differently. Send POSTs to a different backend than GETs.
A single ALB serving three different backends based on the URL path.

What ALB can route to (its "targets")

  • EC2 instances — the classic case.
  • IP addresses — including on-premises servers reached via VPN/Direct Connect.
  • Lambda functions — yes, ALB can invoke Lambda. Useful for serverless backends behind a corporate domain.
  • Containers (ECS / EKS tasks) — automatically registers and deregisters as containers come and go.

Other ALB superpowers

  • HTTPS / SSL termination with certificates from ACM (free, auto-renewing).
  • Authentication built in — integrates with Cognito User Pools, OIDC, SAML. Your servers don't have to handle login at all.
  • WebSocket and HTTP/2 support natively.
  • Sticky sessions via cookies.
  • WAF integration (Web Application Firewall — blocks SQL injection, XSS, etc.).
  • Access logs to S3, request tracing.
The classic ALB trap

ALB does NOT have a static IP address. It has a stable DNS name (something like my-alb-1234.us-east-1.elb.amazonaws.com), and the IP behind it can change at any moment. If a question says "the application requires whitelisting specific IP addresses" or "customers connect to a hardcoded IP" — ALB is wrong. Use NLB (static IP per AZ) or front the ALB with Global Accelerator (gives you 2 static anycast IPs).

Exam decision rule

HTTP / HTTPS + path or host routing + microservices + containers + Cognito auth → ALB. Almost everything web-shaped goes here.

Chapter EightNetwork Load Balancer (NLB) Layer 4

If ALB is the smart concierge, NLB is the toll booth. It does very little — and does it incredibly fast.

Why NLB exists

NLB operates at Layer 4. It looks at IP addresses and ports and forwards packets. It does NOT read your traffic. It cannot do path routing. It cannot terminate HTTPS in the same way ALB does (well, it can terminate TLS, but it doesn't read HTTP).

What it gives you in exchange:

  • Insane performance. Millions of requests per second. Microsecond latency. Built on AWS Hyperplane (the same tech behind PrivateLink).
  • Static IP address per Availability Zone — you can also assign your own Elastic IPs. This is the reason most NLB questions exist.
  • Any TCP / UDP / TLS workload. Game servers, IoT, financial trading, MQTT, video streaming, custom protocols — anything non-HTTP.
  • Source IP preservation — by default, the backend sees the real client IP. ALB does not (it sets X-Forwarded-For).

The trade-offs

  • No path routing. No host routing. No header inspection.
  • No native authentication.
  • No WAF integration directly (WAF is HTTP-aware; you'd put WAF on a CloudFront/ALB in front).
Real-world parallel

ALB is a sorting office. Every package gets opened, read, and routed.

NLB is a high-speed conveyor belt. Packages fly past at 100 km/h. Nobody opens anything. They go to the bay marked on the box and that's it.

Exam triggers for NLB

"Static IP addresses" · "millions of requests per second" · "ultra-low latency" · "TCP / UDP traffic" · "preserve the client's source IP" · "non-HTTP protocol" → NLB.

Chapter NineGateway Load Balancer (GWLB)

GWLB is the strangest of the three because it's not really for load balancing your application. It's plumbing for security tools.

The problem GWLB solves

Big companies often want every packet entering their VPC to be inspected by a third-party firewall, intrusion detection system, or deep packet inspection appliance — products from Palo Alto, Fortinet, Check Point, Trend Micro and so on.

Before GWLB, inserting these appliances into the traffic flow was painful. You'd have to redesign routing, deal with asymmetric flows, manage failover for the appliances yourself.

GWLB makes it transparent: traffic enters the GWLB, gets bounced through a fleet of security appliances, comes back, and continues to its destination — and the appliances appear "in line" without complex routing changes.

Two GWLB-specific facts to remember

  • It uses the GENEVE protocol on port 6081 to encapsulate and forward traffic to appliances. Memorise this — exam loves it.
  • It pairs with a GWLB Endpoint (GWLBe) in your VPC, similar to a VPC endpoint, to route traffic through the appliances.
Exam trigger phrases

"third-party virtual appliances" · "transparent traffic inspection" · "firewalls in line" · "GENEVE / port 6081" → GWLB. If the question is about balancing your own application servers, GWLB is the wrong answer.

Chapter TenThe bits inside every load balancer

Whatever flavour of LB you pick, the internal anatomy is the same. Three components.

1. Listener

A listener is the door the LB opens to the world. It's a protocol + port combination — for example, "HTTPS on port 443" or "TCP on port 25". A listener has rules attached that decide where matching traffic goes.

2. Target Group

A target group is the pool of backends a listener forwards traffic to. Targets can be EC2 instances, IP addresses, Lambda functions, or containers. The target group also defines:

  • The protocol and port the LB uses to reach the targets (often different from what the user uses — e.g., users use HTTPS:443, LB talks to backends on HTTP:80).
  • The health check configuration (next chapter).
  • Stickiness, deregistration delay, slow start.

3. Rules (ALB only)

For ALB, listener rules do the smart routing — "if path is /api/* send to target group A; if host is shop.example.com send to target group B." Rules are evaluated in priority order, with a default rule for everything that doesn't match.

A listener catches traffic, rules decide where it goes, target groups receive it.

Chapter ElevenHealth checks — the heartbeat

The whole reason a load balancer can hide failures is health checks. Every few seconds, the LB asks each target: "are you OK?" If the target keeps saying yes, traffic flows. If it stops saying yes, the LB stops sending traffic.

How health checks work (ALB)

You configure:

  • Protocol & path — typically GET /health over HTTP.
  • Healthy threshold — e.g., 5 consecutive successes = healthy.
  • Unhealthy threshold — e.g., 2 consecutive failures = unhealthy.
  • Interval — how often to check (default 30s).
  • Timeout — how long to wait for a response.
  • Success codes — what HTTP status codes count as healthy (default 200).
Pro tip — make health checks meaningful

A bad health check just returns "200 OK" without checking anything — a server with a broken database connection still passes. A good /health endpoint actually tries to reach the database, the cache, and any critical dependency. That's how you make sure the LB stops sending traffic to a server that's "up" but not actually working.

ELB health checks vs EC2 health checks (a critical distinction for ASG)

An EC2 instance can be "running" (the OS booted) but the application might be broken. By default, ASG only checks EC2 status — so it'll happily keep a broken instance because Linux is up.

You should turn on ELB health checks in the ASG configuration. Then ASG will replace any instance the load balancer marks as unhealthy. We'll come back to this later.

Chapter TwelveSticky sessions, cross-zone, deregistration delay

Sticky sessions (session affinity)

Normally, every request from a user might go to a different backend. That's fine if your app is stateless — the server doesn't remember anything between requests.

But if the server stores the shopping cart in its own memory, sending the user's next request to a different server means the cart vanishes. To prevent this, ALB can use a cookie to "stick" a user to the same backend for a configurable duration.

Why sticky sessions are usually a code smell

Sticky sessions are a workaround for state living in the wrong place. The right fix is to move session data to a shared store — ElastiCache for Redis, DynamoDB, or similar — so any server can serve any request. On the exam, if a question mentions "requests must go to the same backend", sticky sessions might be a valid quick fix, but if the question asks for the best long-term solution, the answer is usually "store session state in ElastiCache/DynamoDB."

Cross-zone load balancing

If you have 4 instances in AZ-A and 1 instance in AZ-B, what happens? Without cross-zone, the LB sends 50% of traffic to each AZ — meaning the 1 instance in AZ-B gets the same total load as the 4 in AZ-A combined. That single instance gets crushed.

With cross-zone load balancing on, the LB distributes evenly across all instances regardless of AZ. Each of the 5 instances gets 20%.

  • ALB — cross-zone is always on, free.
  • NLB — cross-zone is off by default, and turning it on may incur inter-AZ data charges.

Deregistration delay (connection draining)

When you remove a target from a target group (manual deregister, or ASG terminating it), there might be in-flight requests being processed. If the LB cuts the target off instantly, those users get errors.

The deregistration delay tells the LB to stop sending new requests to the target but allow up to N seconds for in-flight ones to complete. Default is 300 seconds (5 minutes). Tune lower for stateless APIs, higher for long-running uploads.

Chapter ThirteenSSL/TLS termination & SNI

SSL/TLS termination

When a user connects via HTTPS, the connection is encrypted. Decrypting on the load balancer (rather than on every backend) is called SSL termination. Benefits:

  • Backends don't waste CPU on decryption.
  • Certificates live in one place — easier to manage and rotate.
  • The LB can read HTTP headers and apply Layer 7 rules (impossible if it can't decrypt).

You attach an ACM certificate (free, auto-renewing) to the listener. Done.

SNI — Server Name Indication

What if you host 5 different websites behind one ALB, each with its own HTTPS certificate? Old SSL only allowed one certificate per IP. SNI fixes that — the client tells the server which hostname it's after, and the LB picks the matching certificate.

ALB and NLB both support SNI. CLB does not — that's one reason CLB is dead.

Exam trigger

"Host multiple HTTPS sites with different certificates on a single load balancer" → SNI on ALB or NLB.

Chapter FourteenAuto Scaling Groups — the smart manager

The load balancer solves distributing traffic. But what about the supply side — having the right number of servers? That's the Auto Scaling Group's job.

💵 The Auto Scaling Group itself is completely free — $0. It's a control loop, not a billed resource; you pay only for the EC2 instances it launches. That's why scaling in aggressively or scaling to zero overnight is pure saving — there's no penalty for the ASG, only for running boxes.

What an ASG is, in one sentence

An Auto Scaling Group is a manager that maintains a target number of identical EC2 instances, replaces unhealthy ones automatically, and grows or shrinks the fleet based on demand.

The three numbers that define an ASG

Minimum
The fewest instances ASG will ever run. If a failure drops you below this, ASG launches replacements until you're back at minimum.
Desired
The current target. ASG will run exactly this many healthy instances. Scaling policies change this number up and down.
Maximum
The cap. ASG will never run more than this, even if a scaling policy says to. Stops a runaway scale-out from bankrupting you.
Restaurant analogy

Min = "we always have at least 2 chefs working, even at 4am." Desired = "right now, given how busy we are, the right number is 5 chefs." Max = "no matter how crazy the rush, we will not employ more than 20 chefs because we can't afford it."

What the ASG actually does

  • Launches instances using a Launch Template (the recipe).
  • Spreads them across Availability Zones for HA.
  • Registers them with target groups automatically — they appear behind the LB without any action.
  • Watches health (EC2 status + optionally ELB health checks).
  • Replaces unhealthy ones by terminating and launching new.
  • Adjusts desired based on scaling policies.

Chapter FifteenLaunch Templates and the moving parts

Launch Template — the recipe

A launch template is a blueprint for what an instance should look like: AMI, instance type, key pair, security groups, IAM instance profile, user data script, EBS volumes, network interfaces, and so on.

You used to use Launch Configurations for this. They're being deprecated. Always pick Launch Template in exam answers and in real life. Reasons:

  • Launch Templates support versioning — you can edit and roll forward/back.
  • Mix On-Demand and Spot instances in the same ASG.
  • Use newer features like T2/T3 unlimited, multiple network interfaces, capacity reservations.

The other ASG settings worth knowing

  • VPC + subnets — choose subnets in multiple AZs for HA.
  • Health check type — EC2 only (default) or EC2 + ELB. Almost always pick both.
  • Health check grace period — how long to wait after launch before checking health (give the app time to start).
  • Cooldown — after a scaling action, wait N seconds before doing another (avoids flapping).
  • Termination policy — when scaling in, which instance to kill first (oldest launch config, oldest instance, closest to next billing hour, etc.).

Chapter SixteenScaling policies — how ASG decides

How does the ASG know when to add or remove instances? Through scaling policies. There are five flavours.

1. Manual scaling

You change the desired capacity yourself. Useful for testing or one-off events. Not really "auto" scaling.

2. Dynamic — Target Tracking (modern, recommended)

You say: "keep average CPU at 50%" or "keep ALB request count per target at 1000". ASG figures out the rest. It creates the necessary CloudWatch alarms behind the scenes and adjusts capacity to hit the target.

This is the simplest and the answer to most exam questions about scaling.

3. Dynamic — Step Scaling

You define alarms with thresholds and step adjustments. "If CPU > 60%, add 1 instance. If CPU > 80%, add 3 instances. If CPU > 90%, add 5 instances." More control, more configuration.

4. Dynamic — Simple Scaling (legacy)

One alarm, one action. Has to wait for cooldown before another action. Step Scaling does everything Simple does, better.

5. Scheduled scaling

Scale based on time, not metrics. "Every Monday at 9am, set min=10. Every Friday at 6pm, set min=2." Good for predictable patterns — business hours, batch jobs, weekly traffic peaks.

6. Predictive scaling

Machine-learning-based. AWS analyses up to 14 days of historical data and forecasts load up to 48 hours ahead, then pre-warms capacity before you need it. Useful when your load has clear patterns (most consumer apps do) but the lead time to launch instances is long.

PolicyPick when
Target TrackingDefault. "Keep CPU at 50%" — simplest and best for most.
Step ScalingYou need granular control over how aggressively to scale at different load levels.
ScheduledLoad is predictable by clock — known peak times.

Exam trigger phrases

"automatically maintain performance" → Target Tracking.
"scale at known busy times" → Scheduled.

Chapter SeventeenLifecycle Hooks, Warm Pools, Instance Refresh

Lifecycle Hooks

An instance launched by ASG normally goes from Pending straight to InService. Sometimes you need to do something before it goes live — install software, register with a DNS, run database migrations, warm a cache.

A Lifecycle Hook pauses the instance in a state called Pending:Wait, runs your custom logic, then continues. Same on the way out — Terminating:Wait lets you flush logs, drain connections, deregister from external systems before termination.

Exam trigger

"Run a script before the instance starts serving traffic" or "clean up before termination" → Lifecycle Hooks.

Warm Pools

Instances take time to launch — sometimes 60+ seconds for big AMIs with bootstrapping. During a sudden traffic spike, that lag matters.

A Warm Pool is a pre-initialised reserve of instances that are stopped (or hibernated) but ready to go. When ASG scales out, instances come from the warm pool first — they're up in seconds, not minutes.

Keeps cost down (stopped instances don't bill compute) while cutting startup time dramatically.

Instance Refresh

You changed the launch template (new AMI, new instance type, new user data). Now you want to roll those changes out to existing instances. Instance Refresh automates this — gradually terminates old instances and replaces them with new ones following the latest template, with a configurable percentage healthy at all times.

Replaces the old, manual process of detaching, draining, and re-launching by hand.

Chapter EighteenMixed instances & Spot inside ASG

One of the most powerful real-world ASG patterns is the mixed instances policy. You can configure an ASG to:

  • Run on multiple instance types (e.g., m5, m5a, m5n) for capacity flexibility.
  • Mix On-Demand and Spot in defined ratios (e.g., "always 30% On-Demand baseline; the rest Spot").
  • Define an On-Demand base that's never interrupted.

This gives you ASG-level reliability with massive cost savings. Many production fleets run 70-90% Spot for stateless workloads — a ~70% cost reduction over pure On-Demand for the variable portion.

Where Spot is wrong

Spot can be reclaimed with 2 minutes' notice. Don't use Spot for stateful workloads, databases, or anything where loss of an instance corrupts data. For stateless web/app servers behind an LB, Spot is excellent — losing one is fine because the LB stops sending traffic to it and the ASG launches a replacement.

Chapter NineteenELB + ASG: the power couple in production

Now we put it all together. This is the reference architecture for a highly available, scalable web application on AWS. If you can sketch this from memory, you understand the core of SAA-C03.

The classic "3-tier web app" front end: Route 53 → ALB → ASG of EC2 across two AZs.

How traffic flows

  1. The user types shop.example.com. Route 53 resolves it to the ALB's DNS name.
  2. The ALB receives the request at one of its AZ-resident nodes.
  3. The ALB picks a healthy EC2 instance from the target group (which is automatically populated by the ASG) and forwards the request.
  4. If that instance is unhealthy, traffic skips to the next.
  5. If load is high, ASG launches more instances; they automatically join the target group and start receiving traffic.
  6. If load drops, ASG terminates instances; the LB drains connections gracefully via deregistration delay.

Why this is the gold standard architecture

  • HA — multiple AZs; one AZ failure = no outage.
  • Scalable — ASG grows and shrinks horizontally.
  • Elastic — Target Tracking + scheduled hits both predictable and unexpected demand.
  • Self-healing — ELB health checks + ASG replacement = unhealthy instances auto-replaced.
  • Zero-downtime deploys — Instance Refresh + deregistration delay = roll out new code with no user impact.
  • Cost-efficient — Mixed instance / Spot policy can cut compute costs 50-70%.

💰 CostWhat it all costs

Before the war stories, price the power couple in your head. Anchored on us-east-1; eu-west-1 (Dublin) noted where it differs. Standard month = 730 hours.

💰 What you're billed on

A load balancer has two meters: a fixed hourly charge (you pay it even with zero traffic) plus capacity units — LCUs for ALB, NLCUs for NLB — that meter new connections, active connections, processed bytes, and rule evaluations. The Auto Scaling Group itself is completely free — it's a control loop, not a billed resource. You only pay for the EC2 instances it launches.

⚠️ The hidden costs

An idle ALB still costs ~$16/month — that fixed hourly charge runs whether one request hits it or none. Forgotten ALBs in dev accounts are a classic waste. And LCUs are driven by processed bytes, so a high-throughput app can pay far more in LCUs than in the base hour. On NLB, cross-zone load balancing bills inter-AZ data ($0.01/GB each way) — it's off by default for exactly that reason.

🎯 The one cost lever

ASG being free is the whole game. Because you only pay for running instances, scaling down to zero at night (scheduled actions) or scaling in aggressively after a spike is pure saving — no penalty for the ASG itself. Stack that with Spot inside a mixed-instances policy (up to 90% off the elastic slice) and you cut the compute bill 50–70% while the load balancer line stays a flat ~$16–25/mo. In the small-prod-app reference stack, the ALB is just ~$24/mo of the ~$240 total — the instances are where the money (and the savings) live.

Chapter TwentyIndustry patterns & war stories

Pattern 1: Blue/Green deployment

You have v1 running. You spin up a parallel ASG running v2 with its own target group. The ALB has weighted routing — start with 100% v1 / 0% v2, gradually shift to 90/10, 50/50, 0/100. Roll back instantly by shifting weight back if v2 misbehaves. Zero downtime, fast rollback.

Pattern 2: Canary releases

Similar but smaller — send 1-5% of traffic to a new version using ALB rules (e.g., based on a header), monitor errors, ramp up if healthy, kill if not. Often combined with feature flags.

Pattern 3: A/B testing

ALB weighted routing between two target groups running different versions of the app. Measure conversion rates between user populations. This is identical machinery to blue/green — just different intent.

Pattern 4: Multi-region active/active for global users

Run the ALB+ASG stack in multiple regions. Use Route 53 latency-based routing or Global Accelerator to send users to the nearest region. Failover is automatic if a whole region dies.

Pattern 5: Public ALB → Private app servers

The ALB sits in public subnets. The EC2 instances sit in private subnets with no public IPs. The ALB is the only thing exposed to the internet. Security group on the EC2 only allows inbound from the ALB's security group. This is the standard secure topology.

War story: the cross-zone bill shock

A team turned on cross-zone load balancing on an NLB without realising AWS charges for inter-AZ data on NLB. With heavy traffic and unbalanced AZs, their data transfer bill jumped 400%. Lesson: ALB cross-zone is free; NLB cross-zone is not — turn it on with eyes open.

War story: the "healthy" but broken servers

A startup had EC2-only health checks on their ASG. A bug in the app left the process running but unable to talk to the database. EC2 said "running"; users saw 500 errors for hours. Fix: use ELB health checks pointing at a meaningful /health endpoint that actually pings dependencies. This is so common it's basically a rite of passage.

War story: the slow startup

Auto-scaling fired during a flash sale. New instances took 4 minutes to launch and bootstrap. By the time they were ready, the spike was over and the original instances had crashed. Fix: pre-bake the AMI with everything installed, plus use Warm Pools to keep a buffer of pre-initialised instances ready to go in seconds.

🏭 Named Dublin example — Ryanair's 3-tier stack

Ryanair (Europe's largest airline, HQ in Swords, Dublin) runs the textbook resilient web architecture for Ryanair.com — all-in on AWS: ALB → Auto Scaling group of EC2 → Aurora Multi-AZ + ElastiCache. Its serverless Ryanair Connect app sits behind an ALB protected by AWS WAF + AWS Shield Advanced. This is exactly the "ALB in public subnets, ASG of private-subnet instances, managed multi-AZ database behind it" pattern the exam rewards — and the model answer to "design a highly available, scalable web tier."

🏭 Named Dublin example — Intercom's hands-off Black Friday scaling

Intercom (customer-messaging SaaS, EU customers served from Ireland) says it best: "Each year on Black Friday, as many of our customers hit their busiest period, our infrastructure scales to match without human intervention." That is target-tracking auto-scaling on their worker fleets — set a target (CPU, or a request/queue metric) and let the ASG add and remove instances automatically. Exam mapping: "automatically hold utilisation near a target / scale with no manual effort" → Target-Tracking policy; if the peak time is known in advance, add Scheduled scaling.

Chapter Twenty-OneThe mental model to take into the exam

Forget the long lists for a second. Here's the model you actually need.

The 60-second mental model

Every web/app workload on AWS that needs HA + scalability has the same shape: Route 53 → ELB → ASG of EC2 (across multiple AZs) → backend (RDS/Aurora/DynamoDB).

The only real questions are:

(1) Which ELB? Default to ALB. Switch to NLB when the question screams static IPs, TCP/UDP, or extreme performance. Switch to GWLB when it's about third-party security appliances.

(2) How does the ASG scale? Default to Target Tracking. Use Scheduled for known peaks, Predictive for forecast-able patterns, Step for fine-grained thresholds.

(3) Where do the instances live? Private subnets, behind the ELB which sits in public subnets. Always multiple AZs.

The decision rules I'd want printed on my desk

Pick ALB when…
HTTP/HTTPS · path or host routing · microservices · containers · Cognito auth · WebSocket · default HTTP answer
Pick NLB when…
Static IP needed · TCP/UDP · millions of req/sec · ultra-low latency · preserve source IP · non-HTTP protocol
Pick Launch Template when…
Always. Modern. Versioned. Required for Spot mix and newer features.
Pick Launch Configuration when…
Never on the exam (it's deprecated for new use).
Pick Target Tracking when…
"Maintain a metric." 90% of scaling questions. Default answer.
Pick Scheduled / Predictive when…
The question describes time-based or forecast-based patterns specifically.

Concepts that show up disguised in questions

  • If they describe an instance "running but not responding to users" — answer involves enabling ELB health checks in ASG.
  • If they describe "I changed the AMI but old instances still run the old version" — answer is Instance Refresh.
  • If they describe "user shopping cart disappears when refreshing" — answer is store sessions in ElastiCache (or Sticky Sessions as a quick fix).
  • If they describe "need a script to run before traffic hits the new instance" — Lifecycle Hook.
  • If they describe "instances take too long to launch during spikes" — Warm Pool or pre-baked AMI.
  • If they say "guarantee 30% On-Demand baseline, rest cheapest" — Mixed instances policy with On-Demand base.
