Part 02

RDS — exam-specific facts

Everything that has actually appeared, with the exam angle.

TopicWhat the exam tests
Engines supported MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Aurora. The exam never asks "which engine"; it tests features that work across them. Aurora only supports MySQL- and Postgres-compatible — not Oracle/SQL Server. If a question mandates Oracle or SQL Server, the answer cannot be Aurora.
RDS is on EC2 underneath Implication: you choose an instance class (e.g. db.r6g.large) and pay per hour. Reserved Instances and Savings Plans apply to RDS. "Reduce cost for a 24×7 RDS database with predictable load" → Reserved Instance.
Storage Autoscaling Storage can grow automatically up to a max you set. Cannot shrink. "Database keeps running out of storage" → enable Storage Autoscaling.
Storage encryption Must be enabled at creation. To encrypt an existing unencrypted DB: snapshot → copy snapshot with encryption → restore. Read replicas inherit the primary's encryption state. Encryption uses KMS.
IAM database authentication Available for MySQL and PostgreSQL only. App uses IAM role to generate a 15-minute auth token instead of a password. Question trigger: "manage DB access without passwords / centralise via IAM."
Network RDS sits in a VPC. Public accessibility = OFF by default and should stay OFF. Security Group on the DB allows port 3306/5432/etc. from the app's SG only.
Reboot vs Failover "Reboot with failover" on a Multi-AZ instance triggers a manual failover — used to test DR and to recover from instance-level issues.
Deletion protection Must be turned off before you can delete the database. Production checklist: ON.
Custom RDS RDS Custom for Oracle/SQL Server lets you SSH into the underlying EC2 — for apps that need OS-level customisation. Rare exam topic but appears.

The single most-tested comparison in this topic. Get it cold.

PurposeHigh AvailabilityScale reads / DRHA + read scaling
ReplicationSynchronousAsynchronousSemi-synchronous
Readable standby?NOYESYES (2 readable)
Auto-failoverYES (~60–120s)NO (manual promote)YES (~35s)
Data loss on failoverZeroPossible (lag)Zero (almost)
LocationOther AZ in same regionAny AZ or regionTwo other AZs same region

The Three-Question Decision

Q1: Does the question say "highly available" or "failover" or "regional outage protection within a region"? → Multi-AZ.
Q2: Does the question say "scale reads" or "offload read traffic" or "cross-region read DR"? → Read Replica.
Q3: Does the question say "Multi-AZ and serve reads from the standby"? → Multi-AZ DB Cluster.

The Trap That Catches Everyone

"We have Multi-AZ. Our reads are slow. How do we fix it?" The wrong answer (which feels right): "Send reads to the standby." The standby in standard Multi-AZ is NOT readable. Correct answer: add Read Replicas, or switch to Multi-AZ DB Cluster, or add ElastiCache.

Part 05

RDS Security & RDS Proxy

Four security pillars + the most common Lambda+RDS exam question.

The Four Security Layers

1. Network: Private subnet, Security Group locked to app SG, no public access.
2. Encryption at rest: KMS-managed, set at creation. Existing DB → snapshot → copy with encryption → restore.
3. Encryption in transit: SSL/TLS for connections.
4. Authentication: Passwords (default) or IAM DB Auth (MySQL/Postgres only). Passwords stored in Secrets Manager.

RDS Proxy — the must-know feature

Trigger
"Lambda + RDS, 'too many connections' errors"
Answer
RDS Proxy (connection pooling)
Trigger
"Reduce failover time when DB primary fails"
Answer
RDS Proxy (cuts failover by up to 66%)
Trigger
"Enforce IAM authentication on DB connections from many Lambdas"
Answer
RDS Proxy with IAM auth required
Trigger
"Pool DB connections without changing app code"
Answer
RDS Proxy (drop-in, same wire protocol)
Don't confuse

RDS Proxy does NOT speed up queries. If the question asks about query latency, the answer is caching (ElastiCache) or indexes — not RDS Proxy.

Secrets Manager vs Parameter Store for DB credentials

Exam says "automatically rotate database credentials" → Secrets Manager.
Exam says "store DB password securely with lowest cost" → Parameter Store SecureString.

Part 06

Aurora — exam-specific facts

Everything that has actually appeared, with exam angle.

TopicWhat the exam tests
Compatibility Aurora MySQL is MySQL-compatible. Aurora PostgreSQL is Postgres-compatible. No Oracle, no SQL Server. If question mentions Oracle or SQL Server, Aurora is wrong.
Storage 6 copies of data across 3 AZs. Self-healing. Auto-grows in 10 GB increments up to 128 TB. Storage is shared between compute instances.
Replicas Up to 15 read replicas. Replication lag typically ~100ms. New replicas attach instantly (shared storage).
Endpoints Cluster (writer) endpoint, Reader endpoint, Custom endpoints, Instance endpoints. "Distribute reads automatically" → Reader endpoint. "Dedicate replicas to analytics" → Custom endpoint.
Failover Under 30 seconds (faster than standard RDS). Promoting a reader to writer doesn't require moving data.
Survival Can lose 2 of 6 storage copies and still write. Can lose 3 of 6 and still read. Survives one entire AZ failure with zero impact.
Performance claim 5× MySQL, 3× PostgreSQL. Memorise these for "improve performance without changing application code."
Cost ~20% more than equivalent RDS. If question is cost-only and features overlap, RDS wins; if features matter, Aurora.
Part 07

Aurora Global Database

THE answer for cross-region database DR. Memorise the numbers.

The Four Numbers

Up to 5 secondary regions · RPO < 1 second · RTO < 1 minute · Replication via storage layer (not app-level), so it doesn't burden the primary's compute.

Trigger phrases

Trigger
"Cross-region disaster recovery for a relational database, RPO of seconds, RTO under 1 minute"
Answer
Aurora Global Database
Trigger
"Read traffic from multiple regions with low latency"
Answer
Aurora Global Database (secondaries are readable)
Trigger
"Promote secondary region to primary in < 1 minute"
Answer
Aurora Global Database (managed failover)
Trigger
"Cross-region DB DR but Oracle / SQL Server"
Answer
NOT Aurora. Use cross-region snapshot copy or Read Replicas (where supported).
The Aurora Multi-AZ vs Global Trap

Aurora has its own Multi-AZ built in (shared storage = 3 AZs). That handles AZ failure. Aurora Global Database handles region failure. The exam will offer both — pick based on the qualifier.

Cross-Region Read Replica vs Aurora Global

Both replicate cross-region. Aurora Global is faster (storage-level, <1s RPO) and supports managed failover. Cross-Region Read Replica has higher lag and manual promotion. If both are options and the question stresses RPO/RTO → Aurora Global.

Part 10

RDS vs Aurora decision tree

Walk this tree in your head when the exam shows both as options.

1. Does the workload require Oracle or SQL Server?
YES → RDS (Aurora doesn't support them)
2. Is cost the dominant constraint, and the workload is well-defined MySQL/Postgres?
YES → RDS (cheaper for same instance class)
3. Is there a need for cross-region DR with RPO/RTO measured in seconds/minutes?
YES → Aurora Global Database
4. Are there many read replicas needed (>5) or large data volumes (>64 TB)?
YES → Aurora
5. Is the workload unpredictable / spiky?
YES → Aurora Serverless v2
6. Need fast failover (< 30s), backtrack, or cloning?
YES → Aurora
7. None of the above strongly applies?
→ RDS (default cheaper option)

Part 14

Top exam traps in this topic

The mistakes that turn 75% test-takers into 65% test-takers. Read each one twice.

Trap #1

Standard Multi-AZ standby is NOT readable. If asked "we have Multi-AZ, can we use the standby for reads?" → No. Use Read Replicas, or Multi-AZ DB Cluster.

Trap #2

Aurora Multi-AZ ≠ Aurora Global Database. Multi-AZ = within one region (built-in). Global = across regions (extra feature, <1s RPO).

Trap #3

Read Replicas use async replication. They can have lag. If the question requires "always reads see latest data," replica is wrong — primary, or strongly-consistent DAX/Redis with write-through.

Trap #4

You can't enable encryption on an existing unencrypted RDS in-place. Snapshot → copy with encryption → restore. Same for replicas — they inherit primary's encryption state.

Trap #5

Aurora doesn't support Oracle or SQL Server. If a question requires those engines, Aurora is wrong no matter how appealing the features.

Trap #6

RDS Proxy fixes connections, NOT query speed. "Lambda DB queries are slow" → caching. "Lambda can't open more connections" → RDS Proxy.

Trap #7

Parameter Store does NOT auto-rotate. If the question says "automatic rotation," it must be Secrets Manager. Parameter Store is for cheap config storage.

Trap #8

DAX is for DynamoDB, not for SQL databases. If a question shows DynamoDB + "microsecond reads" → DAX. If it shows RDS/Aurora + caching → ElastiCache (Redis usually).

Trap #9

ElastiCache is in-memory; it's NOT a primary store. Question scenarios about durable storage or system-of-record always rule out ElastiCache.

Trap #10

Aurora Serverless v2 doesn't scale to zero. If the requirement says "no charges when idle," v2 doesn't fit — you'd need to manually pause/delete the cluster.

Trap #11

RDS Read Replicas have a different endpoint than the primary. The app must be coded to use the replica endpoint for reads — it's not transparent. Aurora's Reader Endpoint is automatic.

Trap #12

Manual snapshots survive DB deletion; automated backups may not. For long-term retention, choose manual snapshots (or set automated retention to keep on delete).

Part 15

24 SAA-style practice questions

Real exam style. Read the question, read all four options, decide, then expand the answer. Each one teaches a different rule.

Question 01
Medium · Resilience

A company runs a critical MySQL database on Amazon RDS. The Solutions Architect must ensure that if the primary database instance fails, another instance takes over automatically with no data loss. The application connects through a single endpoint. Which configuration meets these requirements with the LEAST operational overhead?

  1. Enable RDS automated backups with a 7-day retention period, and create a Lambda function that restores from the most recent snapshot in case of failure.
  2. Configure the RDS database as Multi-AZ, which creates a synchronous standby replica in another AZ.
  3. Create five Read Replicas in different AZs and write a custom script that promotes one to primary on failure.
  4. Migrate to Aurora and enable Aurora Global Database with one secondary region.
Reveal answer & explanation

B — Multi-AZ deployment

Multi-AZ uses synchronous replication so failover has zero data loss. AWS handles the failover automatically (~60–120s), and the RDS endpoint transparently re-points to the new primary — no application code changes. Automated backups (A) don't provide automatic failover. Read Replicas (C) use async replication so data loss is possible, and they don't auto-failover. Aurora Global Database (D) is cross-region DR — overkill and not required by this scenario.

For "automatic failover with zero data loss within a region" → always Multi-AZ.

Question 02
Medium · Performance

An e-commerce application uses an Amazon RDS PostgreSQL database with Multi-AZ enabled. As traffic has grown, reporting and analytics queries from the data team are slowing down the customer-facing app. The CTO wants to keep the customer experience fast while letting analysts run heavy queries. What is the BEST solution?

  1. Send the analytics queries to the Multi-AZ standby instance.
  2. Add Read Replicas and direct analytics queries to them.
  3. Increase the instance size of the primary database.
  4. Enable storage autoscaling on the primary database.
Reveal answer & explanation

B — Read Replicas for analytics

Read Replicas are designed exactly for this — offload heavy or slow reads from the primary. Analytics queries hitting replicas don't impact the customer-facing primary. Option A is the classic trap: standard Multi-AZ standby cannot serve reads. Option C is a brute-force solution that's expensive and doesn't fundamentally separate workloads. Option D addresses disk space, not query competition.

A confuses Multi-AZ with Read Replica functionality. Standard RDS Multi-AZ standby is for failover only, never for reads.

Question 03
Hard · Multi-Region DR

A financial services company requires disaster recovery for its primary relational database. Requirements: data loss must be under 1 second, the secondary region must be readable for analytics, and failover to the secondary region must complete in under 1 minute. Which solution BEST meets these requirements?

  1. Amazon RDS for PostgreSQL with cross-region Read Replicas.
  2. Amazon Aurora PostgreSQL with Aurora Global Database.
  3. Amazon RDS for PostgreSQL Multi-AZ with deletion protection enabled.
  4. Amazon DynamoDB Global Tables.
Reveal answer & explanation

B — Aurora Global Database

Aurora Global Database is purpose-built for cross-region DR of relational databases. It delivers RPO < 1 second, RTO under 1 minute on failover, and secondary regions are readable. RDS cross-region Read Replicas (A) have higher lag and slower failover with manual promotion. RDS Multi-AZ (C) is single-region only. DynamoDB Global Tables (D) is NoSQL — not relational.

"Relational + cross-region DR + RPO seconds + RTO < 1min" → Aurora Global Database. Memorise this triplet.

Question 04
Medium · Lambda + RDS

A serverless API uses AWS Lambda functions that connect to an Amazon RDS MySQL database. During traffic spikes, the application starts returning errors with the message "Too many connections". The team wants a solution that scales connections without changing application code. What should they implement?

  1. Increase the max_connections parameter of the database to the maximum.
  2. Enable Multi-AZ to distribute connections between primary and standby.
  3. Place Amazon RDS Proxy between the Lambda functions and the database.
  4. Migrate from Lambda to ECS Fargate to maintain persistent connections.
Reveal answer & explanation

C — RDS Proxy

RDS Proxy is a managed connection pool that sits between clients (especially Lambda) and the database. It maintains a small pool of long-lived connections to the DB and multiplexes Lambda traffic over them — eliminating "too many connections" errors without code changes. A masks the problem temporarily but doesn't scale. B is wrong: Multi-AZ standby doesn't accept connections. D works but requires significant refactoring.

Lambda + RDS + "too many connections" = RDS Proxy. Reflex answer.

Question 05
Hard · Caching + connections

A serverless app (Lambda) reads rarely-changing user-profile data from RDS MySQL thousands of times per second and needs microsecond reads. Under load it also exhausts the database's max connections — each Lambda invocation opens its own connection — causing "too many connections" errors. Which TWO actions together BEST fix both the latency and the connection exhaustion? (Choose two.)

  1. Put an ElastiCache cluster in front of RDS (lazy loading + TTL) to serve the hot reads in microseconds.
  2. Add RDS Proxy between Lambda and RDS to pool and reuse database connections.
  3. Increase the RDS instance class to raise the connection limit.
  4. Add RDS Read Replicas and route reads to them.
  5. Migrate to Aurora Serverless v2.
Reveal answer & explanation

A and B — ElastiCache + RDS Proxy

Two problems, two targeted fixes. A ElastiCache serves the hot, rarely-changing reads from RAM in microseconds and offloads the DB (read replicas give milliseconds, not microseconds — so D fails the latency goal). B RDS Proxy pools and reuses connections, absorbing Lambda's connection storms — the standard fix for "Lambda + RDS too many connections." C a bigger instance is a costly band-aid that delays the same problem. E Serverless v2 changes scaling, not the connection-storm or the µs-latency requirement.

Microsecond reads → ElastiCache. Serverless/Lambda exhausting DB connections → RDS Proxy (connection pooling).

Question 06
Medium · Cache choice

A real-time gaming platform needs to display a global leaderboard ranking the top 100 players by score, updated within milliseconds of a score change. Which AWS service is MOST appropriate?

  1. Amazon ElastiCache for Memcached storing scores as key-value pairs.
  2. Amazon ElastiCache for Redis using a sorted set.
  3. Amazon DynamoDB with a Global Secondary Index on the score attribute.
  4. Amazon Aurora MySQL with a covering index on score.
Reveal answer & explanation

B — ElastiCache for Redis (sorted set)

Redis sorted sets are specifically designed for leaderboards: O(log N) insert and O(log N + M) retrieval of the top M items, always kept sorted by score. Memcached (A) only supports simple key-value pairs — no sorting. DynamoDB (C) and Aurora (D) would work but are slower and more complex; Redis sorted sets are the canonical leaderboard pattern.

"Leaderboard" → Redis sorted set. Period. This is one of Redis's signature features.

Question 07
Hard · Spiky workload

A startup is launching a SaaS product. They have no historical data to predict database load — traffic could be near zero for weeks, then spike 100× during product demos. They want to avoid over-provisioning capacity and minimise cost during quiet periods, while handling spikes without manual intervention. Which database is BEST?

  1. Amazon Aurora MySQL on a db.r6g.4xlarge instance with Multi-AZ.
  2. Amazon RDS for PostgreSQL with storage autoscaling.
  3. Amazon Aurora Serverless v2 with min 0.5 ACU and max 64 ACU.
  4. Amazon DynamoDB on-demand capacity mode.
Reveal answer & explanation

C — Aurora Serverless v2

Aurora Serverless v2 automatically scales database capacity in fractions of a second between your min and max ACUs based on load. A fixed-size instance (A) wastes money during quiet periods. RDS storage autoscaling (B) handles disk space, not capacity. DynamoDB on-demand (D) is the right idea but it's NoSQL — the scenario implies relational (SaaS application typically). Aurora Serverless gives the elasticity benefits with a relational engine.

"Unpredictable / spiky / no capacity planning" + relational DB → Aurora Serverless v2.

Question 08
Medium · Encryption

A compliance audit requires all production databases to be encrypted at rest. The team realises that one of their production RDS MySQL databases was created two years ago without encryption. What is the correct procedure to encrypt this existing database?

  1. Modify the RDS instance and enable encryption from the AWS Management Console.
  2. Take a snapshot, copy the snapshot with encryption enabled, then restore the encrypted snapshot to a new RDS instance.
  3. Reboot the RDS instance with the "encrypt on reboot" option.
  4. Attach an AWS KMS key to the existing RDS instance through the IAM console.
Reveal answer & explanation

B — Snapshot → copy with encryption → restore

RDS encryption at rest cannot be enabled in-place. The standard procedure is: take a snapshot of the unencrypted database, copy that snapshot specifying a KMS key (this is where encryption is applied), then restore the encrypted snapshot to a new RDS instance. Update your application to point to the new endpoint. Options A, C, D do not exist.

Many candidates expect a "modify and check encryption" option. It does not exist. Encryption must be set at instance creation, period.

Question 09
Medium · Secrets

A security policy requires the database password for an RDS MySQL instance to be automatically rotated every 30 days. The application running on EC2 should fetch the current password at runtime. Which approach is BEST?

  1. Store the password in SSM Parameter Store as a SecureString. Manually update it every 30 days.
  2. Store the password in AWS Secrets Manager and enable automatic rotation using the built-in RDS rotation template.
  3. Hardcode the password in an environment variable and rotate the EC2 instance every 30 days.
  4. Store the password in an S3 bucket with versioning and KMS encryption.
Reveal answer & explanation

B — Secrets Manager with built-in RDS rotation

AWS Secrets Manager has built-in rotation templates for RDS MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server. Enable rotation, set 30 days, done — Secrets Manager updates the password in the DB and in the secret store, atomically. Parameter Store (A) does not natively rotate. C is anti-pattern. D doesn't integrate with RDS for rotation.

"Auto-rotate database password" = Secrets Manager. Always.

Question 10
Hard · Read consistency

An application uses RDS PostgreSQL with two Read Replicas. After users update their profile, they're frequently shown the stale (old) version of the profile when they immediately refresh. Which solution would BEST address this without massively rearchitecting the app?

  1. Route reads to the Read Replicas immediately after writes for a smoother user experience.
  2. Implement read-your-writes consistency by routing reads to the primary for a short period after a user write.
  3. Increase the number of Read Replicas from 2 to 5 to improve replication speed.
  4. Disable Multi-AZ on the primary database to free up resources.
Reveal answer & explanation

B — Read-your-writes routing to the primary

RDS Read Replicas use asynchronous replication, so a replica can be a few milliseconds behind. The standard pattern is to route a user's reads to the primary for a brief period after they perform a write — this guarantees they see their own changes immediately. A makes it worse. C: more replicas don't reduce replication lag (they all replicate independently). D is unrelated and removes HA.

Replica lag is fundamental to asynchronous replication. Adding more replicas doesn't fix it.

Question 11
Medium · Migration to Aurora

A team runs a self-managed MySQL database on EC2 with 8 TB of data. They want to migrate to a managed service with better performance, automatic 6-way replication across AZs, and the ability to scale to 15 read replicas as they grow. They don't want to rewrite any of the application's SQL. Which option is BEST?

  1. Migrate to Amazon RDS for MySQL with Multi-AZ enabled.
  2. Migrate to Amazon Aurora MySQL-compatible.
  3. Migrate to Amazon DynamoDB.
  4. Migrate to Amazon Redshift.
Reveal answer & explanation

B — Aurora MySQL-compatible

Aurora delivers all the asked-for features: MySQL wire compatibility (no SQL changes), 6 copies across 3 AZs automatically, up to 15 read replicas, ~5× MySQL performance. RDS MySQL (A) tops out at 5 replicas and lacks the 6-way replication. DynamoDB (C) is NoSQL — rewriting required. Redshift (D) is a data warehouse, not an OLTP database.

"MySQL compatible + 6 copies + 15 replicas" is Aurora's signature combo. Recognise it instantly.

Question 12
Hard · Backtrack vs snapshot

A developer accidentally ran UPDATE accounts SET balance = 0 against an Aurora MySQL production database 10 minutes ago, zeroing every customer balance. The DBA needs to recover the data with minimum downtime. Which option provides the FASTEST recovery?

  1. Restore the most recent automated snapshot to a new Aurora cluster, then swap the application's connection string.
  2. Use Aurora Backtrack to rewind the cluster to a point 11 minutes ago.
  3. Restore using Point-In-Time Recovery to a new cluster, then re-import data.
  4. Use Aurora Cloning to copy the cluster as it was 11 minutes ago.
Reveal answer & explanation

B — Aurora Backtrack

Aurora Backtrack (MySQL-compatible only) rewinds the existing cluster to a point in time — typically within minutes — without spinning up a new cluster, without restoring data from snapshots. The app keeps using the same endpoint. A, C, D all involve creating a new cluster and modifying the connection string, which is much slower. Backtrack window is up to 72 hours.

"Quickly undo a bad query, Aurora MySQL, minimum downtime" = Aurora Backtrack.

Question 13
Medium · Cloning

An engineering team wants a copy of their 4 TB production Aurora database to test a risky schema migration. They need the copy to be ready quickly, cost as little as possible, and not affect production performance. What should they do?

  1. Restore from the latest automated snapshot into a new cluster.
  2. Use Aurora Database Cloning to create a copy of the production cluster.
  3. Enable cross-region replication and run the test in another region.
  4. Manually export the data using mysqldump and import it into a new cluster.
Reveal answer & explanation

B — Aurora Database Cloning

Aurora Cloning uses copy-on-write: the clone is created instantly with no data copy, sharing storage with the source. You only pay for blocks that change after cloning. Restoring from snapshot (A) takes hours for 4 TB and duplicates storage costs. C is overkill and complex. D would take a long time and stress production during the export.

"Instant copy of production data for testing, minimal cost" = Aurora Cloning.

Question 14
Hard · Memcached vs Redis

An application stores user session data in an ElastiCache cluster. The business requires that sessions must persist across cache node restarts and the cache must remain available even if one AZ goes down. Which engine and configuration is BEST?

  1. Memcached cluster with multiple nodes for horizontal scaling.
  2. Redis cluster with multi-AZ replication and persistence enabled.
  3. Memcached single-node deployment with automatic backups to S3.
  4. Redis without replication for lowest cost.
Reveal answer & explanation

B — Redis with Multi-AZ + persistence

The requirements (survive restarts + survive an AZ outage) eliminate Memcached, which has neither persistence nor multi-AZ replication. Redis offers both — persistent snapshots ensure data survives restarts, and Multi-AZ replication promotes a replica if the primary AZ fails. D fails the AZ requirement.

If the question mentions any of: persistence, multi-AZ, replication, failover → Redis, not Memcached.

Question 15
Medium · Memcached choice

A team needs simple in-memory caching with the highest possible throughput per cache node. They are okay with losing all cached data if a node restarts and don't need replication. Which option is most appropriate?

  1. Amazon ElastiCache for Redis with cluster mode enabled.
  2. Amazon ElastiCache for Memcached.
  3. Amazon DynamoDB DAX.
  4. Amazon Aurora Serverless v2.
Reveal answer & explanation

B — ElastiCache for Memcached

The scenario explicitly states simple caching, highest throughput, no need for replication or persistence — all the Memcached strengths. Memcached is also multi-threaded per node, giving better single-node throughput than Redis. Redis (A) would also work but is overkill and the question's "highest throughput per node" tips toward Memcached.

"Simple cache, no HA, multi-threaded throughput" = Memcached. The rare time Memcached is the answer.

Question 16
Medium · Cost optimisation

A company runs an RDS PostgreSQL Multi-AZ instance 24/7 with steady, predictable workload over the next 3 years. Their AWS bill is dominated by RDS costs. Which approach reduces RDS cost MOST while preserving the existing setup?

  1. Switch to Aurora Serverless v2 with auto-scaling.
  2. Purchase a 3-year Reserved DB Instance with All Upfront payment.
  3. Reduce backup retention to 0 days.
  4. Disable Multi-AZ to use half the resources.
Reveal answer & explanation

B — Reserved DB Instance, 3-year, All Upfront

For predictable, steady, long-running workloads, Reserved Instances offer up to ~69% discount compared to on-demand. 3-year All Upfront has the maximum discount. The scenario's "steady, predictable, 24/7, 3 years" exactly matches RI's sweet spot. A may help variable loads but here it's not optimal. C and D weaken backups and HA — never the right answer for cost.

Steady + predictable + multi-year → Reserved Instance. Always.

Question 17
Hard · Cross-region DR

A regulated industry requires that database backups be retained for 7 years and stored in a different region from the production database. Which solution meets this with the LEAST operational overhead?

  1. Set RDS automated backup retention to 35 days.
  2. Use EventBridge to trigger a Lambda function that takes a manual snapshot and copies it to a different region.
  3. Manually take a snapshot every month and copy it to another region.
  4. Use AWS Backup to manage cross-region backup policies for RDS with a 7-year retention rule.
Reveal answer & explanation

D — AWS Backup with cross-region policy

AWS Backup is a centralised service that manages backups (including for RDS) with cross-region copy and long-term retention rules, all policy-driven. Lowest operational overhead. A doesn't allow 7-year retention (35-day max). B works but requires writing/maintaining custom Lambda. C is manual and error-prone.

"Long-term compliance backups, cross-region, low ops overhead" → AWS Backup. RDS automated backups max out at 35 days.

Question 18
Hard · DAX trap

A team is building a high-traffic mobile game that uses DynamoDB to store player state. They want microsecond read latency for the most-accessed items, with minimal application changes. Which service provides this?

  1. Amazon ElastiCache for Redis in front of DynamoDB.
  2. Amazon DynamoDB Accelerator (DAX).
  3. Amazon ElastiCache for Memcached configured with the DynamoDB Streams trigger.
  4. Amazon Aurora with DynamoDB compatibility mode.
Reveal answer & explanation

B — DAX

DAX is the purpose-built, in-memory cache for DynamoDB, providing microsecond reads with the same API as DynamoDB — so minimal app changes (just point to the DAX endpoint). ElastiCache (A and C) can cache DynamoDB but requires significant application code changes to manage the cache. D doesn't exist.

ElastiCache is for SQL DBs (RDS/Aurora) or generic caching. For DynamoDB specifically, DAX is the AWS-recommended answer.

Question 19
Hard · Multi-AZ DB Cluster

A team needs an RDS MySQL deployment with high availability AND the ability to send read traffic to the standby instances for cost-efficient read scaling — without setting up separate Read Replicas. Which deployment option BEST fits?

  1. RDS Multi-AZ (single-standby) with the read endpoint pointing to the standby.
  2. RDS Multi-AZ DB Cluster, which provides one writer and two readable standbys across three AZs.
  3. Aurora Global Database with regional Read Replicas.
  4. RDS with five Read Replicas spread across AZs.
Reveal answer & explanation

B — Multi-AZ DB Cluster

Multi-AZ DB Cluster is a newer RDS deployment option (MySQL and PostgreSQL only) that gives one writer + two readable standbys across three AZs with semi-synchronous replication. It combines HA and read scaling without managing separate Read Replicas. A is wrong because standard Multi-AZ standbys are NOT readable. C is overkill cross-region. D works but doesn't combine with HA the way the scenario asks.

"Multi-AZ + standbys serve reads" = Multi-AZ DB Cluster (not standard Multi-AZ).

Question 20
Medium · Caching strategy

A reporting application reads slow-changing reference data from RDS thousands of times an hour. The team wants to cache it in ElastiCache. The data is updated by an admin tool once a week. They want the cache to ALWAYS reflect the latest data immediately after admin changes. Which caching strategy is BEST?

  1. Lazy Loading with a 60-second TTL.
  2. Write-Through, so the admin tool updates the cache at the same time as the database.
  3. Lazy Loading with no TTL.
  4. Disable the cache and rely on the database directly.
Reveal answer & explanation

B — Write-Through

Write-Through keeps cache and DB in sync on every write. When the admin tool updates the database, it also updates the cache — readers see fresh data immediately. Lazy Loading with TTL (A) would show stale data for up to 60 seconds after an update. Lazy Loading without TTL (C) is even worse — stale forever until invalidated. D defeats the purpose.

"Cache must always be fresh after updates" → Write-Through.

Question 21
Hard · Engine constraint

An enterprise application requires Oracle database with Oracle-specific features (PL/SQL packages, Oracle Spatial). They want to migrate from an on-premises Oracle deployment to AWS with as little code change as possible, while gaining managed backups, automatic Multi-AZ failover, and read replicas. Which managed service is appropriate?

  1. Amazon Aurora PostgreSQL with the Oracle compatibility plugin.
  2. Amazon RDS for Oracle.
  3. Amazon Redshift configured with Oracle dialect.
  4. Amazon DynamoDB with Oracle SDK.
Reveal answer & explanation

B — RDS for Oracle

Aurora does not support Oracle — that's a hard limit. When the requirement is Oracle (or SQL Server), the answer is RDS, never Aurora. RDS for Oracle provides managed backups, Multi-AZ failover, and Read Replicas (since Oracle EE on RDS Custom or BYOL).

"Aurora PostgreSQL with Oracle plugin" is a fabricated distractor. Aurora supports MySQL- and Postgres-compatible engines only.

Question 22
Hard · Combined architecture

A SaaS company's architecture: ALB → EC2 Auto Scaling Group → Aurora MySQL Multi-AZ cluster. Database CPU is regularly 90%+ during business hours. Profiling shows 95% of queries are SELECTs on a small set of frequently-accessed reference tables that rarely change. The team wants to reduce DB load without redesigning the schema or sharding. Which TWO actions, together, would deliver the BEST result? (Choose two.)

  1. Add Aurora Read Replicas and route SELECT queries to the Reader Endpoint.
  2. Place an ElastiCache for Redis cluster in front of Aurora using a lazy-loading pattern with TTL.
  3. Migrate the database to DynamoDB.
  4. Reduce the Multi-AZ standby's storage size to free up IOPS.
  5. Set up Aurora Global Database to a secondary region.
Reveal answer & explanation

A and B — Read Replicas + Redis cache

A: Adding Aurora Read Replicas spreads the SELECT load across multiple readers, dropping primary CPU. Routing via the Reader Endpoint requires no app changes. B: An ElastiCache Redis cache absorbs the most-requested reads (95% of queries hit a small set of reference tables — a perfect cache candidate), further offloading the database. Lazy loading with TTL keeps it simple. Both together create a layered defense. C is a major rearchitecture. D is nonsensical (Multi-AZ standby isn't even accessible). E is cross-region DR, not a load reduction.

Tiered architecture: cache absorbs hot reads, replicas absorb the rest, primary handles writes + cold reads. This is the canonical pattern.

Question 23
Hard · Upgrades

A production RDS for PostgreSQL database must be upgraded off a major version that is reaching end-of-life. The business requires: minimal downtime, the ability to fully test the new version with production-like data before cutting over, and a fast rollback if anything goes wrong. Building custom replication is off the table. Which approach BEST meets all of these?

  1. Modify the DB engine version in place during a maintenance window.
  2. Create a read replica, upgrade it, and promote it to be the new primary.
  3. Use an RDS Blue/Green Deployment — a synced "green" copy on the new version that you test, then switch over in about a minute with built-in safety checks.
  4. Take a snapshot, restore it to a new instance on the new version, and repoint the application.
Reveal answer & explanation

C — RDS Blue/Green Deployment

Blue/Green creates a fully synced green environment already on the target version, so you can test it with real replicated data, then switch over in ~1 minute with built-in safety checks and easy rollback — exactly "minimal downtime + test-first + fast rollback, no custom replication." A in-place modify takes the DB down and is hard to roll back. B a promoted read replica can run the new version but loses sync at promotion and rollback is manual. D snapshot/restore loses every write made after the snapshot (downtime + data gap).

Major-version upgrade · minimal downtime · test-then-switch · easy rollback → RDS Blue/Green Deployments.

Question 24
Hard · Cost

An Aurora MySQL cluster runs an I/O-intensive analytics workload. The bill shows that I/O charges are about 40% of the total Aurora cost and swing unpredictably month to month, making budgeting hard. The team wants more predictable, ideally lower cost without changing the workload or schema. What should they do?

  1. Switch the cluster to Aurora Serverless v2.
  2. Enable the Aurora I/O-Optimized cluster configuration.
  3. Add more read replicas to spread the I/O.
  4. Move to Aurora Standard and buy Reserved Instances for the writer.
Reveal answer & explanation

B — Aurora I/O-Optimized

Aurora I/O-Optimized removes the separate per-I/O charge and bundles it into compute/storage pricing — giving predictable cost and, when I/O is a large share of the bill (AWS guidance: ~25%+; here it's 40%), usually a lower total. A Serverless v2 changes the compute scaling model, not how I/O is billed. C read replicas add cost and don't change per-I/O billing. D Reserved Instances cut compute cost but leave the volatile per-I/O charges in place.

Aurora I/O is a big / unpredictable share of the bill (≥~25%) → switch to Aurora I/O-Optimized (no per-I/O charges).
