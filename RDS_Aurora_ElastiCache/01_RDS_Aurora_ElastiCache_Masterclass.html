Chapter 01

Before AWS, let's ask: what is a database?

Forget the jargon. Forget RDS and Aurora for a moment. Let's start at the very beginning.

Imagine you run a small bookshop. On day one, you can keep track of your books on a notepad. Title, author, price, how many copies in stock. Easy.

Day 100, you have 500 books, three staff members, an online ordering system, and customers asking "Do you have The Hobbit in stock?" twenty times a day. The notepad is useless. You need a structured place to store information, with the ability to quickly look things up, update them safely (so two people don't sell the last copy at the same time), and not lose anything if your laptop dies.

That structured place is a database. That's it. A database is just an organised system to store, find, update, and protect information.

Mental Model

A database is like a very strict, very fast librarian with a perfect memory. You ask it a question ("Give me all books by Tolkien published before 1960") and it returns the answer in milliseconds. It never loses a book, never lets two people check out the same copy at the same time, and keeps a log of every action.

SQL vs NoSQL — the 30-second version

There are two main families of databases. You'll hear these words constantly:

Relational (SQL) databases

Data is stored in tables, like spreadsheets. Each row is a record (one book, one customer, one order). Columns are fields (title, price, date). Tables can link to each other — an "orders" table references "customers" by ID. You query them with a language called SQL (Structured Query Language), which reads almost like English: SELECT * FROM books WHERE author = 'Tolkien'.

Examples: MySQL, PostgreSQL, Oracle, Microsoft SQL Server, MariaDB. These are what powers your bank, your hospital records, most e-commerce sites, and most accounting systems on Earth.

NoSQL databases

Data is stored more flexibly — sometimes as documents (like JSON), key-value pairs, or graphs. No rigid table structure. Built for massive scale and flexibility, but you give up some of the strict guarantees of SQL.

Examples: DynamoDB, MongoDB, Cassandra, Redis. (Don't worry about NoSQL too much in this guide — that's DynamoDB's territory, a separate topic.)

For this guide

RDS and Aurora are relational (SQL) databases. ElastiCache is technically a NoSQL key-value store used for caching, not as a primary database. We'll cover all three because they almost always work together in real applications.

Chapter 02

The pain that RDS solves

Why anyone pays AWS for databases instead of running their own.

Before AWS, running a database meant doing all of this yourself:

  • Buy a server, install Linux, install MySQL or PostgreSQL.
  • Set up backups. Write scripts to dump the database every night and copy the files somewhere safe.
  • Patch the OS every few weeks for security updates. Restart, hope nothing breaks.
  • Upgrade MySQL when a new version is released. Test, migrate, hope nothing breaks.
  • Plan for failure. If the server dies, what happens? Set up a second server. Configure replication. Test failover. Pray.
  • Monitor everything. Disk usage, CPU, slow queries, connection counts. Get woken up at 3am when things go wrong.
  • Scale it. If reads get too heavy, add read replicas — manually. If writes get too heavy... cry.

This is the world a database administrator (DBA) lived in. Brilliant, expensive, often grumpy, and absolutely necessary.

Amazon RDS is the deal where you give AWS your database, and they handle every painful chore on that list — except writing the queries. You're not buying a database; you're renting a fully-managed service.

What "managed" actually means

When AWS says RDS is "managed," they mean specifically:

What AWS doesWhat you still do
Provisions the server (hardware) Choose the engine (MySQL, Postgres…)
Installs & updates the database software Design your tables and write your SQL
Takes automatic daily backups Choose your backup retention (1–35 days)
Patches the OS and DB engine Pick a maintenance window
Replicates to standby (Multi-AZ) Decide if you want Multi-AZ on or off
Monitors hardware health Monitor your own queries & metrics
Automatic failover if the primary dies Connect your app using the RDS endpoint
Industry Reality

Companies like Airbnb, Lyft, Coinbase, Robinhood, and tens of thousands of startups run their production databases on RDS or Aurora. The trade-off is well understood: you pay a managed-service premium (often 20–40% more than self-managed) to never get paged at 3am because a disk filled up. For most companies, that's a no-brainer.

Don't forget

RDS is "managed," not "magic." You still have to design good schemas, write efficient queries, choose the right instance size, and turn on the right features (Multi-AZ, read replicas, encryption). The exam loves to ask: "the team forgot to enable [X] — what should they do?"

Chapter 03

RDS — the basics

What it actually is, what engines it runs, and how you talk to it.

RDS stands for Relational Database Service. When you create an RDS database, AWS spins up an EC2 instance behind the scenes (you never see it), installs the database engine you chose, attaches storage (EBS), and gives you back an endpoint — a DNS name like mydb.abc123.us-east-1.rds.amazonaws.com.

Your application connects to that endpoint exactly like it would connect to any database. As far as your app knows, it's just MySQL or PostgreSQL — because it is.

The six engines RDS supports

EngineWhat it isIndustry use
MySQL The world's most popular open-source DB Most web apps, WordPress, e-commerce
PostgreSQL Open-source, more advanced features than MySQL Fintech, geospatial apps, modern startups
MariaDB A fork of MySQL, fully compatible Used where MySQL licensing concerns exist
Oracle Commercial, very expensive, very powerful Banks, telcos, government, legacy enterprise
SQL Server Microsoft's commercial database .NET shops, Microsoft-heavy enterprises
Aurora AWS's own engine (we'll spend chapters on this) AWS-native modern apps wanting cloud-scale
Exam Tip

The exam rarely tests "which engine" choice. It tests what features RDS offers regardless of engine — Multi-AZ, Read Replicas, encryption, backups. When the question says "MySQL on RDS," the engine name is just flavour; the real question is about the feature.

How you size an RDS instance

You pick three things when you create an RDS database:

  1. Instance class — like an EC2 size. db.t3.medium for dev, db.r6g.4xlarge for serious production. This sets CPU and RAM.
  2. Storage type and size — gp3 (general purpose SSD), io1/io2 (high-performance SSD with provisioned IOPS), or magnetic (legacy, don't use).
  3. Deployment — Single-AZ (cheap, no failover), Multi-AZ (one standby), or Multi-AZ DB Cluster (two readable standbys — we'll get there).

RDS uses an EC2 instance under the hood — meaning RDS inherits EC2's quirks: it has CPU, RAM, network bandwidth, and it gets billed per hour. You can resize it, but that requires a brief downtime (or use Multi-AZ to fail over to the resized instance with near-zero downtime).

Connect-the-dots

Because RDS runs on EC2 + EBS, the pricing models you already know apply: Reserved Instances work for RDS too. If you know you'll run your database for 1 or 3 years, Reserved RDS = up to 69% discount. The exam loves this combo: "predictable database workload, lowest cost" → Reserved Instances.

Chapter 04

RDS Storage & Backups

How AWS handles the most precious thing — your data on disk.

Storage auto-scaling

You start with, say, 100 GB of storage. As your data grows, RDS can automatically grow the storage when it gets close to full — without any downtime. You just set a maximum (say, 1 TB) and a threshold (say, "grow when 90% full"). This is called Storage Autoscaling.

Storage can grow but it cannot shrink. If you over-provisioned, the only way back is to dump and restore the data to a smaller database.

Automated Backups

RDS automatically backs up your database every day, plus captures transaction logs continuously. This gives you the magical ability called Point-In-Time Recovery (PITR) — you can restore your database to any moment within your retention window, accurate to the second.

  • Retention: 0 to 35 days. Set to 0 = automated backups disabled (don't do this).
  • Backups happen during a daily backup window you specify.
  • Backups are stored in S3 (you never see them — they're managed).
  • When you delete the RDS instance, you can choose to keep or discard automated backups.
Exam Tip

"Restore to 4:32pm yesterday before the bad migration" → Point-in-Time Recovery. This is RDS Automated Backups, not a manual snapshot.

Manual Snapshots

You can also manually trigger a snapshot at any time. Manual snapshots:

  • Are retained forever until you delete them (unlike automated, which expire).
  • Can be copied to another region for disaster recovery.
  • Can be shared with other AWS accounts.
  • Are great for "snapshot before a risky change" workflows.
Industry Pattern

Almost every production team runs this pattern: automated backups with 7–14 day retention, plus a Lambda function that takes a manual snapshot weekly and copies it to a different region for DR. The cost is tiny, the protection is enormous.

Chapter 05

Multi-AZ — High Availability

The single most important RDS feature on the exam. Memorise this chapter cold.

An Availability Zone (AZ) is a physically separate data centre within an AWS region. Think of us-east-1 as a region in Virginia, with us-east-1a, us-east-1b, us-east-1c being three separate buildings, kilometres apart, on different power grids, with different internet links.

If a hurricane takes out one data centre, the others keep running. Multi-AZ deployment uses this to give your database high availability.

How Multi-AZ works

When you enable Multi-AZ on an RDS database, AWS does this:

  1. Creates a second, identical database instance in a different AZ.
  2. Sets up synchronous replication from the primary to this standby. Every write that hits the primary is also written to the standby at the same time — they're always in lockstep.
  3. If the primary fails (hardware, network, AZ outage, even routine maintenance), AWS automatically promotes the standby to be the new primary. This takes about 60 to 120 seconds.
  4. Your application's connection string doesn't change — the RDS endpoint just starts pointing to the new primary.
RDS Multi-AZ: one primary handles all traffic; the standby in another AZ stays in lockstep, ready to take over if the primary fails.
The Three Things to Memorise

1. Replication is synchronous — no data loss on failover.
2. The standby is NOT readable — it just sits there until needed.
3. Multi-AZ is about HA inside one region. It is NOT a disaster recovery solution for an entire region going down.

What Multi-AZ buys you (and doesn't)

Multi-AZ protects you from:

  • Hardware failure of the primary instance.
  • One AZ going dark (power failure, network issue).
  • OS patching or DB engine maintenance — AWS patches the standby first, fails over, then patches the old primary.

Multi-AZ does not protect you from:

  • A whole region failing — use cross-region Read Replicas or Aurora Global Database for that.
  • A developer accidentally running DELETE FROM users — both DBs replicate the disaster instantly. Use backups for this.
  • Performance problems — Multi-AZ doesn't make reads faster, the standby is unused capacity until failover.
Classic Exam Trap

Question: "Our database is hitting read capacity limits. We have Multi-AZ enabled — can we send read queries to the standby?"
Answer: NO. Standard RDS Multi-AZ standby is purely for failover. For read scaling, you need Read Replicas (next chapter) or the newer Multi-AZ DB Cluster (two chapters away).

Industry Reality

Multi-AZ is the default setting for any database that matters. Production checklist at most companies: Multi-AZ ON, automated backups 7+ days, encryption at rest enabled, deletion protection ON. Skipping Multi-AZ on a production DB is the kind of thing that ends careers.

Chapter 06

Read Replicas — Scaling reads

When your database can't keep up with all the people reading from it.

Most databases face the same imbalance: reads vastly outnumber writes. A typical e-commerce site might have one purchase (write) for every hundred page views (reads). All those reads hammer the database.

If you only have one database instance, all those reads — every product page, every user profile load, every search — compete with each other and with the writes. CPU climbs. Queries slow down. Customers wait.

The fix: Read Replicas. You create copies of your primary database that are asynchronously updated. Your application sends writes to the primary and spreads reads across the replicas.

Read Replicas: copies of your primary that handle read traffic. Replication is asynchronous — they can be slightly behind.

Key facts about RDS Read Replicas

  • Up to 5 read replicas per primary on standard RDS (Aurora: 15).
  • Replication is asynchronous — meaning replicas can be milliseconds (or rarely, seconds) behind the primary.
  • Replicas can be in the same AZ, different AZ, or different region.
  • Each replica has its own endpoint — your app must be explicitly coded to send reads there.
  • You can promote a replica to be a standalone database if needed (one-way action; it stops being a replica).
  • Cross-region replicas are used for disaster recovery — you can promote them if a region goes down.

Replica Lag Trap

Asynchronous replication means replicas can be a tiny bit behind. If a user updates their profile and then immediately reloads, the read might hit a replica that hasn't caught up — they see their old data for half a second.

Solutions used in practice: route reads to the primary right after a write (called read-your-writes consistency), or use Aurora's faster replication, or use a cache like ElastiCache to absorb the read.

Industry Pattern

A typical mid-size SaaS architecture: 1 primary RDS PostgreSQL with Multi-AZ ON, plus 2–3 read replicas. Analytics queries and reporting hit the replicas (so heavy queries don't slow the main app), the customer-facing reads hit the primary, all writes go to the primary. As traffic grows, you add more replicas — not bigger instances.

Chapter 07

Multi-AZ DB Cluster — the new one

A 2022 addition to RDS that the SAA-C03 exam loves. Don't confuse it with regular Multi-AZ.

Standard Multi-AZ has a slight wastefulness problem: the standby just sits there. The Multi-AZ DB Cluster fixes this. It's a newer deployment option for MySQL and PostgreSQL on RDS that gives you:

  • One writer instance + two readable standbys, all in different AZs.
  • Semi-synchronous replication — slightly different from standard Multi-AZ's fully synchronous, but writes are confirmed when at least one standby has the data.
  • Faster failover — typically under 35 seconds.
  • The two standbys CAN serve read traffic (unlike standard Multi-AZ).
When to pick this

Multi-AZ DB Cluster is for workloads that want both HA and a bit of read scaling, but don't want the complexity of full Aurora. It's a middle-ground option. Currently only supports MySQL and PostgreSQL.

Exam Trigger

If a question describes "Multi-AZ deployment where standbys also serve reads" or "two readable standby instances," they're testing Multi-AZ DB Cluster, not regular Multi-AZ.

Chapter 08

RDS Security & RDS Proxy

Locking down your database, and solving the most common Lambda + RDS headache.

The four layers of RDS security

1. Network — keep the database off the internet

RDS databases live inside a VPC. Never make them publicly accessible unless absolutely necessary. They should sit in a private subnet, with a Security Group that only allows inbound traffic on the database port (3306 for MySQL, 5432 for PostgreSQL) from your application servers' Security Group.

2. Encryption at rest

RDS uses AWS KMS to encrypt the storage volume. Once enabled at creation, everything stored on disk is encrypted — backups, snapshots, replicas, all of it.

The critical gotcha: encryption can only be enabled at creation time. To encrypt an existing unencrypted database, you have to: snapshot it → copy the snapshot with encryption enabled → restore the encrypted snapshot to a new instance.

3. Encryption in transit

SSL/TLS connections between your app and the database. Just enable it; it's free.

4. IAM database authentication

Normally, your app logs in with a username/password. With IAM auth, your app can use its IAM role to generate a temporary token (15-minute expiry) and log in with that instead. No password lying around in config files. Available for MySQL and PostgreSQL.

Industry Pattern: passwords + Secrets Manager

The most common production pattern is: store the database password in AWS Secrets Manager, which can automatically rotate it every N days. Your application code fetches the current password from Secrets Manager at startup. RDS has built-in integration with Secrets Manager for MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server.

RDS Proxy — the Lambda lifesaver

Here's a real problem: imagine you have an API powered by Lambda functions, each calling RDS. When traffic spikes, hundreds of Lambda instances all try to open database connections at the same time. RDS has a finite connection limit (often 100–200 default), and Lambdas don't share connections — each invocation opens a fresh one.

You get the dreaded "too many connections" error and your API breaks.

RDS Proxy is a fully-managed connection pool. Your Lambda functions connect to the proxy; the proxy maintains a small pool of long-lived connections to RDS and multiplexes all the Lambda traffic over them.

Exam Triggers for RDS Proxy

Any question that mentions Lambda + RDS + "too many connections" / "connection exhaustion" / "improve scalability" → RDS Proxy.
Also: "minimise failover time" → RDS Proxy reduces failover time by up to 66%.
Also: "enforce IAM authentication centrally" → RDS Proxy can require IAM auth.

Trap

RDS Proxy does NOT make queries faster. It's a connection manager, not a performance booster. Don't pick it for "improve query latency" questions — that's caching territory.

Chapter 09

Enter Aurora — AWS's superpower DB

Built by AWS from the ground up. Same SQL as MySQL/PostgreSQL, but rebuilt for the cloud.

In 2014, AWS announced Aurora. The pitch was striking: take MySQL — the most popular database in the world — and re-engineer the parts that don't fit the cloud. Keep the SQL language, the drivers, the tools. Replace the storage layer with something purpose-built for AWS's infrastructure.

The result is a database that looks like MySQL or PostgreSQL to your application (so you don't have to change any code), but underneath is radically different from anything you've used before.

Aurora is to MySQL/PostgreSQL what a Tesla is to a Ford. Same idea of "a car you drive," but the engine is something new.

What "compatible with MySQL/PostgreSQL" means

Aurora comes in two flavours: Aurora MySQL and Aurora PostgreSQL. Your application uses the same MySQL or PostgreSQL drivers, the same SQL, the same connection libraries. You don't write Aurora-specific code. As far as the application knows, it's connecting to MySQL.

But the database storage layer, replication mechanism, and failover behavior are completely different from open-source MySQL.

Chapter 10

Aurora's secret: the storage layer

Understand this, and every other Aurora feature suddenly makes sense.

In a normal database (including standard RDS), each instance has its own copy of the data on EBS. Replication copies changes from primary to replicas. If the primary fails, the replica gets promoted, and AWS reattaches the storage.

Aurora throws this away. Instead:

  • The database compute layer (the part running SQL queries) is separate from the storage layer.
  • All Aurora instances — primary and replicas — share the same storage volume.
  • That shared volume automatically replicates 6 copies of every piece of data across 3 Availability Zones (2 copies per AZ).
  • The storage is self-healing — it constantly scans for corrupted blocks and repairs them from the other copies.
Aurora's architecture: compute instances on top, shared self-healing storage below. The storage is 6 copies, 3 AZs — automatically.

Why this design is brilliant

Because storage is shared:

  • Adding a read replica is instant — no data copying. The new instance just connects to the existing storage.
  • Failover is fast (under 30 seconds) — AWS just promotes a reader to be the writer. Storage doesn't move.
  • You can lose 2 of 6 copies with zero impact on writes, and 3 of 6 copies with zero impact on reads. Aurora survives an entire AZ going down without missing a beat.
  • Storage auto-scales from 10 GB up to 128 TB in 10 GB increments. You don't pre-provision; you pay for what you use.
Memorise this number

6 copies, 3 AZs. This is one of the most-tested Aurora facts. Aurora can lose up to 2 copies without losing write availability, and up to 3 without losing read availability.

Chapter 11

Aurora endpoints & replicas

How your application actually connects to Aurora.

Aurora gives you not one endpoint, but several:

EndpointWhat it does
Cluster (Writer) Endpoint Always points to the current writer. Use for writes. If failover happens, this endpoint automatically updates within seconds to point at the new writer.
Reader Endpoint Load-balances connections across all the readers. Use for read queries — Aurora handles the distribution.
Custom Endpoints You define a group of replicas — say, three of them dedicated to analytics queries — and create a custom endpoint that only routes to those. Great for separating workloads.
Instance Endpoints Each individual node has its own endpoint. Rarely used directly.

Up to 15 read replicas can attach to an Aurora cluster (compared to 5 on standard RDS). And because they share storage, the replication lag is typically under 100 milliseconds.

Industry Pattern

A typical Aurora setup: writer + 2 readers behind the standard reader endpoint, plus 1 replica dedicated to nightly analytics queries via a custom endpoint. The analytics queries can be heavy without affecting customer-facing reads.

Chapter 12

Aurora Global Database

Cross-region disaster recovery for databases. THE exam answer for "DB DR with low RPO/RTO".

Multi-AZ protects against an AZ failing. Aurora Global Database protects against an entire region failing.

You have your primary Aurora cluster in one region — say, us-east-1. You enable Global Database, and AWS replicates the storage to up to 5 secondary regions — say, eu-west-1, ap-southeast-1, and so on. The secondary regions:

  • Receive replicated data from the primary, typically with under 1 second of lag.
  • Are readable — you can run read queries from the secondary regions, great for local read performance for global users.
  • Are NOT writable by default — writes only go to the primary region.
  • Can be promoted to primary in under 1 minute if the primary region fails.
Aurora Global Database: primary region handles writes; up to 5 secondary regions receive replicated data with sub-second lag, ready to be promoted.
THE Exam Trigger

"Cross-region DR for a relational database, RPO of seconds, RTO under 1 minute" → Aurora Global Database. This is THE answer. Memorise the numbers: RPO < 1 second, RTO < 1 minute.

Common Confusion

Don't confuse:
• Aurora Multi-AZ = HA inside one region (built-in, free with Aurora).
• Aurora Global Database = DR across regions (extra feature, extra cost).
They solve different problems. The exam will offer both as answers; the question's qualifier ("region failure" vs "AZ failure") tells you which.

Chapter 13

Aurora Serverless v2

Aurora that scales itself, second by second, based on demand.

Normal Aurora — like normal RDS — runs on a fixed instance size. If your traffic doubles overnight, your db.r6g.large can choke. If your traffic drops to zero on weekends, you're still paying for that db.r6g.large.

Aurora Serverless v2 removes the instance-size decision. You set a minimum and maximum capacity in Aurora Capacity Units (ACUs) — roughly, 2 GB of RAM per ACU — and Aurora automatically scales between those bounds in fractions of a second based on actual load.

  • Scales up in milliseconds when load spikes.
  • Scales down to your minimum when idle.
  • You pay per ACU per second.
  • v2 supports the full Aurora feature set: Multi-AZ, Global Database, read replicas.
Exam Triggers for Aurora Serverless
  • "Unpredictable or spiky workload" → Aurora Serverless.
  • "Don't want to manage instance sizing" → Aurora Serverless.
  • "Cost-effective for variable traffic" → Aurora Serverless.
  • "Dev/test database used only during business hours" → Aurora Serverless.
Industry Pattern

Aurora Serverless v2 is the default choice for new SaaS startups that have no idea what their traffic will look like. It's also popular for multi-tenant apps where each customer has their own database — most are idle most of the time, but a few spike heavily.

Trap

"Serverless" doesn't mean "free when idle." Aurora Serverless v2 always runs at your minimum capacity, so it has a small ongoing cost. (V1, now deprecated, could scale to zero — V2 does not by default in production setups.) If a question says "scale to zero when idle," that's V1 behaviour or implies an architectural workaround.

Chapter 14

Aurora Backtrack & Cloning

Two Aurora-only features that are exam darlings.

Aurora Backtrack

The "oh no" feature. You ran DELETE FROM users without a WHERE clause. In a normal database, your only option is to restore from backup — minutes or hours of downtime.

With Aurora Backtrack (MySQL-compatible only), you can "rewind" the database to a specific point in time — say, 4 minutes ago — without restoring from a snapshot. The cluster stays running, just goes back to its previous state. You can backtrack up to 72 hours.

Trigger

"Quickly recover from a mistaken DELETE/UPDATE without downtime" → Aurora Backtrack. Don't confuse with Point-In-Time Recovery, which creates a new instance.

Aurora Cloning

Need a copy of your 2 TB production database to test a risky migration? Restoring from a snapshot would take hours.

Aurora's database cloning creates a new cluster that shares storage with the source — using copy-on-write. The clone exists instantly, costs almost nothing in storage until changes happen, and as either side writes new data, only the changed blocks are duplicated.

Trigger

"Quickly create a copy of production data for testing without affecting performance" → Aurora Cloning. Much faster and cheaper than snapshot-and-restore.

Chapter 16

ElastiCache — why we need caching

Adding a fast layer in front of your database so it stops being the bottleneck.

Even with Aurora, even with 15 read replicas, even with the most beautifully indexed queries, there's a hard limit to how fast a database can return data. A query that hits disk takes 1–10 milliseconds. A query that hits RAM on the database takes 0.1–1 millisecond. Many websites still need to be faster than that, especially at scale.

The trick used everywhere: caching. Keep the most frequently accessed data in RAM, on a separate dedicated server. When the application needs that data, check the cache first. If it's there (a cache hit), return it in microseconds. If not (a cache miss), fetch from the database and store in the cache for next time.

Mental Model

Your database is the warehouse — vast, organised, complete, but slow to walk through. Your cache is the shelf right behind the cashier — only the top 100 products, instantly grabbable. Most customers want one of those 100 things, so the warehouse barely gets touched.

Amazon ElastiCache is the managed cache service. AWS provisions, patches, monitors, and replicates the cache nodes for you. You pick one of two engines: Redis or Memcached. Both store key-value pairs in memory.

Chapter 17

Redis vs Memcached

The most asked ElastiCache question on the exam. Get this cold.

Memcached — the simple one

Memcached is old (2003) and simple by design. It stores key-value pairs in RAM, and that's basically it. Want to cache an object? SET user:42 → "{name:Alice}". Retrieve it: GET user:42. Done. It's multi-threaded, which means a single Memcached node can fully use a multi-core machine.

Memcached's weaknesses are also fundamental:

  • No persistence. If a node restarts, the cache is empty. Everything has to be rebuilt from the database.
  • No replication. If a node dies, you lose its data.
  • No advanced data types. Just strings.
  • No pub/sub, no transactions, no geospatial.

Redis — the powerhouse

Redis is newer (2009) and far richer. It also stores key-value pairs, but the "value" can be a list, a set, a sorted set, a hash, a geospatial point, a bitmap, a stream, and more. On top of that:

  • Persistence — optionally writes to disk so data survives restarts.
  • Multi-AZ with automatic failover — one primary and read replicas across AZs.
  • Pub/Sub messaging — for real-time notifications.
  • Sorted sets — perfect for leaderboards (sort by score automatically).
  • Transactions and Lua scripting.
  • Geospatial queries — "find all users within 5 km of this point."

The Simple Rule

If the question mentions any of: persistence, replication, multi-AZ, pub/sub, sorted sets, leaderboards, sessions that survive restart, real-time analytics, geospatial — the answer is Redis.

Memcached only wins on questions like "simplest possible cache, highest raw throughput, no need for HA, ephemeral data only."

Industry Reality

In 2026, Redis dominates production caching workloads — well over 80% of new ElastiCache deployments. Memcached lives on in specific high-throughput, single-purpose cache layers (large-scale ad-tech bidders, some legacy systems). The exam reflects this: Redis is the default answer; Memcached only appears as a distractor or when the question's qualifiers explicitly rule out Redis features.

Bonus: ElastiCache Serverless

In 2023 AWS launched ElastiCache Serverless for both Redis and Memcached. Like Aurora Serverless v2, it auto-scales capacity per second and you pay per request + GB-hour. Same exam triggers as Aurora Serverless: "unpredictable workload," "minimise capacity planning."

Chapter 18

Caching strategies

How your application actually uses the cache. Three patterns to know.

1. Lazy Loading (Cache-Aside)

The most common pattern. The application asks the cache first; if not found, it goes to the database, gets the data, and stores it in the cache for next time.

Lazy Loading: only fetch from DB on a cache miss, then store the result for future requests.

Pros: Only requested data is cached (no wasted space). Simple to implement. Resilient to cache failure (just hits DB).

Cons: Cache miss costs 3 trips (cache, DB, cache write). Data can become stale if it changes in the DB and the cache still has old value.

2. Write-Through

Every time the app writes to the database, it also writes to the cache simultaneously. Cache is always up-to-date.

Pros: Data is never stale. Reads are very fast.

Cons: Slower writes (two operations every time). Wastes cache space caching data that may never be read. If the cache is empty (after restart), every read is a miss until data is written again — so often paired with lazy loading.

3. TTL (Time-To-Live)

Not really a separate strategy, but an essential parameter on every cache entry. You attach an expiry to each key — say, 60 seconds. After that, the entry is automatically removed. The next request triggers a fresh fetch from the database.

TTL is how you balance freshness and cache load:

  • Short TTL (5–30 seconds): nearly real-time data, but high DB load.
  • Long TTL (5–30 minutes): much less DB load, but data can be stale.
Strategy Triggers

"Reduce database load for read-heavy workload" → Lazy Loading.
"Ensure cache always has the latest data" → Write-Through.
"Expire cached data automatically after some time" → TTL.

Chapter 19

Real-world ElastiCache use cases

Where caching shows up in actual production systems.

Session storage

When a user logs into your web app, you create a session — a record that proves they're logged in. Instead of looking that up in the database on every request, store it in Redis with a key like session:abc123. Fast, scalable, and if you have multiple web servers, all of them read the same Redis. Used by virtually every major web application.

Database query caching

The product catalog page on an e-commerce site. The same query runs thousands of times per second. Cache the result for 60 seconds. Lazy loading pattern. Reduces DB load by 90%+.

Leaderboards (Redis only)

Real-time gaming or social leaderboards. Redis sorted sets are perfect — adding a score and getting the top 10 are both O(log N) operations. Used by Twitch, gaming companies, Stack Overflow, fitness apps.

Rate limiting

"This API key can make 1000 requests per minute." Redis can track this with INCR + EXPIRE in microseconds. Far faster than checking a database.

Pub/Sub for real-time features

Live chat, notifications, dashboards. Redis pub/sub lets you broadcast a message to all subscribers instantly.

Caching API responses

External APIs you call (weather, stock prices, geocoding) — cache the response for some seconds to avoid hitting the rate limit and to speed up your app.

Industry Pattern

A typical production architecture for a customer-facing app: ALB → EC2 (or ECS/Fargate) → ElastiCache Redis → Aurora MySQL. The Redis layer absorbs 80–95% of all read queries. Aurora handles writes plus the misses. This pattern lets you scale to millions of users without scaling the database tier proportionally — which is what makes it economical.

Chapter 20

Bringing it all together — the full stack

How RDS/Aurora and ElastiCache actually work together in a real architecture.

Let's tie the whole topic together. Imagine you're building a B2C web app — say, an online learning platform with a million users.

A production database tier: ALB → app servers → Redis cache → Aurora MySQL (writer + readers, Multi-AZ) → Aurora Global Database for cross-region DR.

What each piece does

  1. ALB spreads traffic across the app servers.
  2. App servers handle the business logic.
  3. ElastiCache Redis is checked first for every read. User sessions, recent product catalog reads, frequently-accessed user profile data all live here.
  4. On a cache miss, Aurora MySQL serves the read (via the Reader Endpoint, hitting one of the read replicas).
  5. All writes go to the Aurora Writer, which immediately propagates to its readers (~100ms lag) and to the secondary region.
  6. Aurora Global Database replicates everything to eu-west-1 — readable for European users to lower their latency, and ready to be promoted if us-east-1 fails.

Add to that: encryption at rest via KMS, the master password rotating automatically in Secrets Manager, IAM-based authentication on the Aurora cluster, and a Lambda function that takes a weekly cross-region snapshot copy. That's a production-grade database tier.
