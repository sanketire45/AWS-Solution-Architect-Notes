1. EC2 on the Exam — What to Expect

EC2 is woven through every domain of the SAA-C03. Out of 65 questions, expect roughly:

  • 10–15 questions where EC2 is the primary subject (pricing, storage, placement, lifecycle).
  • 20+ more where EC2 is part of a larger scenario (web tier behind ALB, ASG, multi-AZ, integration with S3/RDS).

Questions usually aren't "what is the syntax for X?" They're scenarios: "Company has Y need with Z constraint — pick the best architecture." Two answers will usually be technically correct. The qualifier ("MOST cost-effective", "LEAST operational overhead", "HIGHEST availability") decides between them.

The Two Qualifier-Reading Rules
  1. "MOST cost-effective" or "LEAST cost" — push toward Spot, Reserved, Savings Plans, smaller instance sizes, gp3 over io2, st1/sc1 for cold data, Graviton.
  2. "LEAST operational overhead" — push toward managed services, Savings Plans (over Reserved which require capacity planning), Auto Scaling, AWS-managed AMIs, SSM Session Manager, IAM Roles.

5. The 12 Most Common Traps

These are the wrong answers that feel right. The exam loves them. Learn to recognize and reject them.

EBS Multi-Attach for "shared files between EC2s"EFS (Multi-Attach is for clustered DBs, not general file sharing)
st1 or sc1 as boot volumeHDDs can't boot — use gp3
"Instance Store survives stop because it's a disk"Instance Store data is LOST on stop and terminate. Only survives reboot.

"Use access keys on EC2 for S3 access"Always IAM Role — never hardcoded keys
"Built-in CloudWatch shows memory and disk free"It doesn't — install CloudWatch Agent

"Hibernate on any instance with RAM up to 256 GB"Hibernate limit is 150 GB RAM, encrypted root EBS, supported OS, enabled at launch.

6. Decision Flow — Picking the Right Answer Fast

Flow 1: Which Pricing Model?

  1. Is the workload fault-tolerant and can it survive sudden termination? → Yes → Spot. Done.
  2. Is the workload steady and 24/7, and will the architecture not change? → Yes → Reserved Instance (3-yr all upfront for max discount) or EC2 Instance Savings Plan.
  3. Is it steady but architecture might evolve? → Compute Savings Plan.
  4. Need to bring my own software license? → Dedicated Host.
  5. Need guaranteed capacity in a specific AZ for a launch? → Capacity Reservation (combine with SP for discount).
  6. None of the above (short-term, unpredictable)? → On-Demand.

Flow 2: Which EBS Volume Type?

  1. Is it a boot volume? → Must be SSD (gp3, gp2, io1, io2). Default: gp3.
  2. Database needing 200,000+ IOPS or low single-digit ms latency? → io2 Block Express.
  3. Database needing 32k–64k IOPS, Multi-Attach for cluster? → io1 / io2.
  4. Big sequential workload (logs, Hadoop, data warehouse)? → st1.
  5. Cold data, almost never accessed? → sc1 (or consider S3 Glacier instead).
  6. Everything else? → gp3.

Flow 3: Which Placement Group?

  1. Need lowest possible latency between instances (HPC)? → Cluster + EFA.
  2. Have a small number (≤7 per AZ) of critical instances? → Spread.
  3. Have a distributed system that understands "racks" (Hadoop, Cassandra, Kafka)? → Partition.
  4. Just a web tier? → No placement group. Use Auto Scaling Group across multiple AZs.

Flow 4: How Do Multiple EC2 Share Storage?

  1. Linux, generic file sharing across AZs? → EFS.
  2. Windows / SMB / Active Directory integration? → FSx for Windows.
  3. HPC / ML with massive parallel throughput? → FSx for Lustre (linked to S3).
  4. Clustered DB (Oracle RAC) needing block-level shared volume? → EBS Multi-Attach (io1/io2).
  5. Just shared object storage? → S3 (often the right answer).

7. Practice Questions (25 SAA-style)

Try each on paper or in your head before reading the answer. The explanations tie back to Doc 1 concepts — they're designed to reinforce, not just inform.

Question 1 Single answer
A media company runs a fleet of 200 EC2 instances 24/7 to power its core video streaming service. The instances are m5.4xlarge and the workload has been steady for 2 years. The architecture is mature and unlikely to change. The CFO wants to minimize cost. Which approach is MOST cost-effective?
  1. Purchase 3-year No Upfront Convertible Reserved Instances.
  2. Purchase 3-year All Upfront Standard Reserved Instances for the m5.4xlarge.
  3. Use Spot Instances exclusively to maximize savings.
  4. Stay on On-Demand with Auto Scaling to handle changes.
Answer

BPurchase 3-year All Upfront Standard Reserved Instances. Steady predictable load + known instance type + 3+ year horizon + no architecture changes is the textbook case for Standard RIs. All Upfront gives the maximum discount (up to 72%).

Why not the others: (A) Convertible RIs trade some discount for flexibility you don't need. (C) Spot is wrong — production streaming can't tolerate 2-minute termination warnings. (D) On-Demand wastes money on steady workloads.
Trigger recall: "Steady 24/7, known instance type, never changing" → Standard RI 3-yr All Upfront. If the question hinted at flexibility, you'd switch to Compute Savings Plan.
Question 2 Single answer
A data science team runs nightly batch jobs that process click-stream data from S3. Each job takes ~3 hours, can be safely restarted from a checkpoint, and the team has built robust handling for sudden worker death. They want the LOWEST possible compute cost. Which EC2 option is best?
  1. On-Demand instances scheduled to start at 1 AM nightly.
  2. 1-year Reserved Instances of the largest type they could fully utilize.
  3. Spot Instances launched via a Spot Fleet diversified across instance types.
  4. Dedicated Instances for compliance isolation.
Answer

CSpot Instances via Spot Fleet. The workload is fault-tolerant, has checkpointing, and survives interruption — the exact use case for Spot. Diversifying across instance types in a Spot Fleet raises availability (different pools), so if one type gets reclaimed, others fill the gap.

Why not the others: (A) On-Demand wastes ~80% of the savings Spot would give. (B) RI's commitment is overkill for batch — and RIs are billed even when idle. (D) Dedicated Instances are for compliance, not cost reduction.
Trigger recall: "Fault-tolerant batch + cheapest" = Spot. Doc 1 covered this with the ML training example — same pattern.
Question 3 Single answer
A company is planning to migrate their monolithic application to AWS. They expect to start on EC2, but within 18 months migrate most workloads to AWS Fargate, and some to Lambda. They want to commit to savings now but keep flexibility. Which option fits best?
  1. 3-year Standard Reserved Instances on m5 family.
  2. 1-year EC2 Instance Savings Plan.
  3. 3-year Compute Savings Plan.
  4. Spot Fleet across multiple regions.
Answer

C3-year Compute Savings Plan. The Compute Savings Plan is the only option that automatically applies to EC2, Fargate, AND Lambda. The company can shift workloads freely without losing the discount. Up to 66% off — slightly less than RI but worth the flexibility.

Why not the others: (A) and (B) lock to EC2 — they wouldn't apply once workloads move to Fargate/Lambda. (D) Spot can't run their existing monolith reliably.
Trigger recall: "Might shift to Fargate or Lambda" — the dead giveaway for Compute Savings Plan.
Question 4 Single answer
An enterprise must run Microsoft SQL Server on AWS while reusing its existing per-physical-core SQL Server licenses (BYOL). Compliance requires visibility into the physical hardware. Which EC2 option enables this?
  1. On-Demand Instances with default shared tenancy.
  2. Dedicated Instances.
  3. Dedicated Host.
  4. Spot Instances tagged for BYOL.
Answer

CDedicated Host. Dedicated Host gives you visibility into the physical sockets and cores — required for BYOL per-physical-core licensing of SQL Server, Oracle, and Windows Server. Dedicated Instance doesn't expose hardware details.

Why not the others: (A) Shared tenancy fails BYOL audit. (B) Dedicated Instance gives single-tenancy but no socket/core visibility. (D) Spot doesn't address BYOL at all.
Trigger recall: "BYOL Oracle / SQL Server / Windows" = Dedicated Host.
Question 5 Single answer
A startup is building a SaaS platform. They expect traffic to be moderate but unpredictable — sudden viral spikes possible. They've heard about Reserved Instances but aren't sure their workload is stable enough. What should the architect recommend for the EC2 layer initially?
  1. Buy 3-year Standard Reserved Instances immediately to lock in savings.
  2. Run On-Demand instances behind an Auto Scaling Group; revisit pricing once load patterns stabilize.
  3. Use 100% Spot Instances to minimize cost.
  4. Purchase Dedicated Hosts for predictable cost.
Answer

BOn-Demand with Auto Scaling, revisit later. Unpredictable / viral / unknown patterns = On-Demand is the safe starting point. Once you have 2-3 months of stable utilization data, you can convert the steady baseline to Reserved or Savings Plans.

Why not the others: (A) Locking in 3 years before knowing your baseline is wasted commitment. (C) Pure Spot can drop your platform — bad for a customer-facing SaaS. (D) Dedicated Hosts are expensive and unrelated to traffic patterns.
Trigger recall: "Unpredictable" or "spiky traffic" → On-Demand baseline. Optimize later, not first.
Question 6 Single answer
An EC2 instance hosts a critical Oracle database backed by a single EBS volume. The DBA reports the workload is hitting 80,000 IOPS and needs to scale to 200,000 IOPS with sub-millisecond latency. The volume is currently 6 TB and will grow to 40 TB. Which EBS volume type should be used?
  1. gp3 — set IOPS to 200,000.
  2. io1 with provisioned 64,000 IOPS.
  3. io2 Block Express with provisioned 200,000 IOPS.
  4. st1 throughput-optimized HDD.
Answer

Cio2 Block Express. It's the only EBS volume type that supports up to 256,000 IOPS, sub-ms latency, and 64 TB size. Designed exactly for SAP HANA / large Oracle / huge SQL Server.

Why not the others: (A) gp3 caps at 16,000 IOPS. (B) io1 caps at 64,000 IOPS — not enough. (D) st1 is HDD with only 500 IOPS max — for sequential workloads, not random DB I/O.
Trigger recall: "SAP HANA" or ">64k IOPS" or ">16 TB high-IOPS" = io2 Block Express.
Question 7 Single answer
A development team builds a CI/CD pipeline that processes thousands of small log files daily. The processing job streams logs sequentially, writes summary outputs back to S3, and runs on EC2 for 4 hours nightly. They want the LOWEST-COST EBS storage that can handle the throughput. The volume is 8 TB and is not the boot volume. Which volume type?
  1. gp3 General Purpose SSD.
  2. io2 Provisioned IOPS SSD.
  3. st1 Throughput Optimized HDD.
  4. sc1 Cold HDD.
Answer

Cst1 Throughput Optimized HDD. The workload is large sequential reads/writes (log streaming) — exactly what st1 is built for. Up to 500 MB/s at HDD prices. It's cheaper than gp3 for this kind of throughput workload.

Why not the others: (A) gp3 would work but costs more. (B) io2 is massive overkill — random IOPS isn't the need. (D) sc1 is for infrequent / cold access — too slow for a nightly streaming job.
Trigger recall: "Log streaming, big data, sequential" → st1. "Almost never accessed" → sc1.
Question 8 Single answer
An EC2 instance has an attached Instance Store volume with critical preprocessing data. The team needs to perform routine maintenance and asks if they should stop the instance to save costs during the maintenance window. What should the solutions architect tell them?
  1. Stopping the instance is safe; Instance Store data persists like EBS.
  2. Stopping the instance will permanently delete the Instance Store data; back it up to EBS or S3 first.
  3. Rebooting is the same as stopping, so they can reboot to save costs.
  4. Instance Store data is automatically backed up to S3 every hour.
Answer

BStop deletes Instance Store data; back up first. Instance Store lives on the host's local SSD. A stop moves the instance to potentially different hardware on restart — local SSD contents are gone. Only reboot keeps Instance Store data (same host).

Why not the others: (A) is the textbook trap — Instance Store ≠ EBS. (C) Reboot doesn't save compute costs anyway. (D) AWS doesn't auto-back up Instance Store.
Trigger recall: Stop/Terminate kills Instance Store data. Only reboot preserves it.
Question 9 Single answer
A security review flags that several existing EC2 instances have unencrypted root EBS volumes. The team needs to bring them into encryption-at-rest compliance with the LEAST risk of data loss. What's the correct workflow?
  1. SSH into each instance and run filesystem-level encryption tools.
  2. Modify the existing volume settings to enable encryption in the console.
  3. Take a snapshot, copy the snapshot with encryption enabled, create a new volume from the encrypted snapshot, attach in place of the old one.
  4. Terminate the instances and re-launch with encryption enabled.
Answer

CSnapshot → copy with encryption → create new volume. This is the standard, AWS-documented workflow. The original volume stays untouched (low risk), and the new encrypted volume mirrors it.

Why not the others: (A) Filesystem encryption is OS-level, not the AWS-encryption-at-rest the auditors expect. (B) You cannot toggle encryption on an existing EBS volume — that menu option doesn't exist. (D) Terminating risks data loss.
Trigger recall: "Encrypt existing unencrypted volume" = snapshot, copy with encryption, new volume. Memorize this 3-step path.
Question 10 Single answer
A WordPress deployment uses 4 EC2 instances behind an ALB. The instances need to share user-uploaded images (themes, photos) so that all instances can serve any user's request. Which shared-storage solution should be used?
  1. EBS Multi-Attach volume (io2) shared across the 4 instances.
  2. EFS file system mounted to all 4 instances.
  3. An S3 bucket replicated to a local EBS volume on each instance.
  4. FSx for Lustre file system.
Answer

BEFS. EFS is a Linux NFS file system that mounts to many EC2s simultaneously across AZs. Perfect for shared web content (WordPress uploads, themes). The classic textbook use case.

Why not the others: (A) The trap. EBS Multi-Attach is for clustered DBs (Oracle RAC), not general file sharing — apps would corrupt data without their own locking. (C) Awkward custom solution with sync challenges. (D) FSx for Lustre is HPC-specific, way overpowered for WordPress.
Trigger recall: "Multiple Linux EC2s share files" = EFS. Don't fall for the Multi-Attach trap.
Question 11 Single answer
A high-frequency trading firm runs 80 EC2 instances that need microsecond-level network latency between them, all in the same Availability Zone. Which solution provides the BEST network performance between these instances?
  1. Spread placement group with EFA.
  2. Partition placement group across 7 partitions.
  3. Cluster placement group with ENA-enabled instances.
  4. No placement group; rely on multi-AZ distribution.
Answer

CCluster placement group with ENA-enabled instances. Cluster packs instances tightly on the same physical rack for the lowest possible latency (single-digit microseconds). ENA (Elastic Network Adapter) gives up to 100 Gbps throughput. For even lower latency on HPC workloads, you'd add EFA.

Why not the others: (A) Spread is the OPPOSITE of what's needed — spreads across racks. (B) Partition is for distributed systems, not low-latency. (D) Multi-AZ adds latency.
Trigger recall: "Lowest latency between instances" = Cluster placement group. The trap: confusing Cluster (speed) with Spread (safety).
Question 12 Single answer
A team is deploying a 20-node Apache Cassandra cluster on EC2. Cassandra is rack-aware and replicates data across racks. The team wants to ensure that hardware failure on one rack doesn't take down multiple Cassandra replicas. Which placement strategy is best?
  1. Cluster placement group with 20 instances.
  2. Spread placement group with 20 instances.
  3. Partition placement group with multiple partitions.
  4. No placement group, all in the same AZ.
Answer

CPartition placement group. Cassandra is the prototype use case for Partition. Each partition lives on a different rack. Cassandra's rack-awareness places replicas in different partitions, so a rack failure loses at most one replica per data row — cluster keeps serving.

Why not the others: (A) Cluster puts all 20 on one rack — single point of failure. (B) Spread caps at 7 per AZ — 20 nodes won't fit. (D) No placement group provides no guarantees about physical layout.
Trigger recall: "Cassandra / Hadoop / Kafka" → Partition placement group. ">7 critical instances per AZ" disqualifies Spread.
Question 13 Single answer
A company has 3 critical control-plane EC2 instances running a custom orchestration service. They cannot tolerate any 2 of these failing at the same time. They want maximum protection from correlated hardware failures within a single AZ. Which placement option fits?
  1. Cluster placement group.
  2. Spread placement group.
  3. Partition placement group with 1 partition.
  4. Auto Scaling Group with min 3 instances.
Answer

BSpread placement group. Each of the 3 instances lands on a different physical rack. A single rack failure takes down at most 1 instance. Spread is built exactly for "few critical instances must not share failure domain." 3 is well under the 7-per-AZ limit.

Why not the others: (A) Cluster does the opposite. (C) Partition with 1 partition gives no separation. (D) ASG ensures health but doesn't guarantee different physical racks.
Trigger recall: "Few critical instances, no shared failure" = Spread (max 7 per AZ).
Question 14 Single answer
A team has an EC2 instance that needs to read objects from a specific S3 bucket. A developer suggests creating an IAM user, generating access keys, and storing them in a config file on the instance. The architect rejects this. What should be done instead?
  1. Store the access keys in EC2 user data so they're applied at launch.
  2. Attach an IAM Role with S3 read permissions to the EC2 instance.
  3. Make the S3 bucket public so no credentials are needed.
  4. Use the EC2 root account's credentials.
Answer

BAttach an IAM Role. The instance gets temporary credentials automatically via IMDS, rotated by AWS. No secrets on disk, no compromise risk. This is the security best practice.

Why not the others: (A) User data is visible in metadata — never put secrets there. (C) Making the bucket public is wildly insecure. (D) The "root" account is not an EC2 concept and is a terrible idea anyway.
Trigger recall: "EC2 needs to call AWS API" = IAM Role. Always. Never access keys.
Question 15 Single answer
A security audit reveals that an EC2-hosted web app contains an SSRF vulnerability. An attacker could trick the app into requesting http://169.254.169.254/latest/meta-data/iam/security-credentials/ and steal the role's temporary credentials. The team wants to harden against this attack. What's the BEST mitigation at the EC2 level?
  1. Remove the IAM role from the instance.
  2. Enforce IMDSv2 (session-token-required) on the instance.
  3. Block port 169 on the security group.
  4. Disable the instance metadata service entirely.
Answer

BEnforce IMDSv2. IMDSv2 requires an initial PUT to get a session token before any GET is honored. SSRF attacks issue only GET, so they fail. AWS designed IMDSv2 specifically to neutralize SSRF (after the Capital One incident).

Why not the others: (A) Removing the role breaks the app's legitimate AWS access. (C) Security groups don't filter the special link-local 169.254.169.254 address. (D) Disabling metadata breaks credential delivery to the SDK.
Trigger recall: "SSRF" or "protect metadata service" = IMDSv2. The Capital One 2019 breach story makes this stick.
Question 16 Single answer
A genomics researcher runs a Python analysis pipeline that takes 90 minutes to load reference data into memory, then runs interactively. The researcher uses it 2–3 times a week and would like to avoid the 90-minute warm-up each time. The instance has 64 GB RAM. What's the BEST way to skip warm-up while saving costs when idle?
  1. Stop the instance when idle and start it when needed.
  2. Enable Hibernate on the instance and hibernate when idle.
  3. Run the instance 24/7 to avoid restarts.
  4. Take an AMI of the running instance and launch from it each session.
Answer

BEnable Hibernate. Hibernate saves the in-memory state to the encrypted root EBS volume. On start, RAM is restored — the loaded reference data is already there, no re-loading needed. The 64 GB RAM is under the 150 GB Hibernate limit.

Why not the others: (A) Stop loses RAM contents — 90-min warmup returns every time. (C) 24/7 wastes money for 2–3 sessions/week. (D) An AMI captures disk, not RAM — same problem as Stop.
Trigger recall: "Long warm-up, intermittent use, preserve RAM state" = Hibernate. Remember the 150 GB RAM cap.
Question 17 Single answer
An operations team has set up CloudWatch alarms on EC2 CPU utilization and network traffic, but they need to alert when an instance's memory usage exceeds 85%. They don't see a memory metric in CloudWatch. What's the correct way to enable this?
  1. Enable detailed monitoring (1-minute) — memory metrics will appear.
  2. Install the CloudWatch Agent on the instance and configure it to send memory metrics.
  3. Memory metrics are only available for Nitro instances; switch to one.
  4. Memory cannot be monitored in CloudWatch; use AWS Config.
Answer

BInstall the CloudWatch Agent. AWS can only see what the hypervisor exposes (CPU, network, disk I/O for instance store). Memory and disk-space-free are inside the OS, invisible to the hypervisor. The CloudWatch Agent runs inside the OS and reports those metrics.

Why not the others: (A) Detailed monitoring is 1-min vs 5-min frequency — same metric set. (C) False — applies to all instance types. (D) AWS Config tracks configuration changes, not memory.
Trigger recall: "Memory" or "disk space free" on CloudWatch = install CloudWatch Agent. A classic.
Question 18 Single answer
A company runs a standardized Linux EC2 fleet across us-east-1 and eu-west-1 (Ireland). They want a single hardened "golden image" that's automatically available in both regions for new launches. What's the BEST approach?
  1. Build the AMI in us-east-1 and reference it by AMI ID from eu-west-1 launches.
  2. Build the AMI in us-east-1, then use the AWS CLI / console to copy the AMI to eu-west-1.
  3. Mount the AMI from us-east-1 as a network drive in eu-west-1.
  4. Configure the AMI as a "Global AMI" in the EC2 console.
Answer

BCopy the AMI to eu-west-1. AMIs are region-resident. To use one in another region, you copy it — AWS replicates the underlying snapshots and assigns a new AMI ID in the target region.

Why not the others: (A) AMI IDs aren't valid across regions. (C) AMIs aren't mountable that way. (D) No such thing as a "Global AMI" in AWS.
Trigger recall: "Same image in multiple regions" = copy the AMI. EC2 Image Builder can automate this.
Question 19 Multi-response (Choose 2)
An EC2 instance has a 500 GB gp3 EBS root volume containing critical regulatory data. The company's compliance officer requires backup snapshots to be retained for 7 years, but the data is accessed less than once a year. The team wants the MOST cost-effective long-term snapshot retention strategy. Which TWO actions together achieve this?
  1. Manually delete snapshots after 30 days to save space.
  2. Use AWS Data Lifecycle Manager (DLM) to automate the snapshot schedule and retention.
  3. Move old snapshots to the EBS Snapshot Archive tier after they're no longer in active rotation.
  4. Copy snapshots to S3 Standard for retention.
  5. Convert old snapshots to AMIs to reduce storage cost.
Answer

B+CDLM + Snapshot Archive tier. DLM automates the create/retain/delete lifecycle on a schedule. EBS Snapshot Archive is ~75% cheaper than standard snapshot storage — ideal for rarely-accessed compliance retention (acceptable to wait 24–72 hours to restore).

Why not the others: (A) Deleting violates the 7-year requirement. (D) Snapshots already live in S3-class durability under the hood — you can't directly copy them to S3 Standard. (E) AMIs use the same snapshot storage; no savings.
Trigger recall: "Long retention, rarely accessed" = Snapshot Archive. "Automate snapshot lifecycle" = Data Lifecycle Manager (DLM).
Question 20 Single answer
A development team accidentally terminated a production EC2 instance, losing the root EBS volume in the process. Management asks how to prevent this in the future for production instances. Which is the simplest, most direct safeguard?
  1. Set the IAM policy to deny ec2:TerminateInstances for the entire account.
  2. Enable Termination Protection on each production instance.
  3. Move all production workloads to Reserved Instances which can't be terminated.
  4. Use Stop Protection to prevent termination.
Answer

BEnable Termination Protection. A simple per-instance toggle that blocks console/CLI termination until explicitly disabled. Cheap, easy, targeted.

Why not the others: (A) Account-wide deny breaks legitimate ops. (C) RIs are a billing mechanism — instances are equally terminable. (D) Stop Protection prevents STOPS, not terminations. Common confusion.

Question 21 Single answer
A web application runs on small EC2 instances. Most of the time, CPU utilization sits at 5–10%, but spikes briefly to 70% during user actions. The team picked t3.small instances. Recently, during a marketing campaign, the app slowed dramatically because instances "ran out of CPU credits." What's the right architect response?
  1. Switch to t3.large to get more credits.
  2. Enable T3 Unlimited mode (or migrate to m5.small) — the workload is no longer truly burstable.
  3. Add more t3.small instances to share load.
  4. Switch to Spot Instances for lower cost.
Answer

BEnable T3 Unlimited or move to m5.small. Running out of CPU credits means the workload is no longer a "burstable" fit. T3 Unlimited lets you exceed baseline by paying per CPU-hour above credits (often cheaper than upsizing). If sustained CPU is the new normal, M-family is a better match.

Why not the others: (A) t3.large still earns credits proportionally — bigger but same model. (C) Adding instances helps the load issue but doesn't fix per-instance credit exhaustion. (D) Spot is unrelated.
Trigger recall: "Ran out of CPU credits" = workload isn't burstable anymore. Either Unlimited mode or switch to M/C family.
Question 22 Single answer
A company runs an Oracle RAC clustered database that requires multiple EC2 instances to simultaneously read and write to the same block storage device. They're migrating from on-prem and want to preserve this architecture. Which storage configuration fits?
  1. Mount EFS to all RAC nodes.
  2. Use EBS Multi-Attach with an io2 volume across the RAC nodes (in the same AZ).
  3. Use S3 with PutObject from each node.
  4. Use FSx for Windows.
Answer

BEBS Multi-Attach with io2. Multi-Attach is specifically designed for clustered apps like Oracle RAC that manage their own block-level concurrency. Up to 16 instances in the same AZ can attach to one io1/io2 volume.

Why not the others: (A) Oracle RAC needs block storage, not NFS file system. (C) S3 is object storage — not how a database engine reads/writes blocks. (D) FSx for Windows is SMB, doesn't match RAC's design.
Trigger recall: "Oracle RAC" or "clustered DB shared block volume" = EBS Multi-Attach. The flip side of Q10 — context matters.
Question 23 Single answer
An EC2 instance failed its System Status Check repeatedly — AWS reports the underlying physical host is experiencing degraded hardware. The instance is Nitro-based. What's the SIMPLEST way to restore the instance to a healthy host with minimal downtime?
  1. Terminate the instance and re-launch from the AMI.
  2. Stop the instance and start it again (this moves it to a different host).
  3. SSH in and reboot the OS.
  4. Wait for AWS to migrate the instance automatically.
Answer

BStop and start. When you stop an EBS-backed EC2 instance, it releases the underlying host. On start, AWS places it on different healthy hardware. The instance keeps its instance ID, EBS volumes, and Elastic IP (if attached). This is the standard fix for failed System Status Checks.

Why not the others: (A) Terminate + relaunch loses the instance ID and is more work than needed. (C) Reboot stays on the same broken host — doesn't help. (D) Auto Recovery exists but typically you'd configure it via CloudWatch alarm with EC2 Recover action; "wait passively" isn't reliable for prod.
Trigger recall: System Status Check fail = AWS hardware issue. Stop/start moves you to fresh hardware. For automation, configure CloudWatch alarm + EC2 Recover action.
Question 24 Single answer
A company is planning a major product launch and expects an unpredictable but potentially massive traffic spike. Their architect wants to guarantee that 100 specific m5.4xlarge instances will be available in us-east-1a during the launch window, regardless of region-wide capacity pressure. They do NOT want to commit to a long-term billing reservation. Which option fits?
  1. 3-year All Upfront Reserved Instances for those m5.4xlarge.
  2. On-Demand Capacity Reservation in us-east-1a for 100 m5.4xlarge instances.
  3. Spot Fleet diversified across instance types.
  4. Dedicated Host purchase.
Answer

BOn-Demand Capacity Reservation. Capacity Reservations guarantee that capacity is available when you need it, at On-Demand prices, with no long-term commitment. Perfect for launch events where availability matters more than discount. You can pair it later with a Savings Plan to layer in discounts.

Why not the others: (A) RIs are about discount, not capacity guarantees, and require commitment. (C) Spot doesn't guarantee capacity. (D) Dedicated Hosts solve licensing, not on-demand capacity guarantees.
Trigger recall: "Guarantee capacity in this AZ, no commitment" = On-Demand Capacity Reservation. A subtle but testable concept.
Question 25 Multi-response (Choose 2)
A startup needs to run a machine learning training job on EC2. The job uses GPU-heavy p4d.24xlarge instances and runs for ~24 hours per training run. The team has built robust checkpointing to S3 every 30 minutes. They want to MINIMIZE compute cost while keeping the option to resume from the last checkpoint if interrupted. Which TWO design choices should they adopt?
  1. Use On-Demand instances for the GPU layer to guarantee uninterrupted training.
  2. Use Spot Instances for the GPU layer and design the job to resume from S3 checkpoints on interruption.
  3. Set Spot interruption behavior to Stop or Hibernate where possible to preserve state.
  4. Use Reserved Instances for 3 years.
  5. Mount EBS Multi-Attach so multiple training nodes share state directly.
Answer

B+CSpot for the GPU + Stop/Hibernate interruption behavior. P4d on Spot can be 70–90% cheaper than On-Demand. Combined with S3 checkpointing, interruption is recoverable. Configuring interruption behavior to Stop (or Hibernate, where supported) preserves more state than the default Terminate.

Why not the others: (A) On-Demand is 3–4× more expensive than Spot — wastes the cost reduction goal. (D) Training jobs are bursty, not steady — RIs are wrong fit. (E) Multi-Attach is for clustered DBs, not ML training (which typically uses distributed training over the network).
Trigger recall: "ML training, checkpointing, cheapest" = Spot + careful interruption handling. The combination of choices reflects defense-in-depth thinking.

8. Day-of-Exam Tips for EC2 Questions

Reading the Question
  1. Read the last sentence first. That's where the qualifier lives: "MOST cost-effective," "LEAST operational overhead," "BEST resilience."
  2. Identify the workload character. Steady? Spiky? Fault-tolerant? Production-critical? This sets the pricing model.
  3. Look for hard constraints. "BYOL Oracle license" forces Dedicated Host. "Must guarantee capacity" forces Capacity Reservation. "Sub-millisecond latency between nodes" forces Cluster placement.
  4. Eliminate obviously wrong answers first. Spot for production? Eliminate. Hardcoded keys? Eliminate. IMDSv1 in 2026? Eliminate.
Tie-Breaking Between Two "Correct" Answers
  • If the question emphasizes cost → pick Spot / Reserved / Savings Plan / Graviton / gp3 / st1.
  • If it emphasizes operational overhead → pick managed services, Compute Savings Plan, IAM Role over keys.
  • If it emphasizes resilience / HA → multi-AZ via Auto Scaling Group, Spread or Partition placement, EBS over Instance Store, RDS Multi-AZ.
  • If it emphasizes performance → Cluster placement, io2 Block Express, Instance Store, EFA, ENA.
The Final 10-Point Recall

Before each EC2 question, run this checklist in your head:

  1. What's the qualifier? (cost / ops / HA / performance)
  2. Is the workload fault-tolerant? → Spot in play
  3. Is the workload steady? → Reserved/Savings in play
  4. Does data need to survive instance death? → EBS, not Instance Store
  5. Boot volume? → SSD (gp3/io2), never HDD
  6. Multi-AZ shared file system? → EFS (Linux), FSx (Windows)
  7. Clustered DB shared block volume? → EBS Multi-Attach
  8. Placement: speed (Cluster), few critical (Spread ≤7), many distributed (Partition)
  9. EC2 → AWS API calls = IAM Role, always IMDSv2
  10. Memory/disk-free metric in CW = install CloudWatch Agent
