Part 01

The Problem S3 Solves

Before S3 existed, how did companies store files? Let's go back to 2006 and feel the pain.

Imagine you are running a photo-sharing website in 2004. Your users upload 10,000 photos per day. Where do you put them? You buy a hard drive, plug it into your server, and store the files there. Simple. But then your site grows. You get 100,000 uploads a day. You buy more hard drives. You fill a rack. Then two racks. Then an entire server room. And now you have a new set of nightmare problems:

Problem #1

Capacity Guessing

  • Buy too little → site crashes when full
  • Buy too much → money wasted on idle drives
  • Hard drives take weeks to order and arrive
  • No way to scale instantly for viral moments
Problem #2

Reliability Headaches

  • Hard drives fail — typically 1-4% per year
  • Data centre fire = all your photos, gone
  • Manual backup takes engineers + more hardware
  • Disasters cost companies millions
Problem #3

Global Access Pain

  • Your server room is in Dublin
  • User in Sydney downloads a photo → 300ms+ latency
  • Your server handles every single request
  • Peak traffic → servers explode under load
Problem #4

Operational Burden

  • Hire sysadmins to manage hardware 24/7
  • Write your own backup, replication, redundancy code
  • Manage RAID arrays, storage networks
  • Replace failed drives at 3am on a Sunday
💡 Why S3 Was Built

Amazon solved all four of these problems for themselves first. Amazon.com — the e-commerce site — was dealing with exactly these problems at massive scale. They engineered an internal solution, then in March 2006, they launched it as a service: Simple Storage Service (S3). The idea was radical: what if storage was just an API call? You call PUT object and your file is stored, replicated, and served globally — with zero servers to manage.

S3 didn't just solve Amazon's problem. It kickstarted the entire cloud computing era. Dropbox, Netflix, Instagram, Airbnb — all built on S3 in their early days because it meant they could focus on their product instead of managing storage infrastructure.

The Three Types of Storage — Understanding Where S3 Fits

Before going deeper, you need to understand the three fundamental types of storage. This distinction matters a LOT on the exam and in real interviews.

The Three Storage Types — A Visual Analogy
📋 Exam Key Concept

S3 is object storage — not a file system, not a block device. This means: you CANNOT mount S3 like a hard drive. You CANNOT open a file and write to the middle of it. You retrieve or replace entire objects via HTTP GET/PUT. This is a core architectural constraint that drives exam questions.

Quick rule: Need a database → don't use S3. Need to run an OS → use EBS. Need shared drive across Linux EC2 instances → use EFS. Need to store files, images, videos, backups, logs → use S3.

Part 02

S3 Fundamentals

Buckets, objects, keys — the vocabulary you need before anything else makes sense.

Buckets and Objects — The Mental Model

Think of S3 like a giant, magical filing system with two levels: buckets and objects.

A bucket is like a top-level container — think of it like a postal code district. You create a bucket, give it a name, and choose which AWS Region it lives in. All data inside that bucket physically lives in that region (important for GDPR compliance in Ireland!).

An object is anything you store inside a bucket — a photo, a video, a PDF, a ZIP file, a database backup. The maximum size for a single object is 5 TB. For anything over 5 GB, you must use multipart upload (more on this in Performance).

💡 What "11 Nines" Actually Means

If you store 10,000,000 objects in S3, you'd expect to lose 0.001 objects per year on average. That's basically "never lose anything". AWS achieves this by automatically copying your data across at least 3 Availability Zones within your region. When you PUT an object, it's replicated before S3 confirms success. No action required from you.

Real-world illustration: Netflix stores over 1 petabyte of video content in S3. Dropbox stores billions of user files in S3. Neither has to worry about a failed hard drive wiping their data.

Bucket Naming Rules — The Gotchas

Bucket names are globally unique across all AWS accounts in the world. This is because S3 bucket URLs look like this: https://my-bucket.s3.amazonaws.com/photo.jpg — and there can only be one DNS entry per name.

  • 3–63 characters long, lowercase letters, numbers, hyphens only
  • Must start with a letter or number — NOT a hyphen or uppercase
  • Cannot be formatted as an IP address (e.g., 192.168.1.1)
  • Globally unique — even across different AWS accounts and regions
  • Region-specific data — your bucket lives in one region (e.g., eu-west-1 = Ireland)
📋 Exam Insight

S3 is a global service (appears under "Global" in AWS console) but buckets are regional. Data in a bucket in eu-west-1 (Ireland) physically stays in Ireland unless you configure Cross-Region Replication. This is crucial for data residency requirements — a major concern for Dublin fintech companies under GDPR.

Object Keys — How S3 "Fakes" a Folder Structure

Here's something that trips many people up: S3 has no real folders. It has a flat namespace. An object key is just a string — the full "path" to your object. The key 2025/january/invoice-001.pdf is just a string that happens to contain slashes.

The AWS console and SDKs display these keys as if they were folders — they group keys that share the same prefix. But underneath, it's all flat. This matters for performance: you can distribute requests across prefixes to avoid "hot spots" (more on this in the Performance section).

# These look like folders in the console but are just object keys: s3://my-fenergo-bucket/kyc-docs/2025/january/client-passport-12345.pdf s3://my-fenergo-bucket/contracts/europe/ireland/agreement-xyz.pdf s3://my-fenergo-bucket/audit-logs/2025-01-15-access.log # The key for the first object is literally: "kyc-docs/2025/january/client-passport-12345.pdf"

S3 Strong Consistency — The 2020 Game Changer

💡 Why This Was a Big Deal

Before December 2020, S3 had a confusing split: read-after-write consistency for new PUTs (you could immediately read a newly uploaded file) but only eventual consistency for overwrite PUTs and DELETEs (you might get a stale version for a short period after updating or deleting). This caused real bugs in applications that read immediately after writing.

In December 2020, AWS upgraded S3 to be strongly consistent for ALL operations — at no extra cost, no config needed, no performance trade-off. This was a massive engineering achievement given S3 handles trillions of requests per day.

What strong consistency means in plain language: The moment a PUT (upload) or DELETE succeeds and S3 confirms it, every subsequent GET or LIST will reflect that change. There is no "stale read" window anymore. If you upload a file and immediately list the bucket, the file appears. If you delete a file and immediately GET it, you get a 404. Always. No exceptions.

Before Dec 2020 (Old)

Eventual Consistency

  • Overwrite a file → might read old version briefly
  • Delete a file → might still get it for a short window
  • List bucket after upload → new file might not appear
  • Required workarounds in application code
  • Caused race conditions in distributed systems
After Dec 2020 (Current)

Strong Consistency

  • PUT succeeds → immediately readable everywhere
  • DELETE succeeds → immediately gone, GET returns 404
  • LIST after PUT → new object appears immediately
  • No workarounds needed in application code
  • Applies to ALL S3 operations, ALL storage classes
⚠️ Exam Trap — Old Questions Are Still Around

Many old practice questions and study materials (written before Dec 2020) will mention eventual consistency for S3 overwrites and deletes. These are now wrong. S3 is strongly consistent for all operations since December 2020. If an exam question asks "what consistency model does S3 use?" — the answer is strong consistency. If you see answer choices suggesting eventual consistency for S3 in modern questions, that is a trap/distractor.

Exception to know: S3 strong consistency does NOT apply to S3's integration with Amazon CloudFront edge caches (that's still eventual via TTL-based caching) or to S3 bucket configuration (like policy changes — those may have brief propagation delays). The strong consistency guarantee is for object data operations: GET, PUT, DELETE, LIST.

Part 03

Storage Classes — The Cost Brain

Why pay the same for data you access daily vs. data from 3 years ago? S3's tiered pricing is one of its most powerful features.

💡 Why Storage Classes Were Invented

Imagine a company stores 100TB of data. 5TB is actively used daily — product images, current invoices, logs from the last week. The other 95TB is older data: last year's tax filings, 3-year-old customer records, archived videos. Treating all 100TB identically is massively wasteful.

AWS's answer: storage classes. You pay less to store data you rarely access, but you pay a retrieval fee when you do need it. The less you access data, the cheaper the storage — but the more you pay per retrieval, and the longer it takes to get. Pick the class that matches your actual access pattern.

Tier 1 — Hot

S3 Standard

Your everyday workhorse. Data stored across 3+ AZs. Millisecond access. No minimum storage duration.

✓ 99.999999999% durability | ✓ 99.99% availability | ✓ No retrieval fee

Tier 2 — Warm

Standard-IA

Infrequent Access. Same durability as Standard, multi-AZ. Cheaper storage but you pay per retrieval. Min 30-day storage.

Use for: backups, DR copies, data accessed monthly | Min 30 days | Retrieval fee applies

Tier 3 — Warm+Risky

One Zone-IA

Like Standard-IA but data lives in only ONE Availability Zone. 20% cheaper than Standard-IA but if that AZ fails, your data is gone.

Use for: re-creatable data, secondary backups | Min 30 days | Single AZ risk

Tier 4 — Cold

Glacier Instant Retrieval

Archive data you access maybe once a quarter, but still need in milliseconds when you do. Huge cost savings vs Standard.

Retrieval: milliseconds | Min 90 days | Multi-AZ

Tier 5 — Colder

Glacier Flexible Retrieval

Archive for data accessed maybe 1-2 times per year. Retrieval takes minutes to 12 hours. Much cheaper storage cost.

Retrieval: 1 min–12 hrs (expedited/standard/bulk) | Min 90 days

Tier 6 — Frozen

Glacier Deep Archive

Cheapest storage AWS offers. For compliance archives — 7-year financial records, medical data, legal documents. Retrieve in 12–48 hours.

Retrieval: 12–48 hours | Min 180 days | Lowest $/GB

Tier 7 — Smart

Intelligent-Tiering

AWS watches how often you access each object and automatically moves it between hot, warm, and cold tiers. Small monthly monitoring fee (~$0.0025/1000 objects). No retrieval fee.

Best for: unknown or changing access patterns | No min size for auto-tier

Intelligent-Tiering: The 5 Internal Access Tiers

Intelligent-Tiering isn't just "hot or cold." It has five distinct access tiers that objects move between automatically. The first three are automatic — no configuration needed. The last two are optional archive tiers you opt into for even bigger savings on data you almost never need.

Tier Triggered After Retrieval Time Auto? Real-World Use
Frequent Access Default starting tier Milliseconds Auto Newly uploaded objects, recently accessed data
Infrequent Access No access for 30 days Milliseconds Auto Reports, backups accessed occasionally
Archive Instant Access No access for 90 days Milliseconds Auto Old logs, historical files rarely needed but retrieved instantly
Archive Access No access for 90–730 days Minutes to hours Optional Compliance archives — must enable this tier explicitly
Deep Archive Access No access for 180–730 days Hours (12–48) Optional 7-year regulatory archives — must enable this tier explicitly
💡 Key Insight

If an object that has been moved to Archive Access or Deep Archive Access is retrieved, it moves back to Frequent Access tier automatically. The whole system is self-healing — data returns to the hot tier on access, then cools down again if not accessed further. You never need to manually manage tiers.

When to choose Intelligent-Tiering over manual storage classes: Use it when you genuinely don't know the access pattern of your data. For predictable patterns (logs always cold after 30 days), manual lifecycle rules are cheaper because you avoid the per-object monitoring fee. For unpredictable patterns (user-generated content, documents), Intelligent-Tiering automates the optimisation.

⚠️ Critical Traps — These Appear on the Exam

Trap 1 — Minimum Storage Duration Fees: If you store an object in Glacier Deep Archive and delete it after 30 days, you still pay for 180 days. You're charged for the full minimum regardless of when you delete. Same idea: Standard-IA = pay for 30 days minimum, Glacier Instant/Flexible = 90 days minimum.

Trap 2 — One Zone-IA is NOT the answer for important data: It's 20% cheaper but lives in one AZ. If AWS has an AZ outage, your data could be gone. Only use it for data you can re-create.

Trap 3 — Glacier Instant vs Glacier Flexible: "Instant" means milliseconds. "Flexible" means minutes to hours. The word "instant" vs "flexible" is the clue. The exam will give you a scenario and the retrieval time requirement will tell you which one to pick.

Part 04

Lifecycle Policies — Automating Cost Savings

Why manually move objects between storage classes when AWS can do it for you, automatically?

💡 Why Lifecycle Policies Exist

A company uploads a new product image to S3. For the first 30 days, customers browse it frequently — it must be in S3 Standard (fast access). After 30 days, it's rarely viewed. After 90 days, almost never. After a year, it's only kept for legal archiving.

Without lifecycle rules: Engineers need to manually move objects, write scripts, track dates. Impossible at scale when you have millions of objects.

With lifecycle rules: You define rules once. S3 automatically transitions objects at the right time. Zero ongoing effort. Costs drop automatically as data ages.

Two Types of Actions

Action Type 1

Transition Actions

  • Move objects to a colder storage class after N days
  • Example: Standard → Standard-IA after 30 days
  • Example: Standard-IA → Glacier after 90 days
  • Chain multiple transitions
  • Can only go colder, never back to warmer via lifecycle
Action Type 2

Expiration Actions

  • Delete objects after N days
  • Delete old versions of versioned objects
  • Clean up incomplete multipart uploads
  • Remove expired delete markers automatically
  • Great for log files, temp data, short-lived reports
Typical Lifecycle Transition Flow
⚠️ Lifecycle Traps

Trap 1: You cannot use lifecycle rules to transition objects from a colder class back to a warmer one. E.g., you can't auto-move from Glacier back to Standard. Lifecycle only goes one direction: colder.

Trap 2 — Minimum age requirements: Objects must be in Standard for at least 30 days before transitioning to Standard-IA. S3 enforces this because Standard-IA has a 30-day minimum storage charge anyway — moving earlier would cost more, not less.

Trap 3 — Small objects: As of September 2024, objects smaller than 128KB are NOT transitioned by default lifecycle rules (except to Glacier Flexible and Deep Archive). Be aware of this for small-object workloads.

Part 05

Versioning & Object Lock — Time Machine for Your Data

Protection against the worst kind of mistake: "I accidentally deleted everything."

💡 Why Versioning Was Needed

One Saturday morning, a junior developer at a startup ran a script to clean up old S3 objects. The script had a bug. It deleted the company's entire production database backup instead of the old log files. Without versioning: the data is gone forever. With versioning: every version of every file is preserved. The delete only adds a "delete marker" — the real file is still there, one click away.

This is a real scenario that has killed companies. Versioning is the undo button for S3.

How Versioning Works

When you enable versioning on a bucket, S3 assigns every object a unique version ID. When you "overwrite" a file, S3 doesn't actually replace it — it creates a new version. The old version still exists with its original version ID.

When you "delete" a file, S3 doesn't actually delete it — it adds a delete marker (a special version that makes the object appear deleted). All old versions are still retrievable. To permanently delete an object, you must explicitly delete ALL versions, including the delete marker.

Versioning: How Objects Change Over Time

MFA Delete

Once versioning is enabled, you can add another layer of protection: MFA Delete. This requires a physical MFA (Multi-Factor Authentication) device to permanently delete versions or disable versioning. Even if an attacker steals your IAM credentials, they cannot mass-delete your data without the MFA device in their hands.

📋 Exam Trigger

"Protect S3 data from accidental deletion by employees" → Enable Versioning

"Prevent even root account from deleting data, most secure protection" → Enable Versioning + MFA Delete

"Cannot be deleted for 7 years for regulatory compliance" → Object Lock (Compliance Mode)

Object Lock — WORM Storage

WORM = Write Once, Read Many. Object Lock prevents objects from being deleted or overwritten for a defined retention period — or even indefinitely. This is not "just make it hard to delete" — it is a technical enforcement mechanism where even AWS itself cannot delete your data during the lock period.

This is critical for industries like finance (7-year record retention laws in Ireland), healthcare (patient records), and legal. Dublin-based Fenergo, for example, would use Object Lock to ensure KYC documents cannot be tampered with — satisfying both EU GDPR audit trail requirements and Irish financial regulations.

Mode 1

Compliance Mode

  • Object CANNOT be overwritten or deleted by ANYONE
  • Not even the root AWS account
  • Not even AWS support
  • Retention period cannot be shortened once set
  • Use for: strict regulatory compliance (SEC, FINRA, CBI Ireland)
Mode 2

Governance Mode

  • Most users CANNOT delete or overwrite the object
  • But users with special IAM permission can override
  • Retention period CAN be modified by privileged users
  • Use for: internal policies, accidental deletion prevention
  • Softer compliance where emergency overrides are needed
Legal Hold

Legal Hold

  • No fixed retention period — locked until explicitly released
  • Used during litigation or investigations
  • Any user with s3:PutObjectLegalHold permission can toggle
  • Independent of retention mode — can be combined
  • Real use: "freeze" data during a court case

Glacier Vault Lock — S3 Object Lock's Older Sibling

💡 Why This Is a Separate Thing

S3 Object Lock (covered above) works on S3 buckets. But Amazon S3 Glacier as a standalone service (not S3 Glacier storage class — the original standalone Glacier service) has its own separate locking mechanism called Vault Lock. The exam treats these as distinct concepts.

Think of it this way: Object Lock is for when you store data in S3 buckets and use Glacier as a storage class within S3. Vault Lock is for when you work directly with standalone Glacier vaults (the original Glacier API before it was integrated into S3).

S3 Object Lock

For S3 Buckets

  • Applied to objects inside S3 buckets
  • Two modes: Compliance and Governance
  • Works with any S3 storage class (including Glacier class)
  • Can be set per-object or per-bucket default
  • Managed via S3 API and console
  • Modern standard for new architectures
Glacier Vault Lock

For Glacier Vaults

  • Applied to entire Glacier vaults (not individual archives)
  • Single mode: equivalent to Compliance Mode
  • Policy cannot be changed once locked — permanently immutable
  • Uses a two-step process: initiate lock → validate → complete lock
  • Supports time-based retention AND access control policies
  • WORM guarantee: even AWS cannot override once locked
📋 Exam Trigger

"Prevent compliance data stored in Glacier vaults from being modified even by the account owner" → Glacier Vault Lock Policy

"Enforce WORM directly at the Glacier vault level" → Glacier Vault Lock

Key distinction: S3 Object Lock = object-level, within S3 buckets. Glacier Vault Lock = vault-level, standalone Glacier service. Both are WORM. The question context (bucket vs vault) tells you which one to pick.

Part 06

S3 Replication — Cloning Your Data Across the World

CRR and SRR: why copy your bucket, and what can go wrong if you set it up wrong.

💡 Why Replication Was Needed

A Dublin fintech company stores transaction records in S3 (eu-west-1 region). One day, AWS has a regional incident in Ireland — all services in eu-west-1 become unavailable. The company's customers can't access their data for hours. Revenue stops.

With Cross-Region Replication (CRR), every new object in the Ireland bucket is automatically copied to a backup bucket in Frankfurt (eu-central-1). When Ireland goes down, the Frankfurt copy is immediately available. Zero data loss. Zero manual effort.

Replication Type 1

CRR — Cross-Region Replication

  • Copies objects to a bucket in a DIFFERENT AWS region
  • Primary use: Disaster Recovery (DR)
  • Reduces latency for global users (put copy near them)
  • Meets data residency requirements for regulators
  • Complies with legal requirements to keep copies offsite
Replication Type 2

SRR — Same-Region Replication

  • Copies to another bucket in THE SAME region
  • Use case: separate production and test buckets
  • Use case: aggregate logs from many buckets to one
  • Use case: maintain exact copy for different teams/accounts
  • Same-region = same latency, different IAM/billing account
⚠️ Replication Rules — These Kill Exam Scores

Rule 1 — Versioning is REQUIRED: Both the source AND destination bucket must have versioning enabled before you can set up replication. This is mandatory.

Rule 2 — Replication is NOT retroactive: When you enable replication, only NEW objects uploaded AFTER enabling it will be replicated. Existing objects are NOT copied. To replicate existing objects, you must use S3 Batch Replication separately.

Rule 3 — Delete markers are NOT replicated by default: If you delete an object in the source bucket, the delete marker is NOT automatically replicated to the destination. This is configurable. The reason: to protect against malicious or accidental deletions spreading to your backup.

Rule 4 — No chaining: If bucket A replicates to bucket B, and bucket B replicates to bucket C — an object added to A does NOT automatically go to C. Each replication rule is independent. A to B happens, but B to C is a separate replication rule triggered only if an object is directly added/modified in B.

Part 07

S3 Performance — Speed at Scale

How to move terabytes fast, how to read only part of a file, and how S3 handles millions of requests per second.

S3 Baseline Performance (No Config Needed)

S3 can handle 3,500 PUT/COPY/POST/DELETE requests per second per prefix and 5,500 GET/HEAD requests per second per prefix. These limits are per prefix — not per bucket. A prefix is the path before the object key, like /2025/january/.

The performance trick: If your workload needs more than 5,500 reads/second, spread your objects across multiple prefixes. Each prefix gets its own 5,500/s quota. Ten prefixes = 55,000 reads/second. This is why production S3 architectures use random or hashed prefixes rather than dates.

💡 Industry Pattern — Avoiding Hot Prefixes

A Dublin media company streams video thumbnails from S3. They have 10 million videos. Naming files /thumbnails/2025-01-15/video1.jpg puts all January videos under one prefix → hits the 5,500/s limit instantly. Instead, they hash the video ID: /th/a3/video-abc123.jpg — distributing requests across thousands of 2-character prefixes. Zero bottleneck at any scale.

Multipart Upload — The Must-Use for Large Files

💡 The Problem It Solves

You're uploading a 10GB video file to S3. Halfway through, your network hiccups. The upload fails. You start again from zero. That's 5GB of wasted bandwidth and time. With a slow connection, a 10GB upload might take 2 hours — and if it fails at 90% complete, you restart everything.

Multipart Upload splits your large file into small chunks (minimum 5MB each, up to 10,000 parts). Each chunk is uploaded independently in parallel. If one chunk fails, only that chunk is retried. When all chunks arrive, S3 reassembles them. Result: massive files upload faster and more reliably.

  • Recommended for files over 100MB
  • Required (mandatory) for files over 5GB
  • Parallel uploads = much faster on fast connections
  • Failed parts retry without restarting from zero
  • You can pause and resume large uploads
  • Use lifecycle rules to clean up incomplete multipart uploads — abandoned parts waste money

S3 Transfer Acceleration — Fast Uploads from Across the Globe

💡 The Problem It Solves

Your S3 bucket is in Dublin (eu-west-1). A user in Sydney, Australia needs to upload a 500MB file. Over the public internet, this upload travels through multiple hops, ISPs, and undersea cables — each hop adds latency and potential for packet loss. The upload is slow and unreliable.

With Transfer Acceleration, the upload goes to the nearest CloudFront edge location (probably in Sydney or Singapore). From there, it travels over AWS's private backbone network — which is much faster and more reliable than the public internet — to your bucket in Dublin. Think of it as putting your data on an express motorway instead of city roads.

Transfer Acceleration vs Standard Upload
📋 Exam Trigger

"Global users need to upload large files to a single S3 bucket quickly" → Enable S3 Transfer Acceleration

"Upload from geographically distant location to S3 is slow" → S3 Transfer Acceleration

Note: Transfer Acceleration works for both uploads AND downloads. It has additional cost on top of standard S3 pricing.

Byte-Range Fetches — Downloading Part of a File

What if you have a 10GB log file in S3 and you only need the last 100MB? Normally you'd download the whole 10GB first. That's expensive and slow. Byte-Range Fetches let you request only a specific byte range from S3.

Even better: you can request multiple byte ranges in parallel, just like multipart upload in reverse. This is how Netflix downloads video — not the whole file, just the next few seconds worth of data. And it's how S3 Select works under the hood.

  • Request: GET /my-log-file.gz Range: bytes=0-104857600 — gets first 100MB
  • Parallelise multiple range requests → dramatically faster for large files
  • Use case: retrieve file header only (e.g., first 50 bytes to check file type)
  • Use case: resume interrupted downloads from where they stopped
  • Use case: read only the beginning of large CSV files for data processing

S3 Select & Glacier Select — SQL Inside S3

💡 Why This Exists

You have a 5GB CSV file in S3 containing 100 million rows. You need just 500 rows that match a specific condition. Without S3 Select, you download the entire 5GB, then filter in your application. That costs money (data transfer) and time.

With S3 Select, you send a SQL-like query to S3 itself: SELECT * FROM S3Object WHERE country = 'IE'. S3 runs the filter server-side and returns only the matching 500 rows. Up to 400% faster, 80% cheaper than downloading everything first.

# S3 Select example — query a CSV file without downloading it aws s3api select-object-content \ --bucket fenergo-transactions \ --key 2025/all-transactions.csv \ --expression "SELECT * FROM S3Object s WHERE s.country = 'IE'" \ --expression-type SQL \ --input-serialization '{"CSV": {"FileHeaderInfo": "USE"}, "CompressionType": "NONE"}' \ --output-serialization '{"CSV": {}}' \ output.csv
Part 08

S3 Event Notifications — React to What Happens in Your Bucket

Every upload, delete, or restore can trigger an automated workflow. This is where S3 becomes a first-class event source.

💡 Why Event Notifications Were Needed

A company lets users upload profile photos to S3. After upload, the company needs to: resize the image to 3 different sizes, run it through a content moderation AI to check for inappropriate content, send a notification email to the user, and update a database with the new photo URL.

Without event notifications, you'd need a polling process: "check S3 every 5 seconds for new files" — wasteful and adds latency. With event notifications, the moment an object lands in S3, it immediately fires an event to trigger all these workflows automatically.

Supported Events and Destinations

Event Type Triggers When... Destination Real-World Use
s3:ObjectCreated:* Object uploaded (PUT, POST, Copy, multipart complete) Lambda, SQS, SNS, EventBridge Process uploaded documents, generate thumbnails
s3:ObjectRemoved:* Object deleted or delete marker created Lambda, SQS, SNS, EventBridge Audit deletion, clean up related DB records
s3:ObjectRestore:* Object restored from Glacier Lambda, SQS, SNS, EventBridge Notify when archived document becomes available
s3:Replication:* Replication fails or threshold exceeded Lambda, SQS, SNS, EventBridge Alert when cross-region backup fails
S3 Event Notification Fan-Out Architecture
📋 Exam Insight

If you need to deliver S3 events to MULTIPLE destinations simultaneously → Use SNS fan-out (SNS to multiple SQS queues). Direct S3 notification to SQS goes to one queue only.

EventBridge as destination is the most flexible option — it supports richer content-based filtering, replay, schema registry, and many more AWS/SaaS destinations. When the question mentions "complex routing rules" or "audit all S3 API calls including from the console" → EventBridge.

Versioning for notifications: For non-versioned buckets, if two writes happen simultaneously to the same key, only one notification may fire. Enable versioning to guarantee one notification per write.

Part 09

S3 Encryption — Protecting Data at Rest and in Transit

Four different ways to encrypt S3 objects — each with a different trust model. The exam absolutely loves these.

💡 Why Encryption Matters — Especially in Ireland

Fenergo, the Dublin fintech company, handles KYC documents for 26 of the world's top 50 financial institutions. Those documents contain names, passport numbers, addresses, financial history — extremely sensitive data. Under EU GDPR and the Irish Central Bank of Ireland's regulations, this data must be encrypted at rest and in transit. A data breach can mean fines of up to 4% of global annual revenue.

S3 offers four server-side encryption options. The choice determines who controls the encryption keys — which is the critical security and compliance decision. The more control you want over keys, the more responsibility you take on.

The Four Encryption Options

SSE-S3

AWS Manages Everything

How: AWS creates and manages encryption keys. AES-256 encryption. Default for all new S3 objects since Jan 2023.

Who controls keys: AWS (you have zero visibility into the keys)

Audit trail: None for key usage

Mechanism: Header x-amz-server-side-encryption: AES256

Use when: You need encryption but don't care about key management. Lowest friction.

SSE-KMS

KMS Keys, You're in Control

How: Uses AWS KMS keys (either AWS-managed or customer-managed CMK). Full key rotation control.

Who controls keys: You (through KMS policies + IAM)

Audit trail: Every key usage logged in CloudTrail — who accessed what, when

Mechanism: Header x-amz-server-side-encryption: aws:kms

Use when: Regulatory requirement to control and audit key usage. Financial services, healthcare.

SSE-C

You Provide the Key, AWS Encrypts

How: You provide your own key with every PUT/GET request. AWS encrypts/decrypts then discards your key — never stores it.

Who controls keys: 100% you (AWS never stores the key)

Requirement: Must use HTTPS (keys sent over HTTP = massive security risk)

Mechanism: Send key in request header every time

Use when: You already have an external key management system and can't use KMS. Or extreme compliance requirements.

DSSE-KMS

Double Encryption Layer

How: Applies TWO independent KMS encryption layers on the same object. Extra protection.

Who controls keys: You (through KMS)

Use when: Highest regulatory compliance requirements (some US government, defence, extreme PCI-DSS).

Cost: Twice the KMS API calls → twice the cost. Only use when the compliance standard explicitly requires it.

Client-Side Encryption — Encrypt Before It Leaves Your Machine

💡 The Key Difference From Server-Side Encryption

All four SSE options above (SSE-S3, SSE-KMS, SSE-C, DSSE-KMS) are server-side — meaning your data travels to AWS first, then AWS encrypts it before writing to disk. The data is in plaintext during the network journey from your application to S3 (though TLS protects it in transit).

Client-side encryption flips this: you encrypt the data on your own machine or server before uploading it to S3. S3 receives only encrypted bytes and has no idea what the actual data is. AWS cannot decrypt it — even if they wanted to, even if compelled by a court order. Total data sovereignty.

How It Works

Client-Side Flow

  • Your application generates/loads an encryption key
  • Encrypts the file locally (e.g., AES-256-GCM)
  • Uploads the encrypted blob to S3
  • S3 stores opaque encrypted bytes — no metadata about content
  • To read: download blob, decrypt locally with your key
  • AWS SDKs have built-in client-side encryption helpers
When To Use

Use Cases

  • "Zero trust" architecture — don't trust the cloud provider
  • Defence/intelligence workloads where even AWS access is unacceptable
  • Highly sensitive IP (pharmaceutical formulas, trade secrets)
  • Regulations that prohibit any third party from even theoretically accessing data
  • Multi-cloud strategies where you need portable encryption
Trade-offs

The Costs

  • You own key management entirely — lose the key, lose the data forever
  • S3 features like Select, Batch Operations cannot work on encrypted blobs
  • Search, indexing, and analytics on S3 content become impossible without decrypt
  • More application code and complexity to maintain
  • Key rotation requires re-encrypting all existing objects
📋 Exam Trigger

"Data must be encrypted before it ever reaches AWS" → Client-Side Encryption

"AWS must have absolutely no ability to decrypt data, even theoretically" → Client-Side Encryption

Contrast: SSE-C is often confused with client-side encryption. SSE-C still sends plaintext to AWS (AWS does the encrypting). Client-side means AWS receives already-encrypted bytes. The encryption happens at the client — before the network call.

⚠️ SSE-KMS Has a Hidden Scaling Trap

Every S3 request using SSE-KMS makes a KMS API call. KMS has default quota limits (5,500 to 10,000 API calls per second depending on region). If you have 1,000 clients downloading SSE-KMS encrypted objects simultaneously, you're making 1,000 KMS API calls per second. You can hit KMS throttling limits, causing download failures!

Solution: Enable S3 Bucket Keys. When you enable S3 Bucket Keys, S3 generates a short-lived key from your KMS key and caches it. Instead of one KMS call per object access, the cache handles most requests — dramatically reducing KMS API calls (up to 99% reduction) and cost.

Enforcing Encryption — Deny Unencrypted Uploads

Saying "enable encryption" on a bucket doesn't prevent someone from uploading an unencrypted object if they don't send the encryption header. To enforce encryption, you need a bucket policy.

{ // Deny any PUT that doesn't use SSE-KMS encryption // This forces ALL uploads to use KMS encryption } { "Version": "2012-10-17", "Statement": [{ "Effect": "Deny", "Principal": "*", "Action": "s3:PutObject", "Resource": "arn:aws:s3:::my-secure-bucket/*", "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" } } }] }
📋 Exam Trigger

"Force encryption on all S3 uploads" → Bucket Policy with Deny condition on missing encryption header

"Audit who accessed specific S3 objects using which encryption key" → SSE-KMS + CloudTrail

"Compliance requires you never share encryption keys with AWS" → SSE-C (you provide keys, AWS never stores them)

Encryption in Transit — HTTPS is Mandatory

Data moving between your application and S3 (in transit) is encrypted using TLS/HTTPS. S3 supports both HTTP and HTTPS endpoints, but you should always use HTTPS. To enforce it, add a bucket policy that denies any non-HTTPS request.

// Deny HTTP requests — only allow HTTPS (TLS/SSL) { "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"], "Condition": { "Bool": { "aws:SecureTransport": "false" } } }

S3 Access Control — Who Can Access What

Four different mechanisms, each solving a different access control problem. Understanding when to use which is key.

💡 The Access Control Problem in the Real World

A Dublin e-commerce startup has an S3 bucket with: product images (should be public), customer invoices (only the customer themselves), internal reports (only the finance team), and raw transaction logs (only the analytics pipeline). All in the same AWS account. How do you control access to each type of object differently?

AWS gives you four tools. Each has a different scope and purpose. The exam will test whether you know which tool to reach for in each scenario.

The Four Access Control Mechanisms

1
Block Public Access (Account-Level Safety Net)

This is a master override. When enabled (it's ON by default for all new buckets since April 2023), it prevents any S3 bucket or object in your account from being made public — even if a bucket policy or ACL says "allow public access". Think of it as a circuit breaker. The exam answer for "ensure NO S3 data in this account is ever publicly accessible" is always: Enable Block Public Access at the account level. AWS now enables this by default on all new buckets.

2
IAM Policies (Identity-Based Control)

Attached to IAM users, groups, or roles. Controls what those identities can do. Example: "Alice's IAM policy allows her to GET objects from the finance-reports bucket". IAM policies work great for controlling access within your own AWS account. For cross-account access, you still need a bucket policy on the destination bucket to grant trust to the external account. Key insight: IAM policies and bucket policies are evaluated together — both must allow access for a request to succeed.

3
Bucket Policies (Resource-Based, the Workhorse)

JSON policies attached directly to the S3 bucket. The most powerful and commonly used access control mechanism. Can grant/deny access to specific AWS accounts, IAM users, roles, IP ranges, VPCs, and even anonymous (public) users. Bucket policies are how you grant cross-account access, enforce HTTPS-only, require specific encryption, allow public access for a static website, and much more. For any scenario where you need to control access based on who the caller is (not just what they're doing), bucket policy is your tool.

4
ACLs — Access Control Lists (Legacy, Use Sparingly)

Pre-IAM era access control. Coarse-grained (read/write/full control permissions per bucket or object). AWS now disables ACLs by default on new buckets. You may see ACL questions on the exam in legacy context, but the answer is almost always to use bucket policies instead. The one remaining ACL use case: granting AWS services like CloudFront or S3 replication permissions to access objects — though even this is being replaced by bucket policies and OAI/OAC.

Access Control Decision Flow

S3 Access Points — Simplifying Shared Bucket Access

💡 The Problem Access Points Solve

A company has one S3 bucket with data from multiple applications: the analytics team needs read access to certain prefixes, the data science team needs read/write to a different set, and the compliance team needs read access to audit logs. Managing this via bucket policy becomes a massive JSON blob that's hard to read, audit, and change safely.

S3 Access Points solve this by creating named virtual entry points into the bucket — each with its own hostname, IAM policy, and VPC restrictions. Instead of one complex bucket policy, you have clear, separated access policies per team or application.

  • Each access point has its own DNS hostname: my-ap.s3-accesspoint.eu-west-1.amazonaws.com
  • Each access point has its own IAM policy — much easier to manage per-team or per-app
  • You can restrict an access point to a specific VPC (private access only)
  • Access points can only grant permissions, never more than the bucket policy allows
  • You can have multiple access points per bucket — one per application, department, or partner

Presigned URLs — Temporary Access to Private Objects

💡 The Problem Presigned URLs Solve

Your application stores user documents in a private S3 bucket (no public access). A logged-in user wants to download their own tax statement PDF. You have two bad options: make the bucket public (everyone can download everyone's files — terrible!) or route the download through your application server (your server becomes a bottleneck, extra cost).

The good option: your server generates a presigned URL. This is a special URL that includes an encrypted signature embedding your AWS credentials. Anyone with this URL can download the exact object — but only for a limited time window (you choose: 1 minute to 7 days). After expiry, the URL is dead.

# Generate a presigned URL that expires in 3600 seconds (1 hour) aws s3 presign s3://fenergo-kyc-docs/client-12345/passport.pdf \ --expires-in 3600 \ --region eu-west-1 # Returns something like: https://fenergo-kyc-docs.s3.eu-west-1.amazonaws.com/client-12345/passport.pdf ?X-Amz-Algorithm=AWS4-HMAC-SHA256 &X-Amz-Credential=AKIAIOSFODNN7EXAMPLE/20250115/eu-west-1/s3/aws4_request &X-Amz-Expires=3600 &X-Amz-Signature=....(unique signature)...
⚠️ Presigned URL Traps

Trap 1 — Permissions are checked at USE time, not generation time: If you generate a URL but then revoke the IAM permissions of the user who generated it before the URL expires, the URL immediately stops working even before its expiry time.

Trap 2 — Expiry tied to credential type: If the URL is generated by an IAM role (like an EC2 instance profile), the URL expires when the role's temporary credentials expire — which can be as soon as 1 hour, regardless of the expiry you set. For longer-lived URLs, use IAM user permanent credentials or set a short expiry and regenerate.

Trap 3 — Works for PUT too: Presigned URLs work for both GET (download) and PUT (upload). You can let users upload directly to S3 without proxying through your server — saving bandwidth costs.

CORS — Cross-Origin Resource Sharing

💡 What CORS Is and Why It's Needed

Imagine you have a web app at https://app.fenergo.com. Your JavaScript code wants to fetch a PDF from https://fenergo-docs.s3.amazonaws.com. The browser says: STOP. This is a cross-origin request — the "origin" (app.fenergo.com) is different from the resource origin (s3.amazonaws.com). The browser blocks it by default for security.

CORS lets you configure S3 to tell the browser: "It's OK, requests from app.fenergo.com are allowed." Without this, your web app simply cannot load assets directly from S3.

// S3 CORS configuration example [{ "AllowedHeaders": ["*"], "AllowedMethods": ["GET", "PUT", "POST"], "AllowedOrigins": ["https://app.fenergo.com"], "ExposeHeaders": ["ETag"], "MaxAgeSeconds": 3000 }]
📋 Exam Trigger

"Static website hosted on S3 in one bucket needs to load assets from a second S3 bucket" → Enable CORS on the second bucket to allow requests from the first bucket's origin.

"Browser is getting 'No Access-Control-Allow-Origin' error when loading S3 resources" → CORS policy missing or incorrect on the S3 bucket.

S3 Object Lambda — Transform Data on the Fly

S3 Object Lambda intercepts GET requests to S3 and runs a Lambda function before the data reaches the requester. You don't need to store multiple versions of the same data.

Example: You store customer reports in S3. When the analytics team GETs a report, they get the full data. When the customer GETs the same report via S3 Object Lambda, your Lambda function redacts sensitive fields (like full credit card numbers, showing only last 4 digits) before returning it. One source of truth, multiple filtered views.

  • Automatically watermark documents based on the requesting user
  • Decompress Gzip files on the fly for applications that can't handle compression
  • Enrich data with information from other sources before returning it
  • Filter out rows in a CSV that the requester doesn't have access to see
  • Convert between formats (CSV → JSON) without duplicating storage

S3 Static Website Hosting

S3 can serve HTML, CSS, and JavaScript files directly as a static website. No servers needed. Your website URL becomes: http://my-bucket.s3-website.eu-west-1.amazonaws.com.

This is how many companies serve their landing pages, documentation sites, and single-page applications (React/Vue/Angular built apps). Typically paired with CloudFront for HTTPS and a custom domain. For dynamic content (databases, server-side logic), you still need a backend — S3 is static only.

📋 Exam Trigger

"Host a static website with no servers at minimum cost" → S3 Static Website + CloudFront

"Error 403 on static website" → Usually means Block Public Access is still on, or bucket policy doesn't allow public GetObject.

S3 Batch Operations — Process Millions of Objects at Once

Need to copy 500 million objects from one bucket to another? Encrypt all existing objects in a bucket that weren't encrypted before? Run a Lambda function against every object in a 10TB bucket? S3 Batch Operations handles this at massive scale — no custom code or scripts needed.

You create a "job" that specifies a list of objects (from an S3 Inventory report) and an operation (copy, invoke Lambda, restore from Glacier, tag, etc.). AWS runs the job in parallel across its infrastructure and gives you a completion report.

S3 Requester Pays — Shifting Transfer Costs to the Downloader

💡 The Problem It Solves

A Dublin research institution stores 50TB of public genomics data in S3. Scientists worldwide download from it daily. Without Requester Pays, the research institution pays for all data transfer out of S3 — which can run into thousands of euros per month for popular datasets. They're paying to give data away.

Requester Pays flips the billing model. The research institution pays for storage. The person downloading pays for the data transfer and request costs. This is how large public datasets (government open data, scientific research, geospatial data) are distributed without breaking the publisher's budget.

Normal S3 (Default)

Bucket Owner Pays Everything

  • Bucket owner pays: storage + data transfer + requests
  • Works fine when you control who downloads and how much
  • Problem: public datasets with thousands of external downloaders
  • Transfer costs scale with popularity — can become unsustainable
Requester Pays Enabled

Downloader Pays Transfer

  • Bucket owner pays: storage only
  • Requester pays: data transfer + request costs
  • Requester MUST be authenticated (no anonymous access allowed)
  • Requester must include x-amz-requester-pays header
  • Used by: AWS public datasets, open data initiatives, large media
⚠️ Requester Pays Constraints

Anonymous access is automatically blocked when Requester Pays is enabled. Every requester must have valid AWS credentials — because AWS needs to know who to bill. This means you cannot serve anonymous public content with Requester Pays. Use standard public bucket policy for anonymous access instead.

Exam trigger: "Company hosts a large public dataset in S3, wants external users to pay for their own downloads" → Enable Requester Pays on the bucket.

VPC Gateway Endpoint — Private S3 Access Without the Internet

💡 The Problem It Solves

Your EC2 instances are in a private subnet (no internet gateway, no NAT). They need to read and write to S3 — but S3's endpoint is on the public internet. Without a VPC Endpoint, your private instances simply cannot reach S3 at all.

Even if your instances ARE in a public subnet and CAN reach the internet, all S3 traffic going via the internet means: data transfer costs (egress fees), potential security exposure (traffic on public internet), and latency. A VPC Gateway Endpoint for S3 solves all of this: your EC2 instances talk to S3 over AWS's private network, never touching the internet.

Without VPC Endpoint

S3 via Internet

  • Private subnet → NAT Gateway → Internet → S3
  • OR: Public subnet → Internet Gateway → Internet → S3
  • Data transfer charges apply (NAT Gateway + internet egress)
  • Traffic traverses public internet — security exposure
  • Cannot be used from private subnets without NAT
With Gateway Endpoint

S3 via AWS Private Network

  • EC2 (any subnet) → Route table entry → S3 directly
  • Completely free — no hourly charge, no data charge
  • Traffic stays within AWS network, never public internet
  • Works from private subnets without NAT Gateway
  • Bucket policies can restrict access to only VPC traffic
  • Gateway Endpoints are free — unlike Interface Endpoints (~$0.01/hr/AZ)
  • Only works for S3 and DynamoDB — everything else needs Interface Endpoint (PrivateLink)
  • Works by adding a route table entry in your VPC routing table — not an ENI
  • Scope is regional — one endpoint connects to all S3 buckets in that region
  • You can restrict bucket access to only requests coming through the endpoint (bucket policy condition)
// Bucket policy: only allow access from within your VPC (via Gateway Endpoint) // Blocks all internet access to the bucket — even from valid IAM users outside VPC { "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": ["arn:aws:s3:::my-private-bucket", "arn:aws:s3:::my-private-bucket/*"], "Condition": { "StringNotEquals": { "aws:SourceVpc": "vpc-0a1b2c3d4e5f67890" } } }
📋 Exam Triggers

"EC2 in private subnet needs to access S3 without internet, cheapest option" → VPC Gateway Endpoint for S3 (free)

"Prevent S3 bucket from being accessed outside your VPC" → Bucket policy with aws:SourceVpc condition + Gateway Endpoint

"Connect to S3 privately but need it to work for all AWS services" → Interface Endpoint (PrivateLink) — paid

The key comparison always tested: Gateway Endpoint = free, S3+DynamoDB only, uses route tables. Interface Endpoint = paid, almost every AWS service, uses ENIs in your subnet.

S3 Logging & Monitoring — Knowing What's Happening

Server Access Logging, Storage Lens, and how S3 monitoring differs from CloudTrail — a common exam confusion point.

S3 Server Access Logging — The Detailed HTTP Log

💡 Why This Is Different From CloudTrail

CloudTrail records API calls made to the S3 management plane — bucket creation, policy changes, lifecycle rule updates, replication configuration. It's the "who configured what" log.

S3 Server Access Logging records every individual HTTP request made to the data plane — every GET, PUT, DELETE of individual objects. It's the "who accessed which file, when, from where" log. This is what you need for: security forensics (which IP downloaded this file?), billing analysis (which app is making the most requests?), and detailed access auditing.

CloudTrail S3 Logging

Management Plane Events

  • Bucket-level API calls (CreateBucket, DeleteBucket)
  • Policy changes, ACL changes, lifecycle changes
  • Cross-account access events
  • Free for management events by default
  • Data events (object-level) require additional configuration + cost
  • Near real-time delivery (minutes)
S3 Server Access Logging

Data Plane Events (Every HTTP Request)

  • Every GET, PUT, DELETE of every object
  • Requester IP address, request time, HTTP status code
  • Bytes transferred, referrer, user agent
  • Logs stored in a different S3 bucket (never same bucket)
  • Best-effort delivery (not guaranteed for every request)
  • Can have hours of delay before logs appear
⚠️ Critical Setup Rules

Rule 1 — Never log to the same bucket you're monitoring. This creates an infinite loop: logging a request creates a log file, which creates another log request, which creates another log file... The destination bucket must be a different bucket in the same region.

Rule 2 — Delivery is best-effort, not guaranteed. Some requests may not appear in logs. For guaranteed capture of all access events, use CloudTrail Data Events (with additional cost) instead of Server Access Logging.

Rule 3 — Logs can be hours behind. Don't use Server Access Logs for real-time alerting. Use CloudWatch metrics or EventBridge for that. Access logs are for retrospective analysis.

S3 Storage Lens — Organisation-Wide S3 Analytics

💡 The Problem It Solves

A company has 500 S3 buckets across 12 AWS accounts and 6 regions. The CTO asks: "How much of our S3 data is actually being accessed? Are we wasting money on Standard storage for data no one reads? Which buckets are growing fastest? Are any buckets not encrypted?"

Without Storage Lens: answering these questions requires running custom scripts across every account and region — a multi-week engineering project. With S3 Storage Lens: you get a single interactive dashboard showing all of this automatically, across your entire AWS Organisation, updated daily.

Storage Lens works at three levels of scope: individual account, multi-account (AWS Organizations), and optionally filtered by region, bucket, or prefix. It aggregates dozens of metrics into an interactive console dashboard and can also export data to an S3 bucket for custom analysis with Athena or QuickSight.

Free Tier Metrics

Default Dashboard

  • Total storage used per bucket and region
  • Object count trends over time

  • Buckets missing versioning, MFA Delete, or encryption

  • 15-day data retention in the dashboard
Advanced Metrics (Paid)

Deep Insights

  • Data protection: detailed encryption, replication coverage stats

🏢 Dublin Industry Architecture — Real-World Deep Dive

Fenergo-Style KYC Document Platform in Dublin

Let's build the architecture of a real Dublin-based fintech company's KYC (Know Your Customer) document management system — the type of platform Fenergo (headquartered on Mayor Street, Dublin 1) builds for the world's largest banks. This is a concrete example of every S3 concept you've just learned, applied in production.

📋 The Business Context

Fenergo helps financial institutions onboard corporate clients. When a bank onboards a new corporate client, it needs to collect and verify: a company's certificate of incorporation, directors' passport copies, proof of address, source of funds documentation, and audited financial statements. Under EU GDPR, Irish Central Bank rules, and MiFID II regulations, these documents must be: encrypted at rest, access-logged, retained for at least 7 years, non-deletable during retention period, and geographically restricted to the EU.

🏗️ The S3 Architecture

Primary Storage

Single S3 bucket in eu-west-1 (Dublin). Versioning enabled. All objects encrypted with SSE-KMS using Customer Managed Keys. Block Public Access: ON at account level.

DR Replication

CRR to eu-central-1 (Frankfurt). Meets EU data residency: data never leaves EU. RPO near-zero. If Dublin AWS region fails, Frankfurt has the full copy immediately.

Compliance Lock

S3 Object Lock in Compliance Mode. Retention: 7 years. Even if an attacker compromises the root account, they cannot delete KYC documents during this window. Satisfies Irish law.

Cost Lifecycle

Day 0–365: S3 Standard. Day 365–2555: S3 Glacier Instant Retrieval (cheap, but available in ms for compliance audits). Day 2555+: Glacier Deep Archive. Auto-managed via lifecycle rules.

Access Control

Three S3 Access Points: one for Document Upload Service, one for Compliance Team (read-only), one for Analytics Pipeline (VPC-restricted). Each with separate IAM policies.

Secure Distribution

When a bank officer needs to review a document, the backend generates a Presigned URL valid for 15 minutes. User gets a direct, time-limited download link. No object is ever public. Audit trail in CloudTrail.

Event Processing

Every S3 upload fires an EventBridge event → triggers Lambda for OCR/AI extraction → results stored in DynamoDB → SQS queue for downstream compliance workflow processing.

Audit & Monitoring

S3 Server Access Logging enabled. CloudTrail logs every API call. Macie scans for unexpected PII in non-KYC buckets. CloudWatch Alarms on bucket size and API error rates.

💬 What an Engineer Would Say in an Interview

"We used SSE-KMS with customer-managed keys rather than SSE-S3 because the Central Bank of Ireland requires financial institutions to demonstrate key management control. We rotate the KMS keys annually and every key usage is logged in CloudTrail — giving us a full audit trail of who accessed which document with which key. We also enabled S3 Bucket Keys to avoid KMS throttling at scale, since we have ~50,000 document access events per day."

"For disaster recovery, we configured CRR to Frankfurt specifically because some of our bank clients are domiciled in Germany under German regulatory jurisdiction. Having a German region replica satisfies their local compliance requirements without duplicating the operational stack. The replication is configured with Replication Time Control (RTC) which gives us a guaranteed 15-minute RPO with SLA, rather than the best-effort default replication."

"The Object Lock in Compliance Mode was a non-negotiable. When the European Banking Authority audits a client bank, they need to verify that document records haven't been tampered with. Compliance Mode means that even if our own DBA account is compromised, those 7-year-locked documents cannot be deleted. That's the same guarantee as physical tape archiving, delivered in the cloud."
