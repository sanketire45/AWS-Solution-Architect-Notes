ECS · Part 1

First — What Even Is a Container?

Before ECS makes sense, you need the one idea it orchestrates.

"It works on my machine" is the oldest excuse in software. A container kills it.

A container packages your application together with everything it needs to run — the code, the runtime, libraries, settings — into one standard unit (an image, usually built with Docker). That image runs identically on a laptop, a test server, or a thousand production machines. No more "missing dependency" surprises.

Containers are like lightweight virtual machines, but instead of each carrying a full operating system, they share the host's OS kernel. That makes them tiny, fast to start (seconds), and dense — you can pack many containers on one server. Perfect for microservices.

🔑 The vocabulary

Image = the immutable package (a recipe). Container = a running instance of an image. Registry = where images are stored (AWS's is ECR). Orchestrator = the system that runs, schedules, scales, heals, and networks your containers across a fleet (AWS gives you ECS and EKS).

ECS · Part 2

Why ECS — Running Containers at Scale, the AWS Way

Running one container is easy. Running hundreds across many servers — deciding which server has room, restarting crashed ones, rolling out new versions, load-balancing traffic — is hard. That job is orchestration.

Amazon ECS (Elastic Container Service) is AWS's own container orchestrator. You tell it "run 10 copies of this container and keep them healthy behind a load balancer," and ECS handles placement, health, scaling, and recovery. It's deeply integrated with AWS (IAM, ALB, CloudWatch, VPC) and simpler than Kubernetes — no control plane to learn.

ECS · Part 3

Task Definitions, Tasks & Services

  • Task Definition — the blueprint (JSON): which image, how much CPU/memory, ports, environment variables, the IAM task role, logging. Like a recipe for one unit of your app.
  • Task — a running instance of a task definition (one or more containers running together).
  • Service — keeps a desired number of tasks running, replaces failed ones, and registers them with an ALB target group. This is how you run a long-lived web service.
  • Cluster — the logical group of compute (EC2 instances or Fargate) your tasks run on.
ECS · Part 4

Launch Types — EC2 vs Fargate

The single biggest ECS decision: who manages the servers?

EC2 Launch Type

You manage the nodes

  • You provision & patch a fleet of EC2 instances
  • More control; can use Spot, special instance types, GPUs
  • Cheaper per unit at steady high utilization
  • More operational work
Fargate Launch Type

Serverless — no nodes

  • No EC2 to manage — AWS runs the compute
  • Pay per vCPU + memory your task uses
  • Great for spiky/variable or low-density workloads
  • "No infrastructure to manage" → Fargate
🔑 The default exam answer

"Run containers with no infrastructure to manage, no Kubernetes requirement" → ECS on Fargate. It's the lowest-operational-overhead way to run containers on AWS. (Full Fargate detail in the next tab.)

ECS · Part 5

Scaling, Load Balancing & IAM Roles

  • Service Auto Scaling — target-tracking on CPU, memory, or ALB request count per target; ECS adds/removes tasks automatically. (On EC2 launch type, Cluster Capacity Providers also scale the EC2 layer.)
  • ALB integration — the Service registers tasks with a target group; dynamic port mapping lets many tasks share a host.
  • Task Role (the important one) — IAM permissions for your app inside the container (e.g. read an S3 bucket). The right way to give containers AWS access — never bake credentials into the image.
  • Task Execution Role — lets ECS pull the image from ECR and push logs to CloudWatch.
💵 ECS itself is free — you pay only for the EC2 or Fargate underneath. A small Fargate task (1 vCPU + 2 GB) running 24/7 ≈ $36/month; the same on Fargate Spot is ~70% less.

Fargate · Part 1

Why Fargate — Containers Without Servers

Even with ECS, the EC2 launch type still leaves you babysitting a fleet of instances. Fargate removes them.

With the EC2 launch type you still have to provision the cluster, patch the OS, right-size the instances, and worry about utilization. That's undifferentiated heavy lifting.

AWS Fargate is a serverless compute engine for containers. You hand AWS a task definition and it runs your container — no EC2 instances to provision, patch, or scale. You don't see servers at all; you just specify how much vCPU and memory each task needs, and pay for exactly that, per second, while it runs. Fargate works with both ECS and EKS.

💡 The point

Fargate is to containers what Lambda is to functions: the infrastructure disappears. It's the answer whenever a scenario says "no servers/infrastructure to manage," "fully managed," "least operational overhead" for containers.

Fargate · Part 2

How It Works & What It Costs

  • Pick the vCPU + memory for each task; Fargate provisions isolated compute for it on demand.
  • Each task runs in its own kernel-isolated environment (strong security boundary between tasks).
  • Fargate Spot runs interruptible tasks at up to ~70% off; Graviton (ARM) Fargate is ~20% cheaper.
  • Scales precisely with your tasks — no idle EC2 capacity to pay for.
💵 Fargate ≈ $0.04048 / vCPU-hour + $0.004445 / GB-hour (per-second billing). A 0.5 vCPU + 1 GB microservice 24/7 ≈ $18/month; a 1 vCPU + 2 GB task ≈ $36/month. At steady, high, predictable load, raw EC2 can beat it per-unit — which is the core trade-off.

Fargate · Part 4

Lambda vs Fargate — the other serverless choice

🔑 When Fargate beats Lambda

Both are serverless, but Lambda is for short, event-driven functions (max 15 minutes, 10 GB RAM, package limits). Reach for Fargate when the workload needs: a runtime longer than 15 minutes, more than 10 GB RAM, a GPU, persistent/long-lived connections, a large container image (>10 GB), or steady high utilization. "Long-running containerized service, no servers" → Fargate; "short event-driven function" → Lambda.

ECR · Part 1

Why ECR — Where Your Images Live

Containers run from images. Those images need a secure, managed home that ECS/EKS can pull from.

You build a Docker image. Now where do you store it so your AWS compute can pull and run it — privately, securely, and fast?

Amazon ECR (Elastic Container Registry) is AWS's fully-managed Docker container registry. You push images to it; ECS, EKS, and Fargate pull from it. It's private by default (secured with IAM), encrypted, highly available, and tightly integrated with the rest of AWS — no registry servers to run. (ECR also offers a public gallery.)

ECR · Part 2

How It Works

  • Push: docker push to your private ECR repo (auth via IAM token).
  • Pull: ECS/EKS task execution role pulls the image at launch.
  • Access controlled by IAM (and repo policies); images encrypted at rest (KMS optional).
  • Integrates with the whole CI/CD chain (CodeBuild/CodePipeline, GitHub Actions, etc.).
ECR · Part 3

Scanning, Lifecycle Policies & Replication

  • Image scanning — basic scanning, or enhanced scanning powered by Amazon Inspector to find CVEs in OS packages and language dependencies. "Scan container images for vulnerabilities" → ECR enhanced scanning (Inspector).
  • Lifecycle policies — automatically expire old/untagged images to control storage cost.
  • Cross-region / cross-account replication — keep images close to where they run, or share across accounts.
💵 ECR is $0.10 per GB-month of stored images, plus data transfer out. Lifecycle policies that prune stale images keep this small.

EKS · Part 1

Why EKS — Managed Kubernetes

ECS is AWS-only. Kubernetes is the open-source standard the whole industry uses. EKS gives you Kubernetes, managed.

Kubernetes (K8s) is the dominant open-source container orchestrator. Teams want it for its huge ecosystem, portability across clouds, and existing skills/tooling — but running the Kubernetes control plane yourself is famously complex.

Amazon EKS (Elastic Kubernetes Service) is managed Kubernetes. AWS runs and scales the K8s control plane (the brain) for you across multiple AZs; you run your workloads on worker nodes. You get standard Kubernetes (same APIs, kubectl, Helm charts, operators) without operating the control plane.

🔑 When to pick EKS over ECS

Choose EKS only when a scenario explicitly mentions Kubernetes, the K8s API, multi-cloud portability, or an existing K8s investment. If it just says "run containers, least ops, no K8s requirement," that's ECS on Fargate — simpler and with no control-plane fee.

EKS · Part 2

How It Works — Nodes & Fargate

  • Control plane — managed by AWS, multi-AZ, ~$0.10/hour per cluster.
  • Worker nodes — your pods run on either EC2 nodes (self-managed or Managed Node Groups) or Fargate (serverless pods via Fargate profiles — "Kubernetes + no node management").
  • Node autoscaling — Cluster Autoscaler or Karpenter (fast, flexible, cost-aware node provisioning).
  • Integrates with ALB (AWS Load Balancer Controller), VPC CNI networking, CloudWatch, and ECR.
EKS · Part 3

IRSA / Pod Identity — Securely Giving Pods AWS Access

🔑 Tested security pattern

The most secure way to give a Kubernetes pod permission to call AWS APIs is IRSA (IAM Roles for Service Accounts) — or the newer EKS Pod Identity — which maps a fine-grained IAM role to a specific pod's service account. NOT the node instance role (which would give every pod on that node the same broad permissions). "Least-privilege AWS access for specific pods" → IRSA / Pod Identity.

🏢 Industry — Verified

Coinbase & Flutter run thousands of services on EKS + Graviton + Karpenter

Coinbase migrated 3,500 services from EC2 to EKS on Graviton with Karpenter for node provisioning — AWS reports it "scales 50% faster and cuts costs 62%," with Graviton giving ~20% lower cost and Karpenter ~10% more compute savings. Flutter/PokerStars (Dublin) migrated 1,100 applications and 3 PB to EKS + MSK + Cloud WAN, cutting TCO ~20% and speeding deployments 40%.

Why EKS-on-EC2 here, not Fargate

At thousands-of-services scale with steady utilization, self-managed EC2 nodes (Spot + Graviton + Karpenter bin-packing) are dramatically cheaper per unit than per-task Fargate — the textbook "control & cost at scale" trade-off.

Why Kubernetes at all

Both want the portable K8s ecosystem (Helm, operators, existing skills) across a massive microservice estate — exactly the case where EKS beats ECS.
