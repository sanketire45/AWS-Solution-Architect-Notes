Monitoring & Audit — CloudWatch · CloudTrail · Config

Three services that sound similar but answer three different questions. CloudWatch = "what is happening?" (performance). CloudTrail = "who did it?" (API audit). Config = "is it compliant, and how did it change?" (configuration). Master the distinction — the exam tests it relentlessly.

🔑 One-line memory hook

CloudWatch watches performance. CloudTrail trails API calls. Config checks configuration compliance.

CloudWatch · Part 2

Metrics, Alarms & Dashboards

  • Metrics — time-series data (CPU, network, request count). AWS publishes many automatically; you can push custom metrics.
  • Alarms — watch a metric and trigger an action when a threshold is breached: notify via SNS, trigger Auto Scaling, or run an EC2 action (recover/stop/reboot).
  • Dashboards — visualise metrics across services/regions in one view.
  • Composite alarms combine multiple alarms to cut noise.
🔑 Common pattern

"Alert me / scale when CPU > 70%" → CloudWatch alarm → SNS / Auto Scaling. "Automatically recover a failed EC2 instance" → CloudWatch alarm with an EC2 recover action.

CloudWatch · Part 3

CloudWatch Logs & the Agent

  • CloudWatch Logs — centralise logs from EC2, Lambda, ECS, on-prem; set retention (default = never expire = silent cost).
  • Logs Insights — query logs with a purpose-built query language.
  • Subscription filters → stream logs to Kinesis/Firehose/Lambda in near-real-time (e.g. Logs → Firehose → OpenSearch).
⚠️ The CloudWatch Agent trap

By default, EC2 sends CPU/network/disk-I/O metrics — but NOT memory (RAM) or disk-space-used. To collect those you must install the CloudWatch Agent (and push them as custom metrics). "Monitor EC2 memory utilization" → CloudWatch Agent, not the default metrics.

💵 Logs ≈ $0.50/GB ingested + $0.03/GB-month stored. The silent cost is forgotten retention — set it (e.g. 30/90 days) instead of "never expire."
CloudWatch · Part 4

Events (EventBridge), X-Ray & the Family

  • EventBridge (CloudWatch Events) — react to AWS events or schedules (cron) → trigger Lambda/SNS/Step Functions. "Run a task daily" or "react to a state change" → EventBridge.
  • X-Ray — distributed tracing across microservices ("where is the latency / which call failed?").
  • Synthetics (canary uptime checks), RUM (real-user monitoring), Container/Lambda Insights, Application Signals (unified APM).
🏢 Dublin Industry — Verified

How a Dublin SRE team actually uses CloudWatch in production

At Ryanair, on-call SREs watch CloudWatch alarms & dashboards for the metrics that predict customer pain: Aurora replica lag, ALB 5xx rates, and Kinesis IteratorAge (how far behind stream consumers are). Runbooks tie alarms to actions (e.g. RDS failover, scaling).

The three services map to three real questions the team asks daily: CloudWatch "is it healthy / fast?", CloudTrail "who changed that?", Config "is it still compliant?" — used together for full operational governance.

Observability in practice

Alarms → SNS → on-call; metric filters on logs catch error spikes; an org-wide CloudTrail trail to S3 (management events are free) gives audit; Config rules + auto-remediation keep security groups and encryption compliant across accounts.

CloudTrail · Part 1

Why CloudTrail — The Audit Log of Your Account

Who deleted that bucket? Who changed that security group? CloudTrail answers.

AWS CloudTrail records every API call made in your account — by the console, CLI, SDK, or another AWS service — capturing who did what, when, and from where.

It's the foundation of governance, compliance, and security forensics. After an incident ("who deleted the database?"), CloudTrail is where you look. It's enabled by default for recent management events (90-day console history), but for long-term retention you create a Trail that delivers logs to S3.

CloudTrail · Part 2

Management Events vs Data Events

A tested distinction with a cost angle.

Management EventsData Events
WhatControl-plane ops (create/delete/modify resources, IAM changes)Data-plane ops (S3 object GET/PUT, Lambda invokes)
Default✅ Logged (and free)❌ Not logged by default
CostFreePaid (high volume)
Example"Who deleted this S3 bucket?""Who read this specific object?"
🔑 Tested gotcha

Data events (S3 object-level, Lambda invokes) are NOT logged by default — you must explicitly enable them (and they cost more). "Audit who accessed individual S3 objects" → enable CloudTrail data events. "Audit who changed configuration / deleted resources" → management events (default).

CloudTrail · Part 3

Trails, Log Integrity, Insights & Org Trails

  • Trail → S3 for long-term storage; send to CloudWatch Logs for alerting on specific API calls.
  • Log file validation — cryptographic hashing proves logs weren't tampered with (compliance).
  • CloudTrail Insights — detect unusual API activity (anomalies).
  • Organization Trail — one trail capturing all accounts in AWS Organizations into a central S3 bucket.
🔑 Alert on a specific action

"Alert me whenever someone disables encryption / makes a bucket public / changes IAM" → send CloudTrail to CloudWatch Logs and trigger an alarm/EventBridge rule on that event. (CloudTrail records; CloudWatch/EventBridge react.)

Config · Part 1

Why AWS Config — Configuration History & Compliance

"Was this security group ever open to the world? Is every bucket encrypted right now?" Config answers.

AWS Config records the configuration of your resources over time and evaluates them against rules — giving you a timeline of every change and a continuous compliance picture.

For each resource it keeps a configuration history (what it looked like at each point in time, and what changed) and tells you whether it currently complies with your policies (e.g. "all EBS volumes must be encrypted," "no security group may allow 0.0.0.0/0 on port 22").

Config · Part 2

Config Rules & Auto-Remediation

  • Managed & custom rules — check compliance (e.g. s3-bucket-public-read-prohibited, encrypted-volumes). Custom rules run a Lambda.
  • Auto-remediation — when a resource is non-compliant, run an SSM Automation document to fix it (e.g. re-enable encryption, remove a public ACL).
  • Conformance packs — deploy a collection of rules + remediations as one package (e.g. for PCI/HIPAA).
  • Multi-account/region aggregation via AWS Organizations.
Section 24 · The Big Distinction

CloudTrail vs Config — Don't Confuse Them

CloudTrailConfig
QuestionWho made the API call?What is the resource's configuration & is it compliant?
RecordsAPI activity (actions/events)Resource state over time + compliance
Example"User X called ModifySecurityGroup at 14:03""This SG currently allows 0.0.0.0/0:22 — NON-COMPLIANT; here's its change timeline"
Use forAudit, forensics, who-did-itCompliance, drift, configuration history
🔑 They complement each other

Config tells you a resource is non-compliant and how its config changed; CloudTrail tells you who made the change (the API call). Use both together for full governance. "Continuously check resource compliance / config drift" → Config. "Who performed the action" → CloudTrail.

💵 Config ≈ $0.003 per configuration item recorded + ~$0.001 per rule evaluation. Cost grows with resource churn — scope the recorder to the resource types you must govern.
