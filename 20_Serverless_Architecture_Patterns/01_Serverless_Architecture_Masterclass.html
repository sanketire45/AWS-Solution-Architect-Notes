Serverless Solution Architectures

This section has no new services — it's about assembling the serverless building blocks (Lambda, DynamoDB, API Gateway, Cognito, S3, CloudFront, SQS, Kinesis) into the reference architectures the exam asks you to design. Four patterns you must be able to draw from memory.

Pattern 1 · Part 1

The Serverless Web/Mobile App Backend

The most common serverless architecture — and the one the exam expects you to recognise instantly.

Cognito authenticates → API Gateway validates the token & routes → Lambda runs logic → DynamoDB stores data → S3 holds files. No servers, scales to zero.
  • Cognito User Pool handles sign-up/sign-in and issues JWT tokens.
  • API Gateway (with a Cognito authorizer) is the secure front door; throttles and validates.
  • Lambda runs the business logic; DynamoDB stores high-velocity data; S3 holds user files (often via a presigned URL or Cognito Identity Pool credentials).
Pattern 1 · Part 2

How Requests Flow (and where auth fits)

  • User signs in → Cognito returns a JWT.
  • Client calls API Gateway with the token → the Cognito authorizer validates it.
  • API Gateway invokes Lambda → Lambda reads/writes DynamoDB using its execution role (least privilege).
  • For direct file uploads, the client gets temporary AWS credentials from a Cognito Identity Pool (or an S3 presigned URL) to put objects straight into S3 — bypassing Lambda for large payloads.
Pattern 1 · Part 3

Caching & Performance Levers

  • DynamoDB DAX — microsecond reads for hot data.
  • API Gateway caching (REST) — cache responses to cut Lambda calls.
  • CloudFront in front for global users + static assets.
  • Provisioned Concurrency / SnapStart — kill Lambda cold starts for latency-sensitive APIs.
💵 The whole serverless stack scales to zero: Lambda (1M req + 400k GB-s free), DynamoDB on-demand, HTTP API ($1/M), Cognito (50k MAU free) — a small/medium production app often runs $0–a few dollars/month, the core reason fintechs build serverless-first.
🏢 Dublin Industry — Verified

Fenergo runs a serverless-first, event-sourced platform — the textbook of these patterns

Fenergo (Dublin RegTech for KYC/AML) is single-cloud AWS, serverless-first, running active-active across 3 AZs ("Always On"). It stitches together exactly the patterns in this section: API Gateway + Lambda front-ends, DynamoDB as the event-sourced state/read-model store (with Streams for CDC), EventBridge fan-out, and Step Functions orchestrating cluster upgrades.

Dublin B2B-payments firm AccessPay uses the decoupled event-pipeline pattern — SFTP files land in S3, SQS buffers them, and Lambda processes them for PCI-compliant downstream workflows.

Why serverless-first

Multi-tenant SaaS for regulated banks needs hard tenant isolation and elastic scale — per-tenant IAM condition keys + least-privilege Lambda roles per microservice, with pay-per-use economics that scale to zero between bursts.

Pattern 2 · Part 1

Serverless Static Website Hosting

Host a global website (or SPA) with no web servers at all.

CloudFront serves the SPA from a private S3 bucket (locked with OAC); dynamic API calls go to API Gateway + Lambda. ACM gives HTTPS; WAF protects.
  • S3 stores the HTML/CSS/JS (private, no public access).
  • CloudFront caches globally and is locked to the bucket with OAC.
  • Dynamic parts (login, data) call API Gateway + Lambda + DynamoDB — the Pattern-1 backend.
Pattern 2 · Part 2

Security & HTTPS

  • ACM certificate in us-east-1 for the CloudFront custom domain (the classic trap).
  • OAC keeps the S3 bucket private — users can only reach it through CloudFront.
  • WAF on CloudFront for bot/SQLi/rate protection; Shield Standard is free.
  • Route 53 Alias record (free) points the apex domain at CloudFront.

Pattern 3 · Part 1

Serverless Microservices

Many small, independently-deployable services — each with its own function and data store.

Instead of one big app, you build many small services (orders, payments, inventory…), each owning its logic and data. Serverless makes each one cheap and independently scalable.

  • Each microservice = API Gateway + Lambda + its own DynamoDB table (or other store).
  • Services communicate synchronously (API Gateway / direct invoke) or asynchronously (SQS, SNS, EventBridge).
  • Each scales, deploys, and fails independently — the resilience win.
Pattern 3 · Part 2

Synchronous vs Asynchronous Communication

NeedUse
Direct request/response between servicesAPI Gateway or direct Lambda invoke
Decouple / buffer / absorb spikesSQS
One event → many services (fan-out)SNS (→ SQS per consumer)
Content-based routing / SaaS events / schedulingEventBridge
Orchestrate a multi-step workflow (saga)Step Functions
Cross-VPC/account mixed compute, least opsVPC Lattice
🔑 The golden rule

Prefer asynchronous, event-driven communication between microservices (SQS/SNS/EventBridge) so a slow or failed service doesn't cascade. Use Step Functions when you need to coordinate an ordered, multi-step process (a "saga") with retries, error handling, and a visual workflow.

Pattern 4 · Part 1

S3 Event Processing (thumbnails, transcoding, ETL)

The classic "do something whenever a file lands in S3" pipeline.

Upload fires an S3 event → Lambda processes (resize/transcode) → writes metadata to DynamoDB and the result back to S3. Fully event-driven, scales with uploads.
Pattern 4 · Part 2

Real-Time Data Ingestion & Analytics

  • Kinesis Data Streams ingests high-volume clickstream/IoT in real time (ordered, replayable).
  • Kinesis Data Firehose delivers it (near-real-time, least ops) to S3 / Redshift / OpenSearch.
  • Managed Service for Apache Flink runs real-time SQL/aggregations on the stream.
  • Land in an S3 data lake → query with Athena, visualise with QuickSight.
Pattern 4 · Part 3

Making Event Pipelines Decoupled & Reliable

🔑 The reliability upgrade

For high-throughput or spiky S3 events, insert SQS (or SNS → SQS fan-out) between S3 and the consumers so a slow/failed processor doesn't lose events (durable buffer + retry + DLQ). For "one upload must trigger several independent processors," use SNS → multiple SQS. This combines the event pipeline with the decoupling patterns from Section 17.

⚠️ The 15-minute reminder

If processing a file takes longer than 15 minutes (e.g. large video transcoding), Lambda won't work — use Fargate, AWS Batch, or MediaConvert, often orchestrated by Step Functions.
