Fundamentals · Part 1

What Is a VPC — Your Own Private Data Center

A VPC (Virtual Private Cloud) is your own logically-isolated section of the AWS cloud — a private network where you launch resources (EC2, RDS, etc.) with full control over IP addressing, subnets, routing, and security.

Think of it as building your own data-center network, but software-defined. You choose a private IP range (a CIDR block like 10.0.0.0/16 = ~65,000 addresses), carve it into subnets, and decide exactly what can talk to what.

  • A VPC spans all AZs in one Region; subnets live in a single AZ.
  • Limits: ~5 VPCs/Region (raisable), 200 subnets/VPC, up to 5 CIDRs/VPC. One Internet Gateway per VPC.
  • AWS reserves 5 IPs in every subnet (first 4 + last) — a /28 gives you 11 usable, not 16.
Fundamentals · Part 2

Subnets & CIDR

  • A subnet is a slice of the VPC's IP range, bound to one AZ. You spread subnets across AZs for high availability.
  • CIDR notation: /16 = 65,536 IPs, /24 = 256, /28 = 16. Smaller number = bigger range.
  • A subnet is public or private based purely on its route table — public = has a route to the Internet Gateway.
Fundamentals · Part 3

Route Tables & the Internet Gateway

Public subnet routes to the IGW directly. Private subnet routes outbound through a NAT Gateway (which sits in a public subnet) → IGW. Inbound from the internet to private subnets is blocked.
  • Route table — rules deciding where subnet traffic goes (e.g. 0.0.0.0/0 → igw-xxx). Each subnet is associated with one.
  • Internet Gateway (IGW) — the door to the internet; one per VPC. A subnet with a default route to the IGW (and instances with public IPs) is "public."
Fundamentals · Part 4

Public vs Private Subnet — The Core Pattern

🔑 The 3-tier pattern

The classic secure design: public subnets hold internet-facing things (ALB, NAT Gateway, bastion); private subnets hold app servers and databases (no direct internet route). Private instances reach the internet outbound only via the NAT Gateway. Spread subnets across ≥2 AZs for HA. "Database with no internet exposure" → private subnet.

🏢 Dublin Industry — Verified / Inferred

How Coinbase and Flutter design production VPC networking

Coinbase Ireland runs EKS across 3 AZs with private subnets, a NAT Gateway per AZ for HA egress, and interface VPC endpoints to MSK/DynamoDB to keep traffic on the AWS network (off the internet, and off the NAT bill).

Flutter/PokerStars (Dublin) connects a large multi-account estate with AWS Cloud WAN + Network Firewall — the global, hub-style evolution of Transit Gateway with centralized inspection.

The production pattern

Public subnets for ALB/NAT only; private subnets for app + databases; NAT per AZ for resilient egress; gateway endpoints (free) for S3/DynamoDB and interface endpoints for everything else; Transit Gateway / Cloud WAN to interconnect many VPCs and on-prem with transitive routing.

NAT & Security · Part 1

NAT Gateway — Outbound Internet for Private Subnets

Private instances often need outbound internet (to download patches, call APIs) without being reachable from the internet. A NAT Gateway (in a public subnet) does exactly that: it translates private traffic out to the internet via the IGW, but blocks unsolicited inbound.

⚠️ The #1 NAT trap

A NAT Gateway lives in one AZ. For high availability you must deploy one NAT Gateway per AZ (and route each AZ's private subnet to its local NAT) — otherwise an AZ failure cuts internet for the others. "Highly available outbound internet for private subnets" → NAT Gateway per AZ.

🔑 NAT Gateway vs NAT Instance

NAT Gateway = AWS-managed, scalable, HA-per-AZ (the answer). NAT Instance = legacy self-managed EC2 (you patch it, it's a SPOF). Always prefer NAT Gateway.

💵 NAT Gateway = $0.045/hr (~$33/mo) + $0.045/GB processed — and that per-GB charge stacks on data transfer. Cost lever: route S3/DynamoDB traffic through a free Gateway Endpoint instead of the NAT.
NAT & Security · Part 2

Security Groups vs Network ACLs

Two firewalls at two layers — the most-tested networking distinction.

Operates atInstance (ENI) levelSubnet level
StateStateful (return traffic auto-allowed)Stateless (must allow return traffic explicitly)
RulesAllow onlyAllow AND Deny
EvaluationAll rules evaluatedRules in number order, first match wins
Use forThe primary instance firewallSubnet-wide allow/deny, e.g. block a malicious IP range
NAT & Security · Part 3

The Stateful vs Stateless Trap

⚠️ Why this trips people up

Security Groups are stateful: allow inbound port 443, and the response is automatically allowed out — you don't add an outbound rule. NACLs are stateless: if you allow inbound 443, you must also explicitly allow the outbound ephemeral ports (1024–65535) for the return traffic, or it breaks. "Need explicit deny / subnet-level / block an IP range" → NACL. "Standard instance firewall" → Security Group.

🔑 SG referencing

Security Groups can reference other security groups (e.g. "allow the ALB's SG to reach the app SG on 8080") — cleaner than hard-coding IPs and the standard tier-to-tier pattern.

🔑 Two top triggers

"Private access to S3/DynamoDB from a private subnet, no internet, no NAT cost" → Gateway Endpoint (free). "Private access to another AWS service (SQS, KMS, etc.) or expose your own service across VPCs" → Interface Endpoint / PrivateLink.

Connectivity · Part 2

VPC Peering — Connect Two VPCs

  • Directly connects two VPCs (same or different account/region) so they communicate using private IPs.
  • Non-transitive — if A↔B and B↔C are peered, A cannot reach C through B. You'd need A↔C directly (or a Transit Gateway).
  • No overlapping CIDR blocks allowed.
⚠️ Trap

Peering is non-transitive and becomes a mesh nightmare with many VPCs (N×(N-1)/2 connections). For connecting many VPCs, use Transit Gateway, not a web of peering.

Connectivity · Part 3

Transit Gateway — The Network Hub

Transit Gateway (TGW) is a regional hub that connects thousands of VPCs and on-prem networks through a single, transitive router — replacing a tangle of peering connections. Attach VPCs, VPN, and Direct Connect to one TGW and they can all route to each other (subject to route tables). AWS Cloud WAN extends this globally.

🔑 Trigger

"Connect many VPCs (and on-prem) with transitive routing, least operational overhead" → Transit Gateway. (Two VPCs only, simple → VPC Peering.)

💵 TGW ≈ $0.05/attachment/hr + ~$0.02/GB processed. Gateway endpoints (S3/DynamoDB) stay free; VPC peering has no hourly fee (data transfer only).

Hybrid · Part 1

Site-to-Site VPN — Encrypted Over the Internet

  • Creates an encrypted IPsec tunnel between your on-prem network and your VPC, over the public internet.
  • Fast to set up (minutes), low cost — but latency/throughput depend on the internet.
  • "Quick, encrypted hybrid connectivity" or "temporary/backup link" → Site-to-Site VPN.
Hybrid · Part 2

AWS Direct Connect (DX)

  • A dedicated, private physical fiber connection from your data center to AWS — consistent low latency, high bandwidth, and lower data-transfer costs.
  • Not encrypted by default (it's private, not encrypted) — run a VPN over Direct Connect if you need encryption.
  • Provisioning takes weeks-to-months — so "needed immediately/soon" rules it out.
⚠️ Two DX traps

(1) Not encrypted by default — add a VPN for encryption. (2) Long lead time — if a scenario says "must be working in days," DX alone is wrong (use a VPN now, DX later, or VPN as the answer).

Hybrid · Part 3

🔑 Best of both

The robust enterprise pattern: Direct Connect for performance + a Site-to-Site VPN as encrypted backup (failover). "Consistent low latency AND resilient/encrypted" → DX + VPN.

Hybrid · Part 4

VPC Flow Logs

  • Capture metadata about IP traffic (source/dest, ports, accept/reject) flowing through ENIs, subnets, or the VPC.
  • Send to CloudWatch Logs or S3 (→ Athena) for monitoring, troubleshooting, and security analysis.
  • "Troubleshoot why traffic is blocked / analyze network traffic patterns" → VPC Flow Logs. (They record metadata, not packet contents.)
