Databases in AWS — Choosing the Right One

AWS has a purpose-built database for every data shape — relational, key-value, in-memory, graph, ledger, time-series, document, wide-column, and warehouse. The exam's favourite database question isn't "how does X work" — it's "which database fits this scenario?" Master the decision.

Choosing · Part 1

Why AWS Has So Many Databases

For decades, one relational database did everything. AWS's bet: the right tool for each data shape beats one-size-fits-all.

A relational database is brilliant for structured, related data with complex queries. But forcing everything into tables and JOINs breaks down at scale and for special shapes.

Graph relationships (who-knows-who), append-only ledgers (immutable audit trails), time-series (sensor readings over time), and petabyte analytics each have a database purpose-built to do them faster, cheaper, and simpler than a general-purpose RDBMS. As a Solution Architect, your job is to recognise the data shape and pick the matching engine.

Choosing · Part 2

The Questions That Decide

  • Structured & relational with JOINs / transactions? → RDS / Aurora (SQL).
  • Key-value, massive scale, single-digit ms, serverless? → DynamoDB.
  • Need a cache / session store / sub-ms? → ElastiCache (or DAX for DynamoDB).
  • Relationships are the data (social, fraud, recommendations)? → Neptune (graph).
  • Immutable, cryptographically verifiable history? → QLDB (ledger).
  • Time-stamped measurements at scale (IoT, metrics)? → Timestream.
  • MongoDB / Cassandra compatibility? → DocumentDB / Keyspaces.
  • Petabyte analytics / OLAP / BI dashboards? → Redshift (warehouse).
  • Search & log analytics? → OpenSearch.

Relational · Part 1

Amazon RDS — Managed SQL Engines

(Covered in depth in the RDS/Aurora/ElastiCache topic — here's the selection-level recap.)

  • Managed MySQL, PostgreSQL, MariaDB, Oracle, SQL Server — AWS handles patching, backups, failover.
  • Pick RDS for complex queries, JOINs, transactions, and existing SQL apps (lift-and-shift Oracle/SQL Server → RDS).
  • Multi-AZ = synchronous standby for HA; Read Replicas = async copies to scale reads.
Relational · Part 2

Amazon Aurora — Cloud-Native Relational

  • MySQL/PostgreSQL-compatible, ~5×/3× faster, AWS-engineered storage: 6 copies across 3 AZs, self-healing.
  • Up to 15 read replicas with a reader endpoint; Aurora Global Database for cross-region DR (<1s replication).
  • Aurora Serverless v2 auto-scales (even to near-zero) for spiky/dev; Backtrack rewinds without restore.
  • "Minimal-admin MySQL/Postgres + read scaling + cloud-native HA" → Aurora.
💵 Aurora ~$0.26/hr (r6g.large) + storage + I/O (or I/O-Optimized to remove per-I/O charges); Serverless v2 ~$0.12/ACU-hr scaling to near-zero. RDS db.t3.medium ~$50/mo single-AZ (~$100 Multi-AZ).
Relational · Part 3

Multi-AZ vs Read Replicas — The Eternal Trap

⚠️ Don't confuse them

Multi-AZ = High Availability (a synchronous standby in another AZ that auto-fails-over; the standby is not readable). Read Replicas = read scaling (async copies you send read traffic to; promotion is manual for DR). "Automatic failover / HA" → Multi-AZ. "Offload reporting / scale reads" → Read Replicas. "Cross-region DR database" → cross-region read replica or Aurora Global Database.

NoSQL · Part 1

DynamoDB — Serverless Key-Value at Any Scale

  • Single-digit ms, serverless, virtually unlimited scale; on-demand vs provisioned capacity.
  • GSI (any key, anytime) vs LSI (same partition key, only at creation).
  • Streams (CDC → Lambda), DAX (µs cache), Global Tables (multi-region active-active).
  • Pick for high-velocity, known access patterns, serverless apps.
NoSQL · Part 2

ElastiCache — In-Memory Caching

  • Managed Redis / Valkey (persistence, replication, pub/sub, sorted sets) or Memcached (simple, multi-threaded, no persistence).
  • Use for: cache hot DB reads, session stores, leaderboards, rate limiting.
  • "Cache with persistence + replication + pub/sub" → Redis/Valkey; "simple, multi-threaded cache" → Memcached.

Purpose-Built · Part 1

Amazon Redshift — The Data Warehouse

The one purpose-built database you'll see most: analytics at petabyte scale.

Redshift is a columnar, OLAP data warehouse for running complex analytical queries over huge datasets (BI dashboards, reporting). It's not for high-velocity transactional (OLTP) lookups — that's DynamoDB/RDS. Redshift Spectrum queries data directly in S3; Redshift Serverless removes cluster management.

⚠️ OLTP vs OLAP

"Millions of small transactions / lookups" = OLTP → DynamoDB/RDS. "Complex analytics / aggregations over big data for BI" = OLAP → Redshift. Don't use Redshift for an app's operational database, or RDS for petabyte analytics.

💵 Redshift from ~$0.25/hr (dc2.large) to large clusters; Serverless ~$0.36/RPU-hr. For occasional ad-hoc S3 queries, Athena (serverless, pay-per-TB-scanned) is often cheaper than a running Redshift cluster.

🔑 Ledger vs relational

You could build an audit log in a relational DB, but QLDB gives a built-in, immutable, cryptographically-verifiable history no one can tamper with — the reason it wins "prove the records were never altered" questions. (Note: AWS is phasing QLDB out, but it still appears on the exam.)
