Single Answer
Alias vs CNAME at Zone Apex
Easy
A company has a website hosted behind an Application Load Balancer (ALB). They want users to access the website by typing example.com (without any subdomain) into their browser. The company uses Amazon Route 53 for DNS.

Which DNS record configuration should a solutions architect create?

  • A
    Create a CNAME record for example.com pointing to the ALB's DNS name
  • B
    Create an Alias A record for example.com pointing to the ALB's DNS name
  • C
    Create an A record for example.com with the ALB's IP address
  • D
    Create a CNAME record for www.example.com pointing to example.com
✅ Correct Answer
B — Alias A Record

A CNAME record cannot be created at the zone apex (example.com with no subdomain). This is a fundamental DNS rule (RFC 1912). Route 53's Alias record was invented specifically to solve this — it works at the zone apex, points to AWS resources like ALBs, is free, and automatically follows the ALB's changing IPs.

Why A is wrong: CNAME at zone apex is forbidden by DNS spec. Route 53 won't let you create it.

Why C is wrong: ALB IPs change dynamically — you'd need to update the A record constantly. Alias handles this automatically.

Why D is wrong: That points a subdomain to the root domain, which doesn't solve the original question of making example.com work.
🧠 Memory Hook: "Zone Apex = Always Alias." If it's the naked root domain → Alias A record. Never CNAME.

Single Answer
Canary Deployment — Instant Rollback
Hard
A team wants to send 5% of traffic to a new version of a web app behind a single Application Load Balancer, keeping 95% on the stable version. Critically, if error rates spike they must be able to pull the new version back to 0% within seconds. Which approach gives the MOST reliable near-instant control over the split?

Which design should the solutions architect choose?

  • A
    Route 53 Weighted routing across two records (v1=95, v2=5); set v2's weight to 0 to roll back.
  • B
    ALB weighted target groups (forward rule splitting to v1/v2 target groups); shift weights at the load balancer.
  • C
    Route 53 Weighted routing with a very low TTL (e.g., 1 second).
  • D
    Route 53 Failover routing between v1 and v2.
✅ Correct Answer
B — ALB weighted target groups

Because there is already a single ALB, splitting at the load balancer changes the traffic ratio instantly for every request — rollback to 0% is immediate. This is the right tool when the requirement is near-instant control.

Why A is the trap: Route 53 Weighted routing does split by percentage, and "5% of traffic" reflexively points there — but DNS answers are cached by resolvers and clients for the TTL. After you set the weight to 0, cached clients keep hitting v2 until their cache expires, so rollback is not within seconds. (C) A 1-second TTL reduces the window but many resolvers ignore tiny TTLs, and it hammers DNS — still not reliable instant control. (D) Failover is all-or-nothing on health, not a 5/95 split.
🎯 How to discriminate: "Split by %" alone → Route 53 Weighted is fine. Add "instant rollback / within seconds" and DNS TTL caching disqualifies DNS-level splitting → shift the split to the load balancer (ALB weighted target groups) or a deployment tool (CodeDeploy).

Single Answer
Geolocation — Missing Default Record
Medium
A solutions architect configures Route 53 Geolocation routing for a global application. Records are created for the United States, Germany, and Japan. Users in Australia report they cannot access the application at all, even though the application is running normally.

What is the most likely cause of this issue?

  • A
    The application's servers in Australia are down
  • B
    Australia is not a supported geolocation in Route 53
  • C
    A Default record was not created for unmatched geographic locations
  • D
    Route 53 does not support the Asia Pacific continent for geolocation routing
✅ Correct Answer
C — Missing Default Record

When using Geolocation routing, Route 53 only routes users whose location matches one of your configured records. Users from unmatched locations receive a DNS error (no answer / NXDOMAIN) unless you create a Default record. Australia doesn't match US, Germany, or Japan, so those users get nothing.

Why A is wrong: The question says "application is running normally" — ruling out server issues.

Why B is wrong: Route 53 supports all countries globally.

Why D is wrong: Asia Pacific and Oceania are fully supported.
🧠 Memory Hook: "Geo = Always Default." Geolocation without a Default = some users fall through the floor. The Default catches everyone else.

Failover + Health Check
Medium
A company has a primary web application in us-east-1 and a disaster recovery site in eu-west-1. Route 53 Failover routing is configured with the primary pointing to us-east-1. The primary region goes completely down, but traffic is NOT failing over to eu-west-1.

What is the most likely reason traffic is not failing over?

  • A
    The secondary record in eu-west-1 does not have a health check configured
  • B
    The primary record in us-east-1 does not have a health check configured
  • C
    Failover routing requires a minimum TTL of 60 seconds
  • D
    Route 53 Failover routing does not support cross-region configurations
✅ Correct Answer
B — Health check missing on PRIMARY

Failover routing only triggers when Route 53 detects the primary endpoint is unhealthy. The detection mechanism is a health check on the primary record. Without it, Route 53 assumes the primary is always healthy and never switches to secondary — even if the region is completely down.

Why A is wrong: The secondary doesn't need a health check to receive failover traffic. The HC must be on the primary to detect failure.

Why C is wrong: No such TTL requirement exists for failover.

Why D is wrong: Cross-region failover is one of the primary use cases of Route 53 Failover routing.
🧠 Memory Hook: "Failover needs a smoke detector on PRIMARY." No smoke detector = no alarm = no evacuation to secondary.

Latency vs Geolocation
Medium
A fintech company stores customer data in two regions: eu-central-1 (Frankfurt) for EU customers and us-east-1 (Virginia) for US customers. Due to GDPR regulations, EU customer data must only be processed in the EU region, regardless of network conditions.

Which Route 53 routing policy ensures EU customers always use the EU region?

  • A
    Latency-based routing, as it will route EU users to the closest region
  • B
    Geolocation routing with EU as a continent rule pointing to eu-central-1
  • C
    Weighted routing with higher weights for EU records
  • D
    Geoproximity routing with a positive bias for eu-central-1
✅ Correct Answer
B — Geolocation Routing

This question is about compliance, not performance. Geolocation routing guarantees that users in a specific country or continent ALWAYS go to the configured endpoint — unconditionally. Latency routing could send EU users to Virginia if the US connection is faster, which would violate GDPR.

Why A is wrong: Latency picks the fastest network path. An EU user might get routed to us-east-1 if it has lower latency — violating GDPR data residency.

Why C is wrong: Weighted routing splits traffic by percentage — EU users could still end up in us-east-1.

Why D is wrong: Geoproximity's bias expands/shrinks the catchment area but doesn't guarantee EU users always stay in EU.
🧠 Memory Hook: "Compliance/Data Residency = Geolocation." Any time you see GDPR, HIPAA, data sovereignty, local laws → Geolocation. NOT Latency.

Single Answer
Private VPC Health Check
Hard
A company uses Route 53 Failover routing to switch traffic between a primary and secondary database endpoint. The primary database is an RDS instance in a private subnet with no public IP address. The team wants Route 53 to automatically detect if this private RDS instance is unhealthy and trigger failover.

How should a solutions architect configure the health check for the private RDS instance?

  • A
    Create an endpoint health check using the RDS instance's private IP address
  • B
    Assign a public IP to the RDS instance and create an endpoint health check
  • C
    Create a CloudWatch alarm monitoring the RDS DatabaseConnections metric, then create a Route 53 health check that monitors the CloudWatch alarm state
  • D
    Route 53 cannot perform health checks on resources inside a private VPC; use an NLB instead
✅ Correct Answer
C — CloudWatch Alarm → Route 53 Health Check

Route 53 health checkers are on the public internet — they cannot reach a private subnet directly. The solution is to use CloudWatch to monitor the private resource (using VPC agent or RDS native metrics), create a CloudWatch Alarm, and then configure a Route 53 health check of type "CloudWatch alarm." Route 53 reads the alarm state to determine health.

Why A is wrong: Route 53's public health checkers cannot reach private IPs — they're on the internet, not inside your VPC.

Why B is wrong: Assigning a public IP to an RDS instance is a security anti-pattern and not recommended.

Why D is wrong: Route 53 CAN health-check private resources — just indirectly, via CloudWatch alarms.
🧠 Memory Hook: "Private = CloudWatch bridge." Route 53 can't see inside VPC → CloudWatch is the bridge that carries the health signal out.

Single Answer
TTL and Migration
Medium
A company is planning to migrate their web application to a new set of servers with different IP addresses next weekend. The current DNS A record has a TTL of 86400 seconds (24 hours). After updating the DNS record, they notice that many users are still being directed to the old servers 6 hours after the change.

What should the solutions architect have done BEFORE the migration to prevent this issue?

  • A
    Deleted the old DNS record before creating the new one
  • B
    Created the new record in a different hosted zone
  • C
    Reduced the TTL of the DNS record to a low value (e.g., 60 seconds) several days before the migration
  • D
    Used a CNAME record instead of an A record
✅ Correct Answer
C — Lower TTL days before migration

TTL controls how long DNS resolvers cache an answer. With TTL=86400, resolvers cache the answer for 24 hours. After updating the record, resolvers with a cached answer won't re-query Route 53 until their cache expires — up to 24 hours later. The fix: lower TTL to 60 seconds several days before the migration (so all caches have refreshed to the short TTL). Then after the migration, raise it back.

Why A is wrong: Deleting the record creates an outage. Clients with cached answers might still work, but new queries would fail.

Why B is wrong: A different hosted zone doesn't help with TTL caching.

Why D is wrong: Changing record type doesn't affect TTL caching behaviour.
🧠 Memory Hook: "Migration = TTL prep." Lower TTL 2 days early → wait for old caches to expire → migrate → raise TTL back. Three-step dance.

Route 53 vs Global Accelerator
Hard
A financial trading company runs a TCP-based application that requires very fast, deterministic failover between AWS regions. The application cannot tolerate the DNS TTL delays that occur with standard DNS-based failover. The company also requires the application endpoint to have two static IP addresses that never change, so they can be whitelisted in client firewalls.

Which solution BEST meets these requirements?

  • A
    Route 53 Failover routing with health checks and a very low TTL
  • B
    Route 53 Latency-based routing with health checks
  • C
    AWS Global Accelerator with endpoint groups in each region
  • D
    Route 53 with an Alias record pointing to a Network Load Balancer
✅ Correct Answer
C — AWS Global Accelerator

Three requirements identify Global Accelerator: (1) Static IPs — Global Accelerator gives 2 static anycast IPs that never change; (2) Non-HTTP TCP — supports any TCP/UDP; (3) Instant failover without DNS TTL delays — Global Accelerator routes at the network level (not DNS), so failover is < 30 seconds without any TTL waiting. Route 53 failover always depends on TTL, even at low values.

Why A is wrong: Even with very low TTL (e.g., 60s), clients still have a cached DNS entry that must expire before they see the new routing. Not truly instant. Also, A doesn't provide static IPs.

Why B is wrong: Latency-based doesn't provide static IPs or instant failover.

Why D is wrong: An NLB has static IPs, but only within one region. Doesn't solve the cross-region instant failover requirement.
🧠 Memory Hook: "Static IPs + Instant Failover + TCP = Global Accelerator." Route 53 = DNS game. Global Accelerator = network-level game. Different layers entirely.

Geoproximity Routing Requirements
Medium
A global e-commerce company wants to route users to AWS regions based on their geographic proximity, but they also want the ability to artificially shift more traffic toward ap-southeast-1 during off-peak hours in other regions, to balance infrastructure costs. The team is trying to configure this in the standard Route 53 console but cannot find the option.

What is required to configure this type of routing?

  • A
    Enable Advanced Route 53 features in the AWS console settings
  • B
    Use Geolocation routing with per-country rules instead
  • C
    Use Route 53 Traffic Flow to configure Geoproximity routing with bias values
  • D
    Geoproximity routing is only available via the AWS CLI, not the console
✅ Correct Answer
C — Route 53 Traffic Flow

Geoproximity routing with bias is only configurable through Route 53 Traffic Flow — the visual policy editor. It cannot be set up from the standard DNS record editing interface. Traffic Flow provides a GUI where you can create complex routing policies including Geoproximity with bias adjustments (+/-1 to 99).

Why A is wrong: There is no "Advanced Route 53 features" setting.

Why B is wrong: Geolocation routes by country/continent with hard rules — it doesn't support bias-based shifting.

Why D is wrong: Geoproximity is available in the console — but only through the Traffic Flow interface, not the standard record editor.
🧠 Memory Hook: "Geoproximity = Traffic Flow required." If the exam mentions bias, shifting traffic, or Geoproximity — Traffic Flow is always part of the answer.

Multi-Value vs Load Balancer
Hard
A startup runs 6 identical web servers across different Availability Zones. They want Route 53 to return multiple server IPs to clients so that if one server is unhealthy, clients won't be directed to it. The CTO suggests using "Route 53 Multi-Value Answer routing as a load balancer to eliminate the need for an ALB."

Which statement about this approach is CORRECT?

  • A
    Multi-Value routing is a valid replacement for an ALB as it distributes requests evenly across all healthy instances
  • B
    Multi-Value routing can replace an ALB only when all servers are in the same Availability Zone
  • C
    Multi-Value routing returns up to 8 healthy IPs and removes unhealthy endpoints from responses, but it is NOT a substitute for a load balancer and does not evenly distribute traffic
  • D
    Multi-Value routing is not supported when there are more than 4 records
✅ Correct Answer
C — Multi-Value is NOT a load balancer

AWS explicitly states Multi-Value Answer routing is not a substitute for a load balancer. It returns up to 8 healthy IPs and omits unhealthy ones from DNS responses, but the actual request distribution depends on the client — they pick one IP semi-randomly. There's no session persistence, no connection draining, no algorithm-based balancing. An ALB provides all of these.

Why A is wrong: "Evenly distributes requests" is what load balancers do, not Multi-Value DNS routing.

Why B is wrong: Multi-Value works across AZs. This restriction doesn't exist.

Why D is wrong: Multi-Value supports up to 8 records per response, but you can have more records defined (only 8 returned per query).
🧠 Memory Hook: "Multi-Value = health-aware DNS, NOT a load balancer." The exam uses this as a trap — if they say 'eliminate the ALB' → wrong.

Single Answer
Hybrid DNS — Direction
Medium
A company has connected their on-premises data centre to AWS using Direct Connect. Their on-premises applications need to resolve the DNS names of resources inside their AWS VPC (e.g., orders.internal.aws which resolves to a private EC2 IP). Currently, the on-premises DNS server cannot resolve these names.

What should a solutions architect configure to resolve this?

  • A
    Create a Route 53 Resolver Inbound Endpoint, then configure the on-premises DNS server to forward queries for the AWS domain to the Inbound Endpoint's IP
  • B
    Create a Route 53 Resolver Outbound Endpoint with a forwarding rule for the on-premises DNS server
  • C
    Make the private hosted zone public so on-premises servers can query it
  • D
    Create an Internet Gateway in the VPC to allow on-premises DNS queries to reach Route 53
✅ Correct Answer
A — Resolver Inbound Endpoint

The direction is: on-premises → AWS. For this direction, you need an Inbound Endpoint. It creates ENIs in your VPC with private IPs that the on-premises DNS can reach (via Direct Connect). Configure the on-prem DNS to forward queries for your AWS domain to those IPs, and Route 53 answers them.

Why B is wrong: Outbound Endpoint is for the reverse direction: AWS → on-premises DNS queries.

Why C is wrong: Making a private hosted zone public would expose internal DNS records to the internet — a security issue.

Why D is wrong: Internet Gateway allows public internet access — it doesn't help route private DNS queries over Direct Connect.
🧠 Memory Hook: "On-prem needs AWS DNS → INBOUND endpoint (traffic comes IN to AWS)." Think of it from Route 53's perspective: queries are coming IN from outside.

Single Answer
Alias Record Targets
Medium
A solutions architect needs to configure Route 53 DNS records for several resources. They plan to use Alias records to avoid per-query charges and to support the zone apex.

Which of the following targets is NOT supported as an Alias record target in Route 53?

  • A
    An Application Load Balancer (ALB)
  • B
    An Amazon CloudFront distribution
  • C
    An Amazon EC2 instance
  • D
    An Amazon S3 website endpoint
✅ Correct Answer
C — EC2 instance is NOT a valid Alias target

Route 53 Alias records only work with AWS-managed resources where AWS manages the DNS name: ALB, NLB, CloudFront, Elastic Beanstalk, API Gateway, S3 website endpoints, another record in the same hosted zone. EC2 instances have IPs that may change, but AWS doesn't expose them with a stable DNS name that Alias can track. Use a plain A record with the EC2 IP instead.

A, B, D are all valid Alias targets. ALB, CloudFront, and S3 website endpoints are classic examples taught in every Route 53 resource.
🧠 Memory Hook: "Alias targets = AWS-managed DNS names." EC2 has an IP, not a managed DNS name Route 53 can track. EC2 = A record with IP.

Private Hosted Zone — Not Resolving
Hard
For internal service discovery, a team created a private hosted zone internal, added records like payment-service.internal, and associated the zone with their VPC. Yet EC2 instances in that VPC still get NXDOMAIN (no answer) when resolving the names. What is the MOST likely cause?

Which must be true for the private zone to resolve inside the VPC?

  • A
    The VPC must have both enableDnsSupport and enableDnsHostnames enabled.
  • B
    The instances need public IP addresses to query Route 53.
  • C
    An internet gateway must be attached to the VPC for DNS to work.
  • D
    A private hosted zone cannot use a custom TLD like .internal; it must be a real domain.
✅ Correct Answer
A — Enable both VPC DNS attributes

Private hosted zones are resolved by the VPC's built-in Route 53 Resolver (the .2 address in the VPC range). For that to work, the VPC must have both enableDnsSupport (use the resolver) and enableDnsHostnames (assign DNS names) turned on. If a private zone "isn't resolving," these attributes are the first thing to check.

Why B is wrong: Public IPs are irrelevant — private-zone resolution happens entirely inside the VPC via the .2 resolver, not over the internet.

Why C is wrong: An internet gateway provides internet routing, not internal DNS resolution. The private zone needs no internet at all.

Why D is wrong: Private hosted zones happily use custom TLDs like .internal — that's a common, supported pattern.
🎯 How to discriminate: "Private hosted zone associated but not resolving" → check the two VPC attributes: enableDnsSupport + enableDnsHostnames. Public IPs / IGW / custom-TLD answers are distractors — private-zone resolution is purely a VPC-resolver matter.

Multi-Region Performance Architecture
Hard
A media streaming company serves users in North America, Europe, and Asia Pacific. Their application is deployed in us-east-1, eu-west-1, and ap-southeast-1. They want to automatically route each user to the AWS region that provides the best performance for them. If a region becomes unhealthy, users should be automatically routed to the next best region.

Which Route 53 configuration should a solutions architect use?

  • A
    Geolocation routing with health checks, configured per continent
  • B
    Latency-based routing with health checks on each regional endpoint
  • C
    Weighted routing with equal weights and health checks
  • D
    Failover routing with us-east-1 as primary and the other regions as secondary
✅ Correct Answer
B — Latency-Based Routing with Health Checks

The requirement is "best performance" (lowest latency) with automatic failover if a region is unhealthy. Latency-based routing sends each user to the region with the lowest network latency for them. Adding health checks means Route 53 automatically excludes unhealthy regions from responses and routes to the next-best region.

Why A is wrong: Geolocation routes by country/continent — a US user always goes to us-east-1 regardless of latency. Not "best performance" based.

Why C is wrong: Equal weights distribute traffic evenly, but don't optimise for performance — a Singapore user might get sent to Virginia.

Why D is wrong: Failover routing is primary/secondary — it's binary, not multi-region performance optimisation.
🧠 Memory Hook: "Best performance globally = Latency + Health Checks." Latency finds the fastest; health checks skip the broken ones.

DNS Firewall Use Case
Medium
A security team has discovered that some EC2 instances in their VPC may be infected with malware. The malware communicates with external command-and-control servers by resolving specific domain names. The security team wants to prevent all VPC resources from resolving known malicious domains, without modifying security groups or NACLs.

Which AWS service should be used to block these DNS lookups?

  • A
    AWS WAF with a web ACL to block requests to malicious domains
  • B
    VPC Security Groups with outbound rules blocking DNS queries
  • C
    Route 53 Resolver DNS Firewall with rules blocking known malicious domain names
  • D
    Amazon GuardDuty with a threat list of malicious domains
✅ Correct Answer
C — Route 53 Resolver DNS Firewall

Route 53 Resolver DNS Firewall operates at the DNS resolution layer — it intercepts DNS queries from VPC resources and blocks (or allows) them based on domain name lists. It's the purpose-built service for "prevent VPC resources from resolving malicious/unwanted domains." It works without touching security groups or NACLs.

Why A is wrong: AWS WAF operates at HTTP layer 7 — it doesn't intercept DNS queries.

Why B is wrong: Security groups can block network traffic on port 53, but that would block ALL DNS queries including legitimate ones. It's too blunt and would break the VPC's DNS.

Why D is wrong: GuardDuty detects and alerts on suspicious DNS activity but doesn't block DNS queries. It's threat detection, not prevention.
🧠 Memory Hook: "Block VPC DNS = DNS Firewall." GuardDuty alerts. DNS Firewall blocks. Different jobs.

Architecture — Full DR Setup
Hard
A company runs a critical web application in us-east-1. They have set up a DR environment in eu-west-1. Their requirements are:
• The root domain app.company.com should point to an ALB in us-east-1
• If us-east-1 becomes unavailable, traffic should automatically shift to an ALB in eu-west-1
• The solution must use the cheapest possible DNS record type
• Health checks must detect failure at the application layer (HTTP), not just TCP connectivity

Which combination of Route 53 configurations meets ALL requirements?

  • A
    Two CNAME records with Failover routing and TCP health checks
  • B
    Two A records (with ALB IPs) with Failover routing and HTTP health checks
  • C
    Two Alias A records pointing to each ALB, with Failover routing and HTTP health checks on the primary record
  • D
    Two Alias A records pointing to each ALB, with Weighted routing (100/0) and HTTP health checks
✅ Correct Answer
C — Alias A records + Failover + HTTP Health Check

Breaking down each requirement:
• Root domain pointing to ALB: Must use Alias A record (CNAME not allowed at zone apex)
• Automatic failover: Requires Failover routing policy
• Cheapest DNS record: Alias records are free (vs CNAME which charges per query)
• Application-layer (HTTP) health check: Route 53 health checks support HTTP protocol, which checks for HTTP 2xx/3xx responses — detecting app-level failures not just TCP connectivity

Why A is wrong: CNAME at zone apex is not allowed. Also, TCP health checks don't verify application-level health (app could return 503 and TCP would still pass).

Why B is wrong: ALB IPs change dynamically — hardcoding them in A records would break. Must use Alias for ALBs.

Why D is wrong: Weighted routing with 100/0 mimics failover but doesn't automatically switch when the primary fails health check. Only Failover policy does automatic health-check-triggered switching.
🧠 Memory Hook: "DR checklist: Alias (zone apex + free) + Failover (auto-switch) + HTTP HC (app-layer detection)." Three ingredients, all required.

IP-Based Routing — Known Networks
Medium
A SaaS company has a peering agreement with a large regional Internet Service Provider (ISP). They want all DNS queries that originate from that ISP's known IP address ranges (specific CIDR blocks) to be answered with the IP of a dedicated, co-located endpoint, while everyone else is routed normally. The mapping of networks to endpoints is something the company defines itself.

Which Route 53 routing policy meets this requirement?

  • A
    Geolocation routing
  • B
    Latency-based routing
  • C
    IP-based routing
  • D
    Geoproximity routing
✅ Correct Answer
C — IP-Based Routing

IP-based routing lets you map specific client IP ranges — defined as CIDR blocks grouped into a CIDR collection — to specific answers. That is exactly the "known ISP / known network → specific endpoint" use case, where you control the mapping. It was added in 2022, so it is missing from many older "7 routing policies" lists.

Why A is wrong: Geolocation routes by the user's country or continent, not by the ISP's specific IP ranges.

Why B is wrong: Latency routing picks the fastest region by measured network speed — it cannot target one ISP's network.

Why D is wrong: Geoproximity routes by geographic distance (with an optional bias), again not by IP/CIDR ranges you define.
🧠 Memory Hook: "Known IP ranges / specific ISP → IP-Based. Country → Geolocation. Speed → Latency. Distance → Geoproximity."

Split-Horizon DNS
Hard
A company needs the same hostname, api.example.com, to resolve to a private internal ALB when queried from inside their VPC, but to a different public endpoint when queried by external partners over the internet. How should they configure Route 53?

Which configuration achieves this?

  • A
    One public hosted zone with two A records for api.example.com (the internal IP and the public IP).
  • B
    A public hosted zone for example.com (public endpoint) AND a private hosted zone for example.com associated with the VPC (internal ALB) — split-view DNS.
  • C
    A Route 53 Resolver inbound endpoint that rewrites the record based on the query source.
  • D
    A single private hosted zone, plus a NAT gateway so external partners can reach it.
✅ Correct Answer
B — Public + Private hosted zone with the same name (split-view)

This is split-horizon (split-view) DNS. You create a private hosted zone for example.com associated with the VPC (resolving to the internal ALB) and a public hosted zone for the same domain (resolving to the public endpoint). Queries from inside the VPC hit the private zone first; everyone else gets the public zone.

Why A is wrong: A single public zone with two A records hands both IPs to everyone — it leaks the internal address and gives external users an unreachable private IP. No isolation.

Why C is wrong: Resolver inbound endpoints let on-prem systems query Route 53; they don't rewrite answers based on who's asking.

Why D is wrong: A private hosted zone isn't visible outside the VPC at all, so external partners could never resolve it — a NAT gateway doesn't change DNS visibility.
🎯 How to discriminate: "Same name resolves differently inside the VPC vs outside" → split-view DNS: a private hosted zone (associated with the VPC) and a public hosted zone sharing the same domain name. The VPC always prefers the private zone.

Apex Domain + CloudFront + ACM
Hard
A company must serve its apex domain example.com (no www) over HTTPS through CloudFront, whose origin is an app in eu-west-1. Their first attempt fails: the Route 53 record won't validate and CloudFront refuses the ACM certificate. Which configuration is correct?

Which option fixes BOTH problems?

  • A
    A CNAME record for example.com → the CloudFront domain, with the ACM cert in eu-west-1.
  • B
    An Alias A record for example.com → the CloudFront distribution, with the ACM certificate requested in us-east-1.
  • C
    An Alias A record → CloudFront, with the ACM certificate in eu-west-1 (same region as the origin).
  • D
    An A record containing CloudFront's IP addresses, with the ACM cert in us-east-1.
✅ Correct Answer
B — Alias A record + ACM certificate in us-east-1

Two CloudFront gotchas are bundled here. (1) You can't use a CNAME at the zone apex, and CloudFront has no fixed IPs — so the apex must be an Alias A record pointing at the distribution. Why A is wrong: CNAME is illegal at the apex (the apex rule), and the cert is in the wrong region — two failures.

Why D is wrong: CloudFront has no static IP addresses to hardcode in an A record — you must use an Alias.
