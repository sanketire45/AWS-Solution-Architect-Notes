Section 01

What is DNS? Starting from Zero

🤔 Why Was DNS Even Invented — What Problem Did It Solve?

In the 1970s the internet had a few hundred computers. Every computer kept a single text file called HOSTS.TXT listing every other computer's name and IP. One team at Stanford was manually updating this file. When you wanted to add your server to the internet, you literally emailed Stanford and waited for the next release of the file.

😱 The Problem By 1983 the internet had thousands of machines. The HOSTS.TXT file became huge. Updates had to be pushed to every machine manually. It was like updating every phone book in the country every time someone moved. It was completely unmanageable. Jon Postel and Paul Mockapetris invented DNS in 1983 to solve this.
✅ What DNS Solved Instead of one giant file maintained by one team, DNS became distributed. Every organisation manages their own piece of the namespace. The system scales to billions of records automatically. Nobody emails Stanford anymore.
🏠 Real World Analogy — The Phone Book
Imagine you want to call your friend "Sarah." You don't memorise her 10-digit phone number — instead you look up "Sarah" in your contacts and your phone dials the number for you. DNS is the internet's phone book. You type a name like google.com, and DNS looks up the actual "phone number" (IP address) for that server so your browser knows where to connect.

The Problem DNS Solves

Every device on the internet — your laptop, a web server, your phone — has a unique numerical address called an IP address. Think of it like a house address. For example, Google's server might be at 142.250.80.46.

But nobody wants to type 142.250.80.46 into a browser every time they want to visit Google. That's where DNS (Domain Name System) comes in. It translates human-friendly names (google.com) into machine-friendly IP addresses (142.250.80.46).

How DNS Resolution Works — Step by Step

When you type www.example.com in your browser, here's what actually happens behind the scenes:

DNS Resolution Flow
  • ①
    You type a URL

    Your browser asks: "Hey DNS Resolver, what's the IP address for www.example.com?"

  • ②
    DNS Resolver checks its cache

    If it already knows the answer (from a recent lookup), it returns immediately. If not, it starts asking.

  • ③
    Root Nameserver is asked

    The Root NS is like the top of the phone book. It doesn't know the final answer but says "for .com domains, ask the .com nameserver."

  • ④
    .com TLD Nameserver is asked

    The TLD (Top Level Domain) server says "for example.com specifically, go ask Route 53" (the authoritative nameserver).

  • ⑤
    Route 53 gives the final answer

    The authoritative nameserver returns the actual IP address. The resolver caches it and sends it back to your browser. Done!

💡 Key Terms
Authoritative Nameserver: The final, definitive source of truth for a domain. If Route 53 is your authoritative NS, it holds the official DNS records for your domain.

Recursive Resolver: The helper that does the legwork of asking the Root → TLD → Authoritative chain on your behalf. Usually run by your ISP or services like Google (8.8.8.8) or Cloudflare (1.1.1.1).

Section 02

What is Amazon Route 53?

🤔 Why Does Route 53 Exist — What Pain Did It Solve?

Before managed DNS services like Route 53, every company running a website had to manage their own DNS servers — install BIND (DNS server software) on Linux, set up multiple servers for redundancy, monitor them 24/7, patch them, replicate records across all of them. One misconfiguration and your entire website vanishes from the internet globally.

😱 The Old Way Companies needed to hire DNS specialists. A single misconfigured firewall rule meant DNS servers became unreachable. No managed scaling — traffic spikes meant DNS timeouts. Every region needed its own DNS infrastructure. Total cost: tens of thousands per year in infrastructure and engineering time.
✅ What Route 53 Solved AWS took over the entire operational burden. You add records via a web interface or API — AWS handles global replication, redundancy, auto-scaling, patching, and monitoring. You pay cents per million queries instead of thousands per year. And you get a 100% uptime SLA that no self-managed setup could match.

Amazon Route 53 is AWS's managed DNS service. It does three main things:

📋

1. Domain Registration

Buy and register domain names directly through AWS (like buying myapp.com). Route 53 becomes your registrar.

🗂️

2. DNS Hosting

Hosts your DNS records. Route 53 becomes the authoritative nameserver for your domain — it answers "what IP is this domain?"

🚦

3. Traffic Routing

Routes internet traffic intelligently — to the nearest server, healthiest server, by location, by weight, and more. This is Route 53's superpower.

Why is it called "Route 53"?

✨ Fun Fact
DNS runs on port 53. AWS named the service "Route 53" as a play on this — it routes DNS traffic on port 53. Simple but memorable!

Route 53 vs Other DNS Services

You could use other DNS providers like Cloudflare, GoDaddy, or your own servers. So why use Route 53? Because it's deeply integrated with every AWS service — you can point your DNS directly to an ALB, CloudFront distribution, S3 bucket, or EC2 instance using a special type of record called an Alias (more on that later).

🏢 Industry Context — Who Uses Route 53?

Nearly every large company running workloads on AWS uses Route 53 for production DNS. Netflix, Airbnb, Samsung, and thousands of startups rely on it for global traffic distribution. A typical use case: A US user visiting app.netflix.com gets routed to US servers; an EU user gets routed to EU servers — all via Route 53 routing policies.

The 100% SLA — A Big Deal

AWS guarantees 100% uptime for Route 53 DNS. This is genuinely rare — most services only promise 99.9% or 99.99%. If Route 53 goes down, AWS pays you. This is why Route 53 is trusted for mission-critical global routing.

Section 03

DNS Record Types Explained

🤔 Why Do Multiple Record Types Exist — Can't We Just Have One?

Why not just one record type that points a name to an IP? Because the internet has very different needs depending on what you're trying to route.

😱 The Problem Your website runs on one server. Your email goes to a different server. Sometimes you want "shop.example.com" to be an alias for "store.example.com". Sometimes you need to prove domain ownership by adding a text value. Sometimes you need IPv6 addresses. One record type cannot cleanly handle all of these different purposes.
✅ What Different Record Types Solve Each record type has a specific job. A records are for web servers. MX records route email. CNAME creates nicknames. TXT proves domain ownership (Stripe tells you "add this TXT record to verify you control this domain"). AAAA handles IPv6. Clean separation means each system knows exactly which records to look for.

Think of DNS records as entries in your phonebook. Each type of record stores a different kind of information. Here are the ones that matter most:

Record Type Full Name What it stores Simple Analogy
A Address Maps a domain name → IPv4 address "Sarah's house is at 42 Oak Street"
AAAA IPv6 Address Maps a domain name → IPv6 address Same as A but with a newer format of addresses
CNAME Canonical Name Maps a name → another name (alias). Points to another domain, not an IP. "Sarah's nickname is 'Sal' — same person, different name"
Alias AWS-specific Like CNAME but for AWS resources (ALB, CloudFront, S3). Free. Works at zone apex. AWS-native shortcut that always resolves to the current IP automatically
NS Name Server Lists the authoritative nameservers for the domain "For this neighbourhood, the official address book is kept at City Hall"
MX Mail Exchange Tells where to send email for this domain "All letters for this address go to P.O. Box 12"
TXT Text Stores arbitrary text. Used for domain verification, SPF email security "Sticky note attached to the address entry"
SOA Start of Authority Administrative information about the zone (who manages it, TTL defaults, etc.) "The cover page of the phone book — who printed it and when"

The One You'll See Most: A Record

; Simple A record example in Route 53
www.example.com.   A   TTL=300   1.2.3.4
; This means: "www.example.com points to IP 1.2.3.4 for 300 seconds"

CNAME Limitation — The Zone Apex Rule

⚠️ Critical DNS Rule
A CNAME cannot be created at the zone apex (root domain). The "zone apex" means the naked domain itself — example.com (no subdomain in front). You can create a CNAME for www.example.com, but you cannot create a CNAME for example.com itself. This is an RFC standard, not just a Route 53 rule. This is why AWS invented the Alias record.

Section 04

Hosted Zones — The Container for DNS Records

🤔 Why Do We Need Hosted Zones — Why Not Just One Global DNS Store?

Imagine you own three domains: mycompany.com, mystore.ie, and internal.corp. You have different records for each. Some should be visible to the whole internet. Some should only resolve inside your VPC network.

😱 Without Hosted Zones All DNS records for all your domains in one giant pile. Internal server names like db01.internal.corp would be queryable from the public internet — a security risk. Any mistake in one domain's records could affect another. No clear ownership boundaries for different teams.
✅ What Hosted Zones Give You One folder per domain. All records for mycompany.com live in its zone. Public zones answer global DNS queries. Private zones only answer queries from inside specific VPCs — invisible to the outside world. Different teams own different zones.
📂 Analogy
A Hosted Zone is like a folder that contains all the DNS records for a domain. If you own mycompany.com, your hosted zone holds all the records for that domain: the A record for www, the MX records for email, etc.
🌍

Public Hosted Zone

For domains that are accessible from the public internet. When someone types www.example.com anywhere in the world, Route 53 answers them.

Cost: $0.50/month per zone

🔒

Private Hosted Zone

For domains only resolvable inside your VPC. Like internal company DNS — database.internal or api.corp — invisible to the outside world.

Cost: $0.50/month per zone

Public vs Private — When to Use Which

ScenarioUse
Your public website www.example.com Public Hosted Zone
Internal microservice URL payments.internal.company.com accessed only by other services inside your VPC Private Hosted Zone
Staging environment only accessible to dev team inside VPN/VPC Private Hosted Zone
API that external customers call Public Hosted Zone
🏭 Industry Use — Private DNS
In large enterprises, Private Hosted Zones are extensively used for service discovery inside a VPC. Instead of hardcoding IP addresses in your application config, services call each other by DNS names like orders-service.prod.internal. When the IP changes (e.g., after a deployment), only the DNS record needs updating, not every service that talks to it.

Section 05

TTL — The "Cache Timer"

🤔 Why Does TTL Exist — Why Not Always Query Fresh?

Google gets over 8.5 billion searches per day. Imagine if every browser, for every page load, had to go all the way to Google's authoritative nameserver to ask "what IP is google.com?" fresh every single time.

😱 Without TTL Caching DNS infrastructure would need to handle trillions of fresh queries per day globally. Every DNS server would be overwhelmed. Response times would skyrocket. The whole internet would slow to a crawl. It simply wouldn't scale.
✅ What TTL Solves TTL tells resolvers: "cache this answer for X seconds, don't ask again until it expires." Your ISP's resolver asks for google.com once, caches it for 300 seconds, serves the cached answer to the next 10,000 users without contacting Google's nameserver again. Trade-off: if you change your IP, users see the old IP until their TTL expires — which is why you lower TTL before migrations.
🥛 Analogy — Milk Expiry Date
When a DNS resolver looks up an IP address, it writes it down and remembers it for a while before asking again. TTL (Time To Live) is how long that cached answer is "fresh." Like a milk carton — after the TTL expires, the resolver must go ask Route 53 again for the latest answer.

TTL is measured in seconds. A record with TTL=300 means the cached answer is valid for 5 minutes.

TTL ValueCaching DurationWhen to UseTrade-off
60 1 minute During migrations, or when IPs change often More DNS queries, higher Route 53 cost
300 5 minutes Good balance for most apps —
3600 1 hour Stable production environments Changes take 1 hr to propagate
86400 24 hours Rarely-changed records (e.g. MX, NS) Very slow propagation of changes

The Golden Migration Trick

If you're planning to change an IP address or move traffic somewhere new, lower your TTL to 60 seconds a few days before the migration. This ensures that once you make the change, the whole internet picks it up in 60 seconds rather than waiting hours. Then raise the TTL back after the migration is stable.

🎯 Exam Trigger
If the question mentions DNS changes taking too long to propagate, or asks how to speed up a migration — the answer is lower the TTL before making the change.
🪤 Trap
Alias records do NOT let you control TTL — Route 53 manages the TTL for Alias records automatically. Only regular records (A, CNAME, etc.) let you set TTL.

Section 06

Alias Records vs CNAME — A Critical Difference

This is one of the most important and most-tested distinctions in Route 53. Let's nail it perfectly.

The Problem with CNAME

CNAME just points a name to another name. For example: app.example.com CNAME → my-alb-1234.us-east-1.elb.amazonaws.com

Two problems with this:
① Not allowed at the zone apex — you can't have example.com CNAME pointing to the ALB. It must be an A or AAAA record at the apex.
② Costs money — every CNAME lookup costs a DNS query charge in Route 53.

The Alias Record — AWS's Solution

Route 53's Alias record is a special AWS extension that acts like an A record but automatically resolves to the current IP(s) of an AWS resource. It's smart enough to:

  • Work at the zone apex (example.com pointing to an ALB) ✅
  • Be completely free (no charge for Alias DNS queries) ✅
  • Automatically follow IP changes of the target (ALB IPs change over time — Alias handles this) ✅
  • Perform health checking on the target ✅

Alias Record Targets

Alias records can point to these AWS resources:

AWS ResourceUse Alias?
Application Load Balancer (ALB)✅ YES
Network Load Balancer (NLB)✅ YES
Classic Load Balancer✅ YES
CloudFront Distribution✅ YES
Elastic Beanstalk environment✅ YES
API Gateway (regional)✅ YES
S3 website endpoint✅ YES
Another Route 53 record in the same zone✅ YES
EC2 instance (directly)❌ NO (use A record with the IP)

🎯 Exam Trigger
"Route example.com (root domain) to an ALB" → The answer is always Alias A record pointing to the ALB. CNAME is wrong because the zone apex cannot have a CNAME.
🪤 Trap
You CANNOT create an Alias record pointing to an EC2 instance directly. Use a plain A record with the instance's IP address. The reason: EC2 IPs can change and AWS doesn't manage that lifecycle for Alias records.

Section 07

The 7 Routing Policies — Route 53's Superpowers

🤔 Why Do Multiple Routing Policies Exist — Isn't Basic DNS Enough?

Basic DNS is "type a name, get an IP." But the real world is more complex.

😱 What Basic DNS Can't Do A user in Singapore gets routed to your Virginia server: 200ms+ latency, terrible UX. Your site crashes at 2am: DNS still happily returns the dead server's IP. You want to test new code on 5% of users first: impossible. A German user's data accidentally goes to US servers violating GDPR. Basic DNS has no solution for any of these.
✅ What Routing Policies Add Each policy adds a different intelligence layer: Latency adds "route to fastest server." Geolocation adds "route by user's country." Failover adds "auto-switch when primary dies." Weighted adds "send X% of traffic here." Together they turn DNS from a dumb phone book into a smart global traffic controller. This is why the AWS exam tests these heavily — they solve real architectural problems.
🚦 Analogy — GPS Navigation Modes
Imagine you're a GPS routing cars through a city. You might use different strategies: "Take everyone to the same destination (Simple)," "Send 70% on Highway A, 30% on Highway B for testing (Weighted)," "Find the fastest route for each driver (Latency)," "If the main road is closed, take the backup (Failover)," "Route based on the driver's starting city (Geolocation)." Route 53 routing policies work exactly like different GPS navigation strategies.

Route 53 has 7 routing policies. Each one is a different strategy for deciding which IP address to return when someone queries your domain. Let's master each one:

➡️
1. Simple Routing
One domain → one or more IPs
Beginner

What it does: The most basic policy. You set one domain to point to one (or more) IP addresses. Route 53 just returns the answer — no intelligence, no conditions.

Multiple values? If you specify multiple IPs, Route 53 returns all of them in the response and the client picks one randomly. This is NOT load balancing — it's random selection by the client.

Health Checks? Not supported in Simple routing. If the server is down, Route 53 still returns its IP.

🎯 Exam Trigger
"Route all traffic to a single server, basic setup, no health checks needed" → Simple Routing.
⚖️
2. Weighted Routing
Split traffic by percentage
Common

What it does: You assign a weight to each DNS record. Route 53 distributes traffic in proportion to those weights.

How weights work: If Record A has weight 70 and Record B has weight 30, then 70% of queries go to A and 30% to B. The actual numbers don't have to add up to 100 — Route 53 calculates percentages automatically.

Special case: If you set a record's weight to 0, Route 53 stops sending traffic to it (useful for taking a server offline for maintenance).

Real-world uses:

  • A/B testing — send 10% of users to a new version, 90% to the old version
  • Canary deployments — gradually shift traffic to new servers (10% → 25% → 50% → 100%)
  • Blue-Green deployments — quickly switch from old (blue) to new (green) deployment
  • Load distribution across different server capacities
🎯 Exam Trigger
"Distribute 10% of traffic to a new version" or "A/B testing" or "gradually shift traffic" → Weighted Routing.
⚡
3. Latency-Based Routing
Route to the fastest AWS region
Performance

What it does: Routes users to whichever AWS region has the lowest latency (fastest response time) for that user. AWS measures latency between user locations and regions continuously.

Important nuance: Latency ≠ Geography. A user in London might actually get lower latency to us-east-1 than eu-west-1 if, for example, the East US data centre has a faster connection for that network path. Route 53 uses real measured latency, not geographic proximity.

🎯 Exam Trigger
"Route users to the nearest/fastest region" or "minimize latency for global users" → Latency-Based Routing.
🪤 Trap
Latency ≠ Geolocation. Latency picks by network speed. Geolocation picks by physical country/continent. A US user is not guaranteed to be routed to US servers with Latency routing — they go to whichever region is fastest for them.
🔁
4. Failover Routing
Primary / Secondary DR setup
DR Critical

What it does: You designate one record as Primary and another as Secondary. Route 53 sends all traffic to Primary. If the Primary fails its health check, Route 53 automatically switches to Secondary — without any manual intervention.

Use case: Active-passive disaster recovery. Your production environment is Primary; your DR environment is Secondary. If your primary region goes down, Route 53 automatically redirects users to the DR site.

Health checks are mandatory — without them, Route 53 doesn't know when to failover.

🎯 Exam Trigger
"DR switchover if primary fails," "active-passive setup," "secondary should be used only if primary is unhealthy" → Failover Routing.
🗺️
5. Geolocation Routing
Route based on user's country/continent
Compliance

What it does: Routes users based on their physical location — country or continent. A user in Germany gets routed to one server; a user in the USA gets routed to another.

Important: You must always have a Default record for users who don't match any geographic rule. Without it, Route 53 returns "no answer" for unmatched users.

Real-world uses:

  • Data residency compliance — EU users must be served from EU servers (GDPR)
  • Localised content — serve language-specific versions of your site
  • Legal restrictions — block certain content in specific countries
  • Regional pricing — different features or prices per region
🏭 Industry Example
Netflix uses geolocation-based routing (not directly Route 53 routing policies but the concept) to serve region-locked content. A user in the UK sees UK Netflix content; a user in Japan sees Japanese Netflix content. The routing is based on where the user's request originates from.
🎯 Exam Trigger
"Route European users to EU servers," "comply with data residency laws," "serve language-specific content by country" → Geolocation Routing.
🪤 Trap
Always configure a Default record in Geolocation routing. Without it, users from countries you haven't explicitly configured will get no DNS answer and won't be able to reach your site.
📍
6. Geoproximity Routing
Route based on distance + adjustable bias
Advanced

What it does: Routes users based on the geographic distance between the user and your AWS resources. But here's the unique part — you can add a bias to expand or shrink the geographic area each endpoint serves.

Bias explained:

  • Positive bias (+1 to +99) → expands the region's catchment area (attracts more traffic)
  • Negative bias (-1 to -99) → shrinks the region's catchment area (sends less traffic to it)

Requires: Route 53 Traffic Flow (the visual policy editor). This is important — you cannot set up Geoproximity routing from the standard console, you must use the Traffic Flow feature.

💡 Geoproximity vs Geolocation
Geolocation: "If user is in Germany → send to EU server" (hard country/continent rules)

Geoproximity: "Send user to the nearest resource, but I can shift the boundaries using bias" (distance-based with tuning knobs)
🎯 Exam Trigger
"Shift traffic from one region to another based on geographic distance," "use bias to expand coverage area," "requires Traffic Flow" → Geoproximity Routing.
🔢
7. Multi-Value Answer Routing
Return multiple healthy IPs
Health-Aware

What it does: Returns up to 8 healthy records in response to a DNS query. Clients can then choose any of those IPs. Unlike Simple routing with multiple values, Multi-Value checks health and removes unhealthy endpoints from responses.

Not a load balancer: Route 53 cannot truly balance load between servers — it only influences DNS. Load balancing requires an actual ALB/NLB. But Multi-Value with health checks is often a useful "DNS-level best effort" distribution.

⚠️ NOT a Load Balancer
AWS explicitly states: Multi-Value Answer is NOT a substitute for a load balancer. It improves availability by only returning healthy IPs, but does not evenly distribute requests across them. Use an ALB or NLB if you need true load balancing.
🎯 Exam Trigger
"Return multiple IPs, remove unhealthy ones from DNS responses" → Multi-Value Routing. If the question asks about real load balancing → use ALB/NLB.

Health Checks — How Route 53 Monitors Your Resources

🤔 Why Do Health Checks Exist — Why Doesn't DNS Know Servers Are Down?

DNS is fundamentally just a database. It stores "domain → IP" mappings. It has zero awareness of whether those servers at those IPs are actually working.

😱 The Problem Your server at 1.2.3.4 crashes at 3am. DNS continues returning 1.2.3.4 to every user in the world because that's what's in the database. Every user's browser tries to connect to a dead server and times out. Your website is down — and DNS is happily directing more and more frustrated users to the broken address. Without health checks, DNS is completely blind.
✅ What Health Checks Solve Health checks give Route 53 eyes. Every 10–30 seconds, Route 53's global health checkers probe your servers. If a server stops responding, Route 53 marks it unhealthy and — combined with Failover or Multi-Value routing — stops returning that IP to users. Automatic failover in seconds, not hours. No 2am wake-up call needed.
🏥 Analogy — The Night Security Guard
Imagine a security guard who walks around your building every 10–30 seconds checking if each server room is "responding." If a room goes dark (no response), the guard removes that room's address from the directory. Route 53 health checks are those security guards.

Three Types of Health Checks

🌐

Type 1: Endpoint Health Check

Route 53 directly hits your endpoint (IP or domain) via HTTP, HTTPS, or TCP. Checks every 10 or 30 seconds. An endpoint is considered healthy if it responds with HTTP 2xx or 3xx status codes.

Used when: your servers are publicly accessible from the internet.

🔗

Type 2: Calculated Health Check

Combines the results of multiple other health checks using AND/OR/NOT logic. For example: "healthy only if at least 2 out of 3 endpoints are healthy."

Used when: you want health to be determined by multiple conditions.

☁️

Type 3: CloudWatch Alarm Check

Route 53 treats a resource as healthy or unhealthy based on whether a CloudWatch Alarm is in OK or ALARM state. This is the solution for private VPC resources that aren't internet-accessible.

Used when: your resources are inside a private VPC.

Health Checks for Private Resources

There's a key challenge: Route 53 health checkers are on the public internet. They cannot reach resources inside a private VPC directly. Here's the solution:

Health Check for Private VPC Resource

How Many Health Checkers?

AWS has health checkers in multiple global locations. For an endpoint to be considered healthy, more than 18% of health checkers must report it as healthy. This prevents a single location's network issue from marking your healthy server as down.

🎯 Exam Trigger
"How do you monitor a private VPC resource with Route 53 health checks?" → Create a CloudWatch metric/alarm for the resource, then create a Route 53 health check that monitors the CloudWatch alarm state.

Health Check + Routing = Smart DNS

Health checks become truly powerful when combined with routing policies. In Failover, Weighted, Latency, Geolocation, and Multi-Value policies, Route 53 will automatically exclude unhealthy endpoints from DNS responses. This means if a server goes down, Route 53 stops telling users to go there — without any manual intervention.

Domain Registration with Route 53

Route 53 is also an ICANN-accredited domain registrar. You can buy domain names (like myapp.com) directly from AWS, without needing a third-party registrar like GoDaddy.

How It Works

  • 1
    Search & Buy a Domain

    In the Route 53 console, search for available domain names. Pricing varies by TLD (e.g., .com ≈ $12/year, .io ≈ $39/year).

  • 2
    Route 53 Creates a Hosted Zone Automatically

    When you register a domain, Route 53 automatically creates a Hosted Zone for it and sets the domain's NS records to point to Route 53's nameservers.

  • 3
    Add DNS Records

    Now add A records, CNAME records, etc. to your Hosted Zone to point the domain at your servers.

Migrating an Existing Domain to Route 53

If you already own a domain with another registrar (GoDaddy, Namecheap, etc.) and want to use Route 53 as the DNS provider, you have two options:

OptionWhat it doesKeep existing registrar?
DNS Migration only Create a Hosted Zone in Route 53, then update the NS records at your registrar to point to Route 53's nameservers. Route 53 handles DNS, registrar stays the same. Yes
Full Transfer Transfer the domain registration itself to Route 53. Takes 5–10 days, domain must be unlocked, transfer key (EPP code) required. AWS takes over as registrar. No (transferred to AWS)
💡 Pro Tip
Most companies migrate DNS hosting to Route 53 without transferring the domain registration. This is faster, easier, and gives you all the Route 53 routing benefits while keeping your existing registrar relationship.

Section 11

Hybrid DNS — Connecting On-Premises to AWS

In real enterprises, you often have on-premises data centres connected to AWS via Direct Connect or VPN. Both environments need to resolve each other's DNS names.

🤔 Why Does Hybrid DNS Exist — What Industry Problem Forced This?

Almost no large company moves to cloud 100% overnight. It takes years. During that time they have servers both on-premises AND in AWS, and both sides need to talk to each other by name.

😱 The Problem A bank has 500 internal servers with names like payroll-db.corp.bank.com. They've moved their web tier to AWS but payroll still runs on-prem. AWS servers need to call payroll-db.corp.bank.com — but AWS VPC DNS knows nothing about the bank's internal DNS. Meanwhile, on-prem finance tools need to call reporting-api.aws.internal — but on-prem DNS servers know nothing about AWS private hosted zones. Both sides are completely blind to each other.
✅ What Hybrid DNS Solves Route 53 Resolver Endpoints create a two-way DNS bridge over the existing VPN or Direct Connect connection. On-prem DNS forwards AWS domain queries IN to Route 53 via Inbound Endpoints. VPC resources forward corporate domain queries OUT to on-prem DNS via Outbound Endpoints and Forwarding Rules. Both sides can now resolve each other's names without exposing anything to the public internet.
🏢 Real Scenario
Acme Corp has an on-premises data centre and an AWS VPC. Their on-prem servers are named like db01.corp.acme.com and their AWS resources use orders.aws.internal. Each side needs to resolve the other's names.

Route 53 Resolver — The Bridge

Route 53 Resolver is the built-in DNS resolver for VPCs (at IP 169.254.169.253 or the VPC base + 2 address). It handles all DNS queries within the VPC. But for hybrid scenarios, it uses two special features:

⬇️

Inbound Endpoints

Allows on-premises DNS resolvers to forward queries INTO the VPC. For example, when on-prem servers need to resolve orders.aws.internal, they forward to the Inbound Endpoint's IP, and Route 53 resolves it.

⬆️

Outbound Endpoints + Forwarding Rules

Allows VPC queries to be forwarded OUT to on-premises DNS. You set forwarding rules like "for *.corp.acme.com, forward to the on-prem DNS server."

Hybrid DNS Architecture

"On-premises servers need to resolve AWS VPC hostnames" → Route 53 Resolver Inbound Endpoint.
"AWS EC2 instances need to resolve on-premises DNS names" → Route 53 Resolver Outbound Endpoint with Forwarding Rules.

Section 12

DNSSEC and Route 53 Security

🤔 Why Was DNSSEC Needed — Was DNS Really Insecure?

DNS was invented in 1983 for a small academic network where everyone trusted everyone. Security wasn't designed in. For 25 years, nobody really noticed — until 2008.

😱 The Kaminsky Attack — A Wake-Up Call In 2008, security researcher Dan Kaminsky discovered that DNS had a fundamental flaw: an attacker could forge DNS responses. They could tell your resolver "google.com is at MY server." Your resolver would cache this lie for up to TTL seconds. Every time you typed google.com, you'd go to the attacker's fake server. They show you a fake Google login page. You enter your password. This attack — DNS cache poisoning — affected the vast majority of internet infrastructure and was rated one of the worst security bugs of the 2000s.
✅ What DNSSEC Solved DNSSEC adds cryptographic signatures to every DNS record. When your resolver gets a DNS answer, it verifies the signature proves the answer really came from the authoritative server and hasn't been tampered with. A forged answer fails signature verification and is rejected immediately.

What is DNSSEC?

DNSSEC (DNS Security Extensions) adds cryptographic signatures to DNS records. It prevents DNS spoofing — where an attacker tricks a resolver into returning a fake IP address, sending your users to a malicious server.

🔏 Analogy — Signed Documents
Without DNSSEC, DNS answers are like unsigned letters — anyone could forge them. With DNSSEC, each DNS answer is digitally signed, like a notarised document. The receiver can verify the signature is genuine.

Route 53 DNSSEC Support

  • Route 53 supports DNSSEC for domain registration and DNS hosting
  • Uses AWS KMS to store and manage the Key Signing Key (KSK)
  • When you enable DNSSEC on a hosted zone, Route 53 signs all DNS records automatically
  • Available for public hosted zones only (not private)

Route 53 and AWS WAF Integration

Route 53 itself doesn't do DDoS protection, but it integrates with:

  • AWS Shield Standard — free, automatic DDoS protection for Route 53 endpoints
  • AWS Shield Advanced — paid, enhanced DDoS protection with 24/7 response team

Route 53 Resolver DNS Firewall

Route 53 Resolver DNS Firewall allows you to block or allow DNS queries from resources within your VPCs based on domain names. This prevents EC2 instances from communicating with known malicious domains (command-and-control servers).

All the Traps — Consolidated

🪤 Trap 1 — CNAME at Zone Apex
You CANNOT create a CNAME for example.com (root domain). Use an Alias A record instead to point to AWS resources at the zone apex.
🪤 Trap 2 — Latency ≠ Geographic Proximity
Latency routing picks the fastest region based on actual network measurements. A user in the UK could be routed to us-east-1 if it has lower latency for their network.
🪤 Trap 3 — No Default in Geolocation
Without a Default record in Geolocation routing, users from unspecified countries get no DNS answer. Always create a Default record.
🪤 Trap 4 — Multi-Value ≠ Load Balancer
Multi-Value routing returns multiple healthy IPs but does NOT perform actual load balancing. It relies on the client to pick an IP randomly. Use ALB/NLB for true load balancing.
🪤 Trap 5 — Alias Target Cannot Be an EC2 Instance
Alias records work with AWS-managed resources (ALB, CloudFront, S3, etc.) — NOT directly with EC2 instances. Use an A record with the EC2 IP for EC2 instances.
🪤 Trap 6 — Failover Without Health Checks
Failover routing requires health checks on the Primary record. Without a health check, Route 53 never knows the Primary is down and never switches to Secondary.
🪤 Trap 7 — TTL Delays During Migration
If you forget to lower TTL before migrating, users' DNS resolvers may cache the old answer for hours (up to the old TTL). Lower TTL to 60s at least 48 hours before any planned IP change.
🪤 Trap 8 — Geoproximity Requires Traffic Flow
Geoproximity routing cannot be configured from the standard Route 53 console. It requires the Route 53 Traffic Flow visual policy editor.

The Five Route 53 Layers — How the Architecture Actually Works

⚡ Layer 1 — Global API Performance (Latency Routing)
  • Domainapi.payfast.eu — the main endpoint all merchants call for payment processing
  • PolicyLatency-based routing. Three Alias A records — one per region — each pointing to a regional ALB. Route 53 routes each request to the region with lowest measured latency.
  • ResultDublin merchant at checkout: hits eu-west-1 at ~8ms. Tokyo merchant: hits ap-southeast-1 at ~6ms. New York merchant: hits us-east-1 at ~12ms.
  • Health ChecksHTTP check on /health endpoint every 10 seconds. 3 consecutive failures = mark unhealthy. Route 53 automatically stops routing to that region.
  • TTLManaged by Alias records (AWS-controlled). The ALB IPs change as AWS scales — Alias tracks this automatically.
🔒 Layer 2 — GDPR Compliance (Geolocation Routing)
  • Domaineu-data.payfast.eu — a dedicated endpoint for processing EU cardholder personal data
  • PolicyGeolocation routing. EU continent → always routes to eu-west-1 regardless of latency. Non-EU origin → Default record → returns an appropriate response.
  • Why not Latency?Latency routing could send an EU user to us-east-1 if it has lower network latency. That would transfer EU personal data to US servers — a direct GDPR violation. Geolocation's unconditional country/continent rules prevent this.
  • Interview point"We use Geolocation routing as a defence-in-depth DNS-layer control for GDPR. Even if an application bug sends a request to the wrong endpoint, the DNS layer itself enforces EU data stays in eu-west-1."
🔁 Layer 3 — Disaster Recovery (Failover Routing)
  • Domainpayments.payfast.eu — the critical real-time payment processing endpoint
  • SetupPRIMARY: Alias A → ALB in eu-west-1 (Dublin). SECONDARY: Alias A → ALB in eu-central-1 (Frankfurt). Both on Failover routing policy. Health check on PRIMARY every 10 seconds.
  • ScenarioDublin region has connectivity issues at 2:47am. After 3 failed health checks (30 seconds), Route 53 stops returning the Dublin ALB. All traffic shifts to Frankfurt. Merchants experience a brief slowdown. On-call engineer wakes up to a Slack alert — but the crisis was already auto-handled.
  • Data layerAurora Global Database replicates Dublin data to Frankfurt with ~1 second lag. If needed, Frankfurt can promote to primary write region within 1 minute.
  • TTL60 seconds on these records — fast failover propagation is critical for payment SLAs.
🐤 Layer 4 — Zero-Downtime Deploys (Weighted Routing)
  • Domaincheckout.payfast.eu — deployed multiple times per week
  • Normal stateOne Weighted record pointing to the stable ALB. Weight=100.
  • Deploy dayNew version deploys to a "canary" ALB target group. Second Weighted record added for canary: Weight=5. Route 53 sends 5% of real checkout traffic to new version.
  • MonitoringDataDog tracks error rates and payment success rate per version separately. If canary shows elevated errors → immediately set its weight to 0. Full rollback in seconds, no deployment pipeline needed.
  • GraduatedHealthy after 30 mins at 5% → 20% → 50% → 100%. Total deploy time ~2 hours. Zero downtime. Zero customer complaints.
🔗 Layer 5 — Internal Microservices (Private Hosted Zone)
  • Zonepayfast.internal — Private Hosted Zone associated only with production VPCs
  • Records fraud-detection.payfast.internal → Alias → internal ALB for fraud service
    card-vault.payfast.internal → Alias → PrivateLink endpoint (NLB) for PCI card vault
    payments-db.payfast.internal → CNAME → Aurora cluster writer endpoint
    redis.payfast.internal → CNAME → ElastiCache cluster endpoint
  • Why it mattersThe checkout service calls http://fraud-detection.payfast.internal/check — it never knows the IP. When fraud team deploys to a new ALB, they update one DNS record. Zero checkout service config changes needed.
  • Securitycard-vault.payfast.internal resolves to a PrivateLink endpoint. PCI cardholder data traffic never touches the public internet. Compliance requirement met at DNS layer.
🏢 Layer 6 — On-Premises Integration (Hybrid DNS)
  • SituationLegacy finance accounting system still runs on-premises (finance.corp.payfast.eu). New AWS reporting service needs to call it. On-prem compliance tools need to call AWS services.
  • AWS→PremRoute 53 Resolver Outbound Endpoint + Forwarding Rule: "For *.corp.payfast.eu, forward DNS queries to on-prem DNS server at 10.0.0.53" (reachable via Direct Connect).
  • Prem→AWSRoute 53 Resolver Inbound Endpoint created in eu-west-1 VPC. On-prem DNS configured to forward *.payfast.internal queries to the Inbound Endpoint IP. On-prem compliance tools can now resolve internal AWS service names.

Key Industry Lessons from This Architecture

🎯

DNS Is a Multi-Tool, Not a Phonebook

In production, Route 53 is simultaneously a load balancer, a DR switch, a GDPR enforcer, a deployment mechanism, and a service registry. Each routing policy solves a different business problem.

🏗️

Policies Layer on Top of Each Other

Real architectures combine multiple policies: Latency for performance, Geolocation for compliance, Failover for DR, Weighted for deploys — all running simultaneously on different record sets in the same hosted zone.

🔒

Private Zones = Service Mesh Foundation

In microservice architectures, Private Hosted Zones ARE the service discovery layer. It's the AWS-native, zero-cost, zero-complexity alternative to Consul or Istio for simple service-to-service discovery.

📐

Everything is Terraform / IaC

In real companies, DNS records are managed as code. The console is for debugging. Route 53 record changes are part of the deploy pipeline — not manual steps. No human manually edits DNS records in production.
