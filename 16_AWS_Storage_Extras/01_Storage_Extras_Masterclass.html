Snow Family · Part 1

Why Snow Exists — The Bandwidth Wall

Sometimes the fastest way to move 100 terabytes is not the internet — it's a truck.

You have 500 TB of video archives on-premises and you want them in S3. The obvious plan: upload them. Let's do the maths.

Even on a fast, dedicated 1 Gbps link running flat-out, you move roughly ~10 TB per day in the real world. 500 TB would take ~7 weeks of saturated bandwidth — during which nobody else can use that link, and any blip restarts your pain. For a petabyte, it's months.

This is the bandwidth wall: past a certain data size, the network is simply too slow (and too expensive in egress/time) to be practical. Physics beats fibre.

💡 Why the Snow Family Was Built

AWS ships you a rugged physical device, you copy your data onto it locally (at LAN speed), and you ship it back. AWS plugs it in and imports your data into S3. No internet bottleneck, no egress fees for the inbound transfer. A Snowball Edge holds ~80 TB; you can run several in parallel. What took weeks over the wire takes days by courier.

Bonus: these devices also do edge computing — run EC2 instances or Lambda functions on the device at a remote site (a ship, a factory, a war zone) with no connectivity, then ship the results back.

🔑 Facts the exam checks

Snow data lands in S3 only (then you lifecycle/integrate from there — it can't import straight to EFS/FSx/EBS/Glacier vault). Devices encrypt everything with KMS, use a tamper-evident enclosure, and carry an E-Ink shipping label that auto-updates the return address. The whole job is tracked and managed from the AWS console.

Snow Family · Part 3

How a Job Works

Order → receive → load → return → AWS imports. A courier round-trip, not a download.

Inbound data transfer to AWS is free — you pay for the box and the days you keep it, not per-GB.
Snow Family · Part 4

DataSync vs Snowball — the decision

The single most-tested Snow question: network or courier?

🔑 The rule (do the bandwidth math)

If the data fits down your link inside the deadline → DataSync (online). Rough guide: up to ~tens of TB with decent bandwidth. If it's too big, the bandwidth is too small, or there's a hard deadline the network can't meet → Snowball (offline). The exam often gives you a size + a bandwidth + a deadline and wants you to realise the network can't make it — pick Snowball.

💵 A Snowball Edge Storage-Optimized job is ~$300 (includes ~10 days of use), +~$30/day after, plus shipping. Inbound data transfer is free. DataSync is $0.0125/GB moved — so 100 TB ≈ $1,250 over the wire if your bandwidth can carry it in time.

🏢 Industry — Snow in the wild

Media archives, remote sites, and "no bandwidth" migrations

Media companies use Snowball to seed their initial S3 data lake (decades of footage) without saturating their link for months. Remote/edge operations — ships, mines, defence, field research — use Snowcone/Snowball Compute Optimized to run inference locally where there's no connectivity, then ship results home. The recurring theme: when physics or geography beats the network, you send a box.

Storage Gateway · Part 1

Why Storage Gateway — The Hybrid Bridge

Keep your on-prem apps exactly as they are, but quietly back them with unlimited cloud storage.

Not every company can move everything to the cloud overnight. Old apps expect a local file share, an iSCSI disk, or a tape drive. Ripping those out is risky and expensive.

Storage Gateway is a small virtual appliance (a VM, or a hardware box) that runs in your data centre and presents familiar local storage interfaces — NFS/SMB file shares, iSCSI block volumes, or a virtual tape library — while transparently storing the data in S3, Glacier, or EBS snapshots behind the scenes. Your apps think they're talking to local storage; really, the cloud is the backend, with a local cache for hot data.

🔑 Storage Gateway vs DataSync — don't mix them up

Storage Gateway = ongoing hybrid access (your on-prem apps keep using cloud-backed storage day after day, with a local cache). DataSync = a one-time/scheduled bulk migration (move the data, then you're done). "Hybrid, cached, on-prem app keeps running" → Gateway. "Migrate this dataset over the network" → DataSync.

Storage Gateway · Part 2

File Gateway — NFS/SMB → S3

Presents NFS or SMB file shares to your on-prem servers; files are stored as objects in S3. Recently-used files are cached locally for low latency. You can lifecycle them to S3-IA/Glacier and integrate with S3 features. There's also an FSx File Gateway variant that gives low-latency on-prem access to an FSx for Windows file system.

  • Trigger: "on-prem app needs an NFS/SMB share, but store the data in S3" → File Gateway.
  • Each file = an S3 object you can see and use natively in the bucket.
Storage Gateway · Part 3

Volume Gateway — iSCSI block, cached vs stored

Presents iSCSI block volumes to your servers, backed by S3, with point-in-time backups as EBS snapshots. Two modes — and the exam loves the difference:

Cached Volumes

Primary data in S3

  • The full dataset lives in S3; only hot data cached locally
  • Local storage stays small
  • Use when: you want most data in the cloud, low on-prem footprint
Stored Volumes

Primary data on-prem

  • The full dataset lives locally (low latency to all of it)
  • Asynchronously backed up to S3 as EBS snapshots
  • Use when: you need fast local access to everything + cloud backup/DR
Storage Gateway · Part 4

Tape Gateway — kill the physical tape robot

Presents a Virtual Tape Library (VTL) over iSCSI. Your existing backup software (Veeam, NetBackup, Commvault…) thinks it's writing to physical tapes; really the "tapes" are stored in S3 and archived to Glacier / Glacier Deep Archive. You keep your backup app and workflows, but ditch the physical tape robots, off-site couriers, and tape rot.

🔑 Trigger

"Replace physical tape backups but keep the existing backup application" → Tape Gateway. (Writing to S3/Glacier directly is the trap — the legacy backup app expects a VTL, not an object API.)

🏢 Dublin Industry — Verified

Ryanair replaced physical tape backup with Tape Gateway — 65% savings

Ryanair (Dublin) migrated its backup estate from physical tape to Storage Gateway Tape Gateway, reported by their engineering team as driving ~65% savings — no more tape hardware, courier rotations, or off-site vaulting, while keeping their existing backup application and writing the virtual tapes to S3/Glacier. It's the archetypal "modernise the backend without rewriting the app" hybrid story.

Exam mapping

"Replace unreliable physical tape backup, keep the backup software, cut cost" → Tape Gateway → S3/Glacier Deep Archive. This exact scenario shows up regularly.

FSx · Part 1

Why FSx — Managed, Specialised File Systems

EFS is great Linux NFS. But what if you need Windows SMB, a supercomputer file system, or NetApp features?

Amazon EFS gives you a managed, elastic Linux NFS file system shared across many EC2 instances. But the world runs on more than Linux NFS.

Windows apps want SMB shares with Active Directory and Windows ACLs. HPC and ML teams want a parallel file system delivering hundreds of GB/s at sub-millisecond latency. NetApp shops want ONTAP features (snapshots, clones, multi-protocol). Amazon FSx is a family of fully-managed file systems that give you exactly those specialised technologies, without running the storage servers yourself.

FSx · Part 2

The 4 Flavours — Pick the Right One

FlavourProtocolPick when…
FSx for Windows File ServerSMBWindows apps, Active Directory, Windows ACLs, DFS, home directories
FSx for LustreLustre (POSIX)HPC / ML / analytics — sub-ms, hundreds of GB/s, S3-integrated
FSx for NetApp ONTAPNFS + SMB + iSCSINetApp migration, multi-protocol, SnapMirror, clones, auto-tiering
FSx for OpenZFSNFSZFS workloads, snapshots/clones, low-latency NFS
⚠️ The most common FSx trap

Sub-millisecond / HPC / ML / "parallel file system" → FSx for Lustre, NEVER EFS. EFS is millisecond-class Linux NFS — good for shared web/app data, not high-performance computing. And "Windows SMB + AD" → FSx for Windows, never EFS (EFS is Linux/NFS only).

FSx · Part 3

FSx for Windows File Server

  • Fully-managed SMB file shares — exactly what Windows apps expect.
  • Integrates with Active Directory; enforces Windows ACLs; supports DFS Namespaces for scale-out.
  • Single-AZ or Multi-AZ (synchronous standby) for high availability.
  • Built on SSD or HDD; supports dedup, shadow copies (user-restorable snapshots), and backups.
💵 ~$0.13/GB-month (SSD) — a 1 TB share ≈ $130/mo; HDD is ~10× cheaper (~$0.013/GB-mo). Multi-AZ roughly doubles it.
FSx · Part 4

FSx for Lustre — the HPC/ML speed monster

Lustre is the file system that powers supercomputers. FSx makes it a managed service: sub-millisecond latency, hundreds of GB/s throughput, millions of IOPS — for ML training, genomics, financial modelling, video rendering.

🔑 S3 integration (tested)

FSx for Lustre links to an S3 bucket: it lazy-loads objects from S3 on first access and can write results back to S3. So you keep your data lake in cheap S3, spin up a blazing-fast Lustre scratch file system for a compute burst, then tear it down. Two deployment types: Scratch (temporary, cheapest, no replication — for short-lived processing) and Persistent (durable, replicated within an AZ — for longer-running workloads).

💵 Scratch Lustre ~$0.14/GB-month; Persistent is priced by throughput tier. Pair with S3 so you only pay for the fast tier during the compute burst.
FSx · Part 5

FSx for NetApp ONTAP & OpenZFS

NetApp ONTAP

Multi-protocol & NetApp features

  • NFS + SMB + iSCSI from one file system
  • NetApp-native: SnapMirror replication, instant clones, snapshots
  • Auto data-tiering (hot on SSD, cold to a cheap capacity pool)
  • Trigger: "migrate on-prem NetApp / need multi-protocol"
OpenZFS

ZFS on NFS

  • Managed OpenZFS over NFS
  • Cheap, instant snapshots & clones
  • Low-latency, high-IOPS NFS workloads
  • Trigger: "ZFS / move Linux NFS workloads with snapshots"
🏢 Industry — FSx in the wild

HPC, ML training, and Windows enterprise file shares

ML and analytics teams pair FSx for Lustre + S3 + GPU EC2: the training data sits in S3, Lustre lazy-loads it at hundreds of GB/s so the expensive GPUs never wait on storage, then results write back to S3. Enterprises lift-and-shift Windows file servers to FSx for Windows with AD integration so existing apps and permissions just work. NetApp customers use FSx for ONTAP to migrate with SnapMirror and keep their tooling.

DataSync & Transfer · Part 1

AWS DataSync — Fast, Managed Data Movement

Move large datasets over the network, on a schedule, up to ~10× faster than DIY tools.

DataSync is a managed service for moving large amounts of file/object data over the network — on-prem ↔ AWS, or between AWS storage services.

You run a lightweight agent on-prem (a VM) that reads your NFS/SMB/HDFS/object source and pushes to AWS — and it does so up to 10× faster than open-source copy tools, with built-in scheduling, retries, encryption, and data-integrity validation. It preserves file metadata (permissions, timestamps).

  • Targets: S3, EFS, FSx (all flavours) — unlike Snow, DataSync can land directly in EFS/FSx, not just S3.
  • Use for: one-time or scheduled, recurring bulk migration/replication over the wire.
  • Not for: ongoing low-latency hybrid access (that's Storage Gateway) or petabytes with no bandwidth (that's Snowball).
💵 $0.0125 per GB copied — a 50 TB migration ≈ $640, plus your normal bandwidth. Cheap and simple if the network can carry it in time.
DataSync & Transfer · Part 2

AWS Transfer Family — Managed SFTP/FTPS/FTP/AS2

Plenty of businesses still exchange files with partners over SFTP, FTPS, FTP, or AS2 (the B2B EDI protocol). Running and securing your own SFTP servers is a chore. Transfer Family is a fully-managed file-transfer service that speaks those protocols and lands the files straight into S3 or EFS — with IAM/Directory-based auth, scaling, and no servers to patch.

  • Keep your partners' existing SFTP/FTPS/AS2 workflows; just point them at a managed endpoint.
  • Files land in S3/EFS → trigger Lambda, analytics, etc.
⚠️ Cost trap

Transfer Family bills ~$0.30 per protocol-endpoint per hour (~$216/month) whether or not anyone connects — plus ~$0.04/GB transferred. That always-on endpoint fee surprises people (like an idle ALB). Don't leave dev SFTP endpoints running.

DataSync & Transfer · Part 3

AWS Backup — Centralised, Cross-Service Backup

Instead of scripting snapshots per service, AWS Backup centrally manages backups across EBS, EFS, FSx, RDS, Aurora, DynamoDB, S3, Storage Gateway and more — with backup plans (schedules + retention), cross-region & cross-account copy, point-in-time recovery, and Backup Vault Lock for immutable WORM compliance.

🔑 Triggers

"Retain backups beyond RDS's 35-day automated limit," "centralised cross-service / cross-region backup policy," "immutable compliance backups" → AWS Backup (with Vault Lock for WORM).
