SECTION 01

The Exam-Day IAM Picture

Before any cheat sheet, lock in the framing the exam-writers actually use. IAM lives mostly inside the Security domain — the single largest at 30% — but it bleeds into every other domain too.

The Security domain is ~30% of the exam, and IAM is its backbone. But almost every "service A talks to service B" question anywhere on the test is secretly an IAM-role question. Treat IAM not as one topic worth 30%, but as a lens you apply to half the exam.

The One-Sentence Summary of the Whole Topic On every request, AWS asks two things — who are you (authentication) and what may you do (authorization) — then runs a strict evaluation where an explicit Deny anywhere always wins, and where the right answer for any workload is almost always a role with temporary credentials, never a long-lived access key.

What the SAA-C03 keeps asking about IAM

  • Service-to-service access → IAM role (the most common single pattern on the whole exam).
  • Cross-account access → role with cross-account trust + (for 3rd parties) an External ID.
  • Enterprise human access at scale → IAM Identity Center federated with the corporate IdP.
  • Organization-wide guardrails → SCPs (which restrict, never grant).
  • Safe delegation of IAM creation → permission boundaries.
  • Fine-grained conditions → MFA, source IP, region lock, HTTPS-only, tag matching (ABAC).
  • The "most secure / least overhead" qualifier → temporary credentials & managed federation beat static keys & manual setup every time.

SECTION 05

The Evaluation Logic — Quick Ref CH 08, 16

If you internalise one diagram for the exam, make it this one.

Default deny → explicit Deny beats everything → otherwise need an explicit Allow.

Full pipeline order (all layers must allow; any Deny kills it)

  1. SCPs — every SCP up the org tree must allow (management account is exempt)
  2. Permission boundary — caps the identity if attached
  3. Identity-based policy — on the user/group/role
  4. Resource-based policy — on the target resource
  5. Session policies — passed at assume-time (advanced)
The Three Rules That Win Marks 1) Default = deny. No allow → denied.
2) Explicit Deny always wins — over any number of Allows.
3) SCPs only restrict, never grant. Even an "Allow s3:*" SCP grants nobody anything.

Number Facts to Memorise Temp creds default 1 hour, configurable up to 12 hours for direct assume. Role chaining is hard-capped at 1 hour regardless of max-session-duration. Temp access keys start with ASIA (vs AKIA for permanent). EC2 attaches a role via an instance profile wrapper (EC2-only quirk).
PassRole — the sneaky one To give a role to a service (e.g. attach to an EC2 instance at launch) the caller needs iam:PassRole on that role — separate from assuming it. "Developer can't launch an instance with the app role attached" → missing iam:PassRole. Scope it tightly or a dev could attach an admin role and self-escalate.
External ID — the confused-deputy fix Third-party assuming a role in your account → add sts:ExternalId in the trust policy. Without it, a malicious actor could trick the multi-tenant third party into assuming your role by impersonating one of its other customers. Third party = always External ID.

Cross-account S3 — three ways

  • Bucket policy (resource-based) — most common, preferred.
  • Cross-account IAM role — assume into the bucket's account.
  • ACLs — legacy, now disabled by default on new buckets. Rarely the right exam answer anymore.

The Two SCP Traps That Fail People 1) SCPs don't grant — a user still needs an IAM policy for the actual permission.
2) SCPs don't apply to the management account — never run sensitive workloads there expecting SCP protection.
Classic SCP Use Cases Deny disabling CloudTrail/GuardDuty · restrict allowed regions · block specific services org-wide · enforce mandatory resource tags · cap instance sizes in sandbox OUs. All apply even to a member-account's root user.

Allow-list vs Deny-list SCP strategy: default is deny-list (start from FullAWSAccess, add Deny rules) — lower overhead. Allow-list (only explicitly permitted services) is more secure but high maintenance — chosen by compliance-heavy industries.

SECTION 11

The Top 12 Traps That Fail People

Read this list the morning of the exam. Each one is a wrong answer the exam dangles in front of you.

  1. Access keys on EC2/Lambda. Service-to-service is always a role. Keys = wrong.
  2. SCPs grant permissions. They don't — they only cap. Identity policy still required.
  3. SCPs protect the management account. They don't apply there.
  4. Root is exempt from SCPs. No — a member-account root is capped by SCPs.
  5. Allow can override Deny. Never. Explicit Deny always wins.
  6. Nested groups. Not a thing in IAM.
  7. Forgetting iam:PassRole. Launching with a role attached needs PassRole, not just RunInstances.
  8. Skipping External ID for third parties. Confused-deputy risk; always include it.
  9. s3:GetObject ≠ s3:ListBucket. Different actions on different resources (object vs bucket).
  10. Role chaining keeps the 12h max. No — chaining is capped at 1 hour.
  11. S3 ACLs as the modern answer. Deprecated/disabled by default; prefer bucket policies.
  12. IAM users for enterprise staff across many accounts. Use Identity Center federation instead.

QUESTION 01
Single response
A solutions architect has just created a new AWS account. According to AWS best practices, what should be done first with the root user?
  • Generate access keys for the root user so automation scripts can manage the account
  • Enable MFA on the root user, delete any root access keys, and create an IAM admin identity for daily work
  • Attach the AdministratorAccess policy to the root user
  • Use the root user for all administrative tasks since it is the most powerful identity
Correct: B The root user is all-powerful and cannot be restricted within its own account, so it must be locked down immediately: strong password, MFA, no access keys, and daily work moves to a scoped IAM/Identity Center identity.
A & D use root for routine work — the single most dangerous AWS habit. C is meaningless: root already has full power; attaching a policy adds nothing and signals misunderstanding.

Single response
An application running on an EC2 instance needs to read objects from an S3 bucket. What is the most secure way to grant this access?
  • Create an IAM user, generate access keys, and store them in the application's config file

QUESTION 03
Single response
A user has two attached policies. Policy A allows s3:* on all resources. Policy B explicitly denies s3:DeleteBucket. The user attempts to delete a bucket. What happens?

  • Denied — an explicit Deny always overrides any Allow

QUESTION 07
Single response
A platform team wants to let developers create IAM roles for their own applications, but must guarantee a developer can never create a role with more than read-only S3 permissions. What should they implement?

QUESTION 08
Single response
A developer can run ec2:RunInstances but gets an error when launching an instance with an IAM role attached. What permission is most likely missing?

Which statement about Service Control Policies (SCPs) is true?

  • SCPs apply to the Organization's management account
