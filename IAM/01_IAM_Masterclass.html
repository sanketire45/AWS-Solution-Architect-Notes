CHAPTER 01

The 30-Second Story of IAM

Before we touch a single AWS feature, let's understand the problem IAM exists to solve. Once that clicks, everything else falls into place.

Imagine you own a 60-story office tower in the middle of a city. Inside that tower, there are thousands of rooms — server rooms, finance offices, executive suites, a kitchen, a parking garage, mailrooms. Different people need to enter different rooms: cleaners go everywhere but touch nothing, accountants go to the finance floor only, the CEO can go anywhere.

You can't just hand everyone a master key. That would be a disaster. Instead, you hire a security desk. Every person who walks in shows ID, the desk checks who they are, looks up what they're allowed to do, and either lets them through the right doors — or stops them at the elevator.

AWS is that office tower. Every S3 bucket, every EC2 instance, every database, every Lambda function is a room. IAM is the security desk. It's not optional, it's not bolted on — IAM is literally checked on every single request that anyone or anything makes to AWS. Click a button in the console? IAM check. EC2 instance reads a file from S3? IAM check. Lambda function writes to DynamoDB? IAM check.

IAM doesn't do anything for you in the cloud — it doesn't compute, store, or serve. It just stands at the door and asks two questions on every request: "Who are you?" (authentication) and "What are you allowed to do?" (authorisation). That's it. The rest of this entire chapter is just unpacking the surprisingly elegant machinery behind those two questions.

Why this matters more than any other AWS topic

IAM is the most-tested service family on the SAA-C03 exam. The Security domain alone is 30% of your score, and almost every Security question pivots on IAM. But here's the deeper truth: in the real industry, more AWS breaches happen because of bad IAM than every other cause combined. Leaked access keys in GitHub. Over-permissioned roles. Forgotten admin users from 2019 who still have full access. A single IAM mistake can expose your entire company.

So your job here isn't to memorise — it's to build judgement. By the end of this chapter you should be able to look at any AWS scenario and instinctively think: "who is the principal, what's the action, what's the resource, and what's the policy chain that decides if this is allowed?"

Why This Pays Off on the Exam Every question about S3, EC2, Lambda, RDS — at some level — is also an IAM question. The exam loves to give you a "service A needs to access service B" scenario. The correct answer is almost always "create an IAM role with the right trust and permission policy." If you internalise this instinct, you can confidently eliminate two wrong answers on roughly half the exam.
CHAPTER 02

The AWS Account & The Dangerous Root User

Every story starts somewhere. The AWS story starts with an account, and inside every account is a single all-powerful identity called the root user. Understanding the root user is the first step.

What is an AWS Account?

An AWS account is a container. It holds your resources (S3 buckets, EC2 instances, databases), your billing, and your identities. Every resource in AWS lives inside exactly one account. When you sign up at aws.amazon.com with an email and a credit card, you create an account.

Each account has a unique 12-digit number — for example, 123456789012. You'll see this number embedded in something called an ARN (Amazon Resource Name), which is just a fancy globally-unique address for any resource. An ARN looks like this:

# Structure of an ARN — memorise the shape, you'll see this everywhere
arn:aws:s3:::my-bucket-name
arn:aws:iam::123456789012:user/Alice
arn:aws:ec2:us-east-1:123456789012:instance/i-0abc1234
       └─partition─┘└service┘└─region─┘└──account──┘└────resource────┘

The Root User — the Original Sin

When you create the account, the email and password you used become the credentials for the root user. The root user is godlike. It can do literally anything in that account — delete every resource, close the account, change billing, even ignore any other security control you've set up. There is no policy you can attach to restrict the root user inside its own account.

Trap — Daily-Use Root The single most catastrophic security mistake on AWS is using the root user for day-to-day work. If those credentials leak (laptop stolen, password in a Slack message, phishing email), the attacker owns everything — and there is no upper guard rail to stop them inside that account. The exam tests this idea repeatedly: "what's the first thing you do when you create an AWS account?" The answer is always "lock away the root credentials and create an admin IAM user / Identity Center user for daily work."

What you should actually do with the root user

  1. Set a very strong password and enable MFA on the root user immediately.
  2. Delete any root access keys. Root access keys should never exist. If you find them, delete them. The exam will reward "delete root access keys" as the answer to several questions.
  3. Create your first IAM user with admin permissions for daily use, and lock the root credentials in a vault (1Password, AWS-approved physical safe, whatever).
  4. Use the root user only for tasks that literally require it — and they are few. Closing the account, changing the account email, restoring IAM permissions after a self-lockout, signing up for certain enterprise support plans, and a handful of billing operations.
Industry Reality Large enterprises don't just hide the root password — they store it in a tamper-evident envelope inside a physical safe, with two-person retrieval procedures. Some companies rotate the root password annually and require board-level sign-off to retrieve it. The principle is the same at every scale: root is a "break glass in case of emergency" identity, not a working identity.

So if you can't use root… who do you use?

That's what the next twenty chapters are about. You create other identities — IAM users, IAM roles, Identity Center users — each with carefully scoped permissions. The root user becomes an emergency fallback you hope to never use. With that mental model in place, let's meet your first non-root identity.

CHAPTER 03

IAM Users — Identities for Humans & Apps

An IAM user is a permanent identity that lives inside one AWS account. It represents a person or an application that needs long-term access to AWS.

What an IAM User actually is

Concretely, an IAM user is a record stored by AWS that has:

  • A name (e.g., alice, backup-script)
  • Optionally a console password — for humans who log into the AWS Console UI
  • Optionally one or two access keys — for programs that call AWS APIs (more on these in a moment)
  • A set of policies attached, defining what this user can do
  • An ARN: arn:aws:iam::123456789012:user/alice

An IAM user is created inside a specific AWS account. It does not exist anywhere else. If your company has 5 AWS accounts, you'd traditionally need to create the same person as 5 separate IAM users — once in each account. (This is one of the big reasons IAM Identity Center exists, which we'll cover later — but it's important to understand the basic model first.)

Inside one AWS account: one root (locked away) and however many IAM users you create.

When do you create an IAM user?

Honestly, in 2024+ AWS best-practice, you create fewer IAM users than you think. For humans, AWS strongly recommends using IAM Identity Center (federated sign-in) instead. For workloads (EC2, Lambda, ECS tasks), you should use IAM roles, not users. So IAM users are mainly used for:

  • Very small accounts or learning/sandbox environments
  • Third-party SaaS tools that don't support role assumption (they want a static access key pair)
  • Legacy applications running outside AWS that need long-term programmatic access
  • Emergency "break glass" accounts as backup to federation
Exam Trigger On the exam, if a question describes an EC2 instance, Lambda function, or any AWS service needing to access another AWS service, the answer is never "create an IAM user with access keys." It's always "create an IAM role." The presence of "access keys" or "IAM user" in an answer for a service-to-service scenario is a giant red flag.

Hard limits worth knowing

You can have up to 5,000 IAM users per AWS account. Each user can be a member of up to 10 groups. Each user can have up to 2 access keys at a time (which exists deliberately, to support rotation — you create a new key, switch your app to use it, then delete the old one without downtime).

CHAPTER 04

Passwords, Access Keys, and MFA

An identity is useless without a way to prove who it is. AWS gives you three credential types, each for a different purpose. Knowing which to use when is foundational.

1. Console Password — for humans clicking buttons

This is the password Alice types into aws.amazon.com/console when she signs in. It's only used in the web console — never in code, never in scripts. As an account admin, you can enforce a password policy: minimum length, must include numbers, symbols, uppercase letters, expiration period (e.g., 90 days), and prevention of password reuse.

2. Access Keys — for programs calling AWS APIs

An access key is a pair of strings:

AWS_ACCESS_KEY_ID     = AKIAIOSFODNN7EXAMPLE       # public-ish, ~20 chars
AWS_SECRET_ACCESS_KEY = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY # like a password, 40 chars

Any program that calls AWS APIs — the AWS CLI, the boto3 Python SDK, Terraform, a custom Node.js script — needs an identity. If that program runs as an IAM user, it uses these two strings to sign every request. AWS verifies the signature and identifies the caller.

The #1 Cause of AWS Breaches Hardcoding access keys in source code and accidentally pushing to GitHub is — no exaggeration — the most common way AWS accounts get compromised in the real world. Bots crawl public GitHub repos looking for AWS keys 24/7. The keys are usually weaponised within minutes of being published. AWS even runs its own scanner and will email you (and quarantine the keys) if it detects yours in a public repo, but you should never rely on that net.

Rules for access keys

  • Each IAM user can have up to 2 active access key pairs at a time
  • You see the secret key exactly once, at creation. If you lose it, you must create a new one
  • Rotate access keys regularly (90 days is a common policy)
  • Never commit them to git. Never paste them in chat. Use aws configure to store them locally in ~/.aws/credentials, or use environment variables
  • Even better: avoid them entirely when running on AWS infrastructure — use IAM roles instead

3. MFA — Multi-Factor Authentication

MFA adds a second layer on top of a password. Even if an attacker steals the password, they need the second factor (typically a 6-digit code that refreshes every 30 seconds, or a physical security key) to actually log in.

AWS supports several MFA types:

TypeWhat it isWhen to use
Virtual MFAAn app like Google Authenticator, Authy, or 1Password that generates a 6-digit code refreshed every 30 seconds (TOTP).Default for most users. Free, easy.
Hardware TOTPA small physical device that displays a rolling 6-digit code (Gemalto, SurePassID).When phones aren't allowed (some government / finance shops).
FIDO Security Key / U2FA physical key (e.g., YubiKey) plugged into USB or tapped via NFC. Strongest option — phishing-resistant.High-security users — root user, admins, anyone with sensitive permissions.
PasskeysNewer FIDO2-based MFA bound to your device biometrics (Face ID, fingerprint). Rolled out by AWS in 2024.Modern setup, replacing TOTP gradually for many orgs.
Exam Trigger Anything mentioning the words "strongest MFA", "phishing-resistant", or "FIPS-compliant hardware token" points to FIDO security keys. Anything mentioning "basic MFA on a smartphone" points to virtual MFA. The root user should always have MFA enabled — that's a frequent question.

Forcing MFA via policy

Beyond just requiring MFA at login, you can write IAM policies that refuse to perform sensitive operations unless the calling session was authenticated with MFA. This is done with a condition key called aws:MultiFactorAuthPresent — we'll see exactly how when we get to conditions in Chapter 17.

Industry Reality Most mature companies enforce MFA on every human IAM identity, period. Many require a hardware key (YubiKey) for any role that can read production data or change infrastructure. The cost is ~$50 per developer for the key, which is trivial vs. the risk of an account takeover. If you join a tech company, expect a YubiKey to be shipped to you alongside your laptop.
CHAPTER 05

IAM Groups — Stop Managing People One-by-One

If you have 50 developers and you give each one permissions individually, you have created 50 places where something can go wrong. Groups exist to make this sane.

The simplest possible concept

A group is just a labelled bucket. You put IAM users into the bucket, you attach policies to the bucket, and every user in the bucket inherits those policies. That's it. The whole feature.

A group is like a "Developers" Slack channel. Anyone in the channel gets the same announcements. Add a new dev to the channel, they instantly get all the announcements. Remove them, they stop getting anything. The channel itself has no opinions — it's just a label that gathers people.
Attach policies once, to the group. Membership does the rest.

Rules you must remember

  • A group is not an identity. You cannot "log in as a group" or have a group assume a role. It's purely an administrative tool.
  • Groups cannot be nested. No groups-inside-groups. (This is a difference from corporate Active Directory — exam loves this trap.)
  • A user can be in up to 10 groups.
  • An AWS account can have up to 300 groups.
  • Policies attached to a group apply to every member. Removing a user from the group instantly revokes those permissions.
Trap — Nested Groups Anyone with AD/LDAP background reflexively assumes group nesting works. It doesn't. If the question asks "how do you create a hierarchy of permissions in IAM Groups?", the answer is you can't. For hierarchical access control across accounts, you use AWS Organizations with OUs (organisational units) and SCPs — which we'll meet in Chapter 14.

A real example of group design

A typical startup's IAM layout, before Identity Center is brought in:

Admins
Full admin access. 2–3 users, all with MFA + hardware key.
Developers
Full access to dev/staging environments; read-only in production; can deploy via CI/CD only.
DataScientists
Read S3 data lake, run Athena, read-write SageMaker notebooks. No infrastructure.
Finance
Read-only access to AWS billing console only. Cannot see resources.
Contractors
Scoped to a single project — only resources tagged with their project name.

Notice that even with only 5 groups, you've already covered most real-world permission needs of a small company. Onboarding becomes "create user, add to group(s)" — 30 seconds instead of writing custom policies for everyone.

CHAPTER 06

Policies — The JSON Rulebook

Everything in IAM eventually comes down to a policy. A policy is a JSON document that says: "this principal can / cannot do these actions on these resources, under these conditions." Master the shape and you've mastered IAM.

A minimal real policy, dissected

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid":      "AllowReadingTheReportsBucket",
      "Effect":   "Allow",
      "Action":   ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::company-reports",
        "arn:aws:s3:::company-reports/*"
      ],
      "Condition": {
        "IpAddress": { "aws:SourceIp": "203.0.113.0/24" }
      }
    }
  ]
}

Let's walk through every line. These are the same six concepts you'll see in every IAM policy ever written.

Version
Always "2012-10-17". It's a syntax version, not a date. Just write it.
Statement
A list of one or more rules. Each rule has Effect / Action / Resource (and optional Condition). You can have many statements in one policy.
Sid
"Statement ID" — a label for humans. Optional. Useful for debugging and CloudTrail logs.
Effect
Either "Allow" or "Deny". There is no third value.
Action
What API calls are covered. Format: service:Operation. You can use wildcards: s3:* means every S3 action. s3:Get* means every action whose name starts with "Get".
Resource
Which specific resources the actions apply to. Uses ARNs. "*" means every resource.
Condition
Optional. Adds extra restrictions — only allow from a certain IP, only when MFA is present, only at certain times, etc. We'll deep-dive in Chapter 17.
Principal
(Not shown in this policy — appears only in resource-based policies.) Specifies who the rule applies to. In an identity-based policy, the principal is implied (it's whoever the policy is attached to).

What this specific policy says, in English

"Whoever this policy is attached to is allowed to list and read objects from the company-reports S3 bucket, but only if the request comes from an IP address in the 203.0.113.0/24 range (probably the company office network)."

That's the entire concept of IAM in one document: a list of allow/deny rules with optional fine-grained conditions. Every policy you ever read is just a more elaborate version of this.

Reading actions correctly

Action names follow the pattern service:ApiCallName. You don't have to memorise them — you look them up. But you should recognise the shape:

  • s3:GetObject — download a single object from S3
  • s3:PutObject — upload a single object to S3
  • s3:ListBucket — list the contents of a bucket (note: this is a different permission from reading objects — common exam trap)
  • ec2:RunInstances — launch a new EC2 instance
  • iam:PassRole — give an IAM role to a service (e.g., attach a role to an EC2 instance at launch) — a very dangerous permission
  • kms:Decrypt — decrypt something using a KMS key
Subtle Trap — s3:GetObject vs s3:ListBucket s3:GetObject applies to objects (resource: arn:aws:s3:::bucket/*). s3:ListBucket applies to the bucket itself (resource: arn:aws:s3:::bucket — no /*). Many people write policies that grant s3:GetObject and wonder why aws s3 ls doesn't work. It's because listing requires a separate permission on a different resource. This subtlety shows up on the exam.

Wildcards — short, but dangerous

You can use * to mean "anything" and ? to mean "any single character." So:

  • "Action": "s3:*" — every S3 action
  • "Action": "s3:Get*" — every action starting with "Get"
  • "Resource": "*" — every resource in the account that the action could apply to
  • "Resource": "arn:aws:s3:::logs-*" — every bucket whose name starts with "logs-"

The famous AdministratorAccess AWS-managed policy is just one statement:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow",
    "Action":   "*",
    "Resource": "*"
  }]
}

"Allow all actions on all resources." That's all "admin" means in AWS. Powerful, simple, terrifying.

The principle of least privilege

This phrase will come up in every AWS book, exam question, and security audit you ever encounter. It just means: grant the smallest set of permissions someone needs to do their job — no more. Don't give s3:* when s3:GetObject on one specific bucket is enough. Don't give "Resource": "*" when one specific ARN works.

Why? Because if those credentials are ever compromised, the blast radius is contained. A leaked key with s3:GetObject on one bucket is a manageable incident. A leaked key with admin access is "evening news" territory.

Exam Trigger Any answer choice that grants "*" permissions is almost certainly wrong on the SAA exam — unless the question explicitly says "minimum administrative effort" AND the alternatives are clearly impractical. The exam-writers heavily favour least-privilege answers.
CHAPTER 07

The Four Flavours of Policy

Policies are all JSON, but where you attach them and what they apply to creates a few distinct categories. Knowing the difference is exam gold and real-life essential.

Cut #1: Identity-Based vs Resource-Based

Identity-Based Policies

Attached to a user, group, or role. Say what that identity can do. The principal is implicit (it's whoever the policy is attached to).

Example: Attach S3ReadOnlyAccess to user Alice. Now Alice can read S3 — anywhere.

Resource-Based Policies

Attached to a resource (S3 bucket, KMS key, SQS queue, Lambda function, etc.). Say who can access this specific resource. The principal is explicit — you must list it.

Example: An S3 bucket policy that says "anyone in account 222 can read this bucket."

The huge advantage of resource-based policies: they can grant cross-account access without the source account needing to set anything up on their side beyond an IAM identity. Identity-based policies cannot, by themselves, grant access to a resource in a different AWS account.

Common resources that support resource-based policies:

  • S3 bucket policies
  • SQS queue policies
  • SNS topic policies
  • KMS key policies (actually required for KMS — the key policy is the primary access control)
  • Lambda function resource policies (used for cross-account invokes)
  • API Gateway resource policies
  • ECR repository policies
  • Secrets Manager secret policies
  • EFS file system policies

An EC2 instance does not have a resource-based policy. Neither does a DynamoDB table (DynamoDB doesn't support resource policies — there was finally a resource-based policy added for DynamoDB tables in 2024, but historically it didn't exist; older exam materials reflect "no").

// S3 bucket policy example — note the explicit "Principal"
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Principal": { "AWS": "arn:aws:iam::222222222222:root" },
    "Action":    "s3:GetObject",
    "Resource":  "arn:aws:s3:::my-bucket/*"
  }]
}

This bucket policy says: "any principal in account 222 (root means 'the whole account') can get objects from my-bucket." For the access to actually work, the user in account 222 also needs an identity-based policy that allows s3:GetObject on that bucket — both sides must permit it. We'll see this two-key dance in detail in Chapter 11.

Cut #2: AWS Managed vs Customer Managed vs Inline

For identity-based policies specifically, there are three storage flavours:

FlavourWho created itReusable across identities?When to use
AWS Managed AWS provides them; AWS maintains them Yes — pre-built, share-able to many identities Common roles like AdministratorAccess, ReadOnlyAccess, AmazonS3FullAccess. Convenient, but often too broad.
Customer Managed You write it Yes — attach to many users/groups/roles Your real production permissions. Tailored, least-privilege, versioned.
Inline You write it No — lives inside one user/group/role One-off, tightly bound permissions you never want to detach or share. Rare.
Exam Trigger "Need to apply the same custom permissions to multiple users / make changes once and propagate" → Customer Managed Policy. The Inline option exists, but it's almost always the wrong answer on the exam, because it doesn't scale and isn't reusable.

Two policies that live on every role: trust vs permission

This is a foreshadowing for Chapter 9, but it's worth flagging here so you don't get confused: every IAM role has two policies attached:

  • A trust policy (a resource-based policy on the role itself) — says who can assume this role
  • One or more permission policies (identity-based) — says what this role can do once assumed

The trust policy is unique to roles. We'll unpack it shortly.

CHAPTER 08

How AWS Actually Decides Allow vs Deny

When a request hits AWS, IAM runs an evaluation algorithm. The rules are not complicated, but they are strict, and getting them wrong is one of the most reliable ways to fail the exam. So we'll do this precisely.

The four golden rules — memorise these

  1. Everything is denied by default. If no policy explicitly allows you, you are denied.
  2. An explicit Allow overrides the default deny. This is how you actually get permissions.
  3. An explicit Deny always wins. If any applicable policy contains an "Effect": "Deny" for the action and resource in question, the request is denied — no matter how many Allows exist.
  4. Allows do not stack to "more allowed." Once something is allowed, allowing it again in another policy changes nothing. Conversely, a single Deny anywhere is fatal.

"Deny wins. Always. If you remember nothing else, remember that one sentence — it answers about 40% of all IAM trick questions on the exam."

The simplified evaluation flow. We'll add SCPs and boundaries on top in Chapter 16.

A worked example

Suppose Alice has these policies attached to her IAM user:

  • Policy A: Allow s3:* on *
  • Policy B: Deny s3:DeleteBucket on *

Alice tries to delete a bucket. Walk through the algorithm:

  1. Is there a matching explicit Deny? Yes — Policy B denies s3:DeleteBucket. Stop. Denied.

Even though Policy A's s3:* would normally allow s3:DeleteBucket, the explicit deny in Policy B kills the request.

Now Alice tries to upload a file (s3:PutObject):

  1. Is there a matching explicit Deny? No — Policy B only denies s3:DeleteBucket.
  2. Is there a matching explicit Allow? Yes — Policy A allows s3:*, which includes s3:PutObject. Allowed.
Exam Trigger The phrase "how do you make sure a user can never do X, even if they get other policies later" = attach an explicit Deny policy. The phrase "prevent a region from being used" = an explicit deny with a aws:RequestedRegion condition. The phrase "override permissions inherited via group/role" = explicit deny.

What "matching" means

A policy statement matches a request only if all of these are true:

  • The Action includes the API call being made (e.g., s3:GetObject matches s3:*, s3:Get*, or s3:GetObject)
  • The Resource includes the ARN being targeted
  • The Condition block, if present, evaluates to true
  • For resource-based policies, the Principal includes the caller

If any of those don't match, the statement is simply ignored for this request — it neither helps nor hurts.

That's the core algorithm. We'll come back to it in Chapter 16 once we've added all the layers — SCPs, permission boundaries, and resource policies — to see the full evaluation pipeline.

CHAPTER 09

IAM Roles — The Real Magic

This is the most important chapter in the whole guide. If users are permanent identities like full-time employees, roles are like name-tags that anyone can pick up, wear briefly, and use for one specific purpose. Understanding roles is what separates beginners from people who actually understand cloud security.

Why roles exist — the problem first

Imagine an EC2 instance running a web application that needs to read images from an S3 bucket. The naïve approach: create an IAM user, generate access keys, copy the keys onto the EC2 instance (in a config file or environment variable), let the app use the keys to call S3.

This is bad. Why?

  • The keys sit on disk on the instance. Anyone who SSHs in can read them.
  • If the instance is compromised, the attacker gets long-lived keys.
  • If you want to rotate the keys, you have to update every instance.
  • If you have 200 instances, you have to manage keys on every single one.
  • Snapshots / AMIs of the instance now contain credentials. If you share the AMI, you share the keys.

IAM roles solve all of this. Instead of giving the instance a permanent identity, you give it the ability to temporarily become an identity. AWS automatically rotates the credentials behind the scenes. The instance never sees a long-lived secret.

What a role really is

A role is an IAM identity, just like a user — it has an ARN (arn:aws:iam::123456789012:role/ImageReaderRole), permission policies, the works. But it has no password and no long-term access keys. Instead, it has a trust policy that says: "I can be assumed by this specific entity."

Every role has two policies that answer two very different questions.

An EC2 role end-to-end

Let's solve the original problem properly. Web app on EC2 needs to read images from S3.

  1. Create an IAM role called ImageReaderRole.
  2. Its trust policy says: "The EC2 service can assume me."
  3. Its permission policy says: "Can do s3:GetObject on arn:aws:s3:::images-bucket/*."
  4. Attach this role to the EC2 instance when launching it (or after, via Modify IAM Role).
  5. The application running on EC2 just calls AWS — no credentials in code.
  6. Under the hood, the EC2 metadata service (IMDSv2) provides temporary credentials to the instance automatically. The SDK picks them up. AWS rotates them every few hours. The app never knows.
// The trust policy on ImageReaderRole — note the Principal is a service
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action":    "sts:AssumeRole"
  }]
}
// The permission policy on ImageReaderRole — what it can actually do
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow",
    "Action":   "s3:GetObject",
    "Resource": "arn:aws:s3:::images-bucket/*"
  }]
}

That's it. The EC2 instance can now read from S3 with zero credentials on disk. If the instance is compromised, the attacker gets temporary creds (typically valid 1–6 hours) — not a long-term secret.

Industry Reality Real engineering teams use IAM roles for every AWS workload — EC2, Lambda, ECS tasks, EKS pods (via IRSA), CodeBuild jobs, Step Functions, even on-premises servers (via IAM Roles Anywhere). The presence of access keys in any production codebase is treated as a serious incident at most mature companies.

Instance Profiles — the weird wrinkle

For historical reasons, when you attach a role to an EC2 instance, AWS actually wraps it in something called an instance profile. The instance profile is a container with the same name as the role. The console hides this from you — when you click "attach role" to an instance, it creates and uses the instance profile automatically. But if you use the AWS CLI or Terraform, you might see "instance profile" mentioned and wonder what it is. Now you know: it's a thin wrapper around a role, specifically for EC2.

For Lambda, ECS tasks, and other services, no instance profile is involved — the role is attached directly. So this quirk is EC2-only.

The three big use cases for roles

  1. Service roles: AWS service A needs to act on AWS service B (the EC2-to-S3 example above). The most common type by far.
  2. Cross-account roles: A user/role in account A needs access to resources in account B (Chapter 11).
  3. Federation roles: A user signed into your corporate identity provider (Okta, Active Directory) gets temporary AWS access by assuming a role (Chapter 12).

All three rest on the same underlying mechanism, which is the next chapter.

CHAPTER 10

AWS STS & AssumeRole — Behind the Curtain

Roles aren't magic — there's an AWS service running the machinery underneath them. That service is called STS, the Security Token Service. Understanding how STS works clarifies a lot of fuzzy thinking about roles, federation, and temporary credentials.

What STS does, in one sentence

STS takes a request that says "I want to act as role X" and, if you're allowed, hands you back a set of temporary credentials that act exactly like access keys but expire after a configurable lifetime.

A "temporary credentials" bundle has three parts:

  • AccessKeyId — starts with ASIA… (note: not AKIA like a permanent user's key — that's a giveaway in logs)
  • SecretAccessKey — the secret half
  • SessionToken — a third value that must be included in every signed request
  • An Expiration timestamp

That bundle is what your code uses. When the expiration passes, the credentials stop working — you (or the SDK on your behalf) must request a new set.

The five STS API calls worth knowing

API callWhat it doesTypical caller
sts:AssumeRoleAn IAM identity (user or role) assumes another IAM role within AWS.Cross-account access, dev tools, service-to-service.
sts:AssumeRoleWithSAMLTrade a SAML assertion (from Active Directory / Okta / Azure AD) for temporary AWS credentials.Corporate enterprise federation.
sts:AssumeRoleWithWebIdentityTrade a Google/Facebook/Cognito/OIDC token for temporary AWS credentials.Consumer mobile apps; EKS pods (IRSA).
sts:GetSessionTokenGet short-lived creds for the caller themselves, often combined with MFA, to do MFA-protected calls.Manual workflows that need MFA on API calls.
sts:GetFederationTokenOlder mechanism for federation via a custom identity broker. Mostly legacy now.Pre-IAM-Identity-Center custom federation.

What happens, step-by-step, when EC2 calls S3

EC2 → Metadata → STS → Role → temp creds → S3. The app never sees a long-term secret.

Session duration

By default, AssumeRole issues credentials valid for 1 hour. You can configure the role's maximum session duration up to 12 hours. SDKs automatically renew before expiration, so apps don't usually notice. For roles assumed via the AWS Console "Switch Role" feature, sessions max out at 1 hour and you'll see the "session expired" notice.

One subtle wrinkle for the exam: you cannot chain role assumptions for more than 1 hour total. If role A assumes role B, and B then assumes role C, the resulting session of C cannot exceed 1 hour — even if C's max duration is set higher. This is called the role chaining limit.

Trap — Role Chaining "Why did my role chain session expire after 1 hour even though I set 12 hours?" — because role chaining is hard-capped at 1 hour. You'd need to assume directly from the original identity, not through an intermediate role, to get the full duration.

The "PassRole" permission — a quiet danger

To attach a role to an EC2 instance (or Lambda, or any service), the caller needs the iam:PassRole permission on that specific role. PassRole doesn't let you assume the role yourself — it lets you give it to a service. This is critical: if a developer can PassRole any role to an EC2 instance, they could attach an admin role to an instance and then SSH in and become admin. So you scope PassRole tightly to specific roles.

// "Developer can launch EC2 instances but only attach the dev-app role"
{
  "Effect":   "Allow",
  "Action":   "iam:PassRole",
  "Resource": "arn:aws:iam::123456789012:role/DevAppRole"
}
Exam Trigger Any question about "the developer needs to launch EC2 instances with a specific IAM role attached" — the missing permission is almost always iam:PassRole. Many candidates miss this one.
CHAPTER 11

Cross-Account Access — The Classic Pattern

In a single AWS account, IAM is straightforward. Real companies use many accounts — production, staging, dev, security, audit, per-team — and you need a way for principals in one account to safely access resources in another. The pattern is a workhorse on the exam and in the real world.

Why have multiple accounts at all?

A single AWS account is a fault and security boundary. Putting dev, staging, and production in one account means:

  • A developer mistake in dev (wildcard policy) can affect production resources
  • You can't easily limit blast radius — one credential leak owns everything
  • Billing is mixed together; cost attribution is harder
  • Service quotas (EC2 limits, etc.) are shared across all your workloads

So the modern AWS guidance — strongly encouraged for any non-trivial deployment — is multiple accounts, organised via AWS Organizations, with cross-account access for the legitimate handoffs.

The two-account scenario

Let's say you have Account A (Dev — 111111111111) and Account B (Prod — 222222222222). Alice is a developer in account A. She needs to read CloudWatch logs from production for debugging. You don't want her to have a Prod IAM user — too much friction, and too many doors to keep locked.

Step-by-step

  1. In Account B (Prod), create a role: ProdReadOnlyRole
  2. Trust policy on the role: "I can be assumed by anyone in account A." 
    {
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Principal": { "AWS": "arn:aws:iam::111111111111:root" },
    "Action":    "sts:AssumeRole",
    "Condition": {
      "Bool": { "aws:MultiFactorAuthPresent": "true" }
    }
  }]
}
    (The MFA condition is best-practice — require the assumer to have authenticated with MFA. The exam loves to test this.)
  3. Permission policy on the role: read-only on CloudWatch Logs.
  4. Back in Account A, give Alice (or her group) permission to assume the role: 
    {
  "Effect":   "Allow",
  "Action":   "sts:AssumeRole",
  "Resource": "arn:aws:iam::222222222222:role/ProdReadOnlyRole"
}
  5. Alice now calls aws sts assume-role --role-arn arn:aws:iam::222222222222:role/ProdReadOnlyRole --role-session-name alice-debug or uses the "Switch Role" feature in the console. She gets temp creds for account B and can read prod logs.

Two policies — one in each account — both have to allow the assumption. This "both sides must agree" pattern is the heart of cross-account security.

The External ID — for third-party access

If you're letting a third-party SaaS assume a role in your account (e.g., Datadog, a monitoring tool, a backup vendor), there's a special threat called the "confused deputy": the third-party might have multiple customers, and an attacker could trick it into assuming a role in your account by claiming to be one of its other customers.

The fix: include an ExternalId condition in your trust policy. The third party gives you a unique random string (the External ID); you put it in the trust policy; they must provide it when assuming the role.

{
  "Effect":    "Allow",
  "Principal": { "AWS": "arn:aws:iam::ThirdPartyAccountId:root" },
  "Action":    "sts:AssumeRole",
  "Condition": {
    "StringEquals": { "sts:ExternalId": "a-unique-string-the-vendor-gave-me" }
  }
}
Exam Trigger Any scenario where "a third-party company needs to access your AWS account" — the correct answer mentions both cross-account role AND External ID. The wrong answers usually suggest giving the third party an IAM user with access keys (insecure, and operationally awkward).

S3 cross-account access — three flavours

S3 specifically supports three ways to grant cross-account access. Worth knowing because S3 is the single most-tested service:

  • Bucket Policy (resource-based) — most common. Grants account B's principals access to the bucket.
  • Cross-Account IAM Role — the pattern we just walked through; account B users assume a role in account A.
  • ACLs (Access Control Lists) — legacy mechanism, mostly deprecated. AWS recommends bucket policies instead. Now disabled by default on new buckets.
Trap — S3 ACLs If a question offers "use S3 ACLs" as an answer in 2024+, it's almost certainly wrong. AWS now disables ACLs by default and recommends bucket policies for everything. The exception is some legacy cross-account scenarios where ACLs are the only thing that historically worked — but the exam now prefers bucket policies in nearly every case.
CHAPTER 12

Identity Federation — SAML & OIDC

Companies don't want to maintain a second username and password for every employee just to access AWS. Their employees already have Active Directory / Okta / Azure AD credentials. Identity federation means: log in once at the corporate identity provider, get AWS credentials automatically. No IAM user required.

The core idea

Your corporate identity provider (Okta, Azure AD, Google Workspace, AD FS) and AWS have a pre-arranged trust relationship. When Alice signs into her corporate IdP, the IdP gives her a signed token saying "this is Alice, she's in the Developers group." Alice presents that token to AWS STS. STS verifies the signature (because the trust was pre-arranged), looks up which IAM role corresponds to her group, assumes that role, and gives her temporary credentials.

Result: Alice never had an IAM user. AWS never stored her password. If she leaves the company and is removed from the corporate IdP, her AWS access vanishes the next time her token expires. Single source of truth = the corporate IdP. This is exactly what every large organisation wants.

The two main protocols

SAML 2.0 Federation

The enterprise standard for over 20 years. Used with AD FS, Okta, OneLogin, Ping, Azure AD. The token is a signed XML document (the SAML assertion).

AWS API call: sts:AssumeRoleWithSAML

When you see it on the exam: "corporate users with existing Active Directory want to sign into AWS without IAM users."

OIDC / Web Identity Federation

The modern, lightweight standard. Used by Google, Facebook, Amazon Cognito, GitHub Actions. The token is a signed JSON Web Token (JWT).

AWS API call: sts:AssumeRoleWithWebIdentity

When you see it on the exam: mobile/web apps that let users sign in with social accounts; EKS pods using IRSA (IAM Roles for Service Accounts); GitHub Actions deploying to AWS without long-lived access keys.

A typical SAML flow

  1. Alice clicks "Sign into AWS" on her corporate intranet.
  2. The intranet redirects her to AD FS (or Okta etc.) — the corporate identity provider.
  3. She enters her corporate username + password + MFA. The IdP verifies her.
  4. The IdP responds with a SAML assertion: a signed XML blob saying "Alice, Developers group."
  5. Her browser POSTs the SAML assertion to signin.aws.amazon.com/saml.
  6. AWS verifies the signature (it has the IdP's public key, configured earlier as a SAML identity provider). The trust relationship was set up once, in advance.
  7. AWS looks up the IAM role mapped to "Developers" and assumes it on Alice's behalf via sts:AssumeRoleWithSAML.
  8. Alice lands in the AWS Console with her role's permissions, no IAM user involved.

OIDC for cloud-native workloads — IRSA & GitHub Actions

Two modern uses of OIDC federation are huge in industry right now:

Industry — IRSA (IAM Roles for Service Accounts) Kubernetes pods running on EKS need to call AWS APIs. The old way: shove access keys into the pod. The modern way: IRSA — the EKS cluster runs an OIDC identity provider, each pod gets a JWT that identifies its Kubernetes Service Account, and the pod uses that JWT to assume an IAM role via sts:AssumeRoleWithWebIdentity. No static credentials anywhere.
Industry — GitHub Actions ↔ AWS For years, the standard way to deploy from GitHub Actions to AWS was to store an access key as a GitHub secret. That key was a permanent attack target. The modern pattern: configure GitHub's OIDC provider as a trusted identity provider in AWS, then a GitHub Actions workflow gets a short-lived JWT that it trades for AWS credentials via sts:AssumeRoleWithWebIdentity. No long-lived secrets in GitHub. Every major engineering org migrated to this in 2022–2024.

The relationship to IAM Identity Center

You might wonder: if federation is the answer, why does AWS keep talking about IAM Identity Center? Because Identity Center is essentially federation, plus a centralised UI and management layer that makes it simpler to operate across many AWS accounts. We'll meet it next.

CHAPTER 13

IAM Identity Center — The Modern Way

Previously known as AWS Single Sign-On (AWS SSO) — renamed in 2022. This is AWS's recommended way to give humans access to AWS in 2024. If you read AWS's own best-practice guides, every other paragraph mentions Identity Center.

What problem does it solve?

Imagine your company has 15 AWS accounts (one for each major team / environment, all under one AWS Organization). Before Identity Center, giving developers access was painful:

  • Create an IAM user in each account, or
  • Set up SAML federation in each account separately, configuring the same IdP 15 times, or
  • Create cross-account roles in 14 accounts that trust an "identity hub" account, and have everyone log in there first

All of these work. None of them are pleasant. Identity Center bundles them into a single managed service.

What it actually does

Identity Center is a directory + portal. You either:

  • Connect it to your existing identity provider (Okta, Azure AD, Google Workspace, Active Directory), OR
  • Use the built-in user directory ("Identity Center directory") for smaller orgs that don't have an IdP

You then define Permission Sets (effectively pre-baked IAM roles — like "DeveloperAccess," "ReadOnlyAccess"), and assign users/groups to account × permission set combinations.

End result: each user logs into a single portal (https://yourcompany.awsapps.com/start) and sees a list of AWS accounts they can access, each with one or more permission sets. They click, get temporary credentials, get into AWS. One sign-on.

Exam Trigger Any scenario like "a company has 10+ AWS accounts and wants to provide centralised SSO access to developers using their existing corporate directory" — the answer is IAM Identity Center, integrated with the corporate IdP. Wrong answers will suggest creating IAM users in every account, or setting up SAML federation 10 times.

Identity Center vs Cognito vs IAM Federation

These get confused often. The clean way to remember:

ServiceWho it's forTypical scenario
IAM Identity CenterYour employeesInternal users signing into AWS Console & CLI across multiple accounts.
CognitoYour app's end-usersExternal users of an app you've built (e.g., your mobile app's customers).
IAM SAML/OIDC FederationSingle-account use, or programmatic federationDirect IdP-to-IAM trust, particularly for workloads (GitHub Actions, EKS).
CHAPTER 14

AWS Organizations & Service Control Policies (SCPs)

Once you have more than one AWS account, you need a way to manage them together — billing, security, structure. AWS Organizations is the wrapper service, and SCPs are how you enforce policy across all of them at once.

The Organisation structure

An Organization is a tree of accounts:

  • Root — the top of the tree. Every org has exactly one root.
  • Organisational Units (OUs) — folders inside the root. You can nest OUs.
  • Accounts — the leaves. Every AWS account belongs to exactly one OU (or directly to the root).

You also pick one account to be the management account (formerly "master account") — it's the one that pays the bills and the only one that can administer the Organization.

A typical small Organization. SCPs can attach at the Root, any OU, or any single account.

What is an SCP?

A Service Control Policy is a JSON policy that defines the maximum permissions any IAM identity within an account can have. It looks almost identical to an IAM policy, but its job is different:

  • SCPs do not grant any permissions. They only restrict. Even if an SCP says "Allow s3:*," nobody gets S3 access from that alone — they still need a regular IAM policy granting it.
  • SCPs apply to every principal in the affected account(s), including IAM users, IAM roles, and — yes — even the root user of the member account.
  • An action is allowed in a member account only if every applicable SCP and the IAM identity policy and any resource policy permits it.
  • SCPs do not apply to the management account itself. This is a critical detail. The management account is exempt.

SCP = ceiling on what's possible. IAM policy = what's actually allowed within that ceiling.

Concrete SCP examples

Example 1 — Prevent leaving the US

// "Nobody in this account can do anything outside us-east-1 and us-west-2"
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Deny",
    "Action":   "*",
    "Resource": "*",
    "Condition": {
      "StringNotEquals": {
        "aws:RequestedRegion": ["us-east-1", "us-west-2"]
      }
    }
  }]
}

Example 2 — Sandbox can only use cheap regions and small instances

Imagine a sandbox OU where developers can experiment. You attach an SCP that prevents launching instances larger than t3.medium, and blocks expensive services like Redshift entirely. Even if a developer's IAM role grants ec2:RunInstances, the SCP can still block specific instance sizes via a condition.

Example 3 — Block disabling CloudTrail

{
  "Effect":   "Deny",
  "Action":   ["cloudtrail:StopLogging", "cloudtrail:DeleteTrail"],
  "Resource": "*"
}

This prevents anyone — including admins, including the account's root user — from disabling audit logging. A favourite for security and compliance teams.

Big Trap — SCPs Don't Grant A surprising number of people miss the rule that SCPs only restrict, never grant. If you attach an SCP that allows s3:* to an account where the IAM users have no S3 permissions, nobody gets S3 access. The user/role still needs an identity policy granting it. SCP is a ceiling, not a floor.
Exam Trigger The classic SCP exam scenarios:
• "Prevent any user — including root — in member accounts from deleting CloudTrail" → SCP
• "Restrict which regions developers can use" → SCP
• "Enforce that all EC2 instances must be tagged with a project name" → SCP
• "Block specific services across the entire organisation" → SCP
A common wrong answer is "create an IAM policy in every account" — that doesn't scale and a malicious user could still remove it. SCPs are administered centrally and member-account admins cannot bypass them.

Allow-list vs Deny-list SCP strategies

By default, when you enable Organizations with SCPs, AWS attaches an SCP called FullAWSAccess at the root, which allows everything. Most companies then layer on additional deny SCPs ("deny-list" strategy) to forbid specific risky actions.

An alternative is the allow-list strategy — replace FullAWSAccess with an SCP that explicitly allows only the services you want, denying everything else by default. This is more secure but much higher operational overhead (every time you want a new service, you have to update the allow list). Most companies use deny-list; very security-sensitive companies (defence, finance) sometimes use allow-list.

CHAPTER 15

Permission Boundaries — Delegated Admin Safely

A boundary is like an SCP, but applied to a single user or role instead of a whole account. It exists for one specific reason: letting developers create IAM things without letting them grant themselves admin.

The delegation problem

Developers often need to create IAM roles for their applications — a role for their Lambda function, a role for their ECS task, etc. But if you give a developer iam:CreateRole, they can create an admin role for their function, and then go assume it, and now they're admin. Oops.

You could refuse to delegate IAM at all and force every role creation through a ticket system. But that's slow. Permission boundaries are the better answer.

How a permission boundary works

You define a policy (looks like a normal IAM policy) and attach it as the permission boundary on a user or role. From that point on, the effective permissions of that identity are the intersection of their attached identity policies AND the boundary. Even if their identity policy says s3:*, the boundary can cap them to just s3:GetObject.

Crucially, you can also force the developer to attach a boundary to any role they create. The condition iam:PermissionsBoundary in the developer's identity policy means: "you can create roles, but only if you attach this exact boundary to them." Now anything the developer creates can never exceed the boundary's limits.

// On the developer's user: "You can create roles, but only if they have THIS boundary"
{
  "Effect":   "Allow",
  "Action":   ["iam:CreateRole", "iam:AttachRolePolicy"],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "iam:PermissionsBoundary":
        "arn:aws:iam::123456789012:policy/DevRoleBoundary"
    }
  }
}

Permission Boundary vs SCP — common confusion

CHAPTER 16

The Full Evaluation Pipeline (Put It All Together)

Now we have all the pieces. Let's draw the complete picture of how AWS decides whether a request is allowed.

The full algorithm

For any given request from any principal, AWS checks (roughly in this conceptual order):

  1. SCPs — does every SCP up the Organization tree allow this action? (If the principal is in the management account, skip this step — management is exempt.)
  2. Resource-based policy — if the resource has a resource-based policy and it explicitly allows the principal, that can grant access by itself (more on this nuance below).
  3. Identity-based policy on the principal (user / group / role)
  4. Permission boundary on the principal, if attached — caps the identity-based policy
  5. Session policies — when a role is assumed with extra inline policies passed at assume-time (rare; advanced)

The simplified rule everywhere: an explicit Deny anywhere wins immediately. Beyond that, every layer must allow the request, except that for cross-account access via a resource-based policy, the resource policy alone can grant access without needing both sides to allow.

All layers must allow — and any explicit deny anywhere kills the request.

One thing worth saying clearly: cross-account is special

Within a single account, AWS uses an "OR" between identity and resource policies — if either allows, it's allowed (assuming no deny). But for cross-account access, you must have BOTH sides permit it: identity policy in the source account and resource policy in the destination (or a role being assumed). One side alone isn't enough for cross-account.

This is sometimes summarised as: "same account = OR. Cross account = AND."

CHAPTER 17

Conditions — Where Real Power Lives

A policy with just Effect, Action, Resource is blunt — it either allows the action or doesn't. The Condition block transforms IAM into a precision tool. Most clever, scalable IAM patterns rely on conditions.

The condition structure

{
  "Effect":    "Allow",
  "Action":    "s3:GetObject",
  "Resource":  "arn:aws:s3:::secure-bucket/*",
  "Condition": {
    "Bool":      { "aws:MultiFactorAuthPresent": "true" },
    "IpAddress": { "aws:SourceIp":               "203.0.113.0/24" }
  }
}

Conditions have three nesting levels: a comparison operator (StringEquals, IpAddress, NumericLessThan, Bool, DateGreaterThan, etc.), a condition key (the property to test), and one or more values. Multiple conditions in one block are AND'd — all must be true.

The condition keys you must know

Condition KeyWhat it testsCommon use
aws:SourceIpIP address of callerRestrict to office IP ranges
aws:MultiFactorAuthPresentWas MFA used to authenticate this session?Require MFA for sensitive actions
aws:MultiFactorAuthAgeSeconds since MFA was usedForce re-MFA every N minutes
aws:RequestedRegionWhich region the API call is hittingLock down which regions are usable
aws:PrincipalTag/KeyA tag on the calling user/roleABAC — attribute-based access control
aws:ResourceTag/KeyA tag on the target resourceABAC — match principal tag to resource tag
aws:CurrentTimeTime of the requestBusiness-hours-only access
aws:SecureTransportWas the request made over HTTPS?Force TLS on S3 access
aws:PrincipalOrgIDThe Organization ID of the callerAllow only principals from your own org
aws:UserAgentCaller's user-agent stringDetect SDK / CLI vs console (weak signal)
s3:prefix / s3:VersionIdService-specific keys for S3Restrict listing/access to specific prefixes

Two classic condition recipes

Recipe 1 — Force HTTPS-only on an S3 bucket

{
  "Sid":       "DenyInsecureTransport",
  "Effect":    "Deny",
  "Principal": "*",
  "Action":    "s3:*",
  "Resource":  ["arn:aws:s3:::mybucket", "arn:aws:s3:::mybucket/*"],
  "Condition": { "Bool": { "aws:SecureTransport": "false" } }
}

This is an S3 bucket policy that explicitly denies any request not made over HTTPS. The exam loves this one — it's a frequent answer to "how to enforce TLS for S3."

Recipe 2 — Require MFA for any destructive action

{
  "Effect":    "Deny",
  "Action":    ["ec2:TerminateInstances", "s3:DeleteBucket", "rds:DeleteDBInstance"],
  "Resource":  "*",
  "Condition": { "BoolIfExists": { "aws:MultiFactorAuthPresent": "false" } }
}

Note BoolIfExists — this handles the case where the key isn't present at all. Without it, the policy might not behave the way you expect when the caller isn't using MFA-protected sessions.

Exam Trigger Memorise the four condition patterns above — they cover roughly 80% of condition-related questions:
• Restrict by IP → aws:SourceIp with IpAddress
• Require MFA → aws:MultiFactorAuthPresent with Bool
• Force HTTPS → aws:SecureTransport with Bool
• Lock region → aws:RequestedRegion with StringEquals / StringNotEquals
CHAPTER 18

ABAC vs RBAC — Tags Change Everything

In a small AWS account, you write one policy per use case. In a big one, that approach explodes. The way modern AWS shops scale IAM is by giving up on per-resource policies and instead using tags.

RBAC — Role-Based Access Control (the old way)

You define a role for every job function or project: ProjectAlphaDeveloper, ProjectBetaDeveloper, ProjectGammaDeveloper... each granting access to that project's specific resources. New project? New role, new policy. 50 projects = 50 roles to maintain. This is RBAC.

ABAC — Attribute-Based Access Control (the modern way)

You tag your resources and your principals with attributes (Project = Alpha). You write one policy that says: "You can access any resource whose Project tag matches your Project tag."

{
  "Effect":    "Allow",
  "Action":    "ec2:*",
  "Resource":  "*",
  "Condition": {
    "StringEquals": {
      "aws:ResourceTag/Project": "${aws:PrincipalTag/Project}"
    }
  }
}

Read that carefully. The condition compares the resource's Project tag to the caller's Project tag. The ${...} syntax is a policy variable that's substituted at evaluation time. So Alice (tagged Project = Alpha) can manage only resources tagged Project = Alpha. Bob (tagged Project = Beta) is automatically scoped to Beta resources, using the same policy.

Now add a 51st project? Just tag the new resources with Project = Delta and tag the developers with the same. No policy changes. This is why ABAC scales and RBAC doesn't.

Exam Trigger "Company has 100 projects and growing, each with its own developers. Minimise the number of IAM policies / roles." → ABAC with tags on principals and resources. The wrong answer says "create a role per project."

Where ABAC works and where it doesn't

ABAC works beautifully for services that support resource tagging (EC2, S3 objects, RDS, Lambda, etc.). Some services don't support tags on all resource types, in which case you fall back to ABAC where you can and RBAC where you must. AWS has been expanding tag support across services aggressively for the last few years; in 2024 it's the dominant pattern for new architectures.

The trade-off: ABAC requires discipline around tagging. If resources get created without proper tags, the access model breaks. This is why you'll often see Organizations + SCPs used to enforce tagging — denying resource creation without required tags.

CHAPTER 19

IAM Best Practices (AWS's Modern Gospel)

Internalise this list. The exam asks "what's the best practice?" constantly, and the right answer is almost always lifted directly from here.

  1. Don't use the root user for daily work. Lock root credentials away. Enable MFA. Delete any root access keys.
  2. Use IAM Identity Center for humans. Don't create IAM users for employees if you have more than a couple of accounts. Federate from your corporate IdP.
  3. Use IAM roles for workloads. EC2, Lambda, ECS, EKS — always use a role, never embed access keys.
  4. Apply least privilege. Start from zero and add what's needed. Don't start from admin and try to subtract.
  5. Use groups (or Identity Center groups), not per-user policies. Easier to manage and audit.
  6. Enable MFA on every human identity. Phishing-resistant (hardware keys) for high-privilege accounts.
  7. Rotate credentials regularly. AWS recommends 90 days. For IAM users that must have access keys, automate the rotation.
  8. Use roles for cross-account access, not IAM users with copied keys.
  9. Use SCPs to set guardrails at the Organization level — region restrictions, service restrictions, anti-tampering rules.
  10. Use permission boundaries when delegating IAM creation rights to developers.
  11. Tag resources consistently and prefer ABAC for scaling.
  12. Audit with CloudTrail. Log every API call. Send logs to a separate, write-only account for tamper protection.
  13. Review with IAM Access Analyzer. Detect resources shared externally; refine over-permissive policies.
  14. Use temporary credentials over long-lived secrets wherever possible. STS is your friend.
  15. Don't share credentials between people. Every human gets their own identity.
"Treat IAM like a garden — it grows weeds if you don't tend it. Every quarter, walk through your IAM with fresh eyes and prune."
CHAPTER 20

The IAM Audit & Diagnostic Toolkit

AWS gives you several tools to see what's going on inside your IAM. The exam expects you to know which tool answers which question.

ToolWhat it tells youExam trigger
IAM Access Analyzer Finds resources (S3 buckets, KMS keys, IAM roles, Lambda functions, etc.) that are accessible from outside your account or organisation. Also generates fine-grained policies based on observed CloudTrail activity. "Detect resources unintentionally shared externally" / "generate least-privilege policy from real usage"
Credentials Report A CSV of every IAM user in the account with password age, access-key age, last-used date, MFA status. A periodic must-read for any security team. "Audit all IAM user credentials and find stale ones"
Access Advisor / Last Accessed Shows the last time an identity actually used each service it has permission for. Foundation for "remove permissions nobody is using." "Identify unused permissions on a role to trim"
Policy Simulator Test what would happen if a principal tried to do an action before actually doing it. Useful for debugging "why can't I…?" issues. "Test whether a policy change will break access before deploying"
CloudTrail Logs every API call (and console action) in the account. The authoritative source for "who did what when." Send to S3 + CloudWatch. "Find who deleted the bucket / launched that instance"
IAM Roles Anywhere Lets servers outside AWS get temporary AWS credentials using X.509 certificates. Useful for hybrid / on-premises scenarios. "On-prem workload needs AWS access without long-lived keys"
Exam Trigger The distinction between CloudTrail (who did what) and IAM Access Analyzer (what's exposed externally) is a common trap. CloudTrail is reactive — it tells you about events. Access Analyzer is proactive — it tells you about risky configurations before they're exploited.
CHAPTER 21

How Real Companies Actually Use IAM

Theory is one thing. Knowing how mature organisations actually structure IAM in 2024+ helps the exam scenarios click immediately. Here are five patterns from real environments.

Pattern 1 — The early-stage startup (1 account, 5 people)

One AWS account. Root locked away with a hardware MFA key in a safe. Each engineer has an IAM user with MFA. Maybe two groups: Admins and Developers. CI/CD uses an IAM role assumed by GitHub Actions via OIDC. Total IAM artefacts: maybe 10. Total time spent on IAM per week: 15 minutes.

Pattern 2 — The growth-stage company (5–20 accounts)

AWS Organizations enabled. Accounts split by environment: prod, staging, dev, security audit, log archive, network hub. IAM Identity Center hooked up to Okta or Google Workspace — engineers sign in once, see all accounts they can access. Per-account permission sets: AdminAccess, DeveloperAccess, ReadOnlyAccess, BillingViewOnly. SCPs at the root deny disabling CloudTrail / GuardDuty, restrict regions to two or three, deny use of expensive services like SageMaker in dev OUs.

Pattern 3 — The DevOps / platform team pattern

Application teams ship via CI/CD pipelines that assume deployment roles in each target account. The pipeline (in a CI account) has no permissions itself — it just has sts:AssumeRole to specific deployment roles in dev/staging/prod. Each deployment role in the target account has a permission boundary preventing it from creating IAM identities. Developers can't deploy directly; everything goes through pipeline.

Pattern 4 — The multi-tenant SaaS

You're a SaaS company; your customers each have their own AWS account. Your application needs to access resources in customer accounts (e.g., for backup, monitoring, integration). Each customer creates a cross-account role in their account, with a trust policy that allows your account's principal, gated by an ExternalId. Your service assumes that role to operate in their account. Each customer relationship is exactly one role; no shared credentials. Classic confused-deputy-resistant architecture.

Pattern 5 — The compliance-heavy industry (banking, healthcare, defence)

Allow-list SCPs instead of deny-list — only explicitly approved services can be used. All sensitive operations require MFA and source-IP restrictions to corporate networks. Every IAM action is logged to a separate audit account that the workload accounts can't touch. Quarterly access reviews where managers re-certify each direct report's access. Hardware MFA keys mandatory; phones not allowed in some areas.

Industry — Common Lesson Across All Patterns Notice that in every mature pattern, no humans have direct programmatic credentials in production. Humans get console access via Identity Center; deployments go through CI/CD which assumes roles; workloads use service roles. Long-lived access keys (IAM user keys) only exist for legacy third-party tools that genuinely can't do anything else — and they're rotated automatically and scoped tightly. If you encounter an AWS environment in real life where engineers are sharing access keys via Slack, you've found a security problem in progress.
CHAPTER 22

Mental Models to Carry Into the Exam

Forget memorising. Build these instincts and the exam answers itself.

The five reflexes

1. "Service A needs to access service B" → IAM Role.

Always. Never IAM user with access keys. The trust policy says "Service A can assume me," the permission policy says "what I can do in Service B."

2. "Cross-account access" → role with cross-account trust policy.

Both sides allow. External ID if it's a third party. Never share access keys across accounts.

3. "Enterprise users need AWS access" → IAM Identity Center, federated with their IdP.

Not IAM users in every account. Not SAML configured separately per account.

4. "Prevent something across all accounts no matter what" → SCP.

SCPs apply even to root in member accounts. Use a Deny SCP for guardrails.

5. "Let developers create IAM roles safely" → Permission Boundary.

Force them to attach the boundary on every role they create, via a condition on iam:PermissionsBoundary.

The three traps that decide passes

  1. SCPs don't grant — they only restrict. If an SCP allows S3 but no IAM policy grants S3, nobody has S3.
  2. Explicit Deny always wins. Memorise the evaluation order: SCP → Boundary → Identity Policy → Resource Policy, but any Deny kills it.
  3. The management account is exempt from SCPs. Don't put workloads there.

The four things a SAA-C03 question almost always tests

  • "Which is the most secure way?" → Roles + MFA + temporary credentials. Never long-lived secrets.
  • "Which is the lowest operational overhead?" → Managed identities (Identity Center, IRSA, OIDC federation). Never per-account manual setup.
  • "Cross-account third party?" → Role + External ID condition.
  • "Cross-region or cross-account guardrails?" → SCPs.
"You don't pass the IAM section of SAA-C03 by knowing every condition key. You pass by reflexively reaching for the same three answers — role, role, role — and knowing when role isn't the answer."
